## 1. Hilo conductor

C08 (`embeddings-and-vectors`) te mostró cómo funcionan los embeddings y las bases de datos vectoriales. Aquí los ensamblas en el pipeline RAG completo: el patrón que usa BATUTA para dar al agente de auditoría acceso a los manuales contables de cada cliente (F-CTA-01). C10 (`when-not-to-rag`) es el contrapeso: cuándo RAG no es la respuesta y qué usar en su lugar.

## 2. Idea central

**RAG** (Retrieval-Augmented Generation) es el patrón de insertar contexto recuperado de una base de conocimiento en el prompt del LLM antes de generar la respuesta. En lugar de entrenar el modelo con los datos del cliente (caro, lento), o ponerlos todos en el prompt (costoso, no escala), divides los documentos en chunks, los indexas como vectores, y en tiempo de query recuperas solo los chunks relevantes. El LLM genera la respuesta basándose en esos chunks — con citas, con contexto real, sin alucinaciones inventadas.

<!-- INLINE_QUIZ: chunk_size_tradeoff -->

## 3. Por qué importa

Sin RAG, el agente de ACME responde preguntas sobre los manuales contables desde el entrenamiento del LLM — que no conoce los procedimientos internos de ACME, su plan de cuentas, ni las regulaciones específicas de Colombia o México. Con RAG, el agente cita el párrafo exacto del manual de ACME. La diferencia es entre un asistente genérico y un experto en el cliente.

La ficha **F-CTA-01** (auditoría contable automática) usa RAG sobre los manuales de ACME para contextualizar hallazgos: «la factura #F-001 parece un gasto de capital pero el Manual F-CTA-01 §3.2 dice que debe contabilizarse como gasto operativo».

## 4. Cómo funciona por dentro

El pipeline tiene 7 pasos. Los primeros 4 son offline (al subir documentos); los últimos 3 son online (al hacer una query).

### Paso 1: Chunking — dividir el documento

Un documento de 50 páginas no cabe en un solo embedding. El **chunking** lo divide en fragmentos manejables. Tres estrategias:

**Por tokens** (recursive splitter):
- Chunk size: 256–512 tokens. El estándar.
- Overlap: 10–20% del chunk (los últimos N tokens del chunk anterior se repiten al inicio del siguiente, para no cortar contexto en la frontera).
- LangChain `RecursiveCharacterTextSplitter` implementa esto dividiendo primero por `\n\n`, luego `\n`, luego ` `, luego carácter a carácter hasta alcanzar el tamaño.

**Por estructura** (headers, secciones):
- Divide en secciones naturales del documento (H1, H2, párrafo). Mejor para documentos bien estructurados (manuales, contratos).
- Cada chunk preserva el título de su sección — contexto de jerarquía que mejora la relevancia.

**Por semántica**:
- Agrupa oraciones por similitud de embedding. Más costoso (requiere embedear cada oración), mejor calidad.
- Útil para documentos sin estructura explícita.

**Trampas del chunking**:
- Chunks demasiado grandes (>1024 tokens): mezclan conceptos, el embedding pierde precisión.
- Chunks demasiado pequeños (<64 tokens): sin contexto suficiente, el LLM no puede usarlos.
- Fronteras en medio de una tabla o lista: corta la estructura; usar `keep_separator=True`.
- No preservar metadata (página, sección, fuente): sin metadata, no puedes citar la fuente.

### Paso 2: Embedding — vectorizar los chunks

```python
import voyageai

voyage = voyageai.Client()

def embed_chunks(chunks: list[str], batch_size: int = 128) -> list[list[float]]:
    """Genera embeddings en batches para no saturar la API."""
    embeddings = []
    for i in range(0, len(chunks), batch_size):
        batch = chunks[i : i + batch_size]
        result = voyage.embed(batch, model="voyage-3", input_type="document")
        embeddings.extend(result.embeddings)
    return embeddings
```

`input_type="document"` para indexar; `input_type="query"` para la pregunta. Voyage AI (y Cohere) tienen modos distintos para cada caso — mejoran la calidad de recuperación. OpenAI no distingue.

### Paso 3: Indexing — guardar en la base vectorial

Con pgvector (Postgres), cada chunk queda guardado con su embedding y su metadata:

```sql
INSERT INTO document_chunk (tenant_id, doc_id, chunk_index, chunk_text, embedding, metadata)
VALUES (
    'acme',
    'F-CTA-01',
    42,
    'La revalorización de activos fijos se contabiliza según NIIF 16...',
    '[0.1, 0.2, ...]'::vector,
    '{"section": "3.2", "page": 15, "doc_title": "Manual Contable ACME"}'
);
```

Con Qdrant, el payload (metadata) permite filtrar antes de la búsqueda vectorial.

### Paso 4: Retrieval — recuperar los chunks relevantes

Cuando llega una pregunta, se genera el embedding de la query y se buscan los K chunks más similares:

```python
def retrieve(question: str, tenant_id: str, top_k: int = 20) -> list[dict]:
    """Recupera los top_k chunks más similares para un tenant."""
    query_embedding = voyage.embed(
        [question], model="voyage-3", input_type="query"
    ).embeddings[0]

    # pgvector: búsqueda con filtro de tenant
    rows = db.execute("""
        SELECT chunk_text, metadata,
               1 - (embedding <=> %(q)s::vector) AS similarity
        FROM document_chunk
        WHERE tenant_id = %(tid)s
        ORDER BY embedding <=> %(q)s::vector
        LIMIT %(k)s
    """, {"q": query_embedding, "tid": tenant_id, "k": top_k}).fetchall()

    return [{"text": r.chunk_text, "metadata": r.metadata, "similarity": r.similarity}
            for r in rows]
```

**¿Por qué top-20 y no top-5?** Para el reranker. La búsqueda vectorial es aproximada y puede traer chunks no muy relevantes. Recuperar más y luego rerankar da mejor precisión.

**Hybrid search** (denso + BM25): combinar la búsqueda vectorial (semántica) con búsqueda léxica (BM25, que funciona bien con términos técnicos exactos como «NIIF 16», «artículo 423»). pgvector no hace BM25 nativo; para hybrid search use `pg_search` (ParadeDB) o Qdrant con sparse vectors (SPLADE).

**MMR (Maximal Marginal Relevance)**: si todos los top-20 chunks dicen lo mismo (provienen del mismo párrafo duplicado), el LLM recibe contexto redundante. MMR selecciona chunks que son relevantes para la query y distintos entre sí. Útil cuando el corpus tiene mucha redundancia.

### Paso 5: Reranking — reordenar con un modelo más preciso

El embedding busca por similitud aproximada. El **reranker** recibe la query y cada chunk, y les asigna una puntuación de relevancia más precisa — es un modelo cross-encoder que puede ver ambos textos juntos.

```python
import cohere

co = cohere.Client()

def rerank(question: str, chunks: list[dict], top_n: int = 5) -> list[dict]:
    """Reordena chunks con Cohere Rerank 3.5."""
    response = co.rerank(
        model="rerank-v3.5",
        query=question,
        documents=[c["text"] for c in chunks],
        top_n=top_n,
    )
    return [
        {**chunks[r.index], "rerank_score": r.relevance_score}
        for r in response.results
    ]
```

**Alternativas de reranker (2026)**:
| Modelo | Latencia | Nota |
|--------|----------|------|
| `rerank-v3.5` (Cohere) | ~600ms | Contexto 4K, multilingüe, SOTA |
| `rerank-2.5` (Voyage AI) | ~600ms | Mejor que Cohere en benchmarks 2025; soporta instrucciones |
| `BAAI/bge-reranker-v2-m3` | Depende de GPU | Open-source, multilingüe, auto-hosteable |

El reranking es opcional pero mejora significativamente la precisión cuando el corpus es grande. El costo es bajo (solo top-20 chunks, no el corpus completo).

### Paso 6: Augmentation — construir el prompt

Los chunks rerankeados se insertan en el prompt del LLM. El orden importa: los más relevantes cerca del final (antes de la pregunta) — los LLMs tienen sesgo de recencia:

```python
def build_prompt(question: str, chunks: list[dict]) -> str:
    context_blocks = []
    for i, chunk in enumerate(chunks, 1):
        meta = chunk.get("metadata", {})
        source = f"{meta.get('doc_title', 'Doc')} §{meta.get('section', '?')} p.{meta.get('page', '?')}"
        context_blocks.append(f"[{i}] {source}\n{chunk['text']}")

    context = "\n\n---\n\n".join(context_blocks)
    return f"""Contexto del manual de ACME:

{context}

---

Pregunta: {question}

Responde basándote únicamente en el contexto anterior. Cita las fuentes por número [1], [2], etc.
Si la información no está en el contexto, di explícitamente que no la encontraste."""
```

> [!cuidado]
> **Tamaño del prompt**: 5 chunks × 512 tokens = ~2560 tokens de contexto. Más el system prompt y la pregunta: ~3500 tokens de entrada. Con `claude-sonnet-4-6` a $3/M tokens de entrada, eso es $0.0105 por query. A 1000 queries/día: $10.50/día. Calcula el costo antes de decidir cuántos chunks insertar.

### Paso 7: Generación — el LLM responde con citas

```python
import anthropic

client = anthropic.Anthropic()

def generate_answer(prompt: str, module_context: str) -> str:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system=[{
            "type": "text",
            "text": module_context,
            "cache_control": {"type": "ephemeral", "ttl": "1h"}  # cachea el context del módulo
        }],
        messages=[{"role": "user", "content": prompt}]
    )
    return response.content[0].text
```

El `cache_control` con `ttl="1h"` cachea el system prompt (que incluye las instrucciones del agente). Los chunks del RAG van en el `user` message — cambian por query, no se cachean.

### El pipeline completo

```
PDF / DOCX → [chunk] → chunks de texto
                           ↓
                       [embed] ← modelo voyage-3
                           ↓
                    [index] → pgvector / Qdrant
                    ─────────────────────────── (offline, al subir el doc)

pregunta del usuario
        ↓
    [embed query] ← voyage-3, input_type="query"
        ↓
    [retrieve top-20] ← pgvector + filtro tenant_id
        ↓
    [rerank top-5] ← Cohere rerank-v3.5 o Voyage rerank-2.5
        ↓
    [augment prompt] ← insertar chunks con citas
        ↓
    [generate] ← claude-sonnet-4-6
        ↓
    respuesta con citas [1][2][3]
```

### Determinístico vs agéntico: dónde está la frontera en RAG

El chunking es determinístico: dado el mismo documento y la misma estrategia, los chunks siempre son los mismos. El embedding es determinístico: mismo texto, mismo vector. El retrieval es determinístico (dado el índice). El reranking es determinístico (dado el modelo). **La generación es agéntica**: el LLM decide qué chunks usar, cómo sintetizarlos, qué citar. El mismo contexto puede producir respuestas ligeramente distintas (temperatura > 0). Eso es lo correcto: la síntesis y la redacción para humano no tienen regla cerrada.

## 5. Ejemplo conducido

Pipeline RAG mínimo sin dependencias externas (vectores sintéticos para poder ejecutar en Pyodide):

```python
import math

def cosine_sim(v1, v2):
    dot = sum(a * b for a, b in zip(v1, v2))
    norm = lambda v: math.sqrt(sum(x*x for x in v))
    return dot / (norm(v1) * norm(v2))

# corpus de ACME: (chunk_text, embedding_fake, metadata)
corpus = [
    ("La revalorización de activos fijos se rige por NIIF 16.",
     [0.9, 0.1, 0.05, 0.2], {"section": "3.2", "page": 15}),
    ("Los gastos de capital deben activarse como activo fijo.",
     [0.85, 0.15, 0.1, 0.25], {"section": "3.1", "page": 12}),
    ("El pago de dividendos requiere aprobación de junta.",
     [0.1, 0.9, 0.2, 0.1], {"section": "7.1", "page": 45}),
    ("El inventario se valora por FIFO o costo promedio.",
     [0.1, 0.1, 0.9, 0.2], {"section": "5.3", "page": 28}),
]

# query del auditor
query_embedding = [0.88, 0.12, 0.06, 0.22]  # similar a NIIF 16 / activos fijos

# retrieve top-3
scored = sorted(
    [(text, cosine_sim(query_embedding, emb), meta) for text, emb, meta in corpus],
    key=lambda x: x[1], reverse=True
)[:3]

# augmentation (en producción, aquí iría la llamada al LLM)
context = "\n".join(
    f"[{i+1}] §{r[2]['section']} p.{r[2]['page']}: {r[0]}"
    for i, r in enumerate(scored)
)
prompt = f"""Contexto:
{context}

Pregunta: ¿Cómo se contabiliza la revalorización de activos fijos?
Cita las fuentes por número."""
print(prompt)
```

En cada ciclo de auditoría, para cada factura anómala, hace una query RAG sobre los manuales: recupera los 20 chunks más relevantes, los reranquea a 5, y el LLM genera el hallazgo con la referencia exacta del manual. ## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Chunking del documento por tokens con overlap | Determinístico | Dado el mismo doc y la misma config, los chunks son siempre los mismos. |
| Generación del embedding de cada chunk | Determinístico | Mismo texto + mismo modelo → mismo vector. |
| Retrieval de top-K vectores más cercanos | Determinístico | HNSW es reproducible dado el índice. |
| Reranking de los top-K chunks | Determinístico | El modelo cross-encoder produce el mismo score para el mismo par (query, chunk). |
| Selección final de qué citar y cómo redactar la respuesta | Agéntico | El LLM decide qué es relevante, cómo sintetizar y qué enfatizar — sin regla cerrada. |
| Decidir si la respuesta tiene suficiente evidencia o se necesita más contexto | Agéntico | Requiere razonamiento sobre la calidad del contexto — variable por dominio y pregunta. |

## 10. Errores típicos

**1. Chunks demasiado grandes.**
*Síntoma*: la recuperación devuelve chunks que contienen la respuesta pero también mucho ruido; la respuesta del LLM es imprecisa.
*Causa*: chunks de 2000 tokens mezclan varias secciones. El embedding promedia el contenido y pierde precisión semántica.
*Arreglo*: reducir a 256–512 tokens con 10% de overlap.

**2. No usar `input_type` diferenciado al embedear.**
*Síntoma*: la calidad de recuperación es más baja de la esperada con Voyage AI o Cohere.
*Causa*: usar `input_type="document"` tanto para indexar como para hacer queries. Estos modelos tienen representaciones distintas para documentos y queries — mezclarlas degrada la calidad.
*Arreglo*: `input_type="document"` al indexar chunks; `input_type="query"` al embedear la pregunta del usuario.

**3. Olvidar el filtro de `tenant_id` en la búsqueda vectorial.**
*Síntoma*: el agente de ACME devuelve párrafos de los manuales de Globex.
*Causa*: la query vectorial no filtra por tenant.
*Arreglo*: siempre combinar la búsqueda vectorial con `WHERE tenant_id = :tid` (pgvector) o filtro de payload (Qdrant).

**4. No reranquear y confiar solo en el embedding.**
*Síntoma*: el LLM recibe chunks que hablan del tema pero no responden la pregunta específica.
*Causa*: el embedding mide similitud semántica general; el reranker mide relevancia específica para la pregunta.
*Arreglo*: recuperar top-20 con embedding, reranquear a top-5 con Cohere `rerank-v3.5` o Voyage `rerank-2.5`.

**5. Insertar demasiados chunks en el prompt.**
*Síntoma*: el costo por query se dispara; el LLM «se pierde» en el contexto y produce respuestas menos precisas.
*Causa*: insertar 20 chunks × 512 tokens = 10K tokens de contexto por query.
*Arreglo*: usar reranker y quedarte con 3–5 chunks. El límite de contexto es generosidad, no objetivo.

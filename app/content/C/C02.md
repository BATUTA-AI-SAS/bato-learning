## Hilo conductor

C01 (`sqlalchemy-async-sqlite`) te mostró la capa ORM: cómo los modelos mapean a tablas y cómo una sesión async ejecuta queries. También trazaste la frontera honesta: SQLite para esta app local, Postgres para el servicio multitenant. Aquí viene la decisión de diseño más importante de todo el Track C: cómo garantizar que ACME nunca vea datos de Globex, ni siquiera cuando hay un bug en el código de la app. Esta garantía no puede vivir solo en el código — el código tiene bugs. Necesita vivir en la base de datos. Eso es **Row Level Security**. Al terminar este módulo, C03 (`datos-y-backups`) puede asumir que el aislamiento es robusto y construir la estrategia de durabilidad sobre él.

## Idea central

**Multitenancy** significa que múltiples clientes (tenants) comparten la misma infraestructura. El aislamiento entre ellos puede implementarse de tres formas: una base de datos por tenant, un schema por tenant, o una base de datos compartida con `tenant_id` en cada fila. La tercera es la más económica a escala PYME; el riesgo es que una query sin el filtro correcto filtra datos de otro tenant. **RLS (Row Level Security)** en Postgres es la red de seguridad: una política a nivel de motor que filtra filas automáticamente según una variable de sesión, antes de que el resultado llegue a la app.

## Por qué importa

El caso de uso **F-CTA-02** (auditoría contable) tiene dos clientes — Initech Audit y Globex Logistics — que comparten la misma instancia de Postgres. Un bug en `app/repos/audit.py` que omita el filtro `WHERE tenant_id = :tid` devolvería los hallazgos de Globex a Initech. Con RLS activado, ese bug devolvería cero filas — Postgres filtra antes de devolver. El bug sigue existiendo, pero el leak no ocurre.

Sin RLS, la única garantía de aislamiento es el código de la app. En un equipo que escribe repos a diario, eso es insuficiente. El error ocurrirá. La pregunta es si el motor lo atrapa o si lo atrapas tú tres meses después en producción.

> [!cuidado]
> RLS no reemplaza un buen diseño de repos. Es una red de seguridad, no una excusa para omitir `WHERE tenant_id = :tid` en el código. Defensa en profundidad: filtras en el repo **y** el motor también filtra.

## Cómo funciona por dentro

### Los tres patrones de multitenancy: cuándo usar cada uno

| Patrón | Descripción | Cuándo usarlo |
|--------|-------------|---------------|
| **DB por tenant** | Cada cliente tiene su propia instancia o base de datos | Requerimiento de aislamiento físico (salud, finanzas reguladas); cliente paga por su infra |
| **Schema por tenant** | Un schema por cliente en la misma instancia Postgres | 10–50 tenants con schemas distintos; migraciones algo más complejas |
| **Tabla compartida + RLS** | Todos en las mismas tablas, `tenant_id` en cada fila | La mayoría de SaaS PYME: cientos de tenants, schema idéntico, costo mínimo |

BATUTA usa tabla compartida + RLS. Es la opción correcta hasta que un cliente exija aislamiento físico por regulación — en ese caso, la decisión de moverlo a su propio schema o BD se toma por cliente, no para todos.

### `tenant_id` en cada tabla

Toda tabla que contiene datos de tenant lleva una columna `tenant_id`. Sin excepción. Esta es la regla más importante del schema:

```sql
CREATE TABLE invoice (
    id          BIGSERIAL PRIMARY KEY,
    tenant_id   TEXT        NOT NULL,
    period      TEXT        NOT NULL,
    vendor_id   BIGINT      REFERENCES vendor(id),
    total_eur   NUMERIC(12,2) NOT NULL,
    status      TEXT        NOT NULL DEFAULT 'pending',
    issued_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- índice que RLS y las queries del app usan siempre
CREATE INDEX idx_invoice_tenant ON invoice (tenant_id);
```

En SQLAlchemy ORM (lo que usa la app actual con SQLite, y lo que usarías con Postgres), el modelo lo declara igual:

```python
from sqlalchemy.orm import Mapped, mapped_column
from ..db import Base

class Invoice(Base):
    __tablename__ = "invoice"

    id: Mapped[int] = mapped_column(primary_key=True)
    tenant_id: Mapped[str] = mapped_column(index=True, nullable=False)
    period: Mapped[str] = mapped_column(nullable=False)
    total_eur: Mapped[float] = mapped_column(nullable=False)
    status: Mapped[str] = mapped_column(default="pending")
```

### Habilitar y configurar RLS en Postgres

RLS se activa tabla por tabla. Una tabla sin `ENABLE ROW LEVEL SECURITY` no tiene ninguna política — todas las filas son visibles para cualquier usuario con permisos SELECT.

```sql
-- habilitar RLS en la tabla
ALTER TABLE invoice ENABLE ROW LEVEL SECURITY;

-- política de SELECT: solo ves tus propias filas
CREATE POLICY tenant_isolation_select ON invoice
    FOR SELECT
    USING (tenant_id = current_setting('app.tenant_id'));

-- política de INSERT: solo puedes insertar en tu tenant
CREATE POLICY tenant_isolation_insert ON invoice
    FOR INSERT
    WITH CHECK (tenant_id = current_setting('app.tenant_id'));

-- política de UPDATE y DELETE: igual
CREATE POLICY tenant_isolation_update ON invoice
    FOR UPDATE
    USING (tenant_id = current_setting('app.tenant_id'));

CREATE POLICY tenant_isolation_delete ON invoice
    FOR DELETE
    USING (tenant_id = current_setting('app.tenant_id'));
```

`current_setting('app.tenant_id')` lee una variable de sesión de Postgres. No es un parámetro de query; es parte de la conexión. La app la setea al inicio de cada transacción.

> [!nota]
> El superusuario (`postgres`) y roles con `BYPASSRLS` **no** están sujetos a RLS por defecto. En producción, la app conecta con un usuario sin privilegios de superusuario. Nunca conectes tu app de BATUTA con el usuario `postgres`.

### Cómo la app setea `app.tenant_id`

El lugar correcto es dentro de la transacción, usando `SET LOCAL`. `SET LOCAL` es visible solo dentro de la transacción activa — si la transacción hace rollback, el setting se revierte:

```python
from sqlalchemy import text
from sqlalchemy.ext.asyncio import AsyncSession

async def get_session_for_tenant(
    session: AsyncSession,
    tenant_id: str,
) -> AsyncSession:
    """Configura la sesión Postgres con el tenant_id del request."""
    async with session.begin():
        await session.execute(
            text("SET LOCAL app.tenant_id = :tid"),
            {"tid": tenant_id},
        )
        yield session
```

En FastAPI, esto va en un `Depends` que envuelve al `get_session` estándar. El router recibe una sesión ya configurada con el tenant del request autenticado. El tenant_id viene del JWT o la sesión HTTP — nunca del body del request.

```python
from fastapi import Depends
from app.db import SessionDep
from app.auth import get_current_tenant  # dependency de autenticación

async def get_tenant_session(
    session: SessionDep,
    tenant_id: str = Depends(get_current_tenant),
) -> AsyncSession:
    await session.execute(
        text("SET LOCAL app.tenant_id = :tid"),
        {"tid": tenant_id},
    )
    return session

TenantSessionDep = Annotated[AsyncSession, Depends(get_tenant_session)]
```

### Filtro explícito en repos: defensa en profundidad

Aunque RLS garantiza aislamiento a nivel de motor, cada repo aplica el filtro explícitamente. Si el código se migra a otro motor que no tiene RLS, el filtro del repo sigue funcionando:

```python
from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession
from app.models.invoice import Invoice

async def list_invoices(
    session: AsyncSession,
    tenant_id: str,
    period: str,
) -> list[Invoice]:
    result = await session.execute(
        select(Invoice)
        .where(Invoice.tenant_id == tenant_id)
        .where(Invoice.period == period)
        .order_by(Invoice.issued_at.desc())
    )
    return list(result.scalars().all())
```

### Cómo testear que el aislamiento funciona

Un test de aislamiento tiene dos pasos: insertar datos de dos tenants distintos, luego verificar que con la sesión configurada para ACME no aparecen filas de Globex.

```python
import pytest
from sqlalchemy import text, select
from app.models.invoice import Invoice

@pytest.mark.asyncio
async def test_rls_isolation(db_session):
    """Verifica que RLS impide ver filas de otro tenant."""
    # setup: crear facturas de dos tenants
    for tenant in ("acme", "globex"):
        await db_session.execute(
            text("SET LOCAL app.tenant_id = :tid"), {"tid": tenant}
        )
        db_session.add(Invoice(
            tenant_id=tenant, period="2026-04",
            total_eur=1000.0, status="pending"
        ))
    await db_session.flush()

    # act: configurar sesión como ACME
    await db_session.execute(
        text("SET LOCAL app.tenant_id = :tid"), {"tid": "acme"}
    )
    rows = (await db_session.execute(select(Invoice))).scalars().all()

    # assert: solo filas de ACME visibles
    assert all(r.tenant_id == "acme" for r in rows), \
        "RLS leak: se ven filas de otro tenant"
    assert len(rows) == 1, \
        f"Esperaba 1 fila de acme, obtuvo {len(rows)}"
```

Si el test pasa, el aislamiento es demostrable. Si falla (se ven filas de Globex), o bien RLS no está activado, o bien la política usa `current_setting` con el nombre incorrecto.

## Ejemplo conducido

Situación: F-CTA-02. Initech y Globex comparten infra. Un bug en un repo ad-hoc omite el `WHERE tenant_id`. Con RLS activo, el bug es inofensivo.

```sql
-- setup: RLS activo, sesión configurada como 'initech'
SET LOCAL app.tenant_id = 'initech';

-- query con bug: sin WHERE tenant_id
SELECT id, total_eur, status FROM invoice WHERE period = '2026-04';

-- resultado con RLS activo:
--  id  | total_eur | status
-- -----+-----------+---------
--  341 |  4 820,00 | approved
--  342 |  1 200,00 | pending
-- (2 rows) — solo filas de Initech, aunque el SQL no filtra por tenant
```

Sin RLS, la misma query devolvería también las filas de Globex. Con RLS, Postgres aplica la política antes de devolver cualquier resultado.

## Errores típicos

**1. `current_setting` lanza error si la variable no existe.**
*Síntoma*: `ERROR: unrecognized configuration parameter "app.tenant_id"` al hacer SELECT.
*Causa*: la transacción no ejecutó `SET LOCAL app.tenant_id` antes del primer query. O la política usa `current_setting('app.tenant_id')` sin el segundo argumento `true` (missing_ok).
*Arreglo*: en la política, usa `current_setting('app.tenant_id', true)` — el segundo argumento `true` devuelve `NULL` en lugar de lanzar error si la variable no existe. Luego añade `OR tenant_id IS NULL` no — mejor: asegúrate de que el dependency siempre setee la variable. La versión defensiva de la política:

```sql
CREATE POLICY tenant_isolation_select ON invoice
    FOR SELECT
    USING (tenant_id = current_setting('app.tenant_id', true));
```

**2. Superusuario que bypasea RLS en tests.**
*Síntoma*: el test de aislamiento pasa en local (usas un usuario sin privilegios) pero falla en CI (la CI conecta como `postgres`).
*Causa*: superusuarios ignoran RLS por defecto.
*Arreglo*: en tests e2e, conecta siempre con el mismo usuario de aplicación (sin superusuario). Valida esto en el fixture de la sesión.

**3. `SET` en lugar de `SET LOCAL`.**
*Síntoma*: el tenant_id de una transacción anterior «contamina» la siguiente conexión del pool.
*Causa*: `SET app.tenant_id = 'acme'` persiste en la conexión hasta que se cambia explícitamente. Si esa conexión vuelve al pool y la siguiente transacción no setea el tenant, RLS filtra con el tenant anterior.
*Arreglo*: usa siempre `SET LOCAL`. Su scope es la transacción activa; al hacer commit o rollback, la variable se revierte al valor previo (vacío o el default).

**4. Olvidar `tenant_id` en una tabla nueva.**
*Síntoma*: una feature nueva devuelve datos de todos los tenants sin filtrar.
*Causa*: se añadió una tabla sin `tenant_id` y sin política RLS.
*Arreglo*: hacer que la migración que crea una tabla nueva incluya siempre `tenant_id NOT NULL` y las cuatro políticas RLS. Añadir un check en CI que detecte tablas sin `tenant_id` en el schema.

**5. RLS en tabla pero no en vistas.**
*Síntoma*: la query directa a la tabla respeta RLS, pero la vista `v_invoice_summary` devuelve datos de todos los tenants.
*Causa*: las vistas en Postgres heredan los permisos del creador (security definer) a menos que se configure `security_invoker = true`.
*Arreglo*: en Postgres 15+, crear vistas con `WITH (security_invoker = true)` para que hereden el contexto del usuario que consulta, incluido RLS.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Aplicar `SET LOCAL app.tenant_id` al inicio de cada request | determinístico | Es una regla mecánica que el harness siempre ejecuta. El modelo no decide. |
| Verificar que todas las tablas tienen `tenant_id NOT NULL` | determinístico | Es un check estático sobre el schema; automatizable en un script de CI. |
| Detectar si un bug en producción causó un leak de datos | agéntico | Requiere correlacionar logs, trazas, y conocimiento del dominio del cliente para determinar el alcance real. |
| Decidir si mover a un cliente a schema-per-tenant | agéntico | Depende de requisitos regulatorios del cliente, volumen, y roadmap — no hay regla cerrada. |
| Escribir el test de aislamiento que verifica leak=0 | determinístico | El criterio es exacto: cero filas de otro tenant con la sesión configurada. Automatizable. |

## Caso de uso real

**F-CTA-02 — Auditoría contable compartida.** Initech Audit y Globex Logistics son clientes de BATUTA. Sus contadores usan el mismo panel de auditoría mensual. RLS garantiza que el agente de Initech nunca vea hallazgos de Globex, incluso si un bug de código omite el filtro. El test de aislamiento del módulo es el criterio de aceptación de este caso de uso: si `len(globex_rows_seen_by_initech) == 0`, la feature se puede promover a producción.

> [!profundizar]
> La forma canónica de filtrar por `tenant_id` en la app actual (SQLite) está en `app/repos/_base.py` — si existe — o en cada repo concreto como `app/repos/progress.py`. Busca el patrón `.where(Model.tenant_id == tenant_id)` y compáralo con la política RLS de este módulo: la semántica es idéntica, el motor de enforcement es distinto.

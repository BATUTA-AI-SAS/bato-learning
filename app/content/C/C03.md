## Hilo conductor

C01 (`sqlalchemy-async-sqlite`) estableció la capa ORM y la frontera SQLite→Postgres. C02 (`multitenancy-rls`) garantizó el aislamiento por tenant a nivel de motor. Ahora viene la pregunta que todo el mundo pospone: ¿qué pasa cuando el disco falla a las 3 AM? Un servicio sin estrategia de backup verificada no es un servicio — es un riesgo sin cuantificar. Este módulo cierra el Track C: los backups heredan la separación por tenant que diseñaste en C02, y el aislamiento que garantiza que un restore de ACME no mezcle datos de Globex. D01 (`docker-images-containers`) puede arrancar desde aquí con la confianza de que la capa de datos es durable.

## Idea central

**Backup** no es copiar archivos. Es la capacidad demostrada de restaurar un estado conocido de la base de datos en un tiempo acotado. La diferencia entre «tengo backup» y «puedo restaurar» se mide en el **restore drill**: el ejercicio planificado donde realmente ejecutas el restore en un ambiente separado y verificas que los datos son correctos. Sin el drill, el backup es una promesa no verificada.

El caso de uso **F-LEG-01** (revisión de contratos) implica retención de documentos según ley local — en Colombia, 10 años para contratos mercantiles; en México, 5 años para los mismos. Para BATUTA como proveedor de servicio, la pérdida de datos de un tenant es la peor falla posible. ## Cómo funciona por dentro

### Full vs incremental: cuándo usar cada uno

Un **backup full** es una copia completa de la base de datos en un momento dado. Un **backup incremental** copia solo los cambios desde el último full o desde el último incremental. El trade-off es tiempo de backup vs tiempo de restore.

| Tipo | Tamaño | Tiempo de restore | Cuándo usarlo |
|------|--------|-------------------|---------------|
| Full diario | Grande | Mínimo (1 archivo) | La mayoría de PYMES con BD < 50 GB |
| Full semanal + incremental diario | Medio | Medio (aplicar deltas) | BD > 50 GB con cambios frecuentes |
| <abbr title="Write-Ahead Log: registro continuo de cambios que Postgres escribe antes de modificar los datos reales.">WAL</abbr> archiving continuo (<abbr title="Point-In-Time Recovery: restaurar la BD al estado exacto de un momento pasado usando el WAL archivado.">PITR</abbr>) | Mínimo por archivo | Mayor complejidad | Cuando necesitas restaurar a minuto exacto |

Para una PYME con BATUTA y < 50 tenants, **full diario con `pg_dump`** es la solución correcta. Simple de operar, fácil de verificar, suficiente.

### `pg_dump`: la herramienta para PYMES

`pg_dump` exporta una base de datos a un archivo SQL o a formato binario. Es la herramienta estándar incluida con Postgres — no requiere instalación extra.

```bash
# backup completo en formato comprimido (recomendado)
pg_dump \
  --host=localhost \
  --port=5432 \
  --username=batuta_app \
  --dbname=batuta \
  --format=custom \
  --compress=9 \
  --file=/backups/batuta_$(date +%Y%m%d_%H%M%S).dump

# listar el contenido sin restaurar
pg_restore --list /backups/batuta_20260416_030000.dump | head -30

# restaurar a una base de datos nueva (restore drill)
pg_restore \
  --host=localhost \
  --port=5432 \
  --username=batuta_app \
  --dbname=batuta_restore_test \
  --no-owner \
  --no-acl \
  /backups/batuta_20260416_030000.dump
```

El formato `--format=custom` (`.dump`) es binario comprimido: más pequeño que el SQL y permite restauración selectiva por tabla. Es el que debes usar en producción.

> [!cuidado]
> `pg_dump` captura un snapshot consistente de la base de datos usando una transacción REPEATABLE READ internamente. No necesitas parar Postgres para hacer un backup consistente. Lo que **no debes hacer** es copiar los archivos de datos de Postgres (`/var/lib/postgresql/data/`) mientras el motor está corriendo — esos archivos pueden estar en estado inconsistente.

### Destino del backup: Hetzner Storage Box

Para una PYME LATAM, el servidor de aplicación y el backup **no pueden estar en el mismo disco**. El disco puede fallar. El servidor puede estar comprometido. El backup tiene que estar en otro lugar.

**Hetzner Storage Box** es la opción más económica para esto: desde 1 €/mes por 100 GB, con acceso por SFTP, Samba, o rclone. Desde un servidor Hetzner CX22 (donde correrá BATUTA), la transferencia es gratuita porque está dentro de la red Hetzner.

```bash
# script de backup diario (va en un cron job a las 3 AM)
#!/bin/bash
set -euo pipefail

BACKUP_FILE="/tmp/batuta_$(date +%Y%m%d_%H%M%S).dump"
STORAGE_BOX="u123456@u123456.your-storagebox.de"
REMOTE_PATH="/backups/postgres/"

# 1. dump comprimido
pg_dump \
  --username="${PGUSER}" \
  --dbname="${PGDATABASE}" \
  --format=custom \
  --compress=9 \
  --file="${BACKUP_FILE}"

# 2. copiar al Storage Box vía SFTP (clave SSH pre-configurada)
sftp "${STORAGE_BOX}" <<EOF
put ${BACKUP_FILE} ${REMOTE_PATH}
EOF

# 3. eliminar backup local temporal
rm "${BACKUP_FILE}"

echo "Backup completado: $(date)"
```

Las credenciales van en variables de entorno (`PGUSER`, `PGPASSWORD`, `PGDATABASE`), nunca en el script. La clave SSH para el Storage Box se añade a `~/.ssh/authorized_keys` del box, no a la contraseña.

### Retención: 7/14/30 días según industria

No todos los backups se guardan para siempre. La política de retención define cuántos días de backups se mantienen antes de eliminar los más antiguos.

| Industria / regulación | Retención de backups | Base legal (referencia) |
|------------------------|----------------------|-------------------------|
| Servicios generales | 7–14 días rolling | Buena práctica operativa |
| Contabilidad (PYME LATAM) | 30 días + 1 backup mensual por 5 años | Obligaciones tributarias (DIAN CO, SAT MX) |
| Salud privada | 30 días + archivos indefinidos | Ley 23/81 CO, NOM-024 MX |
| Contratos mercantiles | 30 días + 1 backup anual por 10 años | Código de Comercio CO |
| Servicios financieros regulados | 90 días rolling + archivado 5–7 años | SFC CO, CNBV MX |

Para BATUTA con clientes de manufactura y auditoría, la política mínima es **30 días rolling + 1 backup mensual retenido 12 meses**. Si algún cliente es una cooperativa de crédito o una clínica, la política se endurece a los requisitos regulatorios de ese sector.

### Script de retención con `rclone`

`rclone` simplifica la gestión de backups remotos con reglas de retención:

```bash
# rclone configurado con el Storage Box como remote "hetzner-box"
# limpiar backups más antiguos de 30 días
rclone delete hetzner-box:/backups/postgres/ \
  --min-age 30d \
  --include "*.dump"

# sincronizar el backup mensual a una carpeta de archivado
rclone copy /tmp/monthly_backup.dump \
  hetzner-box:/backups/postgres/monthly/
```

### pgBackRest: cuando el volumen exige WAL archiving

`pg_dump` crea un snapshot puntual. Si la base de datos tiene muchas escrituras y necesitas restaurar al minuto exacto antes de un fallo (Point-In-Time Recovery, PITR), necesitas **WAL archiving**. Postgres escribe continuamente al WAL (Write-Ahead Log); si archivas esos archivos, puedes reproducir cualquier estado de la base de datos.

**pgBackRest** gestiona WAL archiving, full backups, y PITR de forma integrada. Su complejidad es mayor que `pg_dump`, pero es la herramienta correcta cuando:

- La base de datos tiene > 100 GB.
- Necesitas RPO (Recovery Point Objective) de minutos, no de horas.
- Tienes múltiples réplicas o un standby de hot-failover.

Para una PYME LATAM con BATUTA y < 50 tenants activos, `pg_dump` diario es suficiente. El RPO en ese escenario es «las últimas 24 horas», que es aceptable si se comunica al cliente en el contrato de servicio.

### El restore drill: sin esto no tienes backup

Un backup que no se ha restaurado nunca es una promesa no verificada. El restore drill es la única forma de saber que:

1. El archivo de backup no está corrupto.
2. El proceso de restore funciona en el tiempo estimado.
3. Los datos restaurados son correctos (comparar row count, checksums).
4. El equipo sabe ejecutar el restore sin documentación externa.

**Protocolo de restore drill mensual:**

```bash
#!/bin/bash
# restore drill — ejecutar en ambiente de staging, nunca en producción
set -euo pipefail

BACKUP_FILE="$1"  # path al archivo .dump
RESTORE_DB="batuta_drill_$(date +%Y%m%d)"

echo "=== Restore Drill: $(date) ==="
echo "Backup: ${BACKUP_FILE}"

# 1. crear base de datos temporal
psql -U postgres -c "CREATE DATABASE ${RESTORE_DB};"

# 2. restaurar
pg_restore \
  --username=postgres \
  --dbname="${RESTORE_DB}" \
  --no-owner \
  --no-acl \
  --verbose \
  "${BACKUP_FILE}"

# 3. verificar: contar filas de tablas clave
psql -U postgres -d "${RESTORE_DB}" -c "
SELECT
    'invoice'::text      AS table_name, COUNT(*) AS rows FROM invoice
UNION ALL SELECT
    'audit_finding', COUNT(*) FROM audit_finding
UNION ALL SELECT
    'vendor', COUNT(*) FROM vendor;
"

# 4. comparar contra producción (manual o automatizado)
echo "Compara el row count anterior con producción."
echo "Si coincide, el backup es válido."

# 5. limpiar
psql -U postgres -c "DROP DATABASE ${RESTORE_DB};"
echo "=== Drill completado ==="
```

El resultado del drill se documenta: fecha, backup restaurado, row count, tiempo de restore, responsable. Sin esa documentación, el drill no ocurrió.

### Datos sensibles: cifrado en reposo y masking en exports

Los backups contienen datos de producción. Si el archivo de backup llega a manos equivocadas, los datos de todos los tenants están expuestos.

**Cifrado en reposo:** el Storage Box de Hetzner soporta cifrado del lado del servidor (SSE). Para cifrado end-to-end (que Hetzner tampoco pueda leer), usa `gpg` antes de transferir:

```bash
# cifrar el dump con la clave pública del equipo
gpg --recipient batuta-backup-key@batuta.io \
    --output "${BACKUP_FILE}.gpg" \
    --encrypt "${BACKUP_FILE}"

# el archivo sin cifrar no sale del servidor
rm "${BACKUP_FILE}"
sftp hetzner-box <<EOF
put ${BACKUP_FILE}.gpg /backups/postgres/
EOF
```

**Masking en exports de desarrollo:** cuando un developer necesita datos reales para reproducir un bug, **nunca** se exporta el dump de producción sin anonimizar. La herramienta `pg_anonymizer` (extensión Postgres) o un script SQL con `UPDATE` sobre una copia puede reemplazar datos PII (nombres, RUTs, correos, montos exactos) por datos ficticios pero estructuralmente válidos.

```sql
-- en la copia de staging: anonimizar antes de entregar al developer
UPDATE invoice
SET
    vendor_name = 'PROVEEDOR_' || id,
    description = 'DESCRIPCION_ANONIMIZADA'
WHERE tenant_id IS NOT NULL;
```

## Ejemplo conducido

Caso F-LEG-01: Constructora Andina guarda contratos de obra. La ley colombiana exige conservar contratos mercantiles 10 años. El restore drill mensual verifica que los backups de hace 6 meses son restaurables.

```bash
# 1. listar backups disponibles en el Storage Box
sftp hetzner-box:/backups/postgres/ <<EOF
ls -la
EOF
# salida:
# -rw-r--r-- 1 u123456 u123456 142857600 Apr 01 03:01 batuta_20260401_030000.dump.gpg
# -rw-r--r-- 1 u123456 u123456 143201280 Mar 01 03:01 batuta_20260301_030000.dump.gpg
# ...

# 2. descargar y descifrar el backup de hace 6 meses
sftp hetzner-box:/backups/postgres/batuta_20251016_030000.dump.gpg ./
gpg --decrypt batuta_20251016_030000.dump.gpg > batuta_20251016.dump

# 3. ejecutar restore drill
bash restore_drill.sh ./batuta_20251016.dump

# salida esperada del drill:
# table_name    | rows
# --------------|-------
# invoice       | 47823
# audit_finding | 12041
# vendor        |   284
# Compara el row count anterior con producción.
# Si coincide, el backup es válido.
# === Drill completado ===
```

El responsable anota en el log: fecha, backup verificado, row counts, tiempo (13 minutos), estado: OK.

## Errores típicos

**1. Backup en el mismo disco que la base de datos.**
*Síntoma*: el disco falla y el backup también se pierde.
*Causa*: el script de backup guarda el dump en `/var/lib/postgresql/backups/` en el mismo servidor.
*Arreglo*: el destino siempre es externo (Hetzner Storage Box, S3, GCS). El backup local temporal se elimina después de la transferencia.

**2. Nunca ejecutar el restore drill.**
*Síntoma*: el día que hay que restaurar, el archivo de backup está corrupto o el proceso falla.
*Causa*: se asumió que el backup era válido sin verificarlo.
*Arreglo*: restore drill obligatorio mensual. Documentar el resultado. Sin drill documentado, el sistema de backup no está operativo.

**3. Política de retención sin archivado de largo plazo.**
*Síntoma*: el cliente regulado pide datos de hace 18 meses; solo existen los últimos 30 días de backups.
*Causa*: la retención se configuró a 30 días sin considerar obligaciones regulatorias.
*Arreglo*: para clientes de sectores regulados, añadir un backup mensual retenido el período que exige la norma (5, 7 o 10 años según el caso).

**4. Exportar dump de producción sin anonimizar a entorno de desarrollo.**
*Síntoma*: un developer tiene datos reales de clientes en su laptop.
*Causa*: se copió el backup de producción directamente al entorno de desarrollo para reproducir un bug.
*Arreglo*: nunca. El entorno de desarrollo usa datos sintéticos o un dump anonimizado. El proceso de anonimización se ejecuta antes de la transferencia.

**5. Usar `pg_dump` sin `--format=custom`.**
*Síntoma*: el archivo dump ocupa 10× más espacio; la restauración no permite selectividad por tabla.
*Causa*: `pg_dump` sin `--format` produce SQL plano, sin compresión y sin metadatos de índice separados.
*Arreglo*: siempre `--format=custom --compress=9`. El formato custom es binario comprimido y soporta `--table` en el restore para restaurar tablas individuales.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Ejecutar el cron de backup diario a las 3 AM | determinístico | Es un script sin decisiones; siempre hace lo mismo. |
| Aplicar la política de retención (eliminar backups > 30 días) | determinístico | Regla cerrada: fecha del archivo vs fecha actual. |
| Verificar integridad del restore (row count == producción) | determinístico | Comparación numérica. Automatizable. |
| Decidir la política de retención adecuada para un cliente nuevo | agéntico | Depende de la industria, regulación aplicable, y acuerdo contractual — sin regla cerrada universal. |
| Detectar si un fallo parcial de disco causó corrupción silenciosa en los datos | agéntico | Requiere análisis de logs, checksums, y conocimiento del dominio. No es automatizable sin contexto. |

## 2. Idea central

Un **JOIN** combina filas de dos tablas usando una condición — típicamente que una clave foránea coincide con una clave primaria. El resultado es una tabla virtual que contiene columnas de ambas tablas. **GROUP BY** agrupa las filas del resultado por el valor de una o más columnas y aplica una función de agregación (`COUNT`, `SUM`, `AVG`, `MIN`, `MAX`) a cada grupo. Un **CTE** (Common Table Expression) es una subquery nombrada que hace que las queries complejas se lean como código con variables intermedias.

## 4. Cómo funciona por dentro

### Schema de ejemplo: tres tablas relacionadas

```sql
CREATE TABLE client (
    id         INTEGER PRIMARY KEY,
    tenant_id  TEXT NOT NULL,
    name       TEXT NOT NULL,
    country    TEXT NOT NULL DEFAULT 'DE'
);

CREATE TABLE invoice (
    id         INTEGER PRIMARY KEY,
    tenant_id  TEXT NOT NULL,
    client_id  INTEGER NOT NULL REFERENCES client(id),
    vendor_name TEXT NOT NULL,
    total_eur  REAL NOT NULL,
    period     TEXT NOT NULL,
    paid       INTEGER NOT NULL DEFAULT 0,
    issued_at  TEXT NOT NULL  -- YYYY-MM-DD
);

CREATE TABLE payment (
    id         INTEGER PRIMARY KEY,
    tenant_id  TEXT NOT NULL,
    invoice_id INTEGER NOT NULL REFERENCES invoice(id),
    amount_eur REAL NOT NULL,
    paid_at    TEXT NOT NULL  -- YYYY-MM-DD
);
```

### INNER JOIN: solo filas que tienen match en ambas tablas

```sql
-- nombre del cliente junto con cada factura
SELECT
    c.name            AS client_name,
    i.vendor_name,
    i.total_eur,
    i.period
FROM invoice i
INNER JOIN client c ON i.client_id = c.id
WHERE i.tenant_id = 'acme'
ORDER BY i.total_eur DESC;
```

El alias `i` y `c` acortan la referencia. `INNER JOIN … ON condición` incluye solo las filas donde la condición es `TRUE` en las dos tablas. Si una factura tiene `client_id = 99` que no existe en `client`, esa factura **no aparece** en el resultado.

### LEFT JOIN: todas las filas de la tabla izquierda, con o sin match

```sql
-- todos los clientes, con sus facturas si las tienen (o NULL si no)
SELECT
    c.name,
    i.vendor_name,
    i.total_eur
FROM client c
LEFT JOIN invoice i ON i.client_id = c.id AND i.tenant_id = 'acme'
WHERE c.tenant_id = 'acme';
```

Con `LEFT JOIN`, si un cliente no tiene facturas, aparece igual — con `NULL` en todas las columnas de `invoice`. Esto es clave para responder «¿qué clientes no tienen facturas este mes?»:

```sql
SELECT c.name
FROM client c
LEFT JOIN invoice i ON i.client_id = c.id
    AND i.period = '2026-04'
    AND i.tenant_id = 'acme'
WHERE c.tenant_id = 'acme'
  AND i.id IS NULL;  -- los que no tienen factura en 2026-04
```

> [!nota]
> El filtro sobre la tabla derecha (`i.period = '2026-04'`) va en la condición del `ON`, no en el `WHERE`. Si lo pones en el `WHERE`, conviertes el `LEFT JOIN` en un `INNER JOIN` implícito (filtras las filas con `NULL`).

### GROUP BY y funciones de agregación

```sql
-- total facturado por cliente en 2026-04
SELECT
    c.name,
    COUNT(i.id)          AS invoice_count,
    SUM(i.total_eur)     AS total_invoiced,
    AVG(i.total_eur)     AS avg_invoice,
    MAX(i.total_eur)     AS largest_invoice
FROM invoice i
INNER JOIN client c ON i.client_id = c.id
WHERE i.tenant_id = 'acme'
  AND i.period = '2026-04'
GROUP BY c.id, c.name
ORDER BY total_invoiced DESC;
```

`GROUP BY c.id, c.name` agrupa todas las facturas del mismo cliente en una sola fila. Cada función de agregación (`COUNT`, `SUM`, `AVG`, `MIN`, `MAX`) reduce las N facturas del grupo a un solo valor.

**Regla**: en el `SELECT` de una query con `GROUP BY`, solo puedes poner:
- columnas que están en el `GROUP BY`, y
- expresiones de agregación.

Poner `i.vendor_name` sin estar en el `GROUP BY` lanza un error en Postgres y MySQL (SQLite lo permite pero da resultados impredecibles).

### HAVING: filtrar grupos después de la agregación

```sql
-- clientes con más de 2 facturas en el período
SELECT
    c.name,
    COUNT(i.id) AS invoice_count,
    SUM(i.total_eur) AS total
FROM invoice i
INNER JOIN client c ON i.client_id = c.id
WHERE i.tenant_id = 'acme'
  AND i.period = '2026-04'
GROUP BY c.id, c.name
HAVING COUNT(i.id) > 2
ORDER BY total DESC;
```

`WHERE` filtra filas **antes** de la agregación. `HAVING` filtra grupos **después** de la agregación. `HAVING total > 10000` usa el alias del `SELECT` — en SQLite funciona; en Postgres y MySQL estrictos hay que repetir la expresión: `HAVING SUM(i.total_eur) > 10000`.

### Subqueries: queries dentro de queries

Una **subquery** es una query anidada dentro de otra. Puede ir en el `WHERE`, en el `FROM` o en el `SELECT`.

```sql
-- facturas cuyo monto supera el promedio del período
SELECT vendor_name, total_eur
FROM invoice
WHERE tenant_id = 'acme'
  AND period = '2026-04'
  AND total_eur > (
      SELECT AVG(total_eur)
      FROM invoice
      WHERE tenant_id = 'acme'
        AND period = '2026-04'
  )
ORDER BY total_eur DESC;
```

La subquery entre paréntesis calcula el promedio; la query externa filtra con ese valor. Es **no correlacionada**: se ejecuta una sola vez, independientemente de las filas de la query externa.

Una subquery **correlacionada** referencia columnas de la query externa — se ejecuta una vez por cada fila:

```sql
-- para cada cliente, la factura más reciente
SELECT c.name, i.vendor_name, i.total_eur
FROM client c
INNER JOIN invoice i ON i.client_id = c.id
WHERE i.id = (
    SELECT id FROM invoice
    WHERE client_id = c.id       -- referencia a la query externa
    ORDER BY issued_at DESC
    LIMIT 1
);
```

> [!cuidado]
> Las subqueries correlacionadas se ejecutan N veces (una por fila de la query externa). Para tablas grandes, son lentas. La alternativa eficiente es un CTE con window function (`ROW_NUMBER()`).

### CTEs: `WITH … AS`

Un **CTE** (Common Table Expression) nombra un resultado intermedio para reutilizarlo. Hace que queries complejas sean legibles:

```sql
WITH
-- paso 1: calcular el total pagado por factura
paid_per_invoice AS (
    SELECT
        invoice_id,
        SUM(amount_eur) AS paid_amount
    FROM payment
    WHERE tenant_id = 'acme'
    GROUP BY invoice_id
),
-- paso 2: calcular el saldo pendiente
balance AS (
    SELECT
        i.id,
        i.client_id,
        i.total_eur,
        COALESCE(p.paid_amount, 0) AS paid_amount,
        i.total_eur - COALESCE(p.paid_amount, 0) AS pending
    FROM invoice i
    LEFT JOIN paid_per_invoice p ON p.invoice_id = i.id
    WHERE i.tenant_id = 'acme'
)
-- query final: clientes con saldo pendiente > 0
SELECT
    c.name,
    SUM(b.pending) AS total_pending
FROM balance b
INNER JOIN client c ON c.id = b.client_id
WHERE b.pending > 0
GROUP BY c.id, c.name
ORDER BY total_pending DESC;
```

Los CTEs son azúcar sintáctico — el motor los reescribe como subqueries internamente. El valor es la legibilidad: cada paso tiene un nombre que describe su propósito.

### Window functions: `ROW_NUMBER`, `RANK`, `LAG`

Las **window functions** calculan un valor por fila teniendo en cuenta un conjunto de filas relacionadas (la «ventana»), sin colapsar el resultado en un solo grupo:

```sql
-- ranking de facturas por cliente, dentro del período
SELECT
    c.name,
    i.vendor_name,
    i.total_eur,
    ROW_NUMBER() OVER (
        PARTITION BY i.client_id
        ORDER BY i.total_eur DESC
    ) AS rank_in_client
FROM invoice i
INNER JOIN client c ON c.id = i.client_id
WHERE i.tenant_id = 'acme'
  AND i.period = '2026-04';
```

`PARTITION BY client_id` resetea el contador para cada cliente. `ROW_NUMBER()` numera las filas del 1 en adelante dentro de cada partición. Para quedarte con solo la factura más cara por cliente: `WHERE rank_in_client = 1` — pero en una subquery o CTE, porque el `WHERE` no puede referenciar window functions directamente.

`LAG(total_eur) OVER (ORDER BY period)` devuelve el valor de la fila anterior — útil para calcular variación período a período.

## 5. Ejemplo conducido

Cálculo de saldo pendiente por cliente (el core de F-FIN-04):

```python
import sqlite3

con = sqlite3.connect(":memory:")
cur = con.cursor()
cur.executescript("""
    CREATE TABLE client   (id INTEGER PRIMARY KEY, tenant_id TEXT, name TEXT);
    CREATE TABLE invoice  (id INTEGER PRIMARY KEY, tenant_id TEXT,
                           client_id INTEGER, total_eur REAL, period TEXT, paid INTEGER DEFAULT 0);
    CREATE TABLE payment  (id INTEGER PRIMARY KEY, tenant_id TEXT,
                           invoice_id INTEGER, amount_eur REAL);

    INSERT INTO client  VALUES (1,'acme','Globex Logistics'), (2,'acme','Initech Audit');
    INSERT INTO invoice VALUES (1,'acme',1,9100.50,'2026-04',0),
                               (2,'acme',1,4820.00,'2026-04',0),
                               (3,'acme',2,7200.00,'2026-04',0);
    INSERT INTO payment VALUES (1,'acme',1,5000.00), (2,'acme',3,7200.00);
""")

cur.execute("""
    WITH paid_per_invoice AS (
        SELECT invoice_id, SUM(amount_eur) AS paid
        FROM payment WHERE tenant_id = 'acme'
        GROUP BY invoice_id
    )
    SELECT c.name,
           SUM(i.total_eur) AS invoiced,
           SUM(COALESCE(p.paid, 0)) AS paid,
           SUM(i.total_eur - COALESCE(p.paid, 0)) AS pending
    FROM invoice i
    JOIN client c ON c.id = i.client_id
    LEFT JOIN paid_per_invoice p ON p.invoice_id = i.id
    WHERE i.tenant_id = 'acme'
    GROUP BY c.id, c.name
    ORDER BY pending DESC
""")
for row in cur.fetchall():
    print(row)
# ('Globex Logistics', 13920.5, 5000.0, 8920.5)
# ('Initech Audit',    7200.0,  7200.0, 0.0)
```

## 8. Caso de uso real

**F-FIN-04 — Days Sales Outstanding (DSO).** El agente calcula el DSO de cada cliente: `AVG(julianday(paid_at) - julianday(issued_at))` agrupado por `client_id`, con un `JOIN` entre `invoice` y `payment`. Los clientes con DSO > 45 días generan un hallazgo. El tramo de cálculo SQL es determinístico; el tramo agéntico es el texto del hallazgo («el cliente X lleva 60 días de DSO, históricamente paga antes del día 30 — posible problema de liquidez»).

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| JOIN entre `invoice` y `payment` para calcular saldo | Determinístico | Regla cerrada: `total - pagado = pendiente`. |
| `GROUP BY client_id` + `SUM` para agregar por cliente | Determinístico | Función matemática sin ambigüedad. |
| `HAVING pending > 0` para filtrar deudores | Determinístico | Regla numérica. |
| Decidir si el saldo pendiente de un cliente es «preocupante» | Agéntico | Depende del historial, del sector, del acuerdo contractual — sin regla única. |
| Redactar el email al CFO con el resumen de saldos | Agéntico | Salida para humano con contexto del negocio. |

## 10. Errores típicos

**1. Filtro de tabla derecha en `WHERE` en lugar de `ON` en un `LEFT JOIN`.**
*Síntoma*: la query supuestamente devuelve «clientes sin facturas» pero devuelve cero filas.
*Causa*: `LEFT JOIN invoice i ON ... WHERE i.period = '2026-04'` — el `WHERE` convierte el `LEFT JOIN` en `INNER JOIN` porque filtra las filas con `i.period = NULL`.
*Arreglo*: mover el filtro de la tabla derecha al `ON`: `LEFT JOIN invoice i ON ... AND i.period = '2026-04'`.

**2. Columna no agregada en `SELECT` con `GROUP BY`.**
*Síntoma*: en Postgres, error `column "i.vendor_name" must appear in the GROUP BY clause or be used in an aggregate function`.
*Causa*: incluiste `i.vendor_name` en el `SELECT` sin añadirla al `GROUP BY`.
*Arreglo*: o la añades al `GROUP BY` o la eliminas del `SELECT` (o usas `MAX(i.vendor_name)` si quieres un valor representativo).

**3. `HAVING` con alias en Postgres.**
*Síntoma*: `ERROR: column "total" does not exist` al usar `HAVING total > 10000` con `total` como alias.
*Causa*: Postgres evalúa `HAVING` antes de resolver los alias del `SELECT`.
*Arreglo*: repite la expresión: `HAVING SUM(i.total_eur) > 10000`. O usa un CTE.

**4. Subquery correlacionada con N filas.**
*Síntoma*: la query tarda 45 segundos con 10 000 facturas.
*Causa*: la subquery correlacionada se ejecuta una vez por cada fila de la query externa.
*Arreglo*: convertir a CTE con `ROW_NUMBER() OVER (PARTITION BY ...)` — el motor la ejecuta una vez.

**5. `COUNT(*)` vs `COUNT(columna)`.**
*Síntoma*: `COUNT(*)` y `COUNT(paid_at)` dan resultados distintos.
*Causa*: `COUNT(columna)` excluye los `NULL`. `COUNT(*)` cuenta todas las filas.
*Arreglo*: si quieres contar filas con valor en esa columna, usa `COUNT(paid_at)`. Si quieres contar todas las filas del grupo, usa `COUNT(*)`.

## Hilo conductor

Vienes de A07 (`async-await-eventloop`), donde viste que `aiosqlite` no es async real — usa un hilo de fondo. Aquí llegas a la capa de persistencia de esta app: SQLAlchemy 2.x sobre SQLite. El objetivo no es solo aprender la API de SQLAlchemy; es entender qué hace esta app concretamente, y trazar la frontera honesta donde SQLite se queda corto y Postgres toma el relevo. Esa frontera es la motivación de C02 (`alembic-migrations`) y C03 (`multitenancy-rls`), donde operarás con la capa de schema y aislamiento.

## Idea central

Una **base de datos** no es un archivo CSV guardado en disco. La diferencia práctica: un CSV solo lo puede abrir un proceso a la vez de forma segura; si dos usuarios intentan escribir al mismo tiempo, uno pisa los datos del otro. Una base de datos gestiona acceso concurrente — diez usuarios escribiendo simultáneamente sin corrupción. Además, tiene **índices** (como el índice al final de un libro: buscar una factura por ID tarda microsegundos en lugar de leer todo el archivo), **transacciones** (o guardas todo el cambio o no guardas nada — sin medios estados) y **integridad referencial** (no puedes crear una factura que apunte a un proveedor que no existe). Para la app de aprendizaje usamos SQLite, que es una base de datos real en un solo archivo; para producción multitenant usamos Postgres, que añade acceso concurrente real y Row-Level Security.

**SQLAlchemy 2.x** es un ORM (Object-Relational Mapper) que mapea clases Python a tablas SQL. En su API moderna, una tabla es una clase que hereda de `DeclarativeBase`; cada columna es un atributo `Mapped[T]`. El motor async (`AsyncEngine` + `AsyncSession`) envuelve el driver subyacente — `aiosqlite` para SQLite, `asyncpg` para Postgres — con la misma API desde el código de la app. Cambiar de SQLite a Postgres es cambiar la URL y el driver; el código de los repos no cambia. Eso es portabilidad real, no marketing.

## Por qué importa

La app corre sobre SQLite porque no necesita más: un solo usuario, archivo en `data/`, sin servidor que mantener. Para BATUTA multitenant con 10 clientes activos en paralelo, SQLite serializa todas las escrituras en un lock global: el primer tenant que inserta 5 000 facturas en un batch bloquea a los demás. Postgres resuelve eso con MVCC (Multi-Version Concurrency Control): lectores no bloquean escritores. Ese salto se produce en el Capstone (E06); aquí se aprende la capa que hace el salto posible sin reescribir los repos.

El caso de uso **F-VTA-02** (forecast de cierre de ventas) exige queries sobre el histórico de múltiples tenants con joins y filtros. Aprenderás a escribir esa query con SQLAlchemy, entender qué SQL genera, y reconocer cuándo el schema actual es suficiente y cuándo no.

> [!nota]
> Honestidad técnica: `aiosqlite` simula async usando un `ThreadPoolExecutor`. Tu `await session.execute(...)` no libera el event loop mientras el hilo trabaja. Para la app de aprendizaje (1 usuario) esto no importa. Para un servicio multitenant real es la razón técnica de moverse a `asyncpg`, donde el await sí libera el loop entre IOs.

## Cómo funciona por dentro

### El modelo declarativo: Mapped[T]

SQLAlchemy 2.x usa anotaciones de tipo para declarar columnas. Compara el modelo real de la app:

```python
# app/models/content.py — extracto real
from sqlalchemy import String, Text, UniqueConstraint
from sqlalchemy.orm import Mapped, mapped_column
from ..db import Base

class Module(Base):
    __tablename__ = "module"
    __table_args__ = (UniqueConstraint("slug", name="uq_module_slug"),)

    id: Mapped[int] = mapped_column(primary_key=True)
    slug: Mapped[str] = mapped_column(String(128), unique=True, index=True)
    ext_id: Mapped[str] = mapped_column(String(16), unique=True)
    track: Mapped[str] = mapped_column(String(2))
    ord: Mapped[int]
    title: Mapped[str] = mapped_column(String(256))
    body_md: Mapped[str] = mapped_column(Text, default="")
```

`Mapped[int]` sin `mapped_column(...)` explícito usa los defaults: columna NOT NULL del tipo inferido. `Mapped[str | None]` declara una columna nullable. El type checker (mypy, pyright) puede validar los tipos sin ejecutar.

### El engine y la sesión: `app/db.py`

El archivo [`app/db.py`](../../app/db.py) tiene cuatro líneas que importan:

```python
engine: AsyncEngine = create_async_engine(
    settings.database_url,  # sqlite+aiosqlite:////srv/data/learning.db
    echo=False,
    future=True,
)

SessionFactory = async_sessionmaker(
    engine,
    expire_on_commit=False,  # crucial en async: evita reloads implícitos
    class_=AsyncSession,
)

async def get_session():
    async with SessionFactory() as session:
        yield session
```

`expire_on_commit=False` es importante en async: el default SQLAlchemy expira todos los atributos al commit (para forzar un reload). En async, ese reload es un `await` implícito en código que no espera un await — lanza `MissingGreenlet`. Con `False`, los objetos siguen accesibles después del commit con los valores que tenían.

### Unit of work: flush vs commit

SQLAlchemy gestiona los cambios en memoria:

- `session.add(obj)`: registra el objeto como pendiente de inserción.
- `await session.flush()`: envía los cambios a la BD en la transacción actual — sin commit. Otros procesos no los ven aún. Útil para obtener el `id` generado.
- `await session.commit()`: hace commit. Los cambios son visibles para todos.
- `await session.rollback()`: descarta todos los cambios desde el último commit.

Patrón recomendado para escritura:

```python
async def create_invoice(session: AsyncSession, data: dict) -> Invoice:
    invoice = Invoice(**data)
    session.add(invoice)
    async with session.begin():
        await session.flush()   # obtiene el id generado por la BD
        return invoice           # id disponible aquí, antes del commit
```

### Queries con `select()`

SQLAlchemy 2.x usa la función `select()`, no el método `.query()` de v1:

```python
from sqlalchemy import select
from sqlalchemy.orm import selectinload

async def get_module_with_exercises(
    session: AsyncSession,
    slug: str,
) -> Module | None:
    result = await session.execute(
        select(Module)
        .where(Module.slug == slug)
        .options(selectinload(Module.exercises))  # evita N+1
    )
    return result.scalar_one_or_none()
```

`scalar_one_or_none()` devuelve el objeto o `None`. `scalars().all()` devuelve lista. `scalar_one()` lanza si no hay exactamente uno.

### Joins explícitos

Para queries que cruzan tablas:

```python
from sqlalchemy import select

# facturas de ACME con nombre del proveedor
stmt = (
    select(Invoice.id, Invoice.total_eur, Vendor.name)
    .join(Vendor, Invoice.vendor_id == Vendor.id)
    .where(Invoice.tenant_id == "acme")
    .where(Invoice.period == "2026-04")
    .order_by(Invoice.issued_at.desc())
)
rows = (await session.execute(stmt)).all()
```

### La frontera con Postgres: qué cambia y qué no

Cuando en el Capstone migras a Postgres, esto cambia en `app/db.py`:

```python
# cambio 1: URL y driver
engine = create_async_engine(
    "postgresql+asyncpg://batuta:${PGPASSWORD}@db:5432/batuta",
    # cambio 2: pool configurado (SQLite no lo necesita)
    pool_size=10,
    max_overflow=5,
    pool_timeout=30,
    pool_pre_ping=True,
)
```

Los repos, servicios y modelos no se tocan. Esa portabilidad es el punto de diseño central de SQLAlchemy.

| Aspecto | SQLite + aiosqlite | Postgres + asyncpg |
|---------|-------------------|--------------------|
| Async real | No (hilo de fondo) | Sí (event loop nativo) |
| Escrituras concurrentes | Lock global | MVCC (sin bloqueo) |
| RLS | No disponible | `ENABLE ROW LEVEL SECURITY` |
| Pool de conexiones | No aplica | `QueuePool` configurable |
| Código de repos | — | Sin cambios |
| Código de modelos | — | Sin cambios |

## Ejemplo conducido

El content loader necesita saber si un módulo cambió para no re-seeder:

```python
from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession
from app.models.content import Module

async def get_module_hash(session: AsyncSession, ext_id: str) -> str | None:
    result = await session.execute(
        select(Module.content_hash).where(Module.ext_id == ext_id)
    )
    return result.scalar_one_or_none()

async def upsert_module(session: AsyncSession, data: dict) -> Module:
    existing = (await session.execute(
        select(Module).where(Module.ext_id == data["ext_id"])
    )).scalar_one_or_none()

    if existing is None:
        module = Module(**data)
        session.add(module)
    elif existing.content_hash != data["content_hash"]:
        for key, val in data.items():
            setattr(existing, key, val)
        module = existing
    else:
        return existing  # sin cambios, sin escritura

    await session.flush()
    return module
```

## Errores típicos

**1. Olvidar `await` en `session.execute()`.**
*Síntoma*: la variable contiene un objeto coroutine; iterar sobre él da caracteres.
*Causa*: `result = session.execute(stmt)` sin `await`.
*Arreglo*: `result = await session.execute(stmt)`. Python 3.11+ lanza `RuntimeWarning: coroutine was never awaited`.

**2. `MissingGreenlet` al acceder a un atributo post-commit.**
*Síntoma*: `sqlalchemy.exc.MissingGreenlet: greenlet_spawn has not been called` al acceder a `module.exercises` después de `commit()`.
*Causa*: `expire_on_commit=True` (el default). SQLAlchemy marca el objeto como expirado; el reload lazy lanza en async porque no hay sesión activa.
*Arreglo*: `expire_on_commit=False` en `async_sessionmaker` (ya presente en `app/db.py`), o carga las relaciones antes del commit con `selectinload`.

**3. Usar `session.query()` de SQLAlchemy v1.**
*Síntoma*: `LegacyAPIWarning` en los logs.
*Causa*: `session.query(Module).filter(...)` es la API antigua.
*Arreglo*: migrar a `select(Module).where(...) + await session.execute(...)`.

**4. N+1 queries por relaciones lazy.**
*Síntoma*: un endpoint que lista 20 módulos hace 21 queries.
*Causa*: relación con `lazy="select"` (el default). Al acceder a `module.exercises`, SQLAlchemy hace un SELECT adicional por módulo.
*Arreglo*: `selectinload(Module.exercises)` en la query principal. Hace 2 queries totales.

**5. Comparar sin `await` en tests pytest-asyncio.**
*Síntoma*: el test pasa siempre aunque el assertion debería fallar.
*Causa*: `assert session.execute(stmt)` sin `await` compara un objeto coroutine con `None`, que es siempre truthy.
*Arreglo*: siempre `result = await session.execute(stmt)` en tests async.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Mapear columnas a `Mapped[T]` | determinístico | Regla mecánica tipo Python ↔ tipo SQL. Sin decisión contextual. |
| Detectar N+1 y añadir `selectinload` | determinístico | `echo=True` muestra las queries; el patrón N+1 es reconocible automáticamente. |
| Elegir entre SQLite y Postgres para un cliente | agéntico | Depende de tenants, concurrencia, regulación, presupuesto — sin regla única. |
| Decidir qué relaciones cargar con eager loading | agéntico | Depende del patrón de acceso del endpoint — cambia con el cliente y el feature. |
| Generar la migración Alembic para un modelo nuevo | determinístico | `alembic revision --autogenerate` + revisión manual. Proceso cerrado. |

## 2. Idea central

**ORM** (Object-Relational Mapper) es una capa que convierte filas de tabla en objetos Python y viceversa. SQLAlchemy 2.x con `Mapped[]` es la forma moderna: defines las columnas como atributos tipados, SQLAlchemy genera el SQL, y tú trabajas con instancias de clase. La imagen mental: una `AsyncSession` es un carrito de supermercado — acumula objetos (agregados, modificados), y `commit()` los empuja a la BD de golpe o los descarta con `rollback()`.

## 3. Por qué importa

Sin ORM, cada query es SQL crudo con concatenación de strings: la ruta más corta a una inyección SQL y a código ilegible cuando hay joins. Sin entender el ciclo de sesión, el error N+1 (una query por cada fila del resultado) destruye el rendimiento. Sin entender `commit` vs `flush`, guardas datos a medias o pierdes cambios.

La ficha F-FIN-02 (forecast de cash flow leyendo el ledger histórico) lee cientos de filas de transacciones contables con joins entre `invoice`, `vendor` y `audit_run`. Si no conoces `selectinload` y `joinedload`, esa query se convierte en N+1 con N=número de facturas.

## 4. Cómo funciona por dentro

### ORM vs SQL crudo: cuándo usar cada uno

| Criterio | ORM (SQLAlchemy) | SQL crudo |
|----------|------------------|-----------|
| Queries CRUD estándar | Recomendado | Innecesario |
| Queries analíticas complejas (window functions, CTEs) | Posible pero verboso | Más legible |
| Seguridad (parámetros) | Automático | Manual — riesgo de inyección |
| Portabilidad de dialect (SQLite ↔ Postgres) | Alta | Baja |

En bato-learning: todo CRUD usa SQLAlchemy. Si en el futuro se necesitan analytics complejos, se pueden pasar strings SQL con `text()` con parámetros bound — nunca concatenados.

### Declarative base y `Mapped[]`

El modelo ORM moderno de SQLAlchemy 2.x usa anotaciones de tipo:

```python
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, relationship
from sqlalchemy import String, Integer, Text, ForeignKey

class Base(DeclarativeBase):
    pass

class Module(Base):
    __tablename__ = "module"

    id: Mapped[int] = mapped_column(primary_key=True)
    slug: Mapped[str] = mapped_column(String(128), unique=True, index=True)
    ext_id: Mapped[str] = mapped_column(String(16), unique=True)
    track: Mapped[str] = mapped_column(String(2))
    ord: Mapped[int] = mapped_column(Integer)
    title: Mapped[str] = mapped_column(String(256))
    body_md: Mapped[str] = mapped_column(Text, default="")

    exercises: Mapped[list["Exercise"]] = relationship(
        back_populates="module",
        cascade="all, delete-orphan",
        order_by="Exercise.ord",
    )
```

`Mapped[int]` no es solo documentación: SQLAlchemy 2.x usa esa anotación para inferir el tipo de columna. `mapped_column()` añade constraints. La diferencia con el estilo v1 (`Column(Integer, ...)`) es que el estilo tipado detecta errores en tiempo de edición con mypy o Pyright.

Este es exactamente el código de [`app/models/content.py`](../../app/models/content.py) que ya corre en la app.

### Engine y SessionFactory

```python
from sqlalchemy.ext.asyncio import (
    AsyncEngine, AsyncSession, async_sessionmaker, create_async_engine
)

engine: AsyncEngine = create_async_engine(
    "sqlite+aiosqlite:///data/app.db",
    echo=False,
    future=True,
)

SessionFactory = async_sessionmaker(
    engine, expire_on_commit=False, class_=AsyncSession
)
```

`expire_on_commit=False` es importante: sin esto, después de un `commit()`, SQLAlchemy marca todos los atributos del objeto como «expirados» y hace una query extra la próxima vez que los leas. En un contexto async esto causa un `MissingGreenlet` error. Con `expire_on_commit=False`, los atributos se mantienen tal como estaban después del commit.

> [!cuidado]
> **Honestidad técnica**: `aiosqlite` **no es async real**. Internamente abre un hilo del sistema operativo por conexión y hace las operaciones SQLite síncronas en ese hilo. El `await` que ves en tus queries no evita el bloqueo de I/O; lo mueve a ese hilo oculto. Para producción con múltiples usuarios concurrentes, `asyncpg` + Postgres es la opción real. Este curso usa SQLite porque cero servidores para aprender; C03 explica cuándo hacer el salto y qué cambia.

### Sesión async y unit of work

`AsyncSession` implementa el patrón **unit of work**: acumula cambios en memoria y los envía a la BD en batch al hacer `commit()`.

```python
async with SessionFactory() as session:
    # SELECT
    result = await session.execute(select(Module).where(Module.track == "B"))
    modules = list(result.scalars().all())

    # INSERT
    new_mod = Module(slug="test-99", ext_id="B99", track="B", ord=99, title="Test")
    session.add(new_mod)
    await session.commit()   # escribe en BD; genera el id autoincrementado

    # UPDATE — sqlalchemy detecta el cambio de atributo
    new_mod.title = "Test Updated"
    await session.commit()

    # DELETE
    await session.delete(new_mod)
    await session.commit()
```

`flush()` envía los SQL a la BD **sin commit**: útil para obtener el `id` generado antes de necesitarlo en una relación, dentro de la misma transacción. `commit()` hace `flush` + commit de la transacción.

### Queries: `select`, `where`, `order_by`, `join`

```python
from sqlalchemy import select

# query básica
stmt = select(Module).where(Module.track == "B").order_by(Module.ord)
result = await session.execute(stmt)
modules: list[Module] = list(result.scalars().all())

# join explícito — para filtrar por campo de la tabla relacionada
from sqlalchemy.orm import selectinload
stmt = (
    select(Exercise)
    .join(Exercise.module)
    .where(Module.track == "C")
    .order_by(Exercise.ord)
)
result = await session.execute(stmt)
exercises = list(result.scalars().all())
```

`result.scalars().all()` convierte el `Result` en una lista de instancias ORM. Sin `.scalars()`, obtienes tuplas `Row` — útil cuando el SELECT tiene múltiples columnas o expresiones de agregación.

### Eager loading: `selectinload` y `joinedload`

El problema N+1: si tienes 20 módulos y para cada uno accedes a `module.exercises`, SQLAlchemy lanza 20 queries adicionales (una por módulo). El resultado son 21 queries para un listado que debería ser 1 o 2.

La solución: declarar el eager loading en la query:

```python
from sqlalchemy.orm import selectinload

stmt = (
    select(Module)
    .where(Module.slug == slug)
    .options(
        selectinload(Module.exercises),
        selectinload(Module.quizzes).selectinload(Quiz.options),
    )
)
result = await session.execute(stmt)
module = result.scalar_one_or_none()
```

Este es exactamente el código de `app/repos/modules.py:get_by_slug`. Con `selectinload`, SQLAlchemy hace 2 queries adicionales (una para todos los exercises, otra para todos los quizzes+options) en lugar de N.

| Estrategia | Queries | Cuándo usar |
|------------|---------|-------------|
| `lazy` (default) | 1 + N | Nunca en async — lanza `MissingGreenlet` |
| `selectinload` | 1 + 1 por relación | Relaciones 1-N con muchos objetos |
| `joinedload` | 1 (JOIN SQL) | Relaciones muchos-1 o 1-1 |

> [!cuidado]
> En contextos async, `lazy loading` lanza `sqlalchemy.exc.MissingGreenlet`. Siempre debes declarar el eager loading explícitamente o acceder a las relaciones dentro del contexto de la sesión abierta. Si accedes a `module.exercises` después de cerrar la sesión, SQLAlchemy intenta una query lazy que falla.

### Transacciones y `session.begin()`

```python
# forma explícita con begin()
async with session.begin():
    session.add(invoice)
    session.add(audit_run)
    # si cualquier operación lanza excepción → rollback automático
    # si el bloque termina sin excepción → commit automático
```

`async with session.begin()` garantiza rollback si algo falla. El `get_session()` de la app (en `app/db.py`) no usa `begin()` explícito — la sesión hace commit en llamadas manuales a `session.commit()`. Para operaciones multi-paso que deben ser atómicas, usa `begin()`.

## 5. Ejemplo conducido

Replica en miniatura lo que hace `app/repos/modules.py`, con una BD en memoria:

```python
import asyncio
from sqlalchemy import String, Integer, ForeignKey, select
from sqlalchemy.orm import (
    DeclarativeBase, Mapped, mapped_column, relationship, selectinload
)
from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker, create_async_engine


class Base(DeclarativeBase):
    pass


class Vendor(Base):
    __tablename__ = "vendor"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(128))
    tenant: Mapped[str] = mapped_column(String(32))
    invoices: Mapped[list["Invoice"]] = relationship(
        back_populates="vendor", cascade="all, delete-orphan"
    )


class Invoice(Base):
    __tablename__ = "invoice"
    id: Mapped[int] = mapped_column(primary_key=True)
    vendor_id: Mapped[int] = mapped_column(ForeignKey("vendor.id"))
    amount: Mapped[float]
    period: Mapped[str] = mapped_column(String(7))
    vendor: Mapped["Vendor"] = relationship(back_populates="invoices")


engine = create_async_engine("sqlite+aiosqlite:///:memory:", future=True)
Factory = async_sessionmaker(engine, expire_on_commit=False, class_=AsyncSession)


async def main() -> None:
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)

    async with Factory() as session:
        acme = Vendor(name="ACME Manufacturing", tenant="acme")
        session.add(acme)
        await session.flush()       # genera acme.id sin commit aún

        inv = Invoice(vendor_id=acme.id, amount=4820.0, period="2026-04")
        session.add(inv)
        await session.commit()      # persiste vendor + invoice en una transacción

    async with Factory() as session:
        stmt = (
            select(Vendor)
            .where(Vendor.tenant == "acme")
            .options(selectinload(Vendor.invoices))
        )
        result = await session.execute(stmt)
        vendor = result.scalar_one()
        print(f"{vendor.name}: {len(vendor.invoices)} facturas")  # ACME Manufacturing: 1 facturas


asyncio.run(main())
```

## 8. Caso de uso real

**F-FIN-02 — Forecast de cash flow leyendo ledger histórico**: Initech Audit necesita proyectar el flujo de caja de los próximos 90 días. El agente lee las transacciones históricas de los últimos 12 meses desde la tabla `invoice` con un join a `vendor` (para filtrar por categoría de proveedor). Si no se usa `selectinload`, la query por 500 facturas lanza 500 queries adicionales para los vendors — 8 segundos en lugar de 300 ms. El tramo de lectura de datos es completamente determinístico; el tramo agéntico (proyectar tendencias con contexto del negocio) viene después.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Escribir el modelo ORM y mapear columnas | Determinístico | Regla cerrada: el schema de la BD determina el modelo. |
| Ejecutar `select().where().order_by()` | Determinístico | La query es código; el resultado es función de los datos. |
| Detectar una factura duplicada (mismo monto, misma fecha, mismo vendor) | Determinístico | Regla cerrada: igualdad exacta en tres campos. |
| Clasificar si una diferencia de monto es timing o error de contabilización | Agéntico | Requiere contexto del negocio; no hay regla cerrada. |
| Redactar el resumen del cash flow para el CFO | Agéntico | Salida para humano; depende de contexto y estilo de Initech. |

## 10. Errores típicos

**1. Lazy loading en contexto async.**
*Síntoma*: `sqlalchemy.exc.MissingGreenlet: greenlet_spawn has not been called...` al acceder a `module.exercises` fuera de la sesión.
*Causa*: acceder a una relación con lazy loading (el default) después de cerrar la sesión.
*Arreglo*: añadir `selectinload(Module.exercises)` en la query; o acceder a la relación dentro del bloque `async with`.

**2. `expire_on_commit=True` (default) en contexto async.**
*Síntoma*: `MissingGreenlet` al leer un atributo del objeto después de `commit()`.
*Causa*: SQLAlchemy marca los atributos como expirados al hacer commit; en async, la siguiente lectura intenta una query lazy que falla.
*Arreglo*: `async_sessionmaker(engine, expire_on_commit=False, ...)`.

**3. Olvidar `await` antes de `session.execute()`.**
*Síntoma*: el resultado es una corrutina, no un `Result`. Las operaciones sobre él fallan con `AttributeError`.
*Causa*: `session.execute(stmt)` es una corrutina async; sin `await`, solo creas el objeto corrutina.
*Arreglo*: siempre `result = await session.execute(stmt)`.

**4. Problema N+1.**
*Síntoma*: la app es rápida con 5 módulos en dev; lenta con 50 en staging. Con `echo=True` en el engine se ven 51 queries donde debería haber 2.
*Causa*: acceder a `module.exercises` dentro de un bucle sin eager loading declarado.
*Arreglo*: `.options(selectinload(Module.exercises))` en la query que lista módulos.

**5. Concatenar variables en el string SQL con `text()`.**
*Síntoma*: inyección SQL posible; falsos negativos en tests de seguridad.
*Causa*: `text(f"SELECT * FROM module WHERE track = '{track}'")` con `track` del usuario.
*Arreglo*: usar siempre la API ORM (`select(Module).where(Module.track == track)`) o `text("...").bindparams(track=track)`.

## 11. Para profundizar

> [!profundizar]
> [`app/models/content.py`](../../app/models/content.py) tiene los tres modelos reales del catálogo: `Module`, `Exercise`, y `Quiz`. Busca la relación `Module.exercises` y compara su `cascade="all, delete-orphan"` con el `ForeignKey(..., ondelete="CASCADE")` en `alembic/versions/0001_initial.py`. Son dos niveles del mismo concepto: el ORM gestiona el cascade en Python; la BD lo aplica directamente en SQL. Ambos deben estar sincronizados.

> [!profundizar]
> [`app/repos/modules.py`](../../app/repos/modules.py) es la referencia completa. La función `get_by_slug` muestra `selectinload` anidado: `.selectinload(Module.quizzes).selectinload(Quiz.options)`. Eso carga quizzes **y** sus opciones en 2 queries adicionales en lugar de 1+N+M. Cuenta cuántos `SELECT` dispara sin `selectinload` vs con él — la diferencia es la definición del problema N+1.

> [!profundizar]
> [`alembic/versions/0001_initial.py`](../../alembic/versions/0001_initial.py) es la migración inicial. Compara `op.create_table("module", ...)` con la clase `Module` de `content.py`. Cada columna debe aparecer en ambos lugares. Cuando C02 (Alembic) enseñe cómo autogenerar migraciones, entenderás por qué el autogenerador a veces falla: si la columna existe en el modelo pero no en la migración (o viceversa), la BD está desfasada.

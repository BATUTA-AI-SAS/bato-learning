## 1. Hilo conductor

C07 (`indexes-and-explain`) te mostró cómo los índices B-tree aceleran búsquedas exactas. Aquí llega el caso que el B-tree no puede resolver: «dame los documentos más parecidos *en significado* a esta pregunta». Eso es búsqueda semántica, y requiere un tipo de índice completamente distinto. C09 (`rag-end-to-end`) construye el pipeline RAG completo sobre lo que aprendes aquí.

## 2. Idea central

Un **embedding** es un vector de números flotantes (típicamente 512–3072 dimensiones) generado por un modelo de lenguaje. Dos textos similares en significado producen vectores similares — sus puntas apuntan en la misma dirección en el espacio de alta dimensión. La **similitud coseno** mide ese ángulo: 1.0 = idénticos, 0.0 = sin relación, -1.0 = opuestos. Una **base de datos vectorial** indexa esos vectores con un índice HNSW (Hierarchical Navigable Small World) que permite encontrar los K más cercanos en `O(log N)` en lugar de `O(N)`.

## 3. Por qué importa

ACME tiene 2 000 manuales de procedimiento contable (F-CTA-01). El auditor pregunta: «¿qué manual cubre la revalorización de activos fijos en ISO 16?». Una búsqueda por palabras clave falla si el manual dice «depreciación acelerada de inmuebles» — son conceptos relacionados pero sin palabras en común. Un embedding convierte la pregunta y cada manual en vectores; los vectores más cercanos son los más relevantes, sin importar las palabras exactas usadas. Eso es el corazón de RAG.

La ficha **F-CTA-01** (auditoría contable) usa embeddings para encontrar hallazgos similares de períodos anteriores cuando analiza una factura anómala. Sin esto, el agente no tiene contexto histórico.

## 4. Cómo funciona por dentro

### Del texto al vector: el modelo de embedding

Un modelo de embedding toma texto y devuelve un array de flotantes. La misma cadena siempre produce el mismo vector (es determinístico). Lo que cambia entre modelos es la calidad del espacio semántico — cuán bien «separa» conceptos distintos y «agrupa» conceptos similares.

```python
# con el SDK de Voyage AI (voyage-3, el más recomendado para LATAM en español)
import voyageai

client = voyageai.Client()  # lee VOYAGE_API_KEY del entorno
result = client.embed(
    ["revalorización de activos fijos", "depreciación acelerada de inmuebles"],
    model="voyage-3",
    input_type="document",
)
v1 = result.embeddings[0]  # lista de 1024 floats
v2 = result.embeddings[1]  # lista de 1024 floats
# v1 y v2 son parecidos semánticamente → similitud coseno alta
```

### Comparativa de modelos de embedding (2026)

| Modelo | Dimensiones | Input máx | Precio | Nota |
|--------|-------------|-----------|--------|------|
| **voyage-3** (Voyage AI) | 1024 | 32K tokens | $0.06/1M tokens | Mejor relación calidad/costo general; buen rendimiento en español |
| **voyage-3-large** (Voyage AI) | 1024–2048 | 32K tokens | $0.12/1M tokens | Estado del arte en MTEB; soporta Matryoshka (truncar dimensiones) |
| **text-embedding-3-large** (OpenAI) | 3072 | 8K tokens | $0.13/1M tokens | Sólido; 3× más dimensiones que voyage-3 sin ventaja proporcional en calidad |
| **embed-v3** (Cohere) | 1024 | 512 tokens | $0.10/1M tokens | Ventana de contexto corta — limitación real para documentos largos |
| **BGE-M3** (BAAI, open-source) | 1024 | 8K tokens | Gratis (auto-host) | Multilingüe, denso + sparse, excelente para español; requiere GPU propia o infra |

**Recomendación para PYME LATAM**: `voyage-3` como default. Si el corpus tiene documentos largos (manuales, contratos), `voyage-3-large`. Si el presupuesto es crítico y hay GPU disponible, `BGE-M3`.

### Similitud: coseno, dot product, L2

Dado que tienes los vectores, ¿cómo mides si dos son «cercanos»?

```python
import math

def cosine_similarity(v1: list[float], v2: list[float]) -> float:
    dot = sum(a * b for a, b in zip(v1, v2))
    norm1 = math.sqrt(sum(a * a for a in v1))
    norm2 = math.sqrt(sum(b * b for b in v2))
    return dot / (norm1 * norm2)

# similitud coseno ignora la magnitud del vector — solo el ángulo
# útil cuando los textos tienen longitudes muy distintas
```

| Métrica | Fórmula | Cuándo usarla |
|---------|---------|---------------|
| **Coseno** | `(v1·v2) / (‖v1‖ × ‖v2‖)` | La más común para texto. Rango [-1, 1]. Ignora magnitud. |
| **Dot product** | `v1·v2` | Cuando los vectores están normalizados (‖v‖ = 1). Más rápido que coseno. pgvector lo prefiere con HNSW. |
| **L2 (Euclidiana)** | `√Σ(v1ᵢ - v2ᵢ)²` | Para embedding de imágenes o cuando la magnitud importa. |

> [!nota]
> La mayoría de modelos de embedding devuelven vectores ya normalizados. En ese caso, similitud coseno y dot product son equivalentes. `pgvector` con operador `<=>` (coseno) o `<#>` (dot product negativo) — ambos dan el mismo resultado con vectores normalizados.

### pgvector: búsqueda vectorial dentro de Postgres

**pgvector 0.8.2** es la extensión que añade un tipo `vector(N)` a Postgres y los índices HNSW e IVFFlat para búsqueda aproximada. Es la opción más sencilla si ya tienes Postgres.

```sql
-- habilitar la extensión
CREATE EXTENSION IF NOT EXISTS vector;

-- tabla de documentos con embedding
CREATE TABLE document_chunk (
    id         BIGSERIAL PRIMARY KEY,
    tenant_id  TEXT NOT NULL,
    doc_id     TEXT NOT NULL,
    chunk_text TEXT NOT NULL,
    embedding  vector(1024),        -- dimensiones del modelo voyage-3
    metadata   JSONB DEFAULT '{}'
);

-- índice HNSW para búsqueda por similitud coseno
-- m=16 (conectividad del grafo), ef_construction=64 (calidad del índice al construir)
CREATE INDEX idx_chunks_embedding ON document_chunk
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
```

Query de similitud:

```sql
-- los 5 chunks más similares a una pregunta (dado su embedding como parámetro)
SELECT chunk_text, 1 - (embedding <=> $1::vector) AS similarity
FROM document_chunk
WHERE tenant_id = 'acme'
ORDER BY embedding <=> $1::vector
LIMIT 5;
```

`<=>` es el operador de distancia coseno. `1 - distancia` convierte a similitud (0 a 1). El `WHERE tenant_id = 'acme'` filtra antes de la búsqueda vectorial — pgvector 0.8+ soporta **iterative index scans** para no perder resultados cuando el filtro es selectivo.

**Límite práctico**: pgvector funciona bien hasta ~10 M vectores con HNSW en un servidor con suficiente RAM para el índice en memoria. Con más, la latencia sube y consideras Qdrant.

### Qdrant: vector database dedicada, self-host o cloud

**Qdrant** (Rust, open-source) es la opción cuando necesitas escalar más allá de pgvector o quieres features avanzadas (filtros payload, multitenancy nativo, cuantización 4-bit, GPU indexing). Disponible como Docker self-host o en Qdrant Cloud.

```python
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct

client = QdrantClient(url="http://localhost:6333")

# crear colección
client.create_collection(
    collection_name="acme_documents",
    vectors_config=VectorParams(size=1024, distance=Distance.COSINE),
)

# insertar vectores con payload (metadata filtrable)
client.upsert(
    collection_name="acme_documents",
    points=[
        PointStruct(
            id=1,
            vector=[0.1, 0.2, ...],      # 1024 floats
            payload={"tenant_id": "acme", "doc_id": "F-CTA-01", "period": "2026-04"}
        )
    ]
)

# búsqueda con filtro de payload
results = client.query_points(
    collection_name="acme_documents",
    query=[0.1, 0.2, ...],               # embedding de la pregunta
    query_filter={"must": [{"key": "tenant_id", "match": {"value": "acme"}}]},
    limit=5,
)
```

### Pinecone: serverless, sin ops, pero con precio

**Pinecone Serverless** (2026) es la opción más sencilla operativamente: cero infraestructura que gestionar. Tier gratuito hasta 2 GB. Producción desde $20/mes (Builder) con ~$0.33/GB/mes de almacenamiento. Para 10 M vectores de 1024 dims (~40 GB): ~$13/mes de almacenamiento + $16-24 por millón de lecturas.

```python
from pinecone import Pinecone

pc = Pinecone(api_key="PINECONE_API_KEY")
index = pc.Index("acme-documents")

index.upsert(vectors=[{
    "id": "chunk-001",
    "values": [0.1, 0.2, ...],
    "metadata": {"tenant_id": "acme", "doc_id": "F-CTA-01"}
}])

results = index.query(
    vector=[0.1, 0.2, ...],
    filter={"tenant_id": {"$eq": "acme"}},
    top_k=5,
    include_metadata=True,
)
```

### Weaviate: módulos integrados

**Weaviate** incluye módulos para embedding automático (conecta directamente con OpenAI, Cohere, etc.) y tiene un lenguaje de query propio (GraphQL + WQL). Útil si quieres delegar el embedding al servidor. El costo operativo es mayor que Qdrant.

### Decision tree: ¿cuál vector DB elegir?

```
¿Ya usas Postgres?
  └── Sí → Empieza con pgvector
       ├── ¿Corpus < 10 M vectores? → pgvector es suficiente
       └── ¿Corpus > 10 M o necesitas cuantización/GPU? → migra a Qdrant

¿Sin Postgres, quieres cero infraestructura?
  └── Sí → Pinecone Serverless (más simple, más caro a escala)

¿Necesitas multitenancy fuerte, cuantización, billones de vectores?
  └── Qdrant Cloud o self-host

¿Quieres embedding automático integrado + módulos?
  └── Weaviate
```

**Para PYME LATAM que ya tiene Postgres**: `pgvector` es la respuesta en el 90% de los casos. Mismo servidor, mismo backup, mismas herramientas. Qdrant cuando el corpus crece o necesitas features avanzadas.

## 5. Ejemplo conducido

Similitud coseno sin dependencias externas (válido en Pyodide):

```python
import math

def cosine_similarity(v1: list[float], v2: list[float]) -> float:
    dot = sum(a * b for a, b in zip(v1, v2))
    norm = lambda v: math.sqrt(sum(x * x for x in v))
    return dot / (norm(v1) * norm(v2))

def top_k_similar(query: list[float], corpus: list[tuple[str, list[float]]], k: int):
    """Devuelve los k documentos más similares al query vector."""
    scored = [(text, cosine_similarity(query, vec)) for text, vec in corpus]
    return sorted(scored, key=lambda x: x[1], reverse=True)[:k]

# vectores sintéticos de 4 dims para ilustrar
corpus = [
    ("revalorización de activos fijos",     [0.9, 0.1, 0.1, 0.2]),
    ("depreciación acelerada de inmuebles", [0.8, 0.2, 0.1, 0.3]),
    ("política de dividendos",              [0.1, 0.9, 0.2, 0.1]),
    ("inventario de materia prima",         [0.1, 0.1, 0.9, 0.2]),
]
query = [0.85, 0.15, 0.05, 0.25]  # similar al primer par

results = top_k_similar(query, corpus, k=2)
for text, score in results:
    print(f"{score:.3f}  {text}")
# 0.992  revalorización de activos fijos
# 0.978  depreciación acelerada de inmuebles
```

## 8. Caso de uso real

**F-CTA-01 — Auditoría contable automática.** BATUTA indexa los manuales contables de ACME (F-CTA-01) en pgvector cuando el cliente sube sus documentos. Al detectar una factura anómala, el agente genera un embedding de la descripción de la anomalía y recupera los 5 párrafos más similares de los manuales. El auditor ve el hallazgo con las referencias exactas. El chunking y el embedding son determinísticos; la selección de qué párrafos son realmente relevantes para el contexto del cliente es agéntica.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Generar el embedding de un texto con un modelo dado | Determinístico | Mismo texto + mismo modelo → mismo vector siempre. |
| Calcular similitud coseno entre dos vectores | Determinístico | Operación matemática cerrada. |
| Recuperar los top-K vectores más cercanos | Determinístico | HNSW aproximado pero estable dado el índice. |
| Decidir si un resultado de similitud 0.72 es «suficientemente relevante» | Agéntico | El umbral depende del dominio, del cliente y de la calidad del corpus — sin regla única. |
| Sintetizar los chunks recuperados en una respuesta para el auditor | Agéntico | Salida para humano con contexto del negocio. |

## 10. Errores típicos

**1. Dimensiones del vector no coinciden con el índice.**
*Síntoma*: `ERROR: expected 1024 dimensions, not 3072`.
*Causa*: el modelo genera vectores de 3072 dims (text-embedding-3-large) pero el índice pgvector se creó con `vector(1024)`.
*Arreglo*: definir las dimensiones del índice según el modelo que vas a usar. Cambiar de modelo implica re-embedar todo el corpus y recrear el índice.

**2. Comparar embeddings de modelos distintos.**
*Síntoma*: la búsqueda devuelve resultados sin sentido aunque los textos son relevantes.
*Causa*: los documentos se indexaron con `voyage-3` pero la query se embedó con `text-embedding-3-large`. Los espacios semánticos son distintos — no son comparables.
*Arreglo*: usar siempre el mismo modelo para indexar y para hacer queries.

**3. No filtrar por `tenant_id` antes de la búsqueda vectorial.**
*Síntoma*: el agente de ACME devuelve documentos de Globex en los resultados.
*Causa*: la query vectorial no tiene `WHERE tenant_id = 'acme'` (pgvector) o no tiene filtro de payload (Qdrant).
*Arreglo*: siempre combinar filtro exacto de tenant con la búsqueda vectorial. En pgvector 0.8+, usar iterative index scans para no perder resultados cuando el filtro reduce mucho el corpus.

**4. Indexar chunks demasiado grandes.**
*Síntoma*: la búsqueda devuelve el chunk correcto pero incluye texto irrelevante que confunde al LLM.
*Causa*: chunks de 2000 tokens mezclan múltiples conceptos en un solo vector.
*Arreglo*: chunks de 256–512 tokens con solapamiento del 10–20%. El módulo C09 cubre la estrategia de chunking en detalle.

## 1. Hilo conductor

C04 (`what-is-a-database`) estableció qué es una base de datos relacional y por qué es la herramienta correcta para facturas, proveedores y auditorías. Ahora aprendes a hablar con ella. **SQL** (Structured Query Language) es el lenguaje estándar para eso — el mismo desde 1974, presente en Postgres, SQLite, MySQL, DuckDB, BigQuery y docenas más. Lo que aprendes aquí funciona en todos. C06 (`sql-joins-aggregations`) añade joins y agregaciones; C07 (`indexes-and-explain`) explica por qué algunas queries tardan 30 segundos y cómo arreglarlo.

## 2. Idea central

**SQL** es un lenguaje declarativo: describes *qué* quieres, no *cómo* el motor lo busca. «Dame todas las facturas del período 2026-04 cuyo monto supere 5 000» — eso es SQL. El motor decide si usa un índice o recorre la tabla entera. Tú declaras la intención; el motor decide el plan. La imagen mental: SQL es como el `=VLOOKUP` de Excel, pero sin límite de filas, sin pestañas que se rompen y con la capacidad de cruzar varias tablas a la vez.

## 3. Por qué importa

El agente de la ficha **F-CTA-01** (auditoría contable automática) no usa pandas para leer facturas. Usa SQL. Pandas carga todo a memoria; SQL filtra en la base de datos antes de devolver datos. Con 500 000 facturas, la diferencia es 4 GB de RAM vs 2 MB de resultado filtrado.

Sin SQL, el lector que termina Track 0 (Excel→Python) sabe escribir listas y diccionarios, pero no puede consultar datos estructurados sin cargar todo el archivo. SQL cierra ese hueco. C08 y C09 (embeddings y RAG) también usan SQL — pgvector es una extensión de Postgres; sus queries se escriben en SQL con operadores de distancia.

## 4. Cómo funciona por dentro

### Crear una tabla: `CREATE TABLE`

Antes de insertar datos necesitas definir la estructura:

```sql
CREATE TABLE IF NOT EXISTS invoice (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    tenant_id   TEXT    NOT NULL,
    vendor_name TEXT    NOT NULL,
    total_eur   REAL    NOT NULL,
    period      TEXT    NOT NULL,
    paid        INTEGER NOT NULL DEFAULT 0  -- 0=False, 1=True en SQLite
);
```

Cada columna tiene un tipo. SQLite usa tipos flexibles (`INTEGER`, `REAL`, `TEXT`, `BLOB`, `NULL`). Postgres tiene tipos más ricos (`NUMERIC(12,2)`, `BOOLEAN`, `TIMESTAMP WITH TIME ZONE`, `JSONB`). El concepto es el mismo.

`PRIMARY KEY AUTOINCREMENT` genera el `id` automáticamente en cada inserción. `NOT NULL` impide que la columna quede vacía. `DEFAULT 0` asigna el valor cuando no lo especificas.

### Insertar filas: `INSERT INTO`

```sql
INSERT INTO invoice (tenant_id, vendor_name, total_eur, period, paid)
VALUES ('acme', 'Siemens GmbH', 4820.00, '2026-04', 0);

-- insertar varias filas en un solo statement
INSERT INTO invoice (tenant_id, vendor_name, total_eur, period, paid)
VALUES
    ('acme', 'Bosch AG',    9100.50, '2026-04', 0),
    ('acme', 'DHL Express', 1234.00, '2026-03', 1),
    ('globex', 'Maersk',    5500.00, '2026-04', 0);
```

Las columnas que omites usan su `DEFAULT`. El `id` lo genera el motor.

### Leer filas: `SELECT`

```sql
-- todas las columnas de todas las filas
SELECT * FROM invoice;

-- columnas específicas
SELECT id, vendor_name, total_eur FROM invoice;

-- con alias para renombrar en el resultado
SELECT vendor_name AS proveedor, total_eur AS monto FROM invoice;
```

`SELECT *` es conveniente para explorar; en producción, siempre nombra las columnas que necesitas. Devolver columnas innecesarias desperdicia ancho de banda y complica el código que lee el resultado.

### Filtrar: `WHERE`

```sql
-- facturas no pagadas de ACME
SELECT id, vendor_name, total_eur
FROM invoice
WHERE tenant_id = 'acme'
  AND paid = 0;

-- facturas cuyo monto supere 5000
SELECT vendor_name, total_eur
FROM invoice
WHERE total_eur > 5000;

-- facturas del período 2026-04 o 2026-03
SELECT vendor_name, period, total_eur
FROM invoice
WHERE period IN ('2026-04', '2026-03');

-- vendor_name que empieza con 'B'
SELECT vendor_name
FROM invoice
WHERE vendor_name LIKE 'B%';
```

Operadores de comparación: `=`, `!=`, `<`, `>`, `<=`, `>=`. Lógica: `AND`, `OR`, `NOT`. Rango: `BETWEEN 1000 AND 5000`. Texto: `LIKE 'prefijo%'` (el `%` es comodín de cualquier cantidad de caracteres).

> [!cuidado]
> **Nunca concatenes variables del usuario en una query SQL.**
> ```python
> # MAL — inyección SQL
> query = f"SELECT * FROM invoice WHERE tenant_id = '{user_input}'"
>
> # BIEN — parámetros bind
> cursor.execute("SELECT * FROM invoice WHERE tenant_id = ?", (tenant_id,))
> ```
> Con concatenación, si `user_input` es `' OR '1'='1`, el atacante lee toda la tabla. Los parámetros bind escapan automáticamente. SQLAlchemy hace esto por ti; si usas `sqlite3` directo, usa `?` como placeholder.

### Ordenar: `ORDER BY`

```sql
-- de mayor a menor monto
SELECT vendor_name, total_eur
FROM invoice
WHERE tenant_id = 'acme'
ORDER BY total_eur DESC;

-- por período ascendente, y dentro del mismo período por monto descendente
SELECT period, vendor_name, total_eur
FROM invoice
ORDER BY period ASC, total_eur DESC;
```

`ASC` (ascendente) es el default. `DESC` invierte el orden.

### Limitar resultados: `LIMIT` y `OFFSET`

```sql
-- las 5 facturas más caras de ACME
SELECT vendor_name, total_eur
FROM invoice
WHERE tenant_id = 'acme'
ORDER BY total_eur DESC
LIMIT 5;

-- paginación: segunda página de 10 resultados
SELECT vendor_name, total_eur
FROM invoice
ORDER BY id
LIMIT 10
OFFSET 10;
```

`LIMIT` corta el resultado al número indicado. `OFFSET` salta las primeras N filas — útil para paginación, pero costoso en tablas grandes (el motor cuenta desde el inicio). Para paginación eficiente sobre tablas grandes, usar cursor-based pagination (`WHERE id > last_id LIMIT 10`).

### `NULL`: el valor ausente

`NULL` no es cero ni string vacío; es la ausencia de valor. Comparar con `NULL` usando `=` nunca devuelve filas:

```sql
-- MAL: nunca devuelve filas aunque haya NULLs
SELECT * FROM invoice WHERE paid_at = NULL;

-- BIEN
SELECT * FROM invoice WHERE paid_at IS NULL;
SELECT * FROM invoice WHERE paid_at IS NOT NULL;
```

### Modificar filas: `UPDATE`

```sql
-- marcar una factura como pagada
UPDATE invoice
SET paid = 1
WHERE id = 2;

-- subir 10% el monto de todas las facturas de Bosch AG
UPDATE invoice
SET total_eur = total_eur * 1.10
WHERE vendor_name = 'Bosch AG'
  AND tenant_id = 'acme';
```

> [!cuidado]
> Un `UPDATE` sin `WHERE` modifica **todas** las filas de la tabla. En producción, cualquier `UPDATE` o `DELETE` sin `WHERE` debe bloquearse por política o requerir confirmación explícita.

### Borrar filas: `DELETE`

```sql
-- borrar una factura específica
DELETE FROM invoice WHERE id = 3;

-- borrar facturas pagadas anteriores a 2026
DELETE FROM invoice
WHERE paid = 1
  AND period < '2026-01';
```

`DELETE FROM invoice` sin `WHERE` borra todas las filas (no el schema). `DROP TABLE invoice` borra el schema entero.

### El orden de evaluación SQL

SQL evalúa las cláusulas en este orden, aunque las escribas en otro:

```
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

Esto importa porque no puedes usar un alias de `SELECT` en el `WHERE`:

```sql
-- MAL: proveedor_monto no existe aún cuando WHERE evalúa
SELECT total_eur * 1.19 AS proveedor_monto
FROM invoice
WHERE proveedor_monto > 5000;

-- BIEN: repite la expresión en WHERE
SELECT total_eur * 1.19 AS monto_con_iva
FROM invoice
WHERE total_eur * 1.19 > 5000;
```

## 5. Ejemplo conducido

Usando `sqlite3` en Python (el mismo módulo que Pyodide expone):

```python
import sqlite3

con = sqlite3.connect(":memory:")   # BD en memoria, desaparece al cerrar
cur = con.cursor()

cur.executescript("""
    CREATE TABLE invoice (
        id          INTEGER PRIMARY KEY AUTOINCREMENT,
        tenant_id   TEXT NOT NULL,
        vendor_name TEXT NOT NULL,
        total_eur   REAL NOT NULL,
        period      TEXT NOT NULL,
        paid        INTEGER NOT NULL DEFAULT 0
    );
    INSERT INTO invoice (tenant_id, vendor_name, total_eur, period, paid) VALUES
        ('acme', 'Siemens GmbH', 4820.00, '2026-04', 0),
        ('acme', 'Bosch AG',     9100.50, '2026-04', 0),
        ('acme', 'DHL Express',  1234.00, '2026-03', 1),
        ('globex', 'Maersk',     5500.00, '2026-04', 0);
""")

# las facturas no pagadas de ACME, de mayor a menor monto
cur.execute("""
    SELECT vendor_name, total_eur
    FROM invoice
    WHERE tenant_id = 'acme'
      AND paid = 0
    ORDER BY total_eur DESC
""")
for row in cur.fetchall():
    print(row)   # ('Bosch AG', 9100.5)  →  ('Siemens GmbH', 4820.0)

# marcar Bosch AG como pagada
cur.execute("UPDATE invoice SET paid = 1 WHERE vendor_name = ? AND tenant_id = ?",
            ("Bosch AG", "acme"))
con.commit()
con.close()
```

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Escribir un `SELECT` con `WHERE` y `ORDER BY` | Determinístico | Regla cerrada: el SQL es código; el resultado es función de los datos. |
| Filtrar facturas por monto exacto (`total_eur = 4820.00`) | Determinístico | Igualdad numérica — sin ambigüedad. |
| Decidir si dos facturas con montos diferentes son «la misma» | Agéntico | Requiere contexto (¿es un ajuste? ¿una versión parcial?) — sin regla cerrada. |
| Redactar el hallazgo de auditoría a partir de los resultados SQL | Agéntico | Salida para humano; depende del estilo y contexto del cliente. |

## 10. Errores típicos

**1. `UPDATE` o `DELETE` sin `WHERE`.**
*Síntoma*: desaparecen todas las filas de la tabla.
*Causa*: olvidaste añadir el `WHERE`. SQL modifica todas las filas cuando no hay filtro.
*Arreglo*: siempre prueba el `SELECT` equivalente antes del `UPDATE`/`DELETE`: `SELECT * FROM invoice WHERE paid = 1 AND period < '2026-01'` — si el resultado es el correcto, convierte a `DELETE`.

**2. Comparar `NULL` con `=`.**
*Síntoma*: `WHERE paid_at = NULL` devuelve cero filas aunque haya nulos.
*Causa*: `NULL = NULL` es `NULL` (desconocido), no `True`. El `WHERE` descarta las filas donde la condición no es explícitamente `True`.
*Arreglo*: `WHERE paid_at IS NULL` o `WHERE paid_at IS NOT NULL`.

**3. Concatenar variables de usuario en SQL.**
*Síntoma*: un usuario malicioso pasa `' OR '1'='1` y lee toda la tabla.
*Causa*: `f"SELECT * FROM invoice WHERE tenant_id = '{user_input}'"` — el atacante cierra el string y añade su condición.
*Arreglo*: parámetros bind (`?` en sqlite3, `:param` en SQLAlchemy). El módulo C01 muestra cómo SQLAlchemy los maneja automáticamente.

**4. Usar `SELECT *` en código de producción.**
*Síntoma*: el endpoint devuelve campos sensibles que no debería exponer, o el rendimiento cae porque se transfieren columnas `TEXT` largas innecesariamente.
*Causa*: `SELECT *` devuelve todo — incluyendo `body_md` de 20 KB por módulo cuando solo necesitas el título.
*Arreglo*: nombra siempre las columnas: `SELECT id, title, estimated_minutes FROM module`.

**5. `LIKE` con comodín al inicio.**
*Síntoma*: la query `WHERE vendor_name LIKE '%GmbH'` tarda 10 segundos aunque hay un índice en `vendor_name`.
*Causa*: `LIKE '%...'` con comodín al inicio no puede usar un índice B-tree — tiene que recorrer toda la tabla.
*Arreglo*: si necesitas buscar por sufijo o contenido, considera Full Text Search (`tsvector` en Postgres, `FTS5` en SQLite) o una base vectorial para búsqueda semántica.

## 1. Hilo conductor

Vienes del Track B, donde construiste endpoints HTTP y plantillas Jinja2 que devuelven datos. Esos datos venían de SQLite a través de SQLAlchemy — pero nunca explicamos por qué SQLite, por qué no un archivo JSON, y por qué existe algo llamado «base de datos». C05 (`sql-from-scratch`) asume que ya sabes qué es una tabla, un schema y una query. Este módulo establece esa base. Si ya sabes la diferencia entre Postgres y MongoDB, avanza directo a §4. Si no, lee §2 primero.

## 2. Idea central

Una **base de datos** es un sistema que almacena datos de forma estructurada, los indexa para que las búsquedas sean rápidas, y garantiza que las escrituras concurrentes no se corrompan entre sí. La diferencia con un archivo: el archivo te obliga a leerlo todo para encontrar una fila; la base de datos tiene un índice que apunta directamente a ella. Existen seis familias principales, cada una optimizada para un patrón de acceso distinto.

## 3. Por qué importa

Sin base de datos, ACME guarda sus facturas en Excel. El archivo crece; abrir el Excel con 50 000 filas tarda 45 segundos. Dos personas no pueden editarlo a la vez sin sobrescribirse. No hay forma de preguntar «dame todas las facturas del proveedor X cuyo monto supere 10 000 EUR» sin ir fila por fila.

El agente de auditoría de la ficha **F-CTA-01** (revisión contable automática) necesita leer 12 meses de facturas en menos de 2 segundos, correr 10 queries en paralelo desde distintas sesiones de usuario, y garantizar que si el agente escribe dos hallazgos al mismo tiempo no se pierda ninguno. Eso no existe en un archivo.

> [!nota]
> En este curso, la app de aprendizaje usa **SQLite** porque es un archivo — cero servidores. Para BATUTA en producción con múltiples clientes, el destino es **Postgres**. Ambos son bases de datos relacionales; los principios de este módulo aplican a los dos.

## 4. Cómo funciona por dentro

### La unidad básica: tabla, fila, columna, schema

Una **tabla** es una cuadrícula. Cada **fila** es un registro. Cada **columna** tiene un nombre y un tipo (`INTEGER`, `TEXT`, `NUMERIC`, `BOOLEAN`, `TIMESTAMP`). El conjunto de tablas, columnas, tipos y restricciones se llama **schema** (ver SHARED §1.4).

```
tabla: invoice
┌────┬───────────┬────────────────┬──────────────┬────────────┐
│ id │ tenant_id │ vendor_name    │ total_eur    │ period     │
├────┼───────────┼────────────────┼──────────────┼────────────┤
│  1 │ acme      │ Siemens GmbH   │   4 820.00   │ 2026-04    │
│  2 │ acme      │ Bosch AG       │   9 100.50   │ 2026-04    │
│  3 │ globex    │ DHL Express    │   1 234.00   │ 2026-04    │
└────┴───────────┴────────────────┴──────────────┴────────────┘
```

La columna `id` identifica cada fila de forma única — es la **clave primaria**. La columna `tenant_id` implementa multitenancy (ver C02): una consulta sin ese filtro devuelve filas de todos los tenants.

### Las seis familias

#### 1. Relacional (Postgres, SQLite, MySQL)

Datos en tablas con filas y columnas. Las tablas se relacionan entre sí por claves foráneas. Las consultas se escriben en **SQL**. El motor garantiza **transacciones ACID** (Atomic, Consistent, Isolated, Durable): si insertas una factura y su línea de detalle en la misma transacción, o se guardan las dos o no se guarda ninguna.

| Cuándo usarla | Ejemplo |
|---------------|---------|
| Datos estructurados con relaciones entre entidades | Facturas, vendedores, pagos, usuarios |
| Necesitas joins (cruzar dos tablas) | Factura + nombre de proveedor |
| Integridad referencial (no puede existir factura sin proveedor) | `FOREIGN KEY` |
| Transacciones multi-tabla | Registrar pago + actualizar saldo en un solo commit |

Es el caballo de batalla. La mayoría de sistemas que manejan dinero, inventario, usuarios o auditorías usan una base de datos relacional. **Postgres 16+** es la elección de producción para BATUTA.

#### 2. Key-value (Redis, DynamoDB)

Cada registro es un par `clave → valor`. La clave es un string; el valor puede ser cualquier cosa: string, número, JSON, lista, set. La operación fundamental es `GET(clave)` y `SET(clave, valor)`. Sin joins, sin schema, sin SQL.

| Cuándo usarla | Ejemplo |
|---------------|---------|
| Caché de resultados costosos | Caché de un reporte que tarda 3 s en generarse |
| Sesiones de usuario (TTL automático) | `session:{user_id}` con expiración 24 h |
| Contadores atómicos | Número de intentos de login (con `INCR`) |
| Colas simples | Lista de emails pendientes de enviar |

**Redis** es el estándar para caché y colas en arquitecturas web. DynamoDB es la versión gestionada de AWS para key-value serverless.

#### 3. Documento (MongoDB, Couchbase)

Cada registro es un documento JSON. No hay schema fijo: dos documentos en la misma colección pueden tener campos distintos. Los documentos se anidan (un documento puede contener otro objeto JSON dentro).

| Cuándo usarla | Ejemplo |
|---------------|---------|
| Datos con estructura variable | Respuestas de formularios con campos opcionales |
| Documentos ricos con jerarquía | Un contrato con cláusulas anidadas |
| Prototipado rápido sin schema | MVP donde el schema cambia cada semana |
| Sin joins frecuentes | Perfil de usuario con todas sus preferencias en un doc |

> [!cuidado]
> «Sin schema» no significa «sin estructura». Significa que el motor no la valida. Si no la validas tú en la app (con Pydantic), obtienes inconsistencia silenciosa. MongoDB con Pydantic como validador externo es razonable; MongoDB sin ninguna validación es una fuente de bugs difíciles de depurar.

#### 4. Columnar (ClickHouse, DuckDB, BigQuery)

En una base de datos relacional tradicional, los datos se almacenan por fila (todas las columnas de una fila juntas en disco). En una base de datos **columnar**, se almacenan por columna (todos los valores de `total_eur` de todas las filas, juntos). Cuando quieres calcular `SUM(total_eur)` sobre 500 millones de filas, el motor lee solo la columna `total_eur` y salta el resto — es 10–100× más rápido que un OLTP relacional para este patrón.

| Cuándo usarla | Ejemplo |
|---------------|---------|
| Analytics sobre grandes volúmenes | «Ventas por región, por mes, en los últimos 3 años» |
| Dashboards con agregaciones | KPIs que se recalculan cada hora |
| Data warehouse | Consolidar datos de múltiples fuentes para reportes |

**DuckDB** es el columnar embebido (como SQLite, pero para analytics). **ClickHouse** es el columnar distribuido para escala. **BigQuery** es el managed de Google. Para una PYME LATAM que quiere analytics sin servidor, DuckDB sobre archivos Parquet es la opción más sencilla.

#### 5. Time-series (TimescaleDB, InfluxDB)

Optimizada para secuencias de valores indexados por tiempo: telemetría, métricas, logs, series financieras. El patrón de escritura es append-only (siempre insertas el timestamp más reciente) y las queries son casi siempre por rango de tiempo (`WHERE ts BETWEEN ... AND ...`). El motor comprime y particiona automáticamente por tiempo.

| Cuándo usarla | Ejemplo |
|---------------|---------|
| Métricas de sistema | CPU, memoria, latencia de la app cada 10 s |
| Datos de sensores IoT | Temperatura de un almacén cada minuto |
| Ticks financieros | Precio de un activo en tiempo real |
| Logs estructurados | Eventos de auditoría con timestamp obligatorio |

**TimescaleDB** es una extensión de Postgres — si ya usas Postgres, es la ruta más directa. **InfluxDB** es la alternativa especializada con su propio query language (Flux).

#### 6. Vector (pgvector, Qdrant, Pinecone, Weaviate)

Almacena **embeddings**: vectores numéricos de cientos o miles de dimensiones que codifican el significado semántico de un texto, imagen o audio. Las queries no son por igualdad (`WHERE id = 5`) sino por **similitud** (`dame los 10 vectores más parecidos a este`). El índice no es un B-tree sino un HNSW (Hierarchical Navigable Small World) o IVF (Inverted File) optimizado para búsqueda aproximada en alta dimensión. Este tipo es el corazón de los sistemas RAG (Retrieval-Augmented Generation) que cubren C08 y C09.

| Cuándo usarla | Ejemplo |
|---------------|---------|
| Búsqueda semántica | «Encuentra facturas parecidas a esta descripción» |
| RAG sobre documentos | Recuperar los párrafos más relevantes para una pregunta |
| Recomendaciones | «Productos similares a los que compró este cliente» |
| Detección de duplicados semánticos | Dos facturas con descripción diferente pero mismo concepto |

### ¿Y las bases de datos de grafo?

**Neo4j** y similares almacenan nodos y aristas. Son óptimas cuando la relación entre entidades *es* el dato (redes sociales, detección de fraude por patrones de relación, knowledge graphs). Para la mayoría de casos de PYME LATAM no son la primera elección; se mencionan aquí para completar el mapa.

### Decision tree: ¿cuál usar?

```
¿Datos estructurados con relaciones entre entidades?
  ├── Sí → Relacional (Postgres)
  │         ├── ¿También necesitas analytics sobre millones de filas?
  │         │   └── Añade columnar (DuckDB o ClickHouse) para el warehouse
  │         └── ¿También necesitas búsqueda semántica?
  │             └── Añade pgvector como extensión de Postgres
  └── No
      ├── ¿Caché / sesiones / colas simples? → Key-value (Redis)
      ├── ¿Documentos JSON con estructura variable? → Documento (MongoDB)
      ├── ¿Métricas con timestamp? → Time-series (TimescaleDB o InfluxDB)
      └── ¿Búsqueda semántica / RAG? → Vector (Qdrant, Pinecone, o pgvector)
```

**Regla para PYME LATAM**: empieza con Postgres. Cuando tengas el problema concreto que justifica otra familia, añade esa familia. No lo hagas antes.

### Transacciones ACID: por qué importan para el dinero

La propiedad más importante de una base de datos relacional en contextos financieros es la **transacción**. Cuando ACME paga una factura, deben pasar dos cosas atómicamente: decrementar el saldo de la cuenta bancaria e incrementar el registro de pagos. Si el proceso muere entre las dos operaciones, el dinero desaparece. Una transacción garantiza que o pasan las dos o no pasa ninguna.

Las bases NoSQL sacrifican parte del ACID a cambio de escala horizontal. Para manejar dinero, inventario o cualquier dato donde «a medias» es peor que «nada», usa relacional.

## 8. Caso de uso real

**F-CTA-01 — Auditoría contable automática.** El agente de ACME lee facturas del período desde Postgres (relacional), busca las facturas más similares a una descripción anómala usando pgvector (vector), y cachea el reporte generado en Redis por 1 hora para evitar re-generarlo si el CFO recarga la página. Tres familias de bases de datos, tres propósitos distintos. El módulo C08 (`embeddings-and-vectors`) construye la capa vector en detalle.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Elegir el tipo de base de datos para un caso dado | Determinístico | El decision tree es una regla cerrada aplicable con los criterios del caso. |
| Detectar si una factura es duplicada por monto y fecha exactos | Determinístico | Igualdad numérica — regla cerrada. |
| Detectar si dos facturas son «el mismo concepto» por descripción | Agéntico | La similitud semántica requiere embedding + modelo; no hay regla. |
| Decidir si mover un tenant a su propio schema por regulación | Agéntico | Depende de regulación local, industria y acuerdo contractual — sin regla única. |

## 10. Errores típicos

**1. Usar una base de datos relacional para todo.**
*Síntoma*: intentas guardar embeddings en una columna `TEXT` como JSON, y las búsquedas de similitud tardan minutos.
*Causa*: SQL no tiene operador de similitud coseno; estás haciendo el cálculo en Python sobre miles de vectores deserializados.
*Arreglo*: añadir pgvector como extensión o usar Qdrant para la capa vector.

**2. Elegir MongoDB «porque es flexible».**
*Síntoma*: tres meses después, la colección tiene documentos con 12 variantes distintas del campo `amount` (`monto`, `total`, `valor`, `amount`, `importe`...).
*Causa*: sin schema, la disciplina del equipo es la única garantía — y el equipo cambia.
*Arreglo*: si los datos tienen estructura estable, usar relacional. Si son genuinamente variables, usar Pydantic para validar antes de insertar en MongoDB.

**3. Guardar métricas de tiempo en una tabla relacional estándar.**
*Síntoma*: la tabla `metric_log` tiene 200 millones de filas; los `SELECT WHERE ts > ...` tardan 30 s aunque hay un índice en `ts`.
*Causa*: el índice B-tree en una tabla sin particionamiento por tiempo es ineficiente para rangos temporales con alta cardinalidad.
*Arreglo*: TimescaleDB convierte la tabla en hypertable con particionamiento automático; la misma query cae a 200 ms.

**4. Redis como base de datos principal.**
*Síntoma*: al reiniciar Redis sin persistencia, todos los datos del día desaparecen.
*Causa*: Redis es in-memory por defecto. Con `RDB` o `AOF` puedes persistir, pero sigue sin ser para datos críticos.
*Arreglo*: Redis para caché y sesiones (que puedes regenerar); Postgres para datos que no puedes perder.

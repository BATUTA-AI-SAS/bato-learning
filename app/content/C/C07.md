## 1. Hilo conductor

C06 (`sql-joins-aggregations`) te dio herramientas para escribir queries complejas. Ahora esas queries llegan a producción y algunas tardan 30 segundos. El problema no es la query — es que el motor no sabe cómo encontrar las filas rápido. Aquí aprendes a diagnosticar con `EXPLAIN ANALYZE`, a crear los índices correctos, y a reconocer el patrón N+1 que destruye el rendimiento de los endpoints que usan SQLAlchemy. C08 (`embeddings-and-vectors`) usa índices especiales (HNSW) para búsqueda vectorial — la intuición que construyes aquí aplica allá también.

## 2. Idea central

Un **índice** es una estructura de datos auxiliar que el motor mantiene sincronizada con la tabla. Lee la tabla completa una sola vez para construirlo; después, las queries que lo pueden usar saltan directamente a las filas relevantes. Sin índice, el motor hace un **seq scan** (recorre todas las filas). Con índice, hace un **index scan** (salta al nodo del árbol). `EXPLAIN ANALYZE` muestra cuál de los dos eligió el motor y cuánto tardó cada nodo del plan.

## 3. Por qué importa

ACME tiene 500 000 facturas. La query `SELECT * FROM invoice WHERE tenant_id = 'acme' AND period = '2026-04'` sin índice recorre las 500 000 filas para encontrar las 3 000 del período. Con un índice compuesto en `(tenant_id, period)`, el motor salta directo a esas 3 000. La diferencia: 4 segundos vs 8 milisegundos.

Sin entender índices, el lector añade índices al azar o, peor, no los añade nunca. El primero genera overhead de escritura innecesario; el segundo genera queries que no escalan. `EXPLAIN ANALYZE` es la herramienta que convierte la intuición en evidencia.

La ficha **F-CTA-01** (auditoría contable) corre queries de lectura sobre `invoice` en cada ciclo del agente. Sin los índices correctos, el agente que debería correr en 5 segundos tarda 3 minutos.

## 4. Cómo funciona por dentro

### El B-tree: el índice estándar

El índice por defecto en Postgres (y SQLite) es un **B-tree** (Balanced Tree). Imagina un árbol donde cada nodo interno contiene claves ordenadas y punteros a sus hijos. Para buscar `tenant_id = 'acme'`, el motor empieza en la raíz, compara, baja al nodo hijo correcto, y llega a las hojas (que apuntan a las filas) en `O(log N)` operaciones en lugar de `O(N)`.

```sql
-- índice simple en una columna
CREATE INDEX idx_invoice_tenant ON invoice (tenant_id);

-- índice compuesto (el orden importa: leftmost prefix rule)
CREATE INDEX idx_invoice_tenant_period ON invoice (tenant_id, period);

-- índice compuesto con una columna de sort frecuente
CREATE INDEX idx_invoice_tenant_period_amount ON invoice (tenant_id, period, total_eur DESC);
```

**Leftmost prefix rule**: un índice en `(tenant_id, period)` puede usarse para queries con `WHERE tenant_id = ...`, o `WHERE tenant_id = ... AND period = ...`, pero **no** para queries con solo `WHERE period = ...` (sin el prefijo izquierdo).

### Cuándo un índice B-tree ayuda

| Query | ¿Índice útil? | Por qué |
|-------|--------------|---------|
| `WHERE tenant_id = 'acme'` | Sí | Igualdad en columna indexada |
| `WHERE tenant_id = 'acme' AND period = '2026-04'` | Sí (índice compuesto) | Igualdad en prefijo izquierdo |
| `WHERE total_eur > 5000` | Sí | Rango en columna numérica indexada |
| `WHERE vendor_name LIKE '%GmbH'` | No | El comodín al inicio impide usar el árbol |
| `WHERE vendor_name LIKE 'Siemens%'` | Sí | El prefijo puede buscar en el árbol |
| `WHERE LOWER(vendor_name) = 'siemens gmbh'` | No (sin índice de expresión) | La función transforma el valor antes de comparar |

### `EXPLAIN ANALYZE`: leer el plan de ejecución

En **Postgres**, `EXPLAIN ANALYZE` ejecuta la query y muestra el plan con tiempos reales:

```sql
EXPLAIN ANALYZE
SELECT vendor_name, total_eur
FROM invoice
WHERE tenant_id = 'acme'
  AND period = '2026-04'
ORDER BY total_eur DESC
LIMIT 10;
```

Output típico sin índice:
```
Seq Scan on invoice  (cost=0.00..18450.00 rows=3000 width=40) (actual time=0.042..4213.5 rows=3000 loops=1)
  Filter: ((tenant_id = 'acme') AND (period = '2026-04'))
  Rows Removed by Filter: 497000
Planning Time: 0.3 ms
Execution Time: 4221.8 ms
```

Output después de `CREATE INDEX idx_invoice_tenant_period ON invoice (tenant_id, period)`:
```
Index Scan using idx_invoice_tenant_period on invoice  (cost=0.56..312.4 rows=3000 width=40) (actual time=0.02..7.3 rows=3000 loops=1)
  Index Cond: ((tenant_id = 'acme') AND (period = '2026-04'))
Planning Time: 0.5 ms
Execution Time: 8.1 ms
```

**Términos clave del plan**:
- `Seq Scan`: recorrido completo de la tabla. La «scan de la muerte» para tablas grandes.
- `Index Scan`: usa el índice para encontrar las filas. Bueno.
- `Index Only Scan`: todas las columnas necesarias están en el índice — ni siquiera lee la tabla. Muy bueno.
- `cost=inicio..total`: estimación del optimizador. El número después de `..` es el coste total estimado.
- `actual time=inicio..fin`: tiempo real medido (en ms). Si difiere mucho del estimado, el optimizador tiene estadísticas desactualizadas — ejecuta `ANALYZE invoice` para actualizarlas.
- `rows=N`: filas procesadas. Si son muchas más de las que devuelves, hay filtrado tardío.
- `Rows Removed by Filter`: filas leídas y descartadas. Un número alto aquí es señal de que falta índice.

> [!nota]
> **SQLite** no tiene `EXPLAIN ANALYZE` en el mismo formato. Usa `EXPLAIN QUERY PLAN`:
> ```sql
> EXPLAIN QUERY PLAN
> SELECT * FROM invoice WHERE tenant_id = 'acme';
> ```
> Output sin índice: `SCAN invoice` (malo). Con índice: `SEARCH invoice USING INDEX ...` (bueno).

### Índice parcial: indexar solo lo que consultas

Si el 95% de las queries buscan `paid = 0` (facturas no pagadas), un índice parcial es más pequeño y rápido que uno completo:

```sql
CREATE INDEX idx_invoice_unpaid ON invoice (tenant_id, period)
WHERE paid = FALSE;
```

Este índice tiene solo el 5% de las filas del índice completo. Menor tamaño → más cabe en caché → más rápido. Pero solo aplica para queries con `WHERE paid = FALSE`.

### Índice de expresión: indexar una función

```sql
-- consultas por vendor_name en minúsculas (búsqueda case-insensitive)
CREATE INDEX idx_invoice_vendor_lower ON invoice (LOWER(vendor_name));

-- ahora esta query usa el índice
SELECT * FROM invoice WHERE LOWER(vendor_name) = 'siemens gmbh';
```

El índice almacena el resultado de `LOWER(vendor_name)`, no el valor original. Postgres y SQLite modernos soportan índices de expresión.

### GIN para JSONB (Postgres)

Cuando guardas datos semiestructurados en una columna `JSONB` de Postgres, el índice B-tree no puede buscar dentro del JSON. El índice **GIN** (Generalized Inverted Index) indexa todas las claves y valores del documento:

```sql
ALTER TABLE invoice ADD COLUMN metadata JSONB;
CREATE INDEX idx_invoice_metadata ON invoice USING GIN (metadata);

-- ahora esta query es rápida
SELECT * FROM invoice WHERE metadata @> '{"category": "utilities"}';
```

`@>` es el operador «contiene» de JSONB. Sin índice GIN, recorre todas las filas.

### N+1: el problema de SQLAlchemy sin eager loading

El **problema N+1** no es un problema de índices — es un problema de arquitectura de queries. Ocurre cuando cargas N registros y luego haces una query adicional por cada uno para obtener sus relaciones:

```python
# MAL: 1 query para módulos + N queries para ejercicios (una por módulo)
modules = (await session.execute(select(Module))).scalars().all()
for module in modules:
    print(module.exercises)  # lazy load → 1 query por módulo → N queries
```

Con 50 módulos: 51 queries. Con 500 módulos: 501 queries. Con `selectinload`:

```python
# BIEN: 2 queries totales (1 para módulos, 1 para todos los ejercicios)
modules = (await session.execute(
    select(Module).options(selectinload(Module.exercises))
)).scalars().all()
```

El índice en `exercise.module_id` hace que la segunda query (todos los ejercicios de esos módulos) sea eficiente — el motor puede hacer un index scan en lugar de seq scan. Ambas piezas importan: eager loading + índice en la foreign key.

### Cuándo NO indexar

Los índices no son gratuitos:
- **Overhead de escritura**: cada `INSERT`, `UPDATE` o `DELETE` en la tabla debe actualizar también los índices. Una tabla con 8 índices puede ser 3× más lenta para escrituras que la misma tabla sin índices.
- **Espacio en disco**: un índice puede llegar a ser tan grande como la tabla misma.
- **El optimizador puede ignorarlo**: si la tabla tiene pocas filas o el filtro devuelve el 80% de los datos, el motor puede preferir un seq scan.

Reglas prácticas para NO indexar:
1. Columnas que se actualizan con altísima frecuencia (contadores, timestamps que cambian cada segundo).
2. Tablas con menos de 1 000 filas — el seq scan es más rápido que el overhead del índice.
3. Columnas con muy baja cardinalidad (ej. una columna booleana con 50% `TRUE`) — el índice no ayuda porque el motor igual lee muchas filas.
4. Índices redundantes: `(tenant_id, period, paid)` hace redundante a `(tenant_id, period)` para las queries que usan los dos primeros campos.

## 5. Ejemplo conducido

Diagnóstico y solución del N+1 en la app real:

```python
from sqlalchemy import select, event
from sqlalchemy.ext.asyncio import AsyncSession
from app.models.content import Module
from app.db import SessionFactory

# activar echo para ver el SQL
import logging
logging.getLogger("sqlalchemy.engine").setLevel(logging.INFO)

async def diagnose_n_plus_one():
    async with SessionFactory() as session:
        # SIN eager loading: 1 query + N queries (una por módulo)
        modules = (await session.execute(select(Module))).scalars().all()
        for m in modules:
            _ = m.exercises   # lazy load — una query por módulo

        # CON selectinload: 2 queries totales
        from sqlalchemy.orm import selectinload
        modules2 = (await session.execute(
            select(Module).options(selectinload(Module.exercises))
        )).scalars().all()
        for m in modules2:
            _ = m.exercises   # ya cargado — cero queries adicionales
```

Para ver el `EXPLAIN QUERY PLAN` en SQLite desde Python:

import sqlite3

for row in cur.fetchall():

# Sin índice: (0, 0, 0, 'SCAN module')
# Con índice: (0, 0, 0, 'SEARCH module USING INDEX idx_module_track (track=?)')

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Leer `EXPLAIN ANALYZE` e identificar un `Seq Scan` | Determinístico | El output es texto estructurado; el patrón es reconocible con regla. |
| Crear un índice en la columna que aparece en el `WHERE` | Determinístico | Regla cerrada: si la columna filtra y la tabla es grande, necesita índice. |
| Decidir si un índice parcial o completo es más apropiado | Agéntico | Depende de la distribución real de los datos y del patrón de queries del cliente — sin regla única. |
| Diagnosticar si el rendimiento degradado se debe a índice faltante vs hardware saturado | Agéntico | Requiere contexto del entorno de producción y del historial de carga. |

## 10. Errores típicos

**1. Añadir índice en columna con baja cardinalidad.**
*Síntoma*: el índice existe pero `EXPLAIN` sigue mostrando `Seq Scan`.
*Causa*: el motor decidió que un seq scan es más rápido porque el filtro devuelve el 60% de las filas. Los índices B-tree son óptimos cuando la selectividad es alta (el filtro devuelve <20% de las filas).
*Arreglo*: para columnas booleanas o de pocos valores distintos, considerar índice parcial o aceptar el seq scan.

**2. Olvidar el índice en la foreign key.**
*Síntoma*: un `JOIN invoice ON invoice.client_id = client.id` hace seq scan en `invoice`.
*Causa*: Postgres no crea automáticamente índice en las columnas de foreign key (SQLite tampoco).
*Arreglo*: `CREATE INDEX idx_invoice_client_id ON invoice (client_id)`.

**3. `LIKE '%texto%'` con comodín al inicio.**
*Síntoma*: la query `WHERE vendor_name LIKE '%GmbH'` no usa el índice en `vendor_name`.
*Causa*: el comodín al inicio impide al B-tree buscar por prefijo.
*Arreglo*: si necesitas búsqueda de texto completo, usa `pg_trgm` con índice GIN en Postgres, o FTS5 en SQLite.

**4. Ignorar el `EXPLAIN` después de `ANALYZE invoice`.**
*Síntoma*: `EXPLAIN` dice que va a leer 100 filas pero en realidad lee 50 000.
*Causa*: las estadísticas del optimizador están desactualizadas (tabla creció mucho desde el último `ANALYZE`).
*Arreglo*: `ANALYZE invoice` actualiza las estadísticas. En Postgres, el autovacuum lo hace periódicamente; en SQLite, ejecuta manualmente.

**5. Crear demasiados índices en una tabla de escritura intensiva.**
*Síntoma*: los `INSERT` de facturas en batch tardan 10 segundos en lugar de 1.
*Causa*: cada inserción actualiza 7 índices en paralelo.
*Arreglo*: revisar qué índices realmente se usan con `pg_stat_user_indexes`. Dropear los que tienen `idx_scan = 0` después de 30 días de producción.

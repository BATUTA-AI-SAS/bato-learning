## 1. Hilo conductor

C09 (`rag-end-to-end`) construyó el pipeline RAG completo. Este módulo es su contrapeso necesario: RAG tiene costos de ingeniería, costos de inferencia, latencia adicional, y modos de fallo propios. Antes de implementarlo, vale la pena verificar si una de las cuatro alternativas resuelve el problema de forma más simple. Este módulo cierra el Track C — el Track D (`docker-images-containers`) puede empezar.

## 2. Idea central

**RAG es el patrón correcto cuando el corpus es grande, dinámico, y la búsqueda semántica es necesaria.** Si el corpus es pequeño y estable, prompt caching es más simple. Si el patrón es repetible y tienes datos etiquetados, fine-tuning. Si el usuario busca por términos exactos, búsqueda léxica. Si el corpus cabe en el context window, long-context. Elegir RAG cuando no hace falta es complejidad innecesaria — exactamente lo que este curso enseña a evitar.

## 3. Por qué importa

El error más común al construir agentes IA para PYMES es implementar RAG porque «es lo que hacen todos» sin evaluar si es lo correcto para el caso. Un sistema RAG mal configurado tiene peor rendimiento que un sistema sin RAG. La ficha **F-CTA-01** usa RAG para los manuales de procedimiento — pero usa prompt caching para el system prompt del tutor, que es estable y no cambia entre sesiones. Saber cuándo usar cada uno es ingeniería, no preferencia.

## 4. Cómo funciona por dentro

### Alternativa 1: Prompt caching

**Cuándo es la respuesta**: el corpus es pequeño (≤ ~50 páginas / ~100K tokens), **estable** (no cambia más de una vez al día), y se consulta muchas veces.

```python
# Ejemplo: system prompt del tutor de bato-learning
# El contenido del módulo (≈ 5K tokens) se cachea 1 hora
client.messages.create(
    model="claude-sonnet-4-6",
    system=[{
        "type": "text",
        "text": module_body_md,              # ≈ 5000 tokens
        "cache_control": {"type": "ephemeral", "ttl": "1h"}
    }],
    messages=[{"role": "user", "content": question}]
)
```

Con `ttl="1h"`, el primer request de la hora paga tokens completos; los siguientes pagan el 10% del costo de entrada (precio de cache hit de Anthropic). Para un módulo de 5K tokens con 50 consultas/hora: ahorro del 90% vs re-enviar el contexto cada vez.

**Límites**: el mínimo cacheable es 1024 tokens (Sonnet 4.6). El máximo depende del context window. Si el corpus es mayor de lo que cabe en un prompt (~100K tokens para la mayoría de casos prácticos), RAG es necesario.

**Comparativa de costo** (1000 queries/día, corpus de 10K tokens):

| Estrategia | Costo/día (estimado) |
|------------|----------------------|
| Sin caching (re-envía 10K tokens cada vez) | ~$30 |
| Con prompt caching (cache hit 95%) | ~$3 |
| RAG con reranker | ~$5–15 (+ costo de indexación) |

Para corpus pequeño y estable: **prompt caching gana**.

### Alternativa 2: Fine-tuning

**Cuándo es la respuesta**: tienes ≥ 500–1000 ejemplos etiquetados de (input, output) y el patrón se repite con alta frecuencia. El fine-tuning quema el conocimiento en los pesos del modelo.

**Cuándo NO es la respuesta**:
- Corpus de menos de 500 ejemplos — el modelo no generaliza.
- Datos que cambian frecuentemente — el modelo queda desactualizado.
- Cuando quieres que el modelo cite la fuente exacta — el fine-tuning no da referencias.
- Cuando no puedes permitirte el costo de re-entrenar al actualizar el corpus.

**Comparativa fine-tuning vs RAG**:

| Criterio | RAG | Fine-tuning |
|----------|-----|-------------|
| Actualizar el corpus | Re-indexar (minutos) | Re-entrenar (horas/días + costo GPU) |
| Citar fuentes | Sí (los chunks son las fuentes) | No (el conocimiento está en los pesos) |
| Datos necesarios | Ninguno etiquetado | ≥ 500–1000 pares etiquetados |
| Costo de inferencia | Más alto (contexto con chunks) | Más bajo (modelo más pequeño posible) |
| Latencia | Más alta (retrieval + rerank) | Más baja (un solo request) |

**Caso de uso para fine-tuning**: clasificación de facturas en categorías contables. Si ACME tiene 2000 facturas etiquetadas por el contador (categoria: OPEX/CAPEX/etc.), fine-tuning en un modelo pequeño puede hacer esa clasificación con alta precisión y a fracción del costo de RAG.

### Alternativa 3: Búsqueda léxica (BM25 / full-text search)

**Cuándo es la respuesta**: las queries contienen términos técnicos exactos (códigos, números, nombres propios) donde la semántica no añade valor.

Ejemplos donde la léxica gana a la semántica:
- «NIIF 16», «artículo 423», «código SAP 0001234»: términos exactos que un embedding no «entiende» mejor que BM25.
- Búsqueda de facturas por número de referencia.
- Normativa legal por número de artículo.

**Opciones**:
- **SQLite FTS5**: full-text search integrado en SQLite, disponible sin dependencias.
- **Postgres `tsvector` + GIN**: full-text search nativo en Postgres.
- **Meilisearch**: servidor de búsqueda léxica, fácil de operar, buena UX.
- **Elasticsearch**: la opción enterprise, más potente pero más costoso de operar.

```sql
-- Postgres: búsqueda léxica sobre manuales
SELECT chunk_text, ts_rank(search_vector, query) AS rank
FROM document_chunk, to_tsquery('spanish', 'NIIF & 16') query
WHERE tenant_id = 'acme'
  AND search_vector @@ query
ORDER BY rank DESC
LIMIT 5;
```

**Hybrid search** combina BM25 + vectorial con un score fusionado (Reciprocal Rank Fusion). Para corpus mixtos con términos técnicos Y consultas semánticas, el hybrid es mejor que cualquiera de los dos solos.

### Alternativa 4: Long-context

**Cuándo es la respuesta**: el corpus completo cabe en el context window del modelo, el corpus es estable, y la latencia extra no es un problema.

Claude 3.5+ y Claude 4.x tienen context windows de 200K tokens (~150K palabras / ~500 páginas). Para manuales de procedimiento de una PYME típica (<100 páginas), puede ser perfectamente viable poner todo el corpus en el system prompt con `cache_control`.

**Comparativa long-context vs RAG**:

| Criterio | Long-context | RAG |
|----------|-------------|-----|
| Calidad de respuesta | Alta (el modelo ve todo) | Depende del retrieval |
| Costo por query | Alto si no se cachea | Moderado |
| Costo con caching | Bajo si el corpus es estable | Moderado |
| Corpus dinámico | Mala opción | Buena opción |
| Corpus > context window | Imposible | Sí |
| Latencia | Mayor (más tokens de entrada) | Mayor (retrieval + rerank) |

**Regla práctica**: si el corpus cabe en un prompt (< 100K tokens) y cambia menos de una vez al día, long-context + prompt caching puede ser más simple y más preciso que RAG.

### Decision tree: RAG vs alternativas

```
¿El corpus cabe en el context window (< ~100K tokens)?
  ├── Sí → ¿Cambia frecuentemente (>1 vez/día)?
  │        ├── No → Long-context + prompt caching
  │        └── Sí → RAG (re-indexar es más barato que re-cachear)
  └── No → RAG (el corpus no cabe, necesitas retrieval)

¿Las queries son principalmente por términos técnicos exactos?
  └── Sí → Búsqueda léxica (BM25 / FTS) o hybrid search

¿Tienes ≥ 500 ejemplos etiquetados y el patrón no cambia?
  └── Sí → Considera fine-tuning (menor costo de inferencia, no cita fuentes)

¿El corpus es grande, dinámico, y las queries son semánticas?
  └── Sí → RAG
```

## 5. Ejemplo conducido

Calculadora de elección de estrategia:

```python
def choose_retrieval_strategy(
    corpus_tokens: int,
    updates_per_day: float,
    queries_are_semantic: bool,
    have_labeled_examples: int,  # número de pares etiquetados
    context_window: int = 200_000,
) -> str:
    if corpus_tokens < context_window:
        if updates_per_day < 1:
            return "long-context + prompt caching"
        # corpus cambia frecuentemente pero cabe
        return "rag"  # re-indexar es más rápido que re-cachear

    if not queries_are_semantic:
        return "lexical search (BM25 / FTS)"

    if have_labeled_examples >= 500:
        return "fine-tuning (+ fallback to rag for new patterns)"

    return "rag"

# Los manuales de ACME: 40K tokens, actualización mensual
print(choose_retrieval_strategy(40_000, 0.03, True, 0))
# → "long-context + prompt caching"

# Corpus de normativa tributaria colombiana: 800K tokens, actualización anual
print(choose_retrieval_strategy(800_000, 0.003, True, 0))
# → "rag"

# Búsqueda de artículos del código tributario por número
print(choose_retrieval_strategy(200_000, 0.1, False, 0))
# → "lexical search (BM25 / FTS)"
```

## 8. Caso de uso real

**F-CTA-01 — Auditoría contable de ACME.** El agente usa estrategias distintas según la naturaleza del dato:
- **Manuales de procedimiento** (30 páginas, se actualizan trimestralmente): long-context + prompt caching.
- **Corpus de hallazgos históricos** (cientos de auditorías anteriores): RAG con pgvector.
- **Normativa tributaria por número de artículo**: búsqueda léxica (tsvector en Postgres).
- **Clasificación de gastos OPEX/CAPEX**: fine-tuning sobre 2000 ejemplos etiquetados por el contador de ACME.

Cuatro técnicas, cuatro casos distintos, todos en el mismo agente.

## 10. Errores típicos

**1. Usar RAG para un corpus de 5 páginas.**
*Síntoma*: el sistema tiene latencia de 800ms + overhead de retrieval para dar una respuesta que no mejora la que daría long-context.
*Causa*: no se evaluó si el corpus caba en el prompt antes de implementar RAG.
*Arreglo*: calcular `tokens_del_corpus < context_window`. Si sí cabe, usar long-context + caching.

**2. Fine-tuning con 50 ejemplos.**
*Síntoma*: el modelo fine-tuneado tiene peor rendimiento que el base en preguntas no vistas.
*Causa*: con muy pocos ejemplos, el modelo sobreajusta al training set y pierde generalización.
*Arreglo*: fine-tuning solo con ≥ 500 ejemplos, mejor con ≥ 1000. Con menos, usar RAG o few-shot prompting.

**3. Búsqueda semántica para términos legales exactos.**
*Síntoma*: la query «artículo 423 del código tributario» no devuelve el artículo 423 — devuelve artículos semánticamente relacionados.
*Causa*: los embeddings normalizan el significado; «artículo 423» y «artículo 424» son casi idénticos semánticamente.
*Arreglo*: usar BM25 / full-text search para términos exactos. O hybrid search (BM25 + vectorial).

**4. Ignorar el costo del long-context sin caching.**
*Síntoma*: el costo de inferencia sube 10× sin que mejore la calidad.
*Causa*: enviar 50K tokens de corpus en cada query sin `cache_control`.
*Arreglo*: siempre combinar long-context con `cache_control: {"type": "ephemeral", "ttl": "1h"}`. Con caching, pagas la primera query de la hora; las siguientes cuestan el 10%.

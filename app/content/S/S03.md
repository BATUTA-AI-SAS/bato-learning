## 1. Hilo conductor

En E01 aprendiste a declarar tools en el SDK de Anthropic. En E02 viste `interrupt_before` en LangGraph para pausar antes de ejecutar un nodo. En S02 aplicaste spotlighting y output filtering como defensas contra IPI. Este módulo cierra el ciclo: incluso si el modelo fue subvertido, las tools deben tener defensas propias que impidan daño real. Hacia S04 usarás estas capas como insumo del threat modeling.

---

## 2. Idea central

Una **tool** es una función con efectos en el mundo real: envía emails, ejecuta código, transfiere fondos, borra registros. El modelo decide llamarla, pero el harness la ejecuta. Esa separación es donde viven las defensas.

Cuatro capas en orden de implementación:
1. **Allowlist**: solo las tools declaradas en config pueden existir.
2. **Sandbox**: tools que ejecutan código arbitrario corren en entorno aislado.
3. **Dry-run**: tools destructivas tienen un modo de simulación que el harness puede forzar.
4. **Human-in-the-loop**: tools de alto impacto requieren confirmación humana antes de ejecutar.

Imagen mental: el modelo es el empleado que pide hacer algo. La allowlist es su descripción de cargo (solo puede hacer lo que el cargo permite). El sandbox es la sala de pruebas. El dry-run es el borrador del informe. El human-in-the-loop es la firma del gerente.

---

## 3. Por qué importa

El agente de ACME tiene acceso a `send_email`, `sql_query`, `write_report`, y `erp_fetch_transactions`. Sin controles:
- Un prompt injection hace que el agente envíe todos los findings por email a un atacante.
- Un error de lógica hace que el agente ejecute `DELETE FROM invoices` en lugar de `SELECT`.
- Un fallo en el loop hace que el agente envíe el mismo reporte 47 veces.

Ninguno de estos escenarios es hipotético. La ficha **F-CTA-02 (auditoría interna)** lista explícitamente las tools del workflow; cada una tiene un nivel de riesgo distinto y necesita controles distintos.

## 4. Cómo funciona por dentro

### 4.1 Allowlist estricta

Las tools disponibles para el agente se declaran en configuración estática, no en runtime. El modelo no puede «pedir» que exista una tool que no esté en la lista.

```python
# app/services/tool_registry.py
from typing import Literal

ToolName = Literal[
    "sql_query",
    "fetch_excel",
    "parse_invoice_pdf",
    "write_report",
    "send_email",
    "erp_fetch_transactions",
]

TOOL_REGISTRY: dict[str, dict] = {
    "sql_query": {
        "schema": {...},
        "risk_level": "low",
        "requires_dry_run": False,
        "requires_hitl": False,
    },
    "send_email": {
        "schema": {...},
        "risk_level": "high",
        "requires_dry_run": True,
        "requires_hitl": True,
    },
    "write_report": {
        "schema": {...},
        "risk_level": "medium",
        "requires_dry_run": False,
        "requires_hitl": False,
    },
}

def get_tools_for_agent(
    agent_context: str,  # "analysis" | "reporting" | "full"
) -> list[dict]:
    """Devuelve solo las tools autorizadas para el contexto del agente."""
    if agent_context == "analysis":
        allowed = {"sql_query", "fetch_excel", "parse_invoice_pdf"}
    elif agent_context == "reporting":
        allowed = {"sql_query", "write_report", "send_email"}
    else:  # "full" — solo con HITL activo
        allowed = set(TOOL_REGISTRY.keys())
    return [TOOL_REGISTRY[name]["schema"] for name in allowed]
```

El modelo nunca ve tools fuera de las autorizadas para ese contexto. Si es subvertido y pide una tool no listada, el SDK responde con error.

### 4.2 Sandbox para tools de ejecución de código

La app bato-learning ya tiene un runner de ejercicios. Ese mismo patrón aplica a cualquier tool que ejecute código arbitrario:

```python
# app/services/exercise_runner.py — el patrón de sandboxing real de la app
import asyncio
import subprocess
import sys

async def run_backend_exercise(code: str, test_code: str) -> dict:
    """Corre código en subprocess aislado con timeout."""
    combined = f"{code}\n\n{test_code}"
    try:
        proc = await asyncio.create_subprocess_exec(
            sys.executable, "-c", combined,
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.PIPE,
        )
        stdout, stderr = await asyncio.wait_for(
            proc.communicate(), timeout=10.0  # timeout duro
        )
        passed = proc.returncode == 0
        return {"passed": passed, "output": stdout.decode(), "error": stderr.decode()}
    except asyncio.TimeoutError:
        proc.kill()
        return {"passed": False, "output": "", "error": "timeout: 10s excedido"}
```

Para una tool de agente que ejecuta queries o código del cliente:

```python
async def sandboxed_sql_query(query: str, tenant: str) -> dict:
    # 1. validar que es SELECT (no DDL/DML)
    if not query.strip().upper().startswith("SELECT"):
        raise ValueError("solo SELECT permitido en sql_query")
    # 2. ejecutar en conexión de solo lectura con timeout
    async with read_only_connection(tenant, timeout=30) as conn:
        rows = await conn.fetch(query)
    return {"rows": [dict(row) for row in rows], "count": len(rows)}
```

Para tools que ejecutan código Python del cliente: usar `subprocess` + `--no-site` + filesystem readonly (bind mounts en Docker o `seccomp` en Linux). La app ya hace esto para los ejercicios.

### 4.3 Dry-run para tools destructivas

Toda tool que produce efectos irreversibles implementa `dry_run: bool`. Cuando `dry_run=True`, simula la ejecución, devuelve un preview del resultado, pero no ejecuta el side-effect.

```python
async def send_email(
    to: list[str],
    subject: str,
    body_md: str,
    tenant: str,
    dry_run: bool = False,  # el harness puede forzar esto
) -> dict:
    preview = {
        "to": to,
        "subject": subject,
        "body_preview": body_md[:200],
        "tenant": tenant,
    }
    if dry_run:
        return {"status": "dry_run", "would_send": preview}
    # ejecución real solo si dry_run=False y ya pasó HITL
    await email_service.send(to=to, subject=subject, body=body_md)
    await audit_log.record(
        tool="send_email",
        tenant=tenant,
        params=preview,
        outcome="sent",
    )
    return {"status": "sent", "to": to}
```

El harness puede forzar `dry_run=True` para toda una fase del workflow:

```python
# en el loop del agente
result = await dispatch_tool(
    tool_use_block,
    force_dry_run=context.phase == "analysis",  # en análisis, todo es dry-run
)
```

### 4.4 Human-in-the-loop (HITL)

Para tools de impacto alto, el agente no ejecuta directo — pausa y espera confirmación. En LangGraph (ver E02) esto se implementa con `interrupt_before`:

```python
# app/integrations/audit_graph.py
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.sqlite.aio import AsyncSqliteSaver

HIGH_IMPACT_NODES = {"send_findings_email", "write_final_report"}

app_graph = agent_graph.compile(
    checkpointer=checkpointer,
    interrupt_before=list(HIGH_IMPACT_NODES),
)
```

El harness expone un endpoint para aprobar/rechazar el step pendiente:

```python
# app/routers/agent_approvals.py
@router.post("/api/agent/{thread_id}/approve")
async def approve_pending_action(
    thread_id: str,
    caller: dict = Depends(require_api_key),
    session: SessionDep = None,
) -> dict:
    # verificar que el thread pertenece al tenant del caller
    thread_info = await thread_repo.get(session, thread_id)
    if thread_info.tenant != caller["tenant"]:
        raise HTTPException(403)
    # resumir el grafo desde el interrupt
    await app_graph.ainvoke(
        None,
        config={"configurable": {"thread_id": thread_id}},
    )
    return {"status": "resumed", "thread_id": thread_id}

@router.post("/api/agent/{thread_id}/reject")
async def reject_pending_action(
    thread_id: str,
    reason: str,
    caller: dict = Depends(require_api_key),
    session: SessionDep = None,
) -> dict:
    # registrar el rechazo y marcar el thread como cancelado
    await thread_repo.cancel(session, thread_id, reason=reason)
    return {"status": "cancelled", "thread_id": thread_id}
```

> [!cuidado]
> `interrupt_before` con `AsyncSqliteSaver` persiste el estado del grafo — cuando el usuario aprueba, LangGraph retoma exactamente donde se detuvo. Si el aprobador tarda horas, el estado sigue ahí. Pon un TTL: si el interrupt no se resuelve en N horas, el harness lo cancela automáticamente.

### 4.5 Idempotencia como defensa contra replay

Cada tool con side-effect tiene una `idempotency_key` que incluye `tenant_id`:

```python
async def write_report(
    kind: str,
    title: str,
    payload: dict,
    tenant: str,
    period: str,
) -> dict:
    idem_key = f"report:{tenant}:{period}:{title[:30]}"
    existing = await report_repo.get_by_idempotency_key(idem_key)
    if existing:
        return {"status": "already_exists", "report_id": existing.id}
    # crear solo si no existe
    report = await report_repo.create(
        kind=kind, title=title, payload=payload,
        tenant=tenant, idempotency_key=idem_key,
    )
    return {"status": "created", "report_id": report.id}
```

Si el agente en loop llama `write_report` 3 veces por el mismo período, el reporte se crea una sola vez. El ataque de replay (repetir una llamada para duplicar un efecto) también queda bloqueado.

### 4.6 Audit log de tools

Cada invocación de tool queda registrada con `tenant_id`, `tool_name`, `params`, `outcome`, `timestamp`:

```python
# app/repos/tool_audit.py
async def log_tool_use(
    session: AsyncSession,
    tenant: str,
    tool_name: str,
    params: dict,
    outcome: dict,
    duration_ms: int,
) -> None:
    entry = ToolAuditLog(
        tenant_id=tenant,
        tool_name=tool_name,
        params_json=json.dumps(params),   # sin PII en params si aplica
        outcome_json=json.dumps(outcome),
        duration_ms=duration_ms,
        created_at=datetime.utcnow(),
    )
    session.add(entry)
    await session.commit()
```

Este log es la base del threat modeling de S04: sin él, no puedes detectar patrones anómalos de uso de tools.

## 5. Ejemplo conducido

Clasificación de las tools canónicas (SHARED §3.6) por nivel de riesgo:

```python
from enum import Enum
from dataclasses import dataclass

class RiskLevel(str, Enum):
    AUTO = "auto"          # ejecuta automáticamente, con audit log
    DRY_RUN = "dry_run"    # dry-run obligatorio antes del step real
    HITL = "hitl"          # pausa y espera confirmación humana
    FORBIDDEN = "forbidden" # agente nunca puede invocarla

@dataclass
class ToolPolicy:
    name: str
    risk_level: RiskLevel
    rationale: str

TOOL_POLICIES: list[ToolPolicy] = [
    ToolPolicy("sql_query",           RiskLevel.AUTO,    "solo lectura, sin side-effects"),
    ToolPolicy("fetch_excel",         RiskLevel.AUTO,    "solo lectura"),
    ToolPolicy("parse_invoice_pdf",   RiskLevel.AUTO,    "OCR, solo lectura"),
    ToolPolicy("fetch_csv",           RiskLevel.AUTO,    "solo lectura"),
    ToolPolicy("write_report",        RiskLevel.DRY_RUN, "escribe artefacto; revisar preview"),
    ToolPolicy("post_slack_message",  RiskLevel.DRY_RUN, "comunicación externa; revisar texto"),
    ToolPolicy("erp_fetch_transactions", RiskLevel.AUTO, "lectura de ERP; sin escritura"),
    ToolPolicy("send_email",          RiskLevel.HITL,    "comunicación irreversible con terceros"),
    ToolPolicy("bank_open_api",       RiskLevel.HITL,    "acceso a datos bancarios en tiempo real"),
]
```

## 8. Caso de uso real

**F-CTA-02 (auditoría interna)**: el workflow tiene cuatro fases — ingest, analyze, report, deliver. La separación de tools por fase es directa: ingest y analyze usan solo tools de lectura. Report usa `write_report` con dry-run. Deliver usa `send_email` con HITL. Si el HITL no se resuelve en 4 horas, el workflow cancela el delivery y notifica al operador.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Verificar que una query empieza con SELECT | Determinístico | Regla de prefijo — cerrada. |
| Forzar `dry_run=True` para tools destructivas en fase de análisis | Determinístico | Política fija por fase — sin ambigüedad. |
| Evaluar idempotency_key antes de ejecutar el side-effect | Determinístico | Lookup en DB, resultado binario. |
| Decidir el TTL del interrupt de HITL para un caso particular | Agéntico | Depende del SLA con el cliente y la urgencia del caso. |
| Clasificar una nueva tool propuesta en el nivel de riesgo correcto | Agéntico | Requiere juicio sobre el impacto de negocio, la reversibilidad, y el contexto del cliente. |

## 10. Errores típicos

**1. Allowlist en runtime en lugar de en configuración.**
*Síntoma*: el agente puede invocar tools que el operador no autorizó porque el harness las agrega dinámicamente según el prompt.
*Causa*: la lista de tools se construye leyendo el input del usuario para «personalizar» el agente.
*Arreglo*: la allowlist es estática en config. El input del usuario puede cambiar parámetros, nunca la lista de tools disponibles.

**2. Dry-run que no simula fielmente.**
*Síntoma*: el operador aprueba basándose en el preview del dry-run, pero la ejecución real produce un resultado distinto.
*Causa*: el dry-run construye el preview con datos de un estado anterior, no del estado actual del workflow.
*Arreglo*: el dry-run lee los mismos datos que la ejecución real. Si hay diferencia de estado entre el dry-run y la ejecución, la ejecución re-valida antes de actuar.

**3. HITL sin TTL.**
*Síntoma*: un interrupt queda pendiente indefinidamente. El tenant olvidó aprobarlo y semanas después el workflow «se reactiva» con datos desactualizados.
*Causa*: `interrupt_before` sin expiración configurada.
*Arreglo*: el harness registra `interrupt_expires_at = now() + TTL`. Un cron job cancela interrupts expirados. El TTL por defecto: 24 horas para operaciones de reporte, 1 hora para operaciones financieras.

**4. Audit log con PII en `params_json`.**
*Síntoma*: el log de `send_email` contiene los emails de los destinatarios y el contenido del mensaje.
*Causa*: loguear `params` completo sin filtrar.
*Arreglo*: para `send_email`, loguear `to_count` (número de destinatarios), no los emails. El `body_md` → loguear hash o primeros 50 chars sin contenido sensible.

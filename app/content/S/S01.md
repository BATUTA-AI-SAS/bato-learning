## 1. Hilo conductor

En B10 aprendiste a implementar autenticación con `Depends`. En C03 viste Row-Level Security como mitigación de fuga entre tenants. Este módulo usa ambas bases para atacar el problema desde el otro lado: dado un router FastAPI, ¿qué vulnerabilidades tiene antes de que añadas cualquier defensa? Hacia S02 aplicarás estas defensas al vector específico de los agentes IA (prompt injection). Sin entender el API Top 10 primero, el threat modeling de S04 no tiene suelo.

---

## 2. Idea central

La **OWASP API Security Top 10 (2023)** es la lista autoritativa de los diez riesgos más críticos en APIs HTTP. Cada riesgo tiene un identificador estable (`API1:2023`, ..., `API10:2023`), un nombre y una descripción de impacto. No son hipótesis: son patrones extraídos de brechas reales documentadas por la OWASP Foundation.

Imagen mental: cada ítem de la lista es una puerta que, en tu API, está entreabierta por defecto. FastAPI no la cierra sola — tú la cierras con `Depends`, Pydantic, `slowapi`, headers de seguridad y RLS. Este módulo mapea puerta por puerta.

---

## 3. Por qué importa

Un agente IA que opera sobre datos de ACME Manufacturing expone los mismos endpoints HTTP que cualquier API REST — con una diferencia: el agente los llama en nombre del usuario, con privilegios elevados, a alta frecuencia y sin un humano revisando cada llamada. Un endpoint vulnerable no solo afecta a un usuario descuidado; lo explota el propio agente comprometido.

La ficha **F-CTA-02 (auditoría interna multitenant)** es el caso canónico: el agente lee facturas de ACME, Globex e Initech con la misma instancia. Un `GET /invoices/{id}` sin verificación de tenant expone datos de los tres.

## 4. Cómo funciona por dentro

### API1:2023 — Broken Object Level Authorization (BOLA)

El defecto más frecuente y el más costoso. El endpoint recibe un ID en la URL y lo consulta sin verificar que el recurso pertenezca al caller.

**Código vulnerable:**
```python
# app/routers/invoices.py — VULNERABLE
@router.get("/invoices/{invoice_id}")
async def get_invoice(invoice_id: int, session: AsyncSession = Depends(get_session)):
    invoice = await session.get(Invoice, invoice_id)
    if invoice is None:
        raise HTTPException(404)
    return invoice  # devuelve la factura de cualquier tenant si el ID es válido
```

**Cómo se explota:** el atacante itera `GET /invoices/1`, `/invoices/2`, ..., `/invoices/50000`. Con un ID numérico secuencial obtiene todas las facturas de todos los tenants. Se llama IDOR (Insecure Direct Object Reference).

**Fix:**
```python
# app/routers/invoices.py — SEGURO
@router.get("/invoices/{invoice_id}")
async def get_invoice(
    invoice_id: int,
    caller: dict = Depends(require_api_key),   # de B10
    session: AsyncSession = Depends(get_session),
) -> InvoiceOut:
    invoice = await invoice_repo.get_by_id_and_tenant(
        session, invoice_id, tenant=caller["tenant"]
    )
    if invoice is None:
        raise HTTPException(404)  # mismo error para "no existe" y "no es tuyo"
    return invoice
```

> [!cuidado]
> Devolver `403` cuando el recurso existe pero no es del tenant filtra información («sé que la factura 42 existe, solo que no es mía»). Usa siempre `404` para ambos casos.

La mitigación canónica en Postgres es **RLS** (ver C03): la política `tenant_isolation` hace que el filtro por `tenant_id` sea automático en cada query, sin importar si el código lo olvida. Es defensa en profundidad — el código Y el motor filtran.

### API2:2023 — Broken Authentication

JWT con secret débil, sin validar la firma, o sin expiración.

**Código vulnerable:**
```python
# VULNERABLE: secret trivial, sin verificar exp
def decode_token(token: str) -> dict:
    return jwt.decode(token, "secret", algorithms=["HS256"])
    # si el atacante adivina "secret", forja cualquier JWT
```

**Fix:**
```python
import secrets, os
JWT_SECRET = os.environ["JWT_SECRET"]  # 32 bytes random en prod, nunca en código

def decode_token(token: str) -> dict:
    return jwt.decode(
        token,
        JWT_SECRET,
        algorithms=["HS256"],
        options={"require": ["exp", "iat", "sub"]},  # campos obligatorios
    )
```

El secret debe tener entropía ≥ 256 bits. Genera con `python -c "import secrets; print(secrets.token_hex(32))"`. Rótalo cada 90 días como mínimo.

### API3:2023 — Broken Object Property Level Authorization

Devuelves el objeto ORM completo cuando el caller solo debería ver 3 campos.

**Código vulnerable:**
```python
# VULNERABLE: devuelve el modelo ORM con TODOS los campos, incluido password_hash
@router.get("/users/{user_id}")
async def get_user(user_id: int, session: AsyncSession = Depends(get_session)):
    return await session.get(User, user_id)
```

**Fix — schema de salida explícito con Pydantic:**
```python
class UserPublic(BaseModel):
    id: int
    name: str
    email: str
    # NO: password_hash, internal_notes, credit_limit

@router.get("/users/{user_id}", response_model=UserPublic)
async def get_user(user_id: int, ...) -> UserPublic:
    user = await session.get(User, user_id)
    return UserPublic.model_validate(user)
```

`response_model=UserPublic` es la segunda línea de defensa: aunque el handler devuelva más campos, FastAPI los filtra antes de serializar.

### API4:2023 — Unrestricted Resource Consumption

Sin rate limit, sin límite de tamaño de cuerpo, sin timeout. Un atacante o un agente en bucle agota la CPU/DB/presupuesto LLM.

**Fix — `slowapi` (basado en `limits`):**
```python
from slowapi import Limiter
from slowapi.util import get_remote_address

limiter = Limiter(key_func=get_remote_address)

@router.post("/api/audit/run")
@limiter.limit("10/minute")   # máx 10 ejecuciones/minuto por IP
async def run_audit(request: Request, ...) -> dict:
    ...
```

Para tamaño de body, añade en el lifespan de FastAPI:
```python
app = FastAPI()
app.add_middleware(
    TrustedHostMiddleware, allowed_hosts=["*"]
)
# limita body a 1 MB
app.state.max_request_body_size = 1_048_576
```

Para tenants: el límite por IP no distingue entre tenants del mismo servidor. Usa la API key como `key_func` para rate limit por tenant.

### API5:2023 — Broken Function Level Authorization

Endpoints de administración sin verificar que el caller tenga rol admin. Los endpoints existen; basta descubrirlos.

**Código vulnerable:**
```python
# VULNERABLE: cualquiera con una API key válida puede borrar tenants
@router.delete("/admin/tenants/{tenant_slug}")
async def delete_tenant(tenant_slug: str, caller: dict = Depends(require_api_key)):
    await tenant_repo.delete(tenant_slug)
```

**Fix:**
```python
async def require_admin(caller: dict = Depends(require_api_key)) -> dict:
    if "admin" not in caller.get("scopes", []):
        raise HTTPException(403, detail="admin scope required")
    return caller

@router.delete("/admin/tenants/{tenant_slug}")
async def delete_tenant(
    tenant_slug: str,
    caller: dict = Depends(require_admin),  # dependency anidada
) -> dict:
    await tenant_repo.delete(tenant_slug)
    return {"deleted": tenant_slug}
```

### API6:2023 — Unrestricted Access to Sensitive Business Flows

Un competidor obtiene una API key de prueba de bajo costo y scrape el agente completo: todos los findings de auditoría, todos los proveedores, todos los montos. La API funciona correctamente; el abuso es de negocio, no técnico.

Mitigaciones:
- Rate limit por tenant + alertas cuando `requests_per_hour` supera 10× el promedio histórico.
- Fingerprinting de uso: secuencias anómalas (solo GETs de listado, sin POSTs) alertan.
- Scopes diferenciados: `read:summaries` vs `read:full_findings`. El tier barato solo accede a resúmenes.
- CAPTCHA o paso humano para exportación masiva.

### API7:2023 — Server-Side Request Forgery (SSRF)

Crítica para agentes IA: el agente acepta una URL como parámetro (e.g., «parsea este PDF desde esta URL») y la fetcha sin validar. Un atacante pasa `http://169.254.169.254/latest/meta-data/` (metadata de AWS/GCP) y obtiene credenciales de la instancia.

**Código vulnerable:**
```python
# VULNERABLE: el agente fetcha cualquier URL que el modelo proponga
async def fetch_document(url: str) -> bytes:
    async with httpx.AsyncClient() as client:
        resp = await client.get(url, timeout=30)
        return resp.content
```

**Fix — allowlist de dominios + bloqueo de IPs privadas:**
```python
import ipaddress, socket
from urllib.parse import urlparse

ALLOWED_SCHEMES = {"https"}
ALLOWED_DOMAINS = {"storage.acme.com", "docs.globex.com"}

def validate_fetch_url(url: str) -> None:
    parsed = urlparse(url)
    if parsed.scheme not in ALLOWED_SCHEMES:
        raise ValueError(f"scheme {parsed.scheme!r} no permitido")
    if parsed.hostname not in ALLOWED_DOMAINS:
        raise ValueError(f"dominio {parsed.hostname!r} no en allowlist")
    # bloquear IPs privadas/loopback/link-local
    try:
        ip = ipaddress.ip_address(socket.gethostbyname(parsed.hostname))
    except Exception:
        raise ValueError("no se pudo resolver el host")
    if ip.is_private or ip.is_loopback or ip.is_link_local:
        raise ValueError("IP privada bloqueada")
```

> [!cuidado]
> La resolución DNS puede cambiar entre la validación y el fetch (TOCTOU). Solución: resolver la IP una vez, validarla, y pasarla directamente al cliente HTTP con el header `Host` correcto.

### API8:2023 — Security Misconfiguration

Debug mode en producción, CORS `*`, headers de seguridad ausentes, directorio de admin con credenciales default.

**Configuración vulnerable:**
```python
# VULNERABLE
app = FastAPI(debug=True)  # stacktrace detallado al cliente en prod
app.add_middleware(CORSMiddleware, allow_origins=["*"])  # cualquier origen
```

**Fix:**
```python
import os
app = FastAPI(debug=False)  # nunca True en prod

ALLOWED_ORIGINS = os.environ.get("CORS_ORIGINS", "https://app.acme.com").split(",")
app.add_middleware(
    CORSMiddleware,
    allow_origins=ALLOWED_ORIGINS,
    allow_credentials=True,
    allow_methods=["GET", "POST", "PUT", "DELETE"],
    allow_headers=["Authorization", "Content-Type"],
)
```

Headers de seguridad con middleware:
```python
from starlette.middleware.base import BaseHTTPMiddleware

class SecurityHeadersMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        response = await call_next(request)
        response.headers["X-Content-Type-Options"] = "nosniff"
        response.headers["X-Frame-Options"] = "DENY"
        response.headers["Strict-Transport-Security"] = "max-age=31536000"
        response.headers["Content-Security-Policy"] = "default-src 'self'"
        return response
```

### API9:2023 — Improper Inventory Management

Rutas legacy abiertas que nadie recuerda. El router de la versión 1 sigue montado en `app` aunque la v2 lleva 18 meses activa.

```python
# VULNERABLE: ruta legacy sin auth, producción la olvidó
@app.get("/v1/invoices")  # la versión 1 no tenía auth
async def legacy_list_invoices():
    return await invoice_repo.list_all()  # devuelve TODO sin filtro
```

**Prevención:**
- Inventario de rutas con `python -c "import app.main; [print(r.path) for r in app.main.app.routes]"` en CI.
- Deprecate con `deprecated=True` en FastAPI y fecha de retiro en el changelog.
- Política: una ruta legacy sin auth activa es una vulnerabilidad de alta severidad.

### API10:2023 — Unsafe Consumption of APIs

El agente confía en la respuesta de una API de tercero (ERP, banco, proveedor de datos) sin validar forma ni contenido. Un proveedor comprometido inyecta datos maliciosos que el agente ejecuta.

**Código vulnerable:**
```python
# VULNERABLE: confía ciegamente en la respuesta del ERP
async def get_vendor_data(vendor_id: str) -> dict:
    resp = await erp_client.get(f"/vendors/{vendor_id}")
    return resp.json()  # ¿y si el ERP devuelve {"name": "<script>alert(1)</script>"}?
```

**Fix:**
```python
from pydantic import BaseModel, field_validator

class VendorData(BaseModel):
    id: str
    name: str
    tax_id: str

    @field_validator("name")
    @classmethod
    def sanitize_name(cls, v: str) -> str:
        # elimina tags HTML; en un agente también filtra prompt injection
        import html
        return html.escape(v.strip())[:200]

async def get_vendor_data(vendor_id: str) -> VendorData:
    resp = await erp_client.get(f"/vendors/{vendor_id}")
    resp.raise_for_status()
    return VendorData.model_validate(resp.json())  # valida shape + tipo
```

## 5. Ejemplo conducido

Un router con tres vulnerabilidades combinadas:

```python
# app/routers/findings.py — detecta cuántas vulnerabilidades hay aquí
from fastapi import APIRouter, Depends
from sqlalchemy.ext.asyncio import AsyncSession
from ..db import get_session
from ..models import Finding

router = APIRouter()

@router.get("/findings/{finding_id}")
async def get_finding(finding_id: int, session: AsyncSession = Depends(get_session)):
    return await session.get(Finding, finding_id)

@router.get("/findings")
async def list_all_findings(session: AsyncSession = Depends(get_session)):
    result = await session.execute(select(Finding))
    return result.scalars().all()

@router.delete("/admin/findings/{finding_id}")
async def delete_finding(finding_id: int, session: AsyncSession = Depends(get_session)):
    finding = await session.get(Finding, finding_id)
    await session.delete(finding)
    await session.commit()
```

Vulnerabilidades: API1 (sin tenant check en `get_finding`), API1 + API3 (lista toda sin tenant y devuelve objeto ORM completo), API5 (endpoint admin sin rol check). El ejercicio pide detectar y corregir las tres.

## 8. Caso de uso real

**F-CTA-02 (auditoría interna multitenant)**: el agente corre con credenciales de servicio que tienen acceso de lectura a las facturas de todos los tenants. Si el endpoint `/invoices/{id}` tiene BOLA (API1), el agente comprometido por indirect prompt injection puede exfiltrar facturas de tenants distintos al suyo. RLS en Postgres actúa como red de seguridad cuando el código olvida el filtro.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Verificar que `invoice.tenant_id == caller["tenant"]` | Determinístico | Comparación de igualdad — regla cerrada. |
| Validar el shape de la respuesta del ERP con Pydantic | Determinístico | Schema fijo, validación automática. |
| Configurar CORS con lista de orígenes permitidos | Determinístico | Lista fija en variables de entorno. |
| Decidir qué endpoints necesitan rate limit más estricto | Agéntico | Depende del perfil de uso real del tenant — sin regla universal. |
| Detectar abuso de API (API6) por patrones de uso anómalos | Agéntico | Requiere baseline histórico y juicio sobre qué es «anómalo» para ese cliente. |

## 10. Errores típicos

**1. Usar el ID del objeto como la única defensa.**
*Síntoma*: el endpoint devuelve datos correctos en las pruebas porque el QA usa IDs propios; en producción el cliente itera IDs ajenos.
*Causa*: confiar en que «el ID es difícil de adivinar» — con IDs numéricos secuenciales es trivial.
*Arreglo*: siempre añadir `AND tenant_id = :tenant` en la query. RLS como segunda línea.

**2. `debug=True` olvidado en el `.env` de staging que se copió a producción.**
*Síntoma*: los stacktraces de FastAPI incluyen paths internos, variables locales y fragmentos de código.
*Causa*: variable de entorno heredada del ambiente de desarrollo.
*Arreglo*: `debug` siempre desde env var; en el CI de producción, el pipeline falla si `DEBUG=True` está en el `.env` de producción.

**3. CORS `*` «porque es una API pública».**
*Síntoma*: cualquier sitio web puede hacer requests con credenciales a la API.
*Causa*: malentender que CORS protege el servidor — en realidad protege al navegador del usuario.
*Arreglo*: lista explícita de orígenes. Las APIs machine-to-machine no necesitan CORS en absoluto.

**4. No distinguir `404` de `403` para BOLA.**
*Síntoma*: el atacante recibe `403` en el objeto ajeno y `404` en el inexistente — sabe qué IDs existen.
*Causa*: querer dar feedback útil al cliente legítimo.
*Arreglo*: devolver siempre `404` cuando el recurso no existe O existe pero no es del tenant del caller.

**5. Rate limit por IP en un SaaS con proxies.**
*Síntoma*: clientes legítimos detrás del mismo proxy corporativo son bloqueados juntos.
*Causa*: `key_func=get_remote_address` en un entorno con NAT empresarial.
*Arreglo*: para SaaS, el `key_func` debe ser la API key o el `tenant_id`, no la IP.

## Hilo conductor

En 00-18 aprendiste Git: commit, branch, push, pull. En D05 construiste el pipeline de CI/CD con GitHub Actions: lint, tests, build, deploy vía SSH. D09 cierra el loop conectando ambos con el flujo real de trabajo diario. No hay concepto nuevo de Git aquí, ni de GitHub Actions. Lo que hay es el flujo completo y sus variantes: PR normal, hotfix, rollback, branch protection. Cuando llegues a E07 (evals en CI), este flujo ya será tu columna vertebral.

## Idea central

El flujo PR → CI → merge → deploy automático es la secuencia que separa un proyecto con historial trazable de un proyecto con «funciona en mi máquina». Cada paso tiene una responsabilidad:

- **Branch**: aislar el cambio del estado estable.
- **PR**: hacer el cambio visible y revisable antes de que entre.
- **CI**: verificar automáticamente que el cambio no rompe nada.
- **Merge**: integrar el cambio al estado estable solo cuando CI está verde.
- **Deploy automático**: llevar el cambio a producción sin intervención manual.

Ningún paso es opcional. Saltarse uno crea deuda de confianza que se cobra en el peor momento.

## Por qué importa

Sin este flujo, el bug de producción llega así: «subí unos cambios rápido el viernes, no corrí los tests, el agente de ACME lleva 6 horas rechazando todas las facturas y el CFO está en el chat». Con el flujo, ese cambio del viernes nunca llega a producción: el CI lo habría bloqueado porque los tests fallaban.

La ficha **F-OPS-02 (gestión de inventario ACME)** y cualquier caso de producción del curso dependen de este flujo para garantizar que los cambios al agente se validan antes de impactar al cliente.

> [!cuidado]
> El pipeline CI/CD es **100% determinístico**: el YAML dice exactamente qué pasa. Ningún agente toma decisiones de deploy. Un LLM puede generar el YAML inicial, pero tú lo revisas, lo pruebas en un branch y lo mergeas. Ningún agente tiene permiso de `git push` a `main` ni de modificar el pipeline de producción sin revisión humana.

## Cómo funciona por dentro

### El flujo estándar: feature o fix

```bash
# 1. Asegúrate de que main está actualizado
git switch main
git pull

# 2. Crea un branch con nombre descriptivo
git switch -c fix/invoice-matching-with-accents

# 3. Haz los cambios
# ... editas app/services/matching.py ...

# 4. Commit con mensaje Conventional Commits
git add app/services/matching.py
git commit -m "fix: matching para descripciones con tilde y mayúsculas"

# 5. Sube el branch a GitHub
git push -u origin fix/invoice-matching-with-accents
# GitHub muestra el botón "Compare & pull request"

# 6. Abre el PR en GitHub (o con gh CLI)
gh pr create --title "fix: matching para descripciones con tilde" \
  --body "Corrige el bug donde 'PAGO ACME' no matcheaba con 'PAGO ÁCME'."

# 7. CI corre automáticamente
# → lint (ruff)
# → tests (pytest)
# → evals del golden set (desde E07 en adelante)

# 8. Si CI verde + review aprobado: merge
# → deploy automático a staging
# → manual promote a producción (o automático si así se configuró)
```

### Branch protection rules

Las **branch protection rules** de GitHub evitan que alguien haga push directo a `main` y que un PR mergee si el CI está rojo:

En GitHub: Settings → Branches → Add rule → Branch name pattern: `main`

Opciones críticas para activar:
- **Require a pull request before merging**: nadie hace `git push origin main` directo.
- **Require status checks to pass before merging**: selecciona los jobs de CI que deben estar verdes (test, lint, eval).
- **Require at least 1 approving review** (opcional en proyectos en solitario, recomendado en equipo).
- **Do not allow bypassing the above settings**: incluso los admins del repo respetan las reglas.

> [!nota]
> En repos personales o cuando eres el único dev, puedes no exigir review. Pero las status checks de CI son no-negociables: sin ellas, el único que sabe si el push rompió algo eres tú — después de que llegó a producción.

### Estrategias de merge

GitHub ofrece tres estrategias cuando apruebas un PR:

| Estrategia | Qué produce | Cuándo usar |
|-----------|-------------|-------------|
| **Squash and merge** | Un solo commit en main con todo el trabajo del branch | Proyectos de aplicación (como este curso): historial limpio |
| **Merge commit** | Un commit de merge + todos los commits del branch | Cuando quieres preservar el historial completo del branch |
| **Rebase and merge** | Commits del branch reescritos encima de main | Historial lineal sin commit de merge; más limpio en teoria, más propenso a conflictos |

**Recomendación para este curso**: **Squash and merge**. El PR agrupa trabajo relacionado; el historial de `main` queda con un commit por feature/fix, legible de un vistazo con `git log --oneline`.

Configura el repositorio para que solo permita Squash: Settings → General → Pull Requests → Allow squash merging (deja solo este).

### Hotfix flow: urgencia sin saltarse CI

Cuando hay un bug crítico en producción:

```bash
# Branch directo desde main (no desde una rama en desarrollo)
git switch main
git pull
git switch -c hotfix/agent-crash-on-empty-invoice

# Arregla el bug
git commit -m "fix: evitar crash cuando la factura llega sin vendor_id"

# Abre PR marcado como hotfix
gh pr create --title "hotfix: crash con factura sin vendor_id" \
  --label hotfix --base main

# CI corre igual (no saltarse)
# Review rápido o self-approval si eres el único dev
# Merge → deploy automático
```

La etiqueta `hotfix` es solo visual. El flujo es el mismo. La diferencia con el flujo normal es la urgencia — pero el CI sigue corriendo. No existe el escenario correcto de «pusheamos directo a main porque era urgente».

### Revert: deshacer un merge limpiamente

Si un cambio que ya mergeó rompe producción y no tienes un hotfix listo:

```bash
# Encuentra el commit del merge
git log --oneline main

# Opción 1: revert del commit específico (crea un nuevo commit que deshace)
git revert a3f92d1
git push origin main
# → CI corre → deploy automático descarga el revert

# Opción 2: en GitHub: Pull Request → Revert
# GitHub crea automáticamente un PR de revert que pasa por CI
```

**Revert es preferible a rollback manual** porque deja evidencia en el historial: queda registrado qué se revirtió y por qué. Un rollback manual (`git reset --hard` + force push) reescribe el historial y borra el rastro.

### El deploy automático: qué pasa después del merge

El workflow de GitHub Actions que construiste en D05 se dispara con `on: push: branches: [main]`. Después del merge:

```yaml
# .github/workflows/deploy.yml (simplificado)
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Hetzner via SSH
        run: |
          ssh ubuntu@${{ secrets.HETZNER_IP }} "
            cd /srv/audit-agent &&
            git pull &&
            docker compose pull &&
            docker compose up -d --remove-orphans
          "
```

El workflow corre en el runner de GitHub. Se conecta al servidor de Hetzner por SSH (la clave privada está en `secrets.SSH_PRIVATE_KEY`, nunca en el repo). Actualiza el código, descarga la nueva imagen, reinicia los contenedores.

El resultado: entre el momento en que aprueban el PR y el momento en que la nueva versión está corriendo en el servidor, pasan 3-5 minutos sin ninguna intervención manual.

### Predecir si un PR bloquea el merge

Dado un workflow de CI y un conjunto de cambios, puedes predecir si el PR va a bloquear antes de abrirlo:

1. ¿Los tests locales pasan? `uv run pytest` antes de `git push`.
2. ¿El linter pasa? `uv run ruff check .` antes de `git push`.
3. ¿El golden set de evals pasa? `uv run python scripts/run_evals.py --threshold 0.85`.
4. ¿La imagen Docker buildea? `docker build -t test .` si tocaste el Dockerfile.

Si los cuatro pasan local, el CI casi con certeza pasa en el runner. Los fallos en CI que no fallaron local generalmente son: variables de entorno que existen en tu máquina pero no en el runner, o dependencias del sistema que no están en la imagen del runner.

## Ejemplo conducido

El flujo completo para un fix del agente de conciliación de ACME:

```bash
# Punto de partida: main actualizado, bug reportado por el cliente
git switch main && git pull

# Branch
git switch -c fix/conciliacion-edge-case

# Trabajo
# ... editas app/services/matching.py ...
# ... añades test en tests/test_matching.py para el caso reportado ...

# Verifica antes de pushear
uv run pytest tests/test_matching.py -v  # ¿pasan los tests?
uv run ruff check app/                   # ¿linter limpio?

# Commit
git add app/services/matching.py tests/test_matching.py
git commit -m "fix: matching para descripciones con tilde"

# Push y PR
git push -u origin fix/conciliacion-edge-case
gh pr create \
  --title "fix: matching para descripciones con tilde" \
  --body "Cierra #42. El agente fallaba con 'PAGO ÁCME' vs 'PAGO ACME'."

# CI en GitHub: lint ✓ | tests ✓ | evals 91% ✓ (threshold 85%)
# Review aprobado (o self-merge en proyecto en solitario)
# Squash and merge
# Deploy automático en ~4 minutos
```

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Ejecutar `git push` y abrir el PR | determinístico | Operación de control de versiones definida; el resultado es predecible |
| CI: correr lint + tests + evals | determinístico | Pipeline declarativo en YAML; mismo código → mismo resultado |
| Decidir si hacer merge de un PR | agéntico | Requiere leer el contexto del cambio, entender los riesgos, decidir si los tests son suficientes |
| Deploy automático al servidor | determinístico | Los comandos SSH son exactos; no hay decisión en el runtime |
| Decidir si hacer hotfix o rollback | agéntico | Requiere evaluar impacto en producción, tiempo de arreglo, riesgo de la operación |
| Elegir la estrategia de merge (squash vs rebase) | agéntico | Depende del historial del repo, del equipo, y de las convenciones del proyecto |

## Errores típicos

**1. Push directo a `main` porque «era un cambio pequeño».**
*Síntoma*: `git log main` muestra commits directos sin PR ni CI.
*Causa*: las branch protection rules no estaban activadas, o el dev las saltó porque tenía permisos de admin.
*Arreglo*: activar «Do not allow bypassing the above settings» en branch protection. Sin excepciones para admins.

**2. PR abierto con CI fallido, mergeado porque «los tests son conocidos».**
*Síntoma*: producción falla por exactamente el test que estaba rojo.
*Causa*: el estado del CI se ignoró porque el dev «sabía» que era un falso positivo.
*Arreglo*: CI rojo = PR bloqueado sin excepción. Si el test es un falso positivo, la solución es arreglarlo, no ignorarlo.

**3. Un solo commit con 47 archivos cambiados.**
*Síntoma*: el PR tiene un diff enorme, el reviewer no puede entender qué pasó, el rollback es imposible de hacer limpiamente.
*Causa*: el dev trabajó semanas en el branch sin commitear.
*Arreglo*: commits pequeños y frecuentes dentro del branch. Cada commit = un cambio lógico completo.

**4. Secrets en el código del workflow (no en GitHub Secrets).**
*Síntoma*: `ANTHROPIC_API_KEY=sk-ant-...` aparece en el YAML del workflow.
*Causa*: el dev no supo cómo configurar GitHub Secrets y «lo hizo funcionar rápido».
*Arreglo*: Settings → Secrets → Actions → New repository secret. En el workflow: `${{ secrets.NOMBRE }}`. La clave nunca en el YAML.

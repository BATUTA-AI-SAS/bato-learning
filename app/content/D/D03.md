## Hilo conductor

En D02 levantaste la app localmente con `docker compose` y entendiste healthchecks y override files. Ese stack ya funciona; ahora necesita una dirección pública, un certificado TLS, secrets que no estén en el repositorio y un backup que no capture mid-write. D03 cubre el camino completo: servidor Hetzner CX22, DNS, Traefik v3, sops+age, cron de backup y GitHub Actions por SSH. Cuando termines aquí, D04 (observabilidad) asume que la app está en producción y tú quieres saber qué está haciendo.

## Idea central

**Desplegar** no es «copiar archivos». Es crear la unidad de cómputo, apuntar el DNS, gestionar el TLS, separar los secrets del código y garantizar que puedes volver atrás. Cada pieza tiene un contrato claro: Hetzner provee la VM, Traefik v3 provee el reverse-proxy con TLS automático via Let's Encrypt, sops+age cifra los secrets antes de commitearlos, y `sqlite3 .backup` captura la base de datos sin riesgo de corrupción.

## Por qué importa

Sin despliegue real, el agente de cierre mensual de ACME (F-FIN-03) existe solo en tu laptop. El workflow Temporal que corre el agente necesita una dirección estable, un certificado válido para los webhooks y un Postgres o SQLite accesible que no se borre si el servidor se reinicia.

La alternativa «todo en Fly.io / Railway» tiene sentido para prototipos, pero el curso enseña el stack que BATUTA opera en producción: Hetzner porque el costo es real (≈ 4,50 EUR/mes para CX22) y predecible, Traefik porque el certificado se renueva automáticamente sin cron extra, sops+age porque git es la fuente de verdad y los secrets no deben salir de él sin cifrar.

> [!cuidado]
> El agente **no** toca infra. Puede generar el manifest inicial de Compose o el script de backup; tú lo revisas y lo aplicas. Dale permiso de `docker compose up` en un sandbox de staging si quieres, nunca en producción sin revisión humana.

## Cómo funciona por dentro

### 1. El servidor: CX22 en Hetzner Cloud

Hetzner Cloud expone una CLI (`hcloud`) y una API REST. El tipo CX22 tiene 2 vCPU, 4 GB RAM, 40 GB SSD NVMe — suficiente para la app + Traefik + Phoenix en modo ligero.

```bash
# instala la CLI
brew install hcloud          # macOS; en Linux: ver hetzner.com/cli

# autentica (token desde console.hetzner.cloud → proyecto → API tokens)
hcloud context create batuta-prod

# crea el servidor con Ubuntu 24.04 y tu clave SSH
hcloud server create \
  --name bato-prod \
  --type cx22 \
  --image ubuntu-24.04 \
  --ssh-key tu-clave-ssh \
  --location nbg1
```

La IP pública aparece en `hcloud server describe bato-prod`. Esa IP va al DNS.

### 2. DNS: un A record, nada más

En tu proveedor de dominio (Cloudflare, Porkbun, lo que uses) crea:

```
A   learning.tudominio.com   →   <IP del servidor>   TTL 300
```

Con TTL 300 (5 min) puedes cambiar rápido si migras de servidor. Traefik hace el TLS challenge sobre este dominio, así que el A record debe resolver antes de arrancar Traefik.

### 3. Docker en Ubuntu 24.04

```bash
ssh root@<IP>

# Docker oficial (no el de snap)
curl -fsSL https://get.docker.com | sh
usermod -aG docker $USER   # si usas usuario no-root

# verifica
docker compose version     # debe ser >= 2.30
```

### 4. Traefik v3: entrypoints, TLS, labels

El repo ya tiene `infra/traefik/traefik.yml` y `infra/hetzner/compose.prod.yml`. Traefik v3 (imagen `traefik:v3.2`) cambia respecto a v2 en varios puntos — el más relevante para nosotros: `tlsChallenge` ya no requiere configuración extra de `http.routers`; basta con declarar el `certresolver` en las labels del contenedor.

```yaml
# infra/hetzner/compose.prod.yml (extracto real del repo, anotado)
services:
  traefik:
    image: traefik:v3.2
    command:
      - --providers.docker=true
      - --providers.docker.exposedByDefault=false
      - --entrypoints.web.address=:80
      - --entrypoints.websecure.address=:443
      - --certificatesresolvers.le.acme.email=ops@tudominio.com
      - --certificatesresolvers.le.acme.storage=/letsencrypt/acme.json
      - --certificatesresolvers.le.acme.tlschallenge=true
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - traefik_letsencrypt:/letsencrypt

  app:
    image: bato-learning:latest
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.app.rule=Host(`learning.tudominio.com`)"
      - "traefik.http.routers.app.entrypoints=websecure"
      - "traefik.http.routers.app.tls.certresolver=le"
      - "traefik.http.services.app.loadbalancer.server.port=8080"

volumes:
  traefik_letsencrypt:
```

El flujo TLS con `tlsChallenge`:
1. Traefik recibe conexión en `:443`.
2. Si no tiene certificado para ese dominio, hace un TLS-ALPN-01 challenge con Let's Encrypt (sin necesidad de exponer `:80` para el challenge).
3. LE valida, emite el certificado, Traefik lo guarda en `/letsencrypt/acme.json`.
4. El volumen `traefik_letsencrypt` persiste el acme.json entre reinicios.

> [!cuidado]
> `acme.json` debe tener permisos `600`. Traefik lo crea con esos permisos, pero si lo copias manualmente falla. Verifica con `ls -la /var/lib/docker/volumes/traefik_letsencrypt/`.

La redirección HTTP→HTTPS va en el entrypoint `web`:

```yaml
# en traefik.yml (config estática) si usas archivo en vez de CLI flags
entryPoints:
  web:
    address: ":80"
    http:
      redirections:
        entryPoint:
          to: websecure
          scheme: https
```

### 5. Secrets con sops + age

`.env` en el servidor es aceptable para un prototipo. En producción con CI/CD, `.env` commiteado es un secret leak esperando ocurrir. La solución: cifrar con **sops** (Mozilla) + **age** (herramienta de cifrado moderna).

```bash
# instala age y sops (servidor y CI)
apt install age                      # Ubuntu 24.04
wget https://github.com/getsops/sops/releases/download/v3.9.3/sops-v3.9.3.linux.amd64 \
  -O /usr/local/bin/sops && chmod +x /usr/local/bin/sops

# genera un par de claves age
age-keygen -o ~/.age/key.txt
# muestra la clave pública: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

# crea el .sops.yaml en la raíz del repo
cat > .sops.yaml <<'EOF'
creation_rules:
  - path_regex: \.enc\.env$
    age: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
EOF

# crea y cifra los secrets
cp .env.example .env.plaintext       # nunca commitear .env.plaintext
sops --encrypt .env.plaintext > .env.enc.env
git add .env.enc.env .sops.yaml      # .env.plaintext va en .gitignore

# en el servidor (o en CI) descifra con la clave privada
export SOPS_AGE_KEY_FILE=~/.age/key.txt
sops --decrypt .env.enc.env > .env
```

La clave privada (`~/.age/key.txt`) se guarda como **GitHub Secret** (`SOPS_AGE_KEY`) y en el servidor en `~/.age/key.txt`. El repositorio solo tiene la clave pública (en `.sops.yaml`) y el `.env.enc.env` cifrado.

### 6. Backup de SQLite sin corrupción

`cp data/learning.db data/learning.db.bak` captura el archivo en mitad de una escritura si la base está activa. El resultado es un backup corrupto. SQLite provee la API `.backup` para hacer hot-backup atómico:

```bash
#!/usr/bin/env bash
# infra/scripts/backup.sh
set -euo pipefail

DB_PATH=/srv/data/learning.db
BACKUP_DIR=/srv/backups
TIMESTAMP=$(date -u +%Y%m%dT%H%M%SZ)
DEST="${BACKUP_DIR}/learning_${TIMESTAMP}.db"

mkdir -p "$BACKUP_DIR"

# hot-backup sin riesgo de mid-write
sqlite3 "$DB_PATH" ".backup ${DEST}"

# comprime y limpia backups > 7 días
gzip "$DEST"
find "$BACKUP_DIR" -name "*.db.gz" -mtime +7 -delete

echo "Backup ok: ${DEST}.gz"
```

Para Postgres (si migras en C03): usa `pg_dump` con `--format=custom` o `pgbackrest` para backups incrementales hacia un Hetzner Storage Box (SFTP, ~3,81 EUR/mes para 100 GB).

Cron en el servidor:

```bash
# crontab -e
0 2 * * * /bin/bash /srv/bato-learning/infra/scripts/backup.sh >> /var/log/bato-backup.log 2>&1
```

### 7. CI/CD con GitHub Actions por SSH

El pipeline más simple: push a `main` → build de imagen → copia al servidor → `docker compose pull && up`.

```yaml
# .github/workflows/deploy.yml
name: deploy
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Decrypt secrets
        env:
          SOPS_AGE_KEY: ${{ secrets.SOPS_AGE_KEY }}
        run: |
          echo "$SOPS_AGE_KEY" > /tmp/age.key
          SOPS_AGE_KEY_FILE=/tmp/age.key sops --decrypt .env.enc.env > .env

      - name: Build & push image
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
        # necesitas GITHUB_TOKEN con permisos write:packages

      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1.2.0
        with:
          host: ${{ secrets.HETZNER_IP }}
          username: root
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /srv/bato-learning
            git pull origin main
            SOPS_AGE_KEY_FILE=~/.age/key.txt sops --decrypt .env.enc.env > .env
            docker compose -f compose.yaml -f infra/hetzner/compose.prod.yml pull
            docker compose -f compose.yaml -f infra/hetzner/compose.prod.yml up -d --remove-orphans
```

### 8. Healthcheck en compose y smoke test post-deploy

```yaml
# en compose.yaml
services:
  app:
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 10s
```

El smoke test va al final del step de deploy:

```bash
# en el script SSH
curl -fsS https://learning.tudominio.com/health | grep -q '"status":"ok"' \
  || { echo "SMOKE TEST FAILED"; exit 1; }
```

Si falla, el deploy se marca rojo en GitHub Actions y el contenedor anterior sigue corriendo (porque `up -d` no para el anterior si el nuevo no pasa el healthcheck).

## Ejemplo conducido

Secuencia completa desde cero hasta app corriendo con TLS:

```bash
# 1. servidor
hcloud server create --name bato-prod --type cx22 \
  --image ubuntu-24.04 --ssh-key mi-key --location nbg1

# 2. DNS: crea A record learning.tudominio.com → IP (en tu panel DNS)
# espera propagación: dig +short learning.tudominio.com

# 3. Docker en el servidor
ssh root@<IP> "curl -fsSL https://get.docker.com | sh"

# 4. clona el repo en el servidor
ssh root@<IP> "git clone https://github.com/tu-org/bato-learning /srv/bato-learning"

# 5. descifra secrets en el servidor
ssh root@<IP> "
  echo '$SOPS_AGE_KEY_PRIVADA' > ~/.age/key.txt
  cd /srv/bato-learning
  SOPS_AGE_KEY_FILE=~/.age/key.txt sops --decrypt .env.enc.env > .env
"

# 6. levanta
ssh root@<IP> "
  cd /srv/bato-learning
  docker compose -f compose.yaml -f infra/hetzner/compose.prod.yml up -d
"

# 7. smoke test (espera ~30s para que Traefik obtenga el cert)
curl -I https://learning.tudominio.com/health
```

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Crear servidor con `hcloud server create` | determinístico | Mismo comando → mismo recurso. Reproducible. |
| Elegir el tipo de servidor (CX22 vs CX32) | agéntico | Depende de load proyectado, presupuesto, roadmap del cliente. No hay regla cerrada. |
| Generar el manifest `compose.prod.yml` inicial | asistido por agente | El agente propone; un humano revisa antes de `git add`. |
| Aplicar `docker compose up` en producción | **solo humano o CI con revisión** | El agente no ejecuta esto directamente. Consecuencias irreversibles sin backup previo. |
| Calcular TTL del backup (7 días) | determinístico | Regla fija; configurable por variable de entorno, no por LLM. |
| Decidir si un error en el smoke test requiere rollback o hotfix | agéntico | Requiere leer el error, evaluar impacto de usuarios activos, tiempo al siguiente release. |

## Errores típicos

**1. `acme.json` con permisos incorrectos.**
*Síntoma*: Traefik arranca pero no obtiene certificado; en los logs: `file has wrong permissions`.
*Causa*: el archivo fue copiado o recreado con permisos `644`.
*Arreglo*: `chmod 600 /var/lib/docker/volumes/traefik_letsencrypt/_data/acme.json` y reinicia Traefik.

**2. A record no propagado al hacer TLS challenge.**
*Síntoma*: Let's Encrypt retorna `urn:ietf:params:acme:error:dns` o `connection refused`.
*Causa*: el DNS aún no resuelve a la IP correcta cuando Traefik intenta el challenge.
*Arreglo*: `dig +short learning.tudominio.com` desde el servidor. Si no coincide con la IP, espera propagación (TTL). No arrancas Traefik hasta que resuelva.

**3. Backup con `cp` captura mid-write.**
*Síntoma*: el backup restaura en un estado inconsistente; SQLite reporta `SQLITE_CORRUPT`.
*Causa*: `cp` no es atómico respecto a la base de datos activa.
*Arreglo*: usa `sqlite3 "$DB_PATH" ".backup $DEST"` siempre. O litestream si quieres WAL replication continua.

**4. Secret en `docker compose logs`.**
*Síntoma*: la clave `ANTHROPIC_API_KEY` aparece en el log de arranque.
*Causa*: algún `print(os.environ)` o `logging.debug(settings.__dict__)` en el código.
*Arreglo*: nunca loguear el dict completo de settings. En structlog, usa `log.bind(api_key_set=bool(settings.anthropic_api_key))` en lugar del valor.

**5. `docker compose up` en CI sin pull previo.**
*Síntoma*: el servidor usa la imagen anterior aunque el build de CI terminó con éxito.
*Causa*: `up -d` reutiliza la imagen local si el tag no cambió.
*Arreglo*: `docker compose pull` antes de `up`, o usa tags con el SHA del commit (`ghcr.io/org/repo:$SHA`).

## Para profundizar

> [!profundizar]
> El archivo [`infra/hetzner/compose.prod.yml`](../../infra/hetzner/compose.prod.yml) es el override de producción. Compáralo con [`compose.yaml`](../../compose.yaml): el primero añade Traefik y las labels; el segundo define la app base. La forma en que `docker compose -f A -f B` fusiona los dos es lo que D02 enseña. Aquí lo aplicas.

> [!profundizar]
> El archivo [`infra/traefik/traefik.yml`](../../infra/traefik/traefik.yml) configura `tlsChallenge: {}`. Nota que el bloque está vacío — en Traefik v3 basta declararlo; no necesitas `httpChallenge` ni DNS challenge para el caso de un solo servidor con IP pública.

## Hilo conductor

En C03 construiste una app multitenant con SQLAlchemy y Alembic. La app corre en tu máquina con `uv run uvicorn`. El problema: «funciona en local» no es una promesa que puedas hacerle a un cliente. D01 cierra esa brecha. Aprenderás a empaquetar la app en una **imagen Docker** de forma que cualquier máquina con Docker — incluyendo un Hetzner CX22 de 4 €/mes — ejecute exactamente lo mismo que tú. D02 extiende esto a múltiples servicios coordinados; D03 lo despliega en producción.

## Idea central

Una **imagen** es un sistema de archivos en capas, inmutable, que describe el entorno completo de una aplicación: OS base, dependencias, código, configuración. Un **contenedor** es la ejecución de esa imagen: un proceso aislado con su propio sistema de archivos, red y límites de recursos. La imagen es la receta; el contenedor es el plato. Puedes ejecutar cien contenedores desde la misma imagen sin duplicar el espacio en disco — las capas se comparten por hash.

## Por qué importa

Sin Docker, el despliegue de un agente para ACME Manufacturing en el servidor del cliente requiere documentar cada dependencia, versión de Python, variable de entorno y ruta de archivo. Inevitablemente falta algo, y el agente falla silenciosamente o no arranca.

El caso **F-OPS-03 (optimización de inventario)** ilustra la presión: el agente de forecast corre como un cron job en el servidor del cliente. Con Docker, el cron dispara `docker run acme/forecast-worker:2026-04` y eso es todo. La imagen contiene el entorno exacto que probaste. El cliente no necesita instalar Python, `uv`, ni gestionar versiones.

> [!cuidado]
> Docker no elimina el problema de los datos ni de los secretos. Una imagen correcta no contiene credenciales hardcodeadas. Las variables de entorno y los volúmenes son los mecanismos correctos — los verás en D02.

## Cómo funciona por dentro

### Imagen como DAG de capas

Cada instrucción de un `Dockerfile` produce una **capa** (layer). Una capa es un diff del sistema de archivos respecto a la capa anterior. Docker les asigna un hash SHA-256 y las guarda en su cache local.

```
Capa 0 (FROM):  python:3.12-slim         → hash abc123
Capa 1 (RUN):   pip install uv            → hash def456  (sobre abc123)
Capa 2 (COPY):  pyproject.toml            → hash ghi789  (sobre def456)
Capa 3 (RUN):   uv sync --no-dev          → hash jkl012  (sobre ghi789)
Capa 4 (COPY):  app/ alembic/ scripts/    → hash mno345  (sobre jkl012)
```

Cuando haces `docker build` por segunda vez, Docker verifica el hash de cada capa. Si no cambió la instrucción ni sus inputs, reutiliza la capa en cache sin reconstruirla. Esto convierte un build de 3 minutos en un build de 4 segundos en la mayoría de iteraciones de desarrollo.

### El Dockerfile de bato-learning

```dockerfile
FROM python:3.12-slim AS base
ENV PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1 \
    PIP_DISABLE_PIP_VERSION_CHECK=1

RUN pip install --no-cache-dir uv      # instala uv en la imagen base

WORKDIR /srv
COPY pyproject.toml .python-version* ./   # solo el manifest primero
RUN uv sync --no-dev                       # instala dependencias (cacheado)

COPY alembic.ini ./                        # luego el resto del código
COPY alembic ./alembic
COPY app ./app
COPY scripts ./scripts

EXPOSE 8080
CMD ["uv", "run", "uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8080"]
```

El orden importa. `pyproject.toml` cambia pocas veces; `app/` cambia en cada iteración. Si pusieras `COPY app ./app` antes de `RUN uv sync --no-dev`, cada cambio de código haría que Docker reinstalara todas las dependencias. Con el orden correcto, `uv sync` solo se repite cuando cambia `pyproject.toml`.

### Cache de capas — el antipatrón más común

```dockerfile
# MAL: las dependencias se reinstalan en cada cambio de código
COPY . .
RUN uv sync --no-dev
```

```dockerfile
# BIEN: dependencias se cachean; solo el código cambia
COPY pyproject.toml .python-version* ./
RUN uv sync --no-dev
COPY app ./app
```

La regla: copia primero lo que cambia menos. Dependencias antes que código de aplicación.

### Multi-stage builds

El `Dockerfile` de bato-learning es de un solo stage — suficiente para Python con `uv` porque `uv sync --no-dev` ya excluye herramientas de desarrollo. Pero el patrón **multi-stage** es relevante cuando necesitas compilar algo (extensiones C, assets, binarios) y no quieres llevar el compilador a producción.

Ejemplo con la `api/` del proyecto (hipotético para ilustrar el patrón):

```dockerfile
# Stage 1: builder — tiene herramientas de compilación
FROM python:3.12-slim AS builder
RUN pip install uv
WORKDIR /build
COPY pyproject.toml .python-version* ./
RUN uv sync --frozen          # instala todo, incluyendo dev

# Stage 2: runtime — solo lo necesario
FROM python:3.12-slim AS runtime
ENV PYTHONUNBUFFERED=1 PYTHONDONTWRITEBYTECODE=1
WORKDIR /srv
# copia solo las dependencias instaladas del builder
COPY --from=builder /build/.venv /srv/.venv
COPY app ./app
ENV PATH="/srv/.venv/bin:$PATH"
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

El stage `runtime` no tiene `pip`, `uv`, ni código de tests. La imagen final es la del último `FROM` — el `builder` desaparece. Resultado típico: imagen que pasa de 800 MB a 120 MB.

### `.dockerignore`

El build context es el directorio que Docker envía al daemon. Sin `.dockerignore`, incluye `.venv/` (cientos de MB), `data/` (la base de datos SQLite del dev), `.git/` y archivos temporales.

```
# .dockerignore mínimo para bato-learning
.venv/
data/
.git/
__pycache__/
*.pyc
.env
*.db
```

Sin esto, el primer `COPY . .` tarda segundos innecesarios y puede filtrar secretos del `.env` a la imagen.

### `EXPOSE`, `ENTRYPOINT` y `CMD`

- **`EXPOSE 8080`** es documentación, no configuración de red. No abre ningún puerto en el host. El puerto se abre con `-p 8080:8080` al correr el contenedor.
- **`CMD`** define el comando por defecto. Se puede sobreescribir en `docker run` o en `compose.yaml` con el campo `command`.
- **`ENTRYPOINT`** fija el ejecutable; `CMD` son los argumentos. Si defines `ENTRYPOINT ["uvicorn"]` y `CMD ["app.main:app"]`, puedes sobreescribir solo los argumentos con `docker run imagen app.other:app`.
- En bato-learning se usa solo `CMD` — suficiente cuando `compose.yaml` necesita sobreescribir el comando entero para incluir migraciones y seed.

### Tamaños de imagen base

| Base | Tamaño comprimido | Cuándo usarla |
|------|-------------------|---------------|
| `python:3.12` | ~350 MB | Nunca en producción; incluye compiladores, headers, docs |
| `python:3.12-slim` | ~50 MB | Default para Python. Sin compiladores, pero con `gcc` disponible vía apt |
| `python:3.12-alpine` | ~20 MB | Solo si no tienes dependencias con extensiones C. `musl libc` puede romper wheels |

Para `bato-learning` y la `api/`: `python:3.12-slim` es la elección correcta. Alpine rompe algunos wheels de SQLAlchemy/asyncpg.

## Ejemplo conducido

Construye y ejecuta la imagen de bato-learning en modo standalone (sin Compose):

```bash
# 1. construir la imagen
docker build -t bato-learning:local .

# 2. ver las capas y sus tamaños
docker history bato-learning:local

# 3. correr el contenedor (sin volumen, DB efímera)
docker run --rm \
  -e ANTHROPIC_API_KEY="${ANTHROPIC_API_KEY}" \
  -e DATABASE_URL="sqlite+aiosqlite:////tmp/test.db" \
  -p 8080:8080 \
  bato-learning:local

# 4. en otra terminal, verificar que la app responde
curl -s http://localhost:8080/api/modules | python3 -m json.tool | head -20
```

El flag `--rm` elimina el contenedor al parar — útil para pruebas rápidas. Sin él, el contenedor persiste con `docker ps -a` y ocupa espacio.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Construir una imagen desde un Dockerfile | determinístico | Mismo Dockerfile + mismo build context → mismo hash de imagen (con `--no-cache`) |
| Decidir el orden óptimo de capas de un Dockerfile | determinístico | Regla clara: lo que cambia menos va primero |
| Detectar secretos en una imagen ya construida | determinístico | `docker history` + regex; herramientas como `trivy` lo automatizan |
| Elegir la imagen base correcta para un nuevo servicio | agéntico | Depende de las dependencias del proyecto, la arquitectura objetivo, el tamaño límite aceptable — no hay regla cerrada sin contexto |
| Generar un Dockerfile inicial para un proyecto desconocido | agéntico | El LLM puede proponer un draft; tú lo revisas, pruebas y pineas versiones antes del commit |

La infra siempre es determinística en ejecución. La fase agéntica es la **generación del draft** — con guardrails: el Dockerfile que te propone un agente nunca va a producción sin revisión humana y tests de build.

## Errores típicos

**1. `COPY . .` antes de instalar dependencias.**
*Síntoma*: cada cambio de código tarda 2–3 minutos porque reinstala todas las dependencias.
*Causa*: Docker invalida el cache de `RUN uv sync` cuando cambia cualquier archivo del contexto, incluyendo los de `app/`.
*Arreglo*: copiar solo `pyproject.toml` y archivos de lock antes del `RUN uv sync`. Ver el Dockerfile de bato-learning como referencia.

**2. Imagen sin `.dockerignore` incluye `.venv/` o `.env`.**
*Síntoma*: el build context tarda 30 segundos en enviarse; la imagen puede contener tokens de API.
*Causa*: sin `.dockerignore`, Docker envía todo el directorio al daemon.
*Arreglo*: añadir `.dockerignore` en la raíz del proyecto antes del primer `docker build`. Verificar con `docker image inspect` que no hay secretos.

**3. `EXPOSE` confundido con `-p`.**
*Síntoma*: el contenedor arranca pero `curl localhost:8080` falla.
*Causa*: `EXPOSE` no abre el puerto en el host. Es documentación.
*Arreglo*: `docker run -p 8080:8080 imagen` o el campo `ports` en `compose.yaml`.

**4. Imagen con versión `latest` en producción.**
*Síntoma*: el deploy del lunes funciona; el deploy del miércoles cambia comportamiento sin tocar código.
*Causa*: `FROM python:3.12-slim` sin fecha puede actualizarse entre builds. `FROM python:3.12.9-slim` es reproducible.
*Arreglo*: pinear la versión exacta del OS base. En el `Dockerfile` de bato-learning: `FROM python:3.12-slim` es aceptable en dev; en prod conviene `python:3.12.9-slim`.

**5. Instalar `uv` sin fijar versión.**
*Síntoma*: el build local usa `uv 0.5.x` y el CI usa `uv 0.4.x`; el comportamiento de resolución de dependencias difiere.
*Causa*: `pip install uv` toma `latest`.
*Arreglo*: `pip install uv==0.5.x` o usar el instalador oficial con versión: `curl -LsSf https://astral.sh/uv/0.5.x/install.sh | sh`.

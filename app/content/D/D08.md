## Hilo conductor

En D05 usaste `${{ secrets.SSH_PRIVATE_KEY }}` en GitHub Actions. Esos secretos viven en la UI de GitHub, fuera del repo. Pero ¿qué hay del `.env` que vive en el servidor de Hetzner? ¿Cómo llegó ahí? ¿Quién puede verlo? ¿Qué pasa si un ingeniero nuevo necesita acceso? D08 responde esas preguntas con cuatro soluciones ordenadas por complejidad creciente: variable de entorno en el servidor, `sops+age` para cifrar en el repo, HashiCorp Vault para equipos con plataforma, y KMS cloud cuando ya estás en AWS/GCP/Azure. Track E asume que los secretos de los agentes están gestionados correctamente — sin esta base, la seguridad del agente es ilusoria.

## Idea central

Un **secreto** es cualquier valor que da acceso a un recurso: API keys, passwords de DB, tokens SSH, certificados. La propiedad que define su gestión correcta es la **segmentación de conocimiento**: el secreto no debe estar en el código, no debe estar en los logs, no debe estar en las imágenes Docker, y no debe estar en el historial de Git — ni siquiera en el historial antiguo. Si estuvo una vez en el repo, se considera comprometido y hay que rotarlo.

## Por qué importa

El anti-patrón más común en PYME: el repo tiene un `.env` con `ANTHROPIC_API_KEY=sk-ant-...` commiteado porque «el repo es privado». Un repo privado no es una garantía de seguridad: un colaborador que se va lleva acceso al historial; un token de GitHub comprometido expone todos los repos; un fork accidental lo hace público.

El caso **F-FIN-01** (conciliación bancaria de Initech) trabaja con credenciales de acceso a cuentas bancarias vía Belvo. Si esas credenciales se filtran, Initech pierde acceso a sus cuentas y la responsabilidad legal recae sobre quien gestionó mal los secretos.

> [!cuidado]
> Si alguna vez commiteaste un secreto a Git, **no es suficiente borrarlo en un commit posterior**. El historial de Git conserva el commit con el secreto. Debes revocar y rotar el secreto inmediatamente, y si el repo es público (o pudo ser visto), tratar el secreto como comprometido desde ese momento.

## Cómo funciona por dentro

### Nivel 0 — Variables de entorno en el servidor (el mínimo viable)

El enfoque más simple: los secretos viven en un `.env` en el servidor, fuera del repo. El `compose.yaml` los lee con `env_file: .env` o con `${VAR}` interpolado.

```bash
# en el servidor Hetzner, fuera del repo:
cat /srv/bato-learning/.env
# ANTHROPIC_API_KEY=sk-ant-api03-...
# DB_PASSWORD=very_long_random_string_here
# TEMPORAL_NAMESPACE=production
```

```yaml
# compose.prod.yaml (en el repo, sin secretos)
services:
  api:
    env_file: .env    # lee el .env que está en el servidor
    # o equivalente:
    environment:
      ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}
      DB_PASSWORD: ${DB_PASSWORD}
```

**Cómo llega el `.env` al servidor:** la primera vez, manualmente por SSH:

```bash
scp .env deploy@65.21.x.x:/srv/bato-learning/.env
```

O generado en el servidor con un script que pide los valores interactivamente. **Nunca** como parte del pipeline de CI/CD que lo commitea al repo.

**Pros:** cero dependencias externas, cero costo, funciona desde el día 1.
**Contras:** sin auditoría (¿quién cambió el `.env` cuándo?), sin rotación automática, acceso compartido entre todos los que tienen SSH al servidor, no hay manera de saber qué versión del secreto está activa.

**Cuándo es suficiente:** PYME con 1-2 ingenieros en el equipo, sin requisitos de auditoría, servidor gestionado por la misma persona que tiene la key original.

### Nivel 1 — sops + age (secretos cifrados en el repo)

**sops** (Secrets OPerationS) es una herramienta que cifra archivos YAML/JSON/ENV manteniendo la estructura legible. **age** es un algoritmo de cifrado moderno (X25519 / ChaCha20-Poly1305) que genera claves cortas y simples, sin la complejidad de GPG.

La idea: el `.env` **cifrado** vive en el repo. Cualquiera puede ver que existe la variable `ANTHROPIC_API_KEY`, pero no su valor. Solo quien tiene la clave privada de age puede descifrarlo.

**Setup:**

```bash
# 1. instalar sops y age
brew install sops age   # macOS
# o: apt install age && curl -L https://github.com/getsops/sops/releases/download/v3.9.0/sops-v3.9.0.linux.amd64 -o sops

# 2. generar par de claves age (una por desarrollador o por entorno)
age-keygen -o ~/.config/sops/age/keys.txt
# genera algo como:
# Public key: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p

# 3. crear .sops.yaml en la raíz del repo
cat > .sops.yaml << 'EOF'
creation_rules:
  - path_regex: .env.prod$
    age: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
EOF

# 4. cifrar el .env de producción
sops --encrypt .env.prod > .env.prod.enc
git add .env.prod.enc .sops.yaml
git commit -m "chore: add encrypted prod secrets"
# NUNCA: git add .env.prod
```

**Descifrado en el servidor (en el pipeline):**

```yaml
# .github/workflows/deploy.yml
- name: Decrypt secrets
  run: |
    echo "${{ secrets.AGE_PRIVATE_KEY }}" > /tmp/age_key.txt
    SOPS_AGE_KEY_FILE=/tmp/age_key.txt sops --decrypt .env.prod.enc > /tmp/.env
  env:
    AGE_PRIVATE_KEY: ${{ secrets.AGE_PRIVATE_KEY }}

- name: Copy decrypted .env to server
  uses: appleboy/ssh-action@v1
  with:
    host: ${{ secrets.HETZNER_HOST }}
    username: deploy
    key: ${{ secrets.SSH_PRIVATE_KEY }}
    script: |
      # el .env descifrado se copia al servidor antes del docker compose up
      # (en la práctica: usar scp desde el runner al servidor)
```

**Pros:** auditoría via Git (quién cambió qué secreto y cuándo), revisión de secretos en PR (se ve el diff cifrado), cero costo extra, funciona sin infraestructura adicional.
**Contras:** cada desarrollador necesita la clave privada de age. Si la clave privada se filtra, todos los secretos cifrados con esa clave quedan expuestos. Requiere disciplina de equipo.

**Cuándo es el default correcto:** PYME con 2-5 ingenieros, repo compartido, sin equipo de plataforma dedicado. Es la solución que más cubre el espectro de riesgo/complejidad para una PYME LATAM.

### Nivel 2 — HashiCorp Vault (rotación, audit log, dynamic secrets)

**Vault** es un servidor que gestiona secretos con: autenticación por roles/tokens, audit log completo de cada acceso, rotación automática, y **dynamic secrets** (genera credenciales temporales de DB o cloud que expiran).

```python
# app que obtiene la API key de Vault en tiempo de ejecución
import hvac

def get_anthropic_key(tenant: str) -> str:
    client = hvac.Client(url="https://vault.internal:8200")
    client.auth.approle.login(
        role_id=VAULT_ROLE_ID,
        secret_id=VAULT_SECRET_ID,
    )
    secret = client.secrets.kv.v2.read_secret_version(
        path=f"anthropic/{tenant}",
        mount_point="secret",
    )
    return secret["data"]["data"]["api_key"]
```

**Dynamic secrets para Postgres:** en lugar de tener una password fija en el `.env`, Vault genera una password temporal que dura 1 hora:

```bash
# Vault genera credenciales temporales de Postgres
vault read database/creds/bato-readonly
# Key             Value
# username        v-approle-bato-abc123
# password        A1b-randomly-generated
# lease_duration  1h
```

La app que terminó de procesar no puede volver a usar esas credenciales. Si alguien roba las credenciales de un contenedor comprometido, expiran en 1 hora.

**Pros:** audit log completo, rotación automática, secrets por tenant posible, dynamic secrets eliminan el problema de passwords estáticas.
**Contras:** requiere un equipo de plataforma para operar Vault. Vault es otro servicio que puede fallar — si Vault está caído, tu app no puede arrancar. Requiere gestionar la alta disponibilidad de Vault.

**Cuándo tiene sentido:** equipo de 5+ ingenieros con al menos 1 dedicado a plataforma/infra, requisitos de compliance (SOC2, ISO 27001), múltiples entornos (dev/staging/prod) con secretos distintos, equipos de múltiples clientes con aislamiento de secretos por cliente.

### Nivel 3 — KMS cloud (AWS / GCP / Azure)

Si tu stack ya corre en una cloud pública, usa su servicio nativo de gestión de secretos:

| Cloud | Servicio | Precio orientativo |
|-------|---------|-------------------|
| AWS | AWS Secrets Manager | $0.40/secreto/mes + $0.05/10k llamadas |
| GCP | Secret Manager | $0.06/versión activa/mes + $0.03/10k accesos |
| Azure | Key Vault | $0.03/10k operaciones |

La integración es nativa: las políticas de IAM controlan quién puede leer qué secreto. No necesitas gestionar infraestructura de Vault.

```python
# ejemplo con AWS Secrets Manager
import boto3, json

def get_secret(secret_name: str) -> dict:
    client = boto3.client("secretsmanager", region_name="us-east-1")
    response = client.get_secret_value(SecretId=secret_name)
    return json.loads(response["SecretString"])

api_key = get_secret("prod/anthropic/api_key")["value"]
```

**Para PYME LATAM en Hetzner:** los servicios KMS de AWS/GCP son innecesarios si no estás en esas clouds. Añaden latencia de red (tu servidor en Hetzner llamando a AWS), costos de egress y una dependencia en otra plataforma. La regla: usa el KMS de la cloud donde ya estás. Si estás en Hetzner, no uses AWS KMS.

### Rotación de API keys sin downtime

El caso concreto: rotar `ANTHROPIC_API_KEY` en producción sin interrumpir el servicio.

El problema: si cambias la key en el `.env` y reinicias el contenedor, hay 2-5 segundos donde las requests en vuelo pueden fallar.

**Patrón de rotación con doble key:**

1. Crea una nueva API key en la consola de Anthropic (manteniendo la vieja activa).
2. Actualiza el `.env` del servidor con ambas keys: `ANTHROPIC_API_KEY_NEW` y `ANTHROPIC_API_KEY_OLD`.
3. Actualiza el código para intentar primero con `_NEW`, fallback a `_OLD`:

```python
import anthropic

def get_client() -> anthropic.AsyncAnthropic:
    # intenta con la key nueva primero
    key = os.environ.get("ANTHROPIC_API_KEY_NEW") or os.environ.get("ANTHROPIC_API_KEY")
    return anthropic.AsyncAnthropic(api_key=key)
```

4. Despliega el código con doble key. Monitorea en Phoenix que los requests funcionen.
5. Después de 24 horas sin errores, revoca la key vieja en Anthropic.
6. Limpia el `.env`: elimina `ANTHROPIC_API_KEY_OLD`, renombra `_NEW` a `ANTHROPIC_API_KEY`.

**Rotación para DB password (la más delicada):**

Postgres acepta múltiples passwords para un mismo usuario en RDS/Neon managed. En self-hosted, la rotación requiere:

1. Crear un nuevo usuario de DB con los mismos permisos.
2. Actualizar `DATABASE_URL` en el `.env` al nuevo usuario.
3. Reiniciar la app (brief downtime de 2-5 seg) o usar blue-green para cero downtime.
4. Revocar el usuario viejo después de verificar que la conexión funciona con el nuevo.

## Ejemplo conducido

Cifrar el `.env` de producción de bato-learning con sops+age:

```bash
# 1. instalar herramientas
curl -LO https://github.com/getsops/sops/releases/download/v3.9.0/sops-v3.9.0.linux.amd64
chmod +x sops-v3.9.0.linux.amd64 && sudo mv sops-v3.9.0.linux.amd64 /usr/local/bin/sops

age-keygen -o ~/.config/sops/age/keys.txt
# Public key: age1abc...xyz (copiar este valor)

# 2. configurar sops para usar age
cat > .sops.yaml << 'EOF'
creation_rules:
  - path_regex: \.env\.prod$
    age: age1abc...xyz
EOF

# 3. crear el .env.prod con los secretos reales (solo en local, nunca en git)
cat > .env.prod << 'EOF'
ANTHROPIC_API_KEY=sk-ant-api03-real-key-here
DB_PASSWORD=random_strong_password_here
TEMPORAL_NAMESPACE=production
EOF

# 4. cifrar y commitear solo el archivo cifrado
sops --encrypt .env.prod > .env.prod.enc
git add .env.prod.enc .sops.yaml
git commit -m "chore: add encrypted production secrets"

# el .env.prod sin cifrar NUNCA va al repo
echo ".env.prod" >> .gitignore

# 5. descifrar cuando se necesita (en el servidor o en CI)
SOPS_AGE_KEY_FILE=~/.config/sops/age/keys.txt sops --decrypt .env.prod.enc > .env.prod
```

El archivo `.env.prod.enc` que va al repo se ve así (extracto):

```yaml
ANTHROPIC_API_KEY=ENC[AES256_GCM,data:abc123encrypted==,tag:xyz==,type:str]
DB_PASSWORD=ENC[AES256_GCM,data:def456encrypted==,tag:abc==,type:str]
```

El nombre de la variable es visible (sin cifrar); el valor está cifrado. En un PR, puedes ver que se añadió una nueva variable sin revelar su valor.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Cifrar un `.env` con `sops --encrypt` | determinístico | Misma clave + mismo input → mismo ciphertext (con nonce determinista). |
| Descifrar en CI y copiar al servidor | determinístico | Comando con parámetros fijos; mismo resultado siempre. |
| Decidir qué secretos necesita cada tenant | agéntico | Depende del contrato, los servicios activados y la configuración de onboarding — sin regla cerrada. |
| Decidir cuándo rotar una API key | agéntico | Requiere evaluar señales: ¿hay alertas de uso anómalo? ¿el proveedor notificó compromiso? ¿cumplimos la política de 90 días? |
| Detectar un secreto hardcodeado en el código con grep/trufflehog | determinístico | Patrón regex fijo: si coincide con `sk-ant-api03-`, es una key de Anthropic en el código. |

La gestión de secretos es mayoritariamente determinística en ejecución. La parte agéntica es la **decisión de política**: qué secretos existen, cuándo rotarlos, quién tiene acceso.

## Errores típicos

**1. `.env` commiteado en Git (el error más común).**
*Síntoma*: `git log --all --oneline | grep -i env` muestra un commit con secretos.
*Causa*: el desarrollador no tenía `.env` en `.gitignore` o hizo `git add .` sin revisar.
*Arreglo*: revocar **inmediatamente** todos los secretos del archivo. No es suficiente borrarlo en un nuevo commit — el historial lo conserva. Usar `git filter-repo` para purgar el historial si el repo es privado y se puede notificar a todos los colaboradores.

**2. Secreto visible en los logs de CI porque se usa `echo ${{ secrets.X }}`.**
*Síntoma*: el log de GitHub Actions muestra `***` pero hay una herramienta en el pipeline que loguea todas las variables de entorno.
*Causa*: algunas herramientas (Docker build, npm scripts) loguean el entorno completo en modo verbose.
*Arreglo*: nunca pasar secretos como argumentos posicionales; pasarlos como variables de entorno del step. Activar `--no-verbose` o equivalente en herramientas que loguean entorno.

**3. Misma API key para todos los tenants.**
*Síntoma*: si la key de Anthropic se filtra, todos los tenants quedan expuestos. No hay forma de revocar el acceso de un tenant sin afectar a todos.
*Causa*: simplicidad inicial que no escala.
*Arreglo*: usar una key por tenant (o por entorno) con gestión independiente. sops puede cifrar un archivo por tenant: `.env.acme.prod`, `.env.globex.prod`.

**4. Vault no disponible → app no arranca.**
*Síntoma*: el servidor de Vault está caído por mantenimiento y la app no puede obtener sus secretos para arrancar.
*Causa*: la app busca secretos en Vault en tiempo de startup sin fallback.
*Arreglo*: cachear los secretos en la app al arrancar (con TTL). Si Vault no responde en el startup, intentar con el `.env` local como fallback. Vault es la fuente de verdad, pero no el único path de arranque.

**5. sops con una sola clave age → bus factor 1.**
*Síntoma*: el único ingeniero con la clave age deja el equipo. Nadie puede descifrar los secretos.
*Causa*: `.sops.yaml` configurado con una sola clave.
*Arreglo*: configurar múltiples destinatarios en `.sops.yaml`. sops cifra el mismo archivo para todas las claves listadas:
```yaml
creation_rules:
  - path_regex: \.env\.prod$
    age: >-
      age1abc...xyz,
      age1def...uvw
```
Mínimo 2 claves age (2 personas del equipo) más 1 clave de emergencia guardada en un gestor de passwords compartido (Bitwarden, 1Password Teams).

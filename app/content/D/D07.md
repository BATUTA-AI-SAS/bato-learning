## Hilo conductor

En D05 automatizaste el deploy con GitHub Actions. El pipeline hace `docker compose up -d` en el servidor y la nueva versión reemplaza la vieja. Para una PYME con pocos tenants, eso es suficiente. Pero ¿qué pasa cuando tienes 50 tenants activos y el deploy del viernes rompe un endpoint crítico? ¿Cómo vuelves atrás sin 30 minutos de downtime? D07 enseña las cuatro estrategias que existen, cuándo cada una es la correcta para una PYME LATAM, y el patrón expand-contract para que las migraciones de base de datos no sean el punto de fallo del deploy.

## Idea central

Una **estrategia de despliegue** controla cómo la nueva versión del software reemplaza a la anterior en producción. La variable que controla es el **porcentaje de tráfico** que ve la nueva versión y **durante cuánto tiempo**. En un extremo: big-bang (todo el tráfico cambia instantáneamente). En el otro: canary (1 % del tráfico ve la nueva versión durante horas antes de escalar). Entre medio: rolling y blue-green. Ninguna es universal; cada una tiene un costo en infraestructura, complejidad operativa y tiempo de rollback.

## Por qué importa

El caso **F-OPS-01** (agente de planificación de producción de ACME) tiene un SLA: si el agente está caído más de 15 minutos durante el turno de producción, ACME pierde capacidad de respuesta. Un big-bang deploy que tarda 3 minutos en reiniciar y tiene un bug que requiere rollback es un evento de 6-8 minutos de downtime — dentro del SLA pero peligroso. Un blue-green deploy permite volver en 30 segundos.

> [!nota]
> Las estrategias avanzadas (canary con weights, feature flags dinámicos) tienen un costo de complejidad que no se justifica para todos los equipos. La tabla de decisión al final del módulo es el punto de partida, no el punto de llegada.

## Cómo funciona por dentro

### Estrategia 0 — Big-bang (lo que no hacer)

**Big-bang** significa: para el servicio viejo, levanta el servicio nuevo, done. Es lo que hace `docker compose up -d` por defecto cuando no hay múltiples réplicas.

```bash
# big-bang: hay un momento de 2-5 segundos donde no hay servicio
docker compose stop api
docker compose up -d api
```

**Problemas:**
- Downtime garantizado durante el reinicio (2-10 segundos típicamente).
- Si el nuevo contenedor falla al arrancar, el servicio está caído hasta que alguien lo arregla.
- Rollback manual: vuelves a hacer el proceso con la imagen anterior.

**Cuándo es aceptable:** entornos de desarrollo, apps internas sin SLA, o en las 3 AM cuando el tráfico es cero y tienes acceso directo. No para producción con clientes.

### Estrategia 1 — Rolling deploy

**Rolling** significa: reemplaza las instancias de la versión vieja con la nueva, **una a la vez** (o en batches), sin downtime.

Para que funcione, la app debe ser **stateless**: cualquier instancia puede manejar cualquier request. Si tienes estado en memoria (sesiones, cache local, locks), rolling no funciona sin coordinación extra.

En Docker Compose con 1 servidor (el caso PYME típico), el rolling deploy nativo no existe — Compose levanta un contenedor nuevo y mata el viejo. Para rolling real necesitas al menos 2 réplicas. En Kubernetes o en Fly.io, el rolling es el comportamiento por defecto.

**Alternativa práctica con Docker en un server:** usar `docker compose up -d --no-deps --scale api=2 api` temporalmente, esperar a que el nuevo contenedor esté healthy, luego escalar de vuelta a 1:

```bash
# paso 1: levantar la nueva versión junto a la vieja (2 réplicas)
docker compose up -d --no-deps --scale api=2 api
# paso 2: esperar a que el nuevo contenedor esté healthy
docker compose ps   # verificar status
# paso 3: eliminar la réplica vieja
docker compose up -d --no-deps --scale api=1 api
```

Esto es rolling manual. Tiene downtime cero si el load balancer (Traefik) distribuye el tráfico entre las dos réplicas durante el overlap.

**Rollback:** correr el mismo proceso con la imagen anterior. Tarda lo mismo que el deploy original.

### Estrategia 2 — Blue-green deploy

**Blue-green** mantiene dos entornos idénticos: **blue** (la versión que sirve tráfico hoy) y **green** (la nueva versión). El swap es instantáneo: se cambia el routing del load balancer de blue a green.

```
Antes del deploy:
  Internet → Traefik → api-blue (v1.0)
  api-green (v1.1) — corriendo, pero sin tráfico

Swap (< 1 segundo):
  Internet → Traefik → api-green (v1.1)
  api-blue (v1.0) — corriendo, pero sin tráfico (listo para rollback)

Rollback (< 1 segundo):
  Internet → Traefik → api-blue (v1.0)
```

**Implementación con Docker Compose + Traefik:**

Dos servicios en el compose, solo uno con el label de Traefik activo:

```yaml
services:
  api-blue:
    image: ghcr.io/acme/audit-api:${BLUE_TAG}
    labels:
      - "traefik.enable=${BLUE_ACTIVE:-true}"   # active by default
      - "traefik.http.routers.api.rule=Host(`api.acme.com`)"
      - "traefik.http.services.api.loadbalancer.server.port=8000"

  api-green:
    image: ghcr.io/acme/audit-api:${GREEN_TAG}
    labels:
      - "traefik.enable=${GREEN_ACTIVE:-false}"  # inactive by default
      - "traefik.http.routers.api.rule=Host(`api.acme.com`)"
      - "traefik.http.services.api.loadbalancer.server.port=8000"
```

Para hacer el swap: cambiar `BLUE_ACTIVE=false GREEN_ACTIVE=true` y recargar Traefik. El tiempo de switch es < 1 segundo. Rollback: invertir las variables.

**Costo:** 2x uso de memoria/CPU durante el periodo de overlap (normalmente horas). En un Hetzner CX22 con 2 GB de RAM, dos réplicas de una API FastAPI ligera caben cómodamente.

**Cuándo usarlo:** cuando el rollback rápido (< 1 min) es más valioso que el costo de 2x recursos. Para sistemas con SLA o con flujos de negocio críticos (procesamiento de pagos, agentes que modifican datos).

### Estrategia 3 — Canary deploy

**Canary** envía un pequeño porcentaje del tráfico a la nueva versión y lo incrementa gradualmente si no hay anomalías. El nombre viene de los canarios en las minas de carbón.

```
                 ┌──────── api-stable (v1.0): 95% del tráfico
Internet → LB ──┤
                 └──────── api-canary (v1.1): 5% del tráfico

Después de 1 hora sin errores:
                 ┌──────── api-stable (v1.0): 50%
Internet → LB ──┤
                 └──────── api-canary (v1.1): 50%

Después de 2 horas:
  Internet → api-canary (v1.1): 100% → renombrar a stable
```

**Implementación:** requiere un load balancer con soporte de weights. Traefik soporta weighted services con la `traefik.http.services.*.loadbalancer.weighted.services` en v3+. Cloudflare también lo soporta con Traffic Control. Para PYME en un solo server, es complejo de implementar correctamente.

**Cuándo tiene sentido para PYME:** cuando tienes > 500 tenants activos y quieres que solo el 5 % experimente la nueva versión durante las primeras horas. Para apps con < 100 tenants diarios, canary no aporta — el tamaño de muestra es demasiado pequeño para detectar errores estadísticamente.

**Para K8s:** Argo Rollouts implementa canary de forma nativa. Pero si tu stack no está en K8s, el overhead de Argo Rollouts no se justifica.

### Estrategia 4 — Feature flags

Un **feature flag** es un interruptor en código que activa una funcionalidad para un subconjunto de usuarios sin necesidad de un nuevo deploy. El código llega a producción desactivado; se activa para % de usuarios o para tenants específicos.

```python
# en la lógica de negocio
from app.flags import is_enabled

async def run_audit(tenant: str, period: str) -> AuditReport:
    if is_enabled("new_anomaly_detector", tenant=tenant):
        # nueva lógica en prueba
        return await _new_detector(tenant, period)
    else:
        # lógica estable
        return await _legacy_detector(tenant, period)
```

**Para BATUTA: flag por tenant.** Activas la nueva feature para `acme` primero, verificas con ellos, luego la activas para `globex`, y finalmente para todos. Es el control de riesgo más fino posible.

**Implementaciones:**

| Herramienta | Precio | Cuándo |
|------------|--------|--------|
| Variable en DB / tabla `feature_flags` | $0 | Default para PYME. Suficiente para 10-20 flags. |
| Unleash (self-hosted) | $0 | Cuando necesitas UI de gestión y audit log. |
| LaunchDarkly | Desde $8/usuario/mes | Cuando el equipo de producto gestiona los flags sin ingeniería. |
| PostHog | Free tier; desde $0 | Si ya usas PostHog para analytics. Los flags son un bonus. |

**Implementación mínima en bato-learning** (tabla en SQLite):

```python
# app/repos/flags.py
async def is_feature_enabled(
    session: AsyncSession,
    flag: str,
    tenant: str,
) -> bool:
    result = await session.execute(
        select(FeatureFlag)
        .where(FeatureFlag.name == flag)
        .where(
            (FeatureFlag.tenant_id == tenant) |
            (FeatureFlag.tenant_id == "*")   # wildcard para todos
        )
    )
    flag_row = result.scalar_one_or_none()
    return flag_row.enabled if flag_row else False
```

### El patrón expand-contract para migraciones

El problema más peligroso en un deploy con base de datos: la nueva versión del código asume una columna que la migración aún no creó (o que la migración ya borró). El resultado es un error 500 en producción mientras la migración corre.

**Expand-contract** divide cada cambio de schema en tres fases desplegadas en tres releases separados:

```
Release 1 — EXPAND:
  Migración añade la columna nueva (nullable, sin eliminar la vieja).
  El código viejo la ignora. El código nuevo puede escribirla.

Release 2 — MIGRATE:
  Proceso de backfill: copia datos de la columna vieja a la nueva.
  Ambas versiones del código coexisten. Ningún downtime.

Release 3 — CONTRACT:
  Migración elimina la columna vieja.
  El código ya no la referencia. Safe to drop.
```

**Ejemplo concreto:** cambiar `invoice_date` (string) a `invoice_date_ts` (timestamp).

```sql
-- Release 1 (expand): añadir columna nueva nullable
ALTER TABLE invoices ADD COLUMN invoice_date_ts TIMESTAMP;

-- Release 2 (backfill): llenar la nueva columna desde la vieja
UPDATE invoices SET invoice_date_ts = invoice_date::TIMESTAMP
WHERE invoice_date_ts IS NULL;

-- Release 3 (contract): eliminar la columna vieja
ALTER TABLE invoices DROP COLUMN invoice_date;
```

**La regla operativa:** nunca hacer `DROP COLUMN` o `RENAME COLUMN` en el mismo deploy que la nueva versión del código que lo asume. Siempre un release de por medio.

### Tabla de decisión: qué estrategia para qué escenario

| Escenario | Estrategia recomendada | Por qué |
|-----------|----------------------|---------|
| PYME < 50 tenants, app interna, sin SLA | Big-bang controlado | La simplicidad supera el riesgo. El downtime de 5 seg es aceptable. |
| PYME 50-500 tenants, SLA informal | Blue-green | Rollback en < 1 min sin complejidad de weights. El 2x de recursos es asumible. |
| SaaS B2C con > 500 tenants activos | Canary + feature flags | El tamaño de muestra justifica la estadística. Los flags dan control fino. |
| K8s / microservicios | Rolling nativo | K8s hace rolling por defecto; es el menor costo. |
| Feature experimental para 1 cliente | Feature flag por tenant | Deploy a todos, activa solo para el tenant piloto. Cero infra extra. |
| Migración de schema riesgosa | Expand-contract | Ninguna estrategia de deploy elimina el riesgo de un DROP mal timed. |

## Ejemplo conducido

Implementar un blue-green swap manual en bato-learning con dos variables de entorno:

```bash
# en el servidor, el .env actual tiene:
# BLUE_TAG=abc1234  BLUE_ACTIVE=true
# GREEN_TAG=def5678  GREEN_ACTIVE=false

# paso 1: levantar la nueva versión en green (sin tráfico)
GREEN_TAG=xyz9999 GREEN_ACTIVE=false docker compose up -d api-green

# paso 2: verificar que green arrancó correctamente
docker compose exec api-green curl -s http://localhost:8000/api/health

# paso 3: swap — green recibe tráfico, blue queda en standby
BLUE_ACTIVE=false GREEN_ACTIVE=true docker compose up -d api-blue api-green

# paso 4: si todo va bien después de 30 min, cleanup
docker compose stop api-blue
docker compose rm -f api-blue

# rollback inmediato (si hay problema en paso 3):
BLUE_ACTIVE=true GREEN_ACTIVE=false docker compose up -d api-blue api-green
```

El rollback tarda < 5 segundos. El tiempo entre el paso 3 y el rollback puede ser días — blue queda corriendo en standby, listo.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Ejecutar el swap blue-green cambiando variables de entorno | determinístico | Acción puntual con resultado binario: el tráfico va a blue o a green. Sin contexto externo. |
| Decidir si escalar el canary del 5 % al 50 % | agéntico | Requiere leer las métricas de error rate, latencia, feedback de usuarios piloto. Sin regla cerrada universal. |
| Aplicar una migración expand-contract en el orden correcto | determinístico | Los tres releases tienen un orden fijo: expand → migrate → contract. No hay varianza. |
| Decidir qué estrategia usar para un nuevo cliente | agéntico | Depende del SLA, número de tenants, criticidad, presupuesto y madurez del equipo del cliente. |
| Evaluar si un feature flag debe activarse para todos | agéntico | Requiere contexto: ¿cuánto tráfico vio el flag? ¿Hay bugs reportados? ¿El cliente piloto dio OK? |

La operación de deploy en sí siempre es determinística: ejecutas el plan o no lo ejecutas. La decisión de cuándo y con qué estrategia es donde entra el juicio humano.

## Errores típicos

**1. `DROP COLUMN` en el mismo deploy que el código que ya no la usa.**
*Síntoma*: la nueva versión de la app arranca bien, pero durante el tiempo en que la migración corre, los contenedores viejos que aún sirven tráfico fallan porque intentan leer la columna que ya no existe.
*Causa*: violación del patrón expand-contract. No hay release intermedio.
*Arreglo*: separar siempre en dos releases: el primero elimina la referencia en código; el segundo hace el `DROP COLUMN`.

**2. Rollback de blue-green que olvida revertir la migración.**
*Síntoma*: el rollback a blue funciona, pero blue es incompatible con el schema que dejó la migración de green.
*Causa*: el blue-green gestiona la capa de aplicación, no el schema. Si green aplicó migraciones incompatibles con el código de blue, el rollback falla.
*Arreglo*: diseñar migraciones compatibles hacia atrás (expand primero). Si la migración es incompatible, blue-green no garantiza rollback seguro — necesitas also hacer rollback de la migración (`alembic downgrade`).

**3. Feature flag sin fecha de expiración.**
*Síntoma*: el código base tiene 40 flags, la mitad inactivos, nadie sabe cuáles eliminar.
*Causa*: los flags se crean con vida indefinida y nunca se limpian.
*Arreglo*: cada flag se crea con una fecha de expiración estimada. Un linter o un test falla si hay flags con fecha vencida activos en el código. Política: un flag que ha estado 100 % activo durante 30 días se promueve a código estable y se elimina el flag.

**4. Canary con muestra demasiado pequeña para ser estadísticamente significativa.**
*Síntoma*: el canary al 5 % corre 1 hora sin errores; se hace el rollout al 100 %; hay un bug que afecta a tenants con cierta configuración que el 5 % no incluyó.
*Causa*: canary sin segmentación intencional de los tenants piloto.
*Arreglo*: en PYME, si tienes < 100 tenants activos, usa feature flags por tenant en lugar de canary por porcentaje. Selecciona tenants piloto representativos explícitamente en lugar de confiar en la aleatoriedad estadística.

## Idea central

Un workflow m├¡nimo:

```yaml

  push:
```

### Pipeline can├│nico de bato-learning

Dos workflows separados con responsabilidades distintas:

**`ci.yml`** ÔÇö corre en cada push y en cada PR:
1. Checkout del c├│digo.
2. Setup Python 3.12 con `actions/setup-python@v5`.
3. `uv sync --dev` ÔÇö instala dependencias incluyendo dev.
4. `ruff check app/` ÔÇö lint.
5. `pytest tests/` ÔÇö tests unitarios.
6. Si falla cualquier step ÔåÆ el PR no se puede mergear (con branch protection).

**`deploy.yml`** ÔÇö corre solo en push a `main` **despu├®s** de que `ci` pase:

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  build-and-push:
    runs-on: ubuntu-24.04
    permissions:
      packages: write    # necesario para GHCR
    outputs:
      image_tag: ${{ steps.meta.outputs.tags }}
    steps:
      - uses: actions/checkout@v4

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push
        id: build
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    needs: build-and-push
    runs-on: ubuntu-24.04
    steps:
      - name: SSH deploy
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.HETZNER_HOST }}
          username: deploy
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /srv/bato-learning
            export IMAGE_TAG=ghcr.io/${{ github.repository }}:${{ github.sha }}
            docker pull "$IMAGE_TAG"
            IMAGE_TAG="$IMAGE_TAG" docker compose up -d --no-build
            docker image prune -f
```

Puntos cr├¡ticos de este YAML:

- **`needs: build-and-push`**: el job `deploy` solo corre si `build-and-push` termin├│ con ├®xito.
- **`cache-from/to: type=gha`**: usa el cache de GitHub Actions para las capas de Docker. El segundo build tarda 30 segundos en lugar de 4 minutos.
- **`${{ github.sha }}`**: el tag de la imagen es el SHA del commit. Reproducible, auditable, sin `latest`.
- **`GITHUB_TOKEN`**: token autom├ítico que GitHub inyecta en cada run. No necesitas crear uno manual para GHCR dentro del mismo repo.

## Errores t├¡picos

**1. Usar `@latest` en lugar de versi├│n pinada.**
*S├¡ntoma*: el workflow funcionaba ayer y hoy falla sin cambios de c├│digo.
*Causa*: la action de tercero lanz├│ una versi├│n incompatible y `@latest` apunta a ella.
*Arreglo*: siempre pinear: `actions/checkout@v4`, `docker/build-push-action@v6`. Para m├íxima seguridad, pinear al SHA del commit de la action, no solo al tag.

**2. Secreto expuesto en un `run: echo ${{ secrets.X }}`.**
*S├¡ntoma*: el log muestra `***` pero en un script que hace `echo $MY_VAR` el valor puede aparecer si la shell expande antes de que GitHub lo enmascare.
*Causa*: `${{ secrets.X }}` se eval├║a antes de que el runner interprete el script. Si lo pasas a un subcomando que loguea, puede filtrarse.
*Arreglo*: pasar secretos via variables de entorno del step, no interpolados en el script: `env: MY_VAR: ${{ secrets.X }}` y luego usar `$MY_VAR` en el script.

**3. El deploy corre aunque los tests fallaron.**
*S├¡ntoma*: una versi├│n rota llega a producci├│n.
*Causa*: `deploy.yml` no tiene `needs: ci` o el job de CI est├í en un workflow distinto sin dependencia declarada.
*Arreglo*: usar `needs: [nombre-del-job-de-test]` en el job de deploy. O usar branch protection rules que requieran que el CI pase antes de mergear a `main`.

**4. `docker compose up -d` sin `--no-build` en el servidor.**
*S├¡ntoma*: el servidor construye la imagen localmente en lugar de usar la que subiste al registry. Si el servidor no tiene acceso a internet o es lento, el build falla o tarda 10 minutos.
*Causa*: sin `--no-build`, compose intenta construir si el servicio tiene `build:` declarado.
*Arreglo*: en el `compose.prod.yaml`, no incluir el campo `build`. O usar `--no-build` expl├¡cito. La imagen ya viene del registry.

**5. SSH key sin permisos correctos en el servidor.**
*S├¡ntoma*: `appleboy/ssh-action` falla con `Permission denied (publickey)`.
*Causa*: la clave p├║blica no est├í en `~/.ssh/authorized_keys` del usuario `deploy` en el servidor, o los permisos del directorio `.ssh/` son incorrectos.
*Arreglo*: `chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys` en el servidor. La clave privada en el secreto de GitHub debe coincidir con la p├║blica registrada.

## Hilo conductor

En D05 automatizaste el pipeline de CI/CD para la app entera como un bloque. En la práctica, un stack BATUTA tiene seis tipos de piezas con propiedades radicalmente distintas: la UI estática no tiene estado; la API es stateless pero necesita proceso; el worker corre en background sin puerto; la base de datos tiene estado y necesita backups; el vector store depende del volumen de embeddings; y la observabilidad es un servicio interno que no escala con el tráfico de usuarios. D06 enseña a desplegar cada una, con la opción simple para empezar y el upgrade cuando el caso lo justifica. D07 añade estrategias de rollout; D08 gestiona los secretos que cada pieza necesita.

## Idea central

Cada pieza del stack tiene un **perfil de despliegue** diferente: estaticidad, estado, escalabilidad, criticidad de datos y costo. Tratar todas igual — meterlas en el mismo servidor, desplegarlas todas a la vez — es el error más común en equipos pequeños. El segundo error es lo contrario: sobre-arquitectar desde el día 1 con Kubernetes, servicios separados y load balancers cuando un solo Hetzner CX22 con Docker Compose es suficiente para los primeros 1 000 tenants.

## Por qué importa

El caso **F-CX-03** (base de conocimiento para atención al cliente) necesita, como mínimo: UI, API, worker de indexado, Postgres con pgvector, y Phoenix para observabilidad. Si no sabes cómo se despliega cada pieza, terminas con todo en un solo contenedor, sin backups, sin healthchecks, y con la observabilidad apagada «para después». «Después» nunca llega.

> [!cuidado]
> La tabla de costos de este módulo asume precios de mayo 2026. Los precios de Hetzner, Cloudflare, Neon y Fly.io cambian. Antes de tomar decisiones de arquitectura, verifica los precios actuales en sus sitios. Lo que no cambia es el razonamiento de cuándo separar vs cuándo consolidar.

## Cómo funciona por dentro

### Pieza 1 — UI estática (HTML + JS)

La UI de bato-learning es Jinja2 server-rendered: no es estática. Pero muchos productos BATUTA tienen un frontend separado en React, Vue o HTML plano.

Una **UI estática** es cualquier conjunto de archivos que el servidor no transforma en tiempo de request: `.html`, `.js`, `.css`, imágenes. Un CDN puede servirlos desde nodos distribuidos globalmente sin una API detrás.

**Opciones de deploy:**

| Plataforma | Free tier | Cuándo |
|-----------|-----------|--------|
| **Cloudflare Pages** | Ilimitado en bandwidth, 500 builds/mes | Default para PYME. Sin egress fees. Red de 300+ PoPs. |
| **Netlify** | 300 build-minutes/mes, 100 GB bandwidth | Si ya usas Netlify por sus formularios o funciones edge. |
| **Vercel** | 100 GB bandwidth, 6 000 build-min/mes | Si el stack es Next.js. Over-engineered para HTML plano. |
| **GitHub Pages** | Ilimitado para repos públicos | Para documentación técnica o demos. Sin CI/CD propio. |

**Deploy como push:** con Cloudflare Pages conectas el repo de GitHub. Cada push a `main` dispara un build y deploy automático. El dominio `*.pages.dev` queda disponible en 30 segundos. Conectas tu dominio propio gratis.

```bash
# Deploy manual con Wrangler CLI (alternativa al push automático)
npx wrangler pages deploy ./dist --project-name=acme-dashboard
```

**Costo para PYME LATAM:** $0/mes para la mayoría de los casos. Si superas 500 builds/mes (imposible para una PYME normal), el plan Pro cuesta $5/mes.

**CDN y cache:** Cloudflare cachea todos los assets estáticos en sus PoPs automáticamente. Para invalidar el cache en un nuevo deploy, Cloudflare Pages usa fingerprinting de archivos: el nombre del archivo incluye un hash del contenido, así que nunca hay conflicto entre versión vieja y nueva.

### Pieza 2 — API stateless

Una API FastAPI **stateless** no guarda estado en memoria entre requests. Todo el estado vive en la base de datos. Esto es lo que la hace desplegable en múltiples instancias sin coordinación.

**Opciones de deploy:**

| Opción | Precio orientativo | Cuándo |
|--------|-------------------|--------|
| **Hetzner CX22 + Docker** | ~4-5 EUR/mes | Default. Control total, precio mínimo. |
| **Fly.io** | Desde $0 (free machines) | Si quieres multi-región sin ops. Buen DX. |
| **Railway** | Desde $5/mes | Si valoras el DX sobre el precio. Sin SSH. |
| **Hetzner CX32 / CCX23** | 8-20 EUR/mes | Cuando el CX22 ya no alcanza (> 50 req/s sostenidos). |

El default de bato-learning es Hetzner CX22 con Docker Compose (D02, D03). Para la API del stack completo de ACME, el mismo servidor sirve API + worker + Phoenix. Separas solo cuando una pieza necesita escalar de forma independiente.

**Rolling deploy sin downtime (stateless):** porque la API no tiene estado en memoria, puedes levantar la nueva versión mientras la vieja sigue sirviendo, luego matar la vieja. Docker Compose no hace esto automáticamente — lo verás en D07.

**Auto-scale:** Fly.io tiene auto-scale por número de requests concurrentes. Hetzner no tiene auto-scale: subes manualmente a un plan mayor. Para PYME LATAM, el auto-scale rara vez se justifica; el tráfico es predecible y un CX32 aguanta picos.

### Pieza 3 — Worker / background jobs

Un **worker** es un proceso que consume tareas de una cola (Redis, Temporal, RabbitMQ) o corre un loop de polling. No tiene puerto HTTP. No recibe requests del exterior. Procesa en background: indexar documentos, mandar emails, correr el agente de auditoría mensual.

**Deploy en Docker Compose:**

```yaml
services:
  api:
    image: ghcr.io/acme/monthly-audit-api:${IMAGE_TAG}
    ports:
      - "8000:8000"

  worker:
    image: ghcr.io/acme/monthly-audit-api:${IMAGE_TAG}   # misma imagen
    command: python -m app.worker                          # distinto punto de entrada
    environment:
      WORKER_CONCURRENCY: "4"
    # sin ports — no escucha en ningún puerto
    restart: unless-stopped
```

El worker usa la **misma imagen** que la API pero con un `command` distinto. Esto elimina la necesidad de mantener dos Dockerfiles. El proceso de build en CI produce una sola imagen; el `command` en compose decide si arranca como API o como worker.

**Healthcheck del worker:** sin puerto HTTP, el healthcheck no puede hacer un GET. Opciones:
- El worker escribe un archivo de heartbeat cada N segundos: `healthcheck: { test: ["CMD", "test", "-f", "/tmp/worker-alive"] }`.
- El worker expone un endpoint HTTP mínimo solo para healthcheck (`/health`) aunque no procese requests externos.
- Si usas Temporal, el worker registra sus workflows y activities — la UI de Temporal muestra si el worker está conectado.

**Escalado de workers:** para añadir concurrencia, escala el servicio `worker`:

```bash
docker compose up -d --scale worker=3
```

Los tres workers comparten la misma cola. Temporal gestiona la distribución automáticamente.

### Pieza 4 — Base de datos relacional

Esta es la pieza con más estado y la que más daño hace si se gestiona mal.

**Opciones:**

| Opción | Precio orientativo | Cuándo |
|--------|-------------------|--------|
| **Self-hosted Postgres en el mismo server** | $0 extra (usa el CX22 existente) | Dev y primeros clientes. Backups manuales con `pg_dump`. |
| **Neon** (serverless Postgres) | Free tier hasta 0.5 GB; desde $19/mes | Cuando quieres 0 ops y branching de DB para tests. |
| **Supabase** | Free tier; desde $25/mes | Si quieres storage + auth + realtime de regalo. |
| **Hetzner Managed Postgres** | Desde ~25 EUR/mes | Cuando quieres self-hosted pero con backups automáticos garantizados. |

**Self-hosted en el mismo server** es la opción más barata y la más frágil si no tienes backups automatizados. Para bato-learning con SQLite, el backup es un `cp`. Para Postgres:

```bash
# backup diario en cron (en el servidor)
0 2 * * * pg_dump -U bato bato_learning | gzip > /srv/backups/db-$(date +%F).sql.gz
```

**Migraciones en CI/CD:** las migraciones corren como step del pipeline (D05, sección de migraciones). La regla es: la migración se aplica **antes** de levantar la nueva versión de la app. El patrón expand-contract (D07) garantiza que ambas versiones (vieja y nueva) sean compatibles durante el deploy.

**Replicación:** para PYME LATAM con < 500 tenants, una instancia Postgres sin réplica es suficiente. La réplica de lectura tiene sentido cuando el 80 % del tráfico es queries de lectura y la primaria está saturada — raramente el cuello de botella en este rango.

### Pieza 5 — Vector database

El vector store guarda **embeddings** (vectores numéricos) y sirve búsquedas de similitud para RAG. La decisión de arquitectura es: ¿pgvector integrado en Postgres, o un servicio separado?

**Cuándo usar pgvector (default):**

```sql
-- extensión en Postgres
CREATE EXTENSION IF NOT EXISTS vector;

-- tabla con columna de embedding
ALTER TABLE documents ADD COLUMN embedding vector(1536);
CREATE INDEX ON documents USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
```

pgvector es la opción más simple: mismo Postgres, misma conexión, mismas transacciones. Para PYME con < 1 M documentos y búsquedas < 100 ms de latencia requerida, pgvector aguanta perfectamente.

**Cuándo separar:**

| Criterio | pgvector | Qdrant Cloud / Pinecone |
|----------|----------|------------------------|
| Volumen de vectores | < 5 M | > 5 M o crecimiento rápido |
| Latencia requerida | < 200 ms | < 20 ms (búsquedas en tiempo real) |
| Equipos que lo usan | 1 (la API) | Múltiples servicios independientes |
| Costo | $0 extra | Qdrant Cloud: desde $25/mes; Pinecone: desde $70/mes |
| Ops necesarios | Ninguno extra | Un servicio más para monitorear |

Para bato-learning y la mayoría de casos BATUTA en PYME LATAM: **pgvector hasta que duela**. «Hasta que duela» significa que las queries de similaridad aparecen en el top de latencia en Phoenix y superan el SLA del cliente.

### Pieza 6 — Observabilidad (Phoenix)

Phoenix self-hosted corre como un contenedor en el mismo servidor (D04). No escala con el número de usuarios — escala con el volumen de spans exportados por los agentes.

**En producción Hetzner:**

```yaml
# en compose.prod.yaml
services:
  phoenix:
    image: arizephoenix/phoenix:latest
    restart: unless-stopped
    environment:
      PHOENIX_WORKING_DIR: /phoenix_data
    volumes:
      - phoenix_data:/phoenix_data
    # NO exponer el puerto 6006 en internet — solo acceso interno
    # acceder por SSH tunnel: ssh -L 6006:localhost:6006 deploy@servidor
```

Phoenix no se expone públicamente. Para acceder a la UI en producción:

```bash
# desde tu laptop
ssh -L 6006:localhost:6006 deploy@65.21.x.x
# luego abre http://localhost:6006 en el navegador
```

**Cuándo migrar a Grafana Cloud:** cuando el volumen de spans supera lo que SQLite maneja bien (aproximadamente 500 k spans/día), o cuando necesitas alertas automáticas basadas en métricas de agentes.

### Pieza 7 — LLM provider

El LLM provider (Anthropic, OpenAI, etc.) no es algo que tú despliegas — es un servicio externo. Pero sí gestionas tres cosas críticas:

1. **API keys:** rotación periódica, mínimo cada 90 días. El D08 cubre esto.
2. **Quotas y rate limits:** Anthropic tiene límites por Tier. En producción, monitorea `x-ratelimit-remaining-requests` en los headers de respuesta.
3. **Multi-provider failover:** si Anthropic tiene un outage, ¿tienes fallback? Para PYME, la respuesta honesta es «no, y está bien». Para SaaS con SLA, considera un wrapper que intente con OpenAI si Anthropic falla con 529 (overloaded).

```python
# patron de failover minimo — solo cuando el SLA lo justifica
async def call_with_fallback(prompt: str, tenant: str) -> str:
    try:
        return await anthropic_client.call(prompt)
    except anthropic.APIStatusError as e:
        if e.status_code == 529:  # overloaded
            return await openai_client.call(prompt)
        raise
```

### Diagrama de despliegue: dos configuraciones BATUTA

**Configuración A — Todo en un server (primeros 500 tenants):**

```
Internet
    │
Cloudflare (DNS + CDN)
    │         │
    │    UI estática
    │    (Pages — gratis)
    │
Traefik (Hetzner CX22 — 4 EUR/mes)
    ├── api:8000     (FastAPI + agentes)
    ├── worker       (sin puerto — Temporal activities)
    ├── postgres     (named volume — backups diarios por cron)
    ├── phoenix:6006 (solo acceso interno por SSH tunnel)
    └── temporal:7233 (workflow engine)
```

**Configuración B — Split (a partir de 500 tenants o SLA exigente):**

```
Internet
    │
Cloudflare Pages → UI estática (gratis, CDN global)
    │
    └── api.acme.com → Fly.io (multi-región, auto-scale)
                           │
               ┌───────────┴───────────┐
               │                       │
         Neon Postgres          Qdrant Cloud
         (managed, branch)      (si > 5M vectores)
               │
         Phoenix Cloud / Grafana Cloud
         (si > 500k spans/día)
```

### Tabla de costo mensual orientativo — PYME LATAM (USD)

| Componente | Configuración A | Configuración B | Notas |
|-----------|----------------|----------------|-------|
| UI estática | $0 | $0 | Cloudflare Pages free |
| API + worker | ~$4 | $5-20 | Hetzner CX22 vs Fly.io |
| Base de datos | $0 | $19-25 | Self-hosted vs Neon/Supabase |
| Vector store | $0 | $0-25 | pgvector incluido vs Qdrant Cloud |
| Observabilidad | $0 | $0-30 | Phoenix self-hosted vs Grafana Cloud |
| Temporal | $0 | $0 | Self-hosted en mismo server |
| LLM (Anthropic) | $10-100 | $10-100 | Depende del uso de tokens |
| **Total infra/mes** | **~$4-104** | **~$24-175** | Sin contar LLM |

La Configuración A es viable para los primeros 500 tenants de una PYME. La Configuración B aplica cuando un componente específico se convierte en cuello de botella — no se migra todo a la vez.

## Ejemplo conducido

Compose completo para el stack de ACME en Configuración A (un server Hetzner):

```yaml
# compose.prod.yaml para acme/monthly-audit
services:
  api:
    image: ghcr.io/acme/monthly-audit-api:${IMAGE_TAG}
    restart: unless-stopped
    environment:
      DATABASE_URL: postgresql+asyncpg://bato:${DB_PASSWORD}@postgres:5432/audit
      ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}
      TEMPORAL_HOST: temporal:7233
    depends_on:
      postgres:
        condition: service_healthy
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.api.rule=Host(`api.acme.com`)"
      - "traefik.http.routers.api.entrypoints=websecure"
      - "traefik.http.routers.api.tls.certresolver=le"

  worker:
    image: ghcr.io/acme/monthly-audit-api:${IMAGE_TAG}
    command: python -m app.worker
    restart: unless-stopped
    environment:
      DATABASE_URL: postgresql+asyncpg://bato:${DB_PASSWORD}@postgres:5432/audit
      ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}
      TEMPORAL_HOST: temporal:7233
    depends_on:
      - postgres
      - temporal

  postgres:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      POSTGRES_USER: bato
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: audit
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD", "pg_isready", "-U", "bato", "-d", "audit"]
      interval: 10s
      timeout: 5s
      retries: 5

  phoenix:
    image: arizephoenix/phoenix:latest
    restart: unless-stopped
    environment:
      PHOENIX_WORKING_DIR: /phoenix_data
    volumes:
      - phoenix_data:/phoenix_data
    # sin label de Traefik — solo acceso interno

  temporal:
    image: temporalio/auto-setup:1.25
    environment:
      - DB=postgres12
      - POSTGRES_SEEDS=postgres
      - POSTGRES_USER=bato
      - POSTGRES_PWD=${DB_PASSWORD}
      - DBNAME=temporal
    depends_on:
      postgres:
        condition: service_healthy

volumes:
  pgdata:
  phoenix_data:
```

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Configurar servicios en compose.yaml | determinístico | YAML declarativo: el mismo archivo produce el mismo stack siempre. |
| Elegir entre pgvector y Qdrant Cloud para un nuevo cliente | agéntico | Depende del volumen proyectado, la latencia requerida, el presupuesto y la madurez del equipo — sin regla cerrada. |
| Configurar healthchecks de Postgres | determinístico | `pg_isready` es binario: funciona o no. La regla es cerrada. |
| Decidir si un componente específico necesita escalar | agéntico | Requiere leer las trazas de Phoenix, el uso de CPU/memoria, las quejas de latencia de los tenants — no hay umbral universal. |
| Correr `pg_dump` para hacer backup | determinístico | Comando con parámetros fijos. Se automatiza con cron. |
| Diseñar la estrategia de failover de LLM provider | agéntico | Depende del SLA contratado, el costo de un outage, las alternativas disponibles y la madurez del equipo de infra. |

## Errores típicos

**1. Mismo puerto para dos servicios distintos.**
*Síntoma*: `docker compose up` falla con `port is already allocated`.
*Causa*: dos servicios en el mismo `compose.yaml` mapean al mismo puerto del host (ej. dos servicios con `- "8080:8080"`).
*Arreglo*: cada servicio en puertos distintos del host. En producción con Traefik, los servicios no exponen puertos directamente — Traefik hace el routing por hostname, todos en 80/443.

**2. Worker sin `restart: unless-stopped`.**
*Síntoma*: el worker muere silenciosamente (excepción no capturada, OOM) y no hay nadie que lo reinicie.
*Causa*: sin política de restart, un contenedor que falla queda en estado `exited`.
*Arreglo*: añadir `restart: unless-stopped` a todos los servicios que deben estar siempre corriendo. El worker incluido.

**3. pgvector sin índice en la columna de embedding.**
*Síntoma*: las búsquedas de similitud tardan segundos aunque la tabla tiene pocos miles de filas.
*Causa*: sin índice, pgvector hace un full-scan de la tabla calculando la distancia con cada fila.
*Arreglo*: crear el índice `ivfflat` o `hnsw` sobre la columna de embedding. Para colecciones < 100 k filas, `hnsw` tiene mejor recall; para colecciones > 1 M, `ivfflat` es más eficiente en memoria.

**4. Phoenix expuesto en internet sin autenticación.**
*Síntoma*: cualquiera con la IP puede ver el historial completo de prompts de todos los tenants.
*Causa*: se añadió un label de Traefik a Phoenix sin middleware de autenticación.
*Arreglo*: no exponer Phoenix con label de Traefik. Acceso solo por SSH tunnel. Si necesitas acceso remoto sin SSH, usa Tailscale o un middleware BasicAuth en Traefik.

**5. Backup de Postgres en el mismo disco que la base de datos.**
*Síntoma*: el disco falla y se pierden tanto la base de datos como el backup.
*Causa*: `pg_dump` escribe en `/srv/backups/` que está en el mismo volumen que `/var/lib/postgresql/`.
*Arreglo*: enviar los backups a Hetzner Storage Box (S3-compatible, desde 1 EUR/mes por 100 GB) o a cualquier almacenamiento externo. En CI/CD, el backup como step que sube a S3 antes del deploy.

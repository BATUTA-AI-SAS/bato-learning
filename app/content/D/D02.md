## Hilo conductor

En D01 construiste y corriste la imagen de bato-learning con `docker run`. Funciona, pero el comando es largo y frágil: tienes que recordar `-p`, `-e`, `-v` en cada ejecución. En cuanto añades un segundo servicio (una base de datos, un worker, un mailcatcher) el problema se multiplica. **docker compose** (v2, sin guion) es la solución: declara todos los servicios en un archivo YAML y los levanta con un solo comando. D02 enseña a leer y modificar el `compose.yaml` real de bato-learning, y a compararlo con el `docker-compose.yml` del proyecto `api/` que orquesta cinco servicios. D03 toma este stack y lo despliega en Hetzner.

## Idea central

**docker compose** (versión 2, comando `docker compose`) lee un archivo YAML que declara servicios, redes y volúmenes, y los crea y conecta como un grupo coherente. La característica clave que no da `docker run` es la **red implícita por nombre de proyecto**: todos los contenedores del mismo compose file pueden hablarse por el nombre de su servicio como hostname — sin configurar nada extra. Un servicio `api` puede conectarse a `postgres:5432` porque el servicio se llama `postgres`. Sin IP hardcodeada. Sin DNS externo.

## Por qué importa

Un agente de base de conocimiento (caso **F-CX-03**) necesita como mínimo tres piezas: una API FastAPI, un vector store, y un worker que procesa documentos en background. Levantarlos uno a uno con `docker run` no es reproducible: el orden importa, las redes no se conectan automáticamente, y los volúmenes hay que crearlos a mano. Con un `compose.yaml`, `docker compose up` levanta el stack en el orden correcto, con las redes y volúmenes ya declarados.

> [!cuidado]
> Compose v2 es `docker compose` (subcomando de Docker). Compose v1 era `docker-compose` (binario separado, deprecated). En Ubuntu 24.04 con Docker Engine 25+, `docker-compose` puede no estar instalado. Usa siempre `docker compose` con espacio.

## Cómo funciona por dentro

### Anatomía del `compose.yaml` de bato-learning

```yaml
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
    image: bato-learning:dev
    container_name: bato-learning
    environment:
      ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY:-}
      DATABASE_URL: sqlite+aiosqlite:////srv/data/learning.db
      LOG_LEVEL: ${LOG_LEVEL:-info}
      ANTHROPIC_MODEL: ${ANTHROPIC_MODEL:-claude-sonnet-4-6}
    ports:
      - "8080:8080"
    volumes:
      - ./data:/srv/data
      - ./app/content:/srv/app/content:ro
    command: >
      sh -c "
      uv run alembic upgrade head &&
      uv run python -m scripts.seed &&
      uv run uvicorn app.main:app --host 0.0.0.0 --port 8080 --reload
      "
```

Analiza cada campo:

- **`build`**: le dice a Compose que construya la imagen usando el `Dockerfile` local. Sin `build`, el campo `image` solo descarga de un registry.
- **`image: bato-learning:dev`**: el tag que se asigna a la imagen construida. Permite referenciarla en otros servicios con `image: bato-learning:dev`.
- **`environment`**: variables de entorno inyectadas al contenedor. La sintaxis `${VAR:-default}` lee del entorno del host y cae al default si no está definida. **Las credenciales nunca van hardcodeadas aquí** — van en el entorno del host o en un `.env`.
- **`ports: ["8080:8080"]`**: mapea el puerto 8080 del contenedor al 8080 del host. Formato: `host:container`.
- **`volumes`**:
  - `./data:/srv/data` — bind mount del directorio `data/` local. La base de datos SQLite persiste entre reinicios del contenedor.
  - `./app/content:/srv/app/content:ro` — bind mount en modo read-only. El contenedor lee el contenido de los módulos desde tu máquina; editas los `.md` y el cambio es inmediato sin reconstruir la imagen.
- **`command`**: sobreescribe el `CMD` del Dockerfile. En dev, corre migraciones y seed antes de arrancar uvicorn con `--reload` (hot-reload). En prod, no se usa este override.

### Redes implícitas

Compose crea automáticamente una red llamada `{proyecto}_default` (donde `{proyecto}` es el nombre del directorio o el valor de `name:` en el yaml). Todos los servicios del compose file se unen a esa red. Esto significa que si añades un servicio `mailcatcher`:

```yaml
services:
  app:
    # ... ya existente
  mailcatcher:
    image: maildev/maildev:2.1
    ports:
      - "1080:1080"
      - "1025:1025"
```

El servicio `app` puede enviar emails a `mailcatcher:1025` sin ninguna configuración adicional. El nombre `mailcatcher` resuelve como hostname dentro de la red del compose.

### Volúmenes: named vs bind mount

| Tipo | Sintaxis | Cuándo usarlo |
|------|----------|---------------|
| Bind mount | `./local/path:/container/path` | Desarrollo: hot-reload de código, datos locales accesibles |
| Named volume | `pgdata:/var/lib/postgresql/data` | Producción: datos que Docker gestiona, sin path explícito en el host |
| Anónimo | `/container/path` (sin nombre) | Temporal, desaparece con el contenedor. Evitar |

En bato-learning usamos bind mount para `data/` porque el archivo SQLite vive en el repo y queremos manipularlo directamente. El `docker-compose.yml` de la `api/` usa named volumes para Postgres (`pgdata`) y Phoenix (`phoenix_data`) porque son datos gestionados que no necesitamos tocar directamente.

### `depends_on` y el problema de la salud

```yaml
services:
  api:
    depends_on:
      - postgres
```

`depends_on` en su forma simple garantiza que el contenedor `postgres` **ha arrancado** antes de `api`. No garantiza que Postgres esté listo para aceptar conexiones. La diferencia importa: Postgres tarda ~2 segundos en inicializar su cluster después de que el contenedor arranca.

La solución correcta es `depends_on` con condición:

```yaml
services:
  api:
    depends_on:
      postgres:
        condition: service_healthy

  postgres:
    image: postgres:16-alpine
    healthcheck:
      test: ["CMD", "pg_isready", "-U", "bato", "-d", "bato_learning"]
      interval: 5s
      timeout: 3s
      retries: 5
      start_period: 10s
```

`condition: service_healthy` hace que Compose espere a que el healthcheck de `postgres` devuelva éxito antes de arrancar `api`. Sin esto, `api` puede intentar conectarse a Postgres mientras este aún inicializa y fallar con `Connection refused`.

### El `docker-compose.yml` de la `api/` — cinco servicios

El proyecto `api/` (directorio `./api/`) tiene un `docker-compose.yml` más complejo, real de producción:

```yaml
services:
  api:          # FastAPI — depende de postgres, redis, phoenix, temporal
  postgres:     # base de datos principal con named volume pgdata
  redis:        # cache / broker de tareas (para Temporal en algunos modos)
  phoenix:      # observabilidad (Arize Phoenix self-hosted) con named volume
  temporal:     # workflow engine; depende de postgres para su propio schema
  temporal-ui:  # UI web de Temporal; depende de temporal
```

Notas de este stack:

- `temporal` usa la misma instancia de Postgres que `api`, pero en una base de datos distinta (`DBNAME=temporal`). Es un patrón válido en dev; en prod se separan.
- `temporal-ui` tiene `TEMPORAL_CORS_ORIGINS=http://localhost:8080`, que apunta al puerto de la `api`. Si bato-learning corre en 8080 también, hay conflicto. En local, uno de los dos debe cambiar de puerto.
- Los volúmenes `pgdata` y `phoenix_data` son named volumes — declarados en la sección `volumes:` al final del archivo. Docker los gestiona y persisten entre `docker compose down` y `docker compose up` salvo que hagas `docker compose down -v` (que los borra).

### `env_file` vs `environment`

```yaml
services:
  api:
    env_file: .env             # lee todas las variables del archivo
    environment:               # override de variables específicas
      LOG_LEVEL: debug
```

`env_file` es útil cuando tienes muchas variables. El archivo `.env` **nunca va al repositorio**: añádelo al `.gitignore`. Las variables que sí pueden ir en el compose (sin secretos) van en `environment` directamente o con `${VAR}` interpolado desde el entorno del host.

> [!nota]
> En producción Hetzner (D03), los secretos van en un `.env` en el servidor, fuera del repo. El `compose.yaml` usa `${SECRET_KEY}` y Docker Compose lo resuelve en tiempo de `up`. Sin secretos hardcodeados, sin secretos en la imagen.

### Profiles — servicios opcionales

```yaml
services:
  app:
    # servicio default — siempre arranca
  mailcatcher:
    profiles: [dev]
    image: maildev/maildev:2.1
```

Con `profiles`, `docker compose up` solo levanta `app`. Para levantar también el mailcatcher: `docker compose --profile dev up`. Útil para servicios que solo necesitas a veces (debugging de email, profilers, adminer para la base de datos).

## Ejemplo conducido

```bash
# 1. arrancar el stack de bato-learning en dev
docker compose up --build

# 2. ver qué contenedores corren y su estado
docker compose ps

# 3. ver logs de la app en tiempo real
docker compose logs -f app

# 4. correr un comando dentro del contenedor en ejecución
docker compose exec app uv run alembic current

# 5. parar y limpiar (conserva volúmenes)
docker compose down

# 6. parar y borrar volúmenes (reset completo)
docker compose down -v
```

Para añadir un mailcatcher al `compose.yaml` de bato-learning:

```yaml
# añadir debajo del servicio app:
  mailcatcher:
    image: maildev/maildev:2.1.0
    ports:
      - "1080:1080"   # UI web
      - "1025:1025"   # SMTP
    profiles: [dev]
```

Luego: `docker compose --profile dev up`. El servicio `app` puede enviar correos a `mailcatcher:1025`.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Declarar servicios, redes y volúmenes en `compose.yaml` | determinístico | Infra como código: mismo YAML → mismo stack, siempre |
| Orden de arranque con `depends_on` + healthcheck | determinístico | Regla cerrada: Compose espera el healthcheck antes de arrancar el dependiente |
| Interpolar `${VAR:-default}` en el compose file | determinístico | El valor lo da el entorno del host; si no está, cae al default declarado |
| Decidir si un nuevo servicio necesita named volume o bind mount | agéntico | Depende de si los datos deben ser accesibles en el host, la frecuencia de reset, y la estrategia de backup — no hay regla sin contexto |
| Diseñar el compose inicial de un proyecto nuevo con N servicios | agéntico | Un agente puede proponer un draft; un humano revisa dependencias, puertos y estrategia de datos antes de commitearlo |

La infra declarada es 100% determinística. La fase agéntica es el **diseño inicial** y las decisiones de arquitectura de almacenamiento — pero el commit lo haces tú después de revisar.

## Errores típicos

**1. `depends_on` sin healthcheck — la app arranca antes de que la DB esté lista.**
*Síntoma*: `sqlalchemy.exc.OperationalError: connection refused` en los primeros segundos de `docker compose up`.
*Causa*: `depends_on` solo espera que el contenedor esté corriendo, no que el proceso interno esté listo.
*Arreglo*: añadir `healthcheck` al servicio de DB y usar `condition: service_healthy` en el `depends_on` del servicio dependiente.

**2. Bind mount de `./app` hace que los cambios del host sobreescriban la imagen.**
*Síntoma*: funciona en dev; en producción (sin volumen) falla porque falta un archivo.
*Causa*: en dev, el volumen sobreescribe el contenido de la imagen. Si un archivo existe en la imagen pero no en el host, desaparece.
*Arreglo*: en prod no montes el código; usa la imagen construida. Separa los compose files: `compose.yaml` para dev, `compose.prod.yaml` para prod (D03).

**3. `docker-compose` con guion en Ubuntu 24.04.**
*Síntoma*: `command not found: docker-compose`.
*Causa*: Compose v1 no está instalado. Ubuntu 24.04 con Docker Engine solo trae Compose v2 como subcomando.
*Arreglo*: usa `docker compose` (sin guion). Si necesitas compatibilidad con scripts legacy, instala el plugin: `apt install docker-compose-plugin`.

**4. Named volume borrado accidentalmente con `docker compose down -v`.**
*Síntoma*: la base de datos Postgres está vacía después de un `down`.
*Causa*: `-v` borra todos los volúmenes declarados en el compose, incluyendo datos persistentes.
*Arreglo*: usar `docker compose down` (sin `-v`) para preservar datos. Solo usar `-v` para reset intencional de entorno. Para backups de datos críticos, no depender solo del volumen.

**5. Secretos en `environment:` hardcodeados.**
*Síntoma*: el `compose.yaml` contiene `ANTHROPIC_API_KEY: sk-ant-...` commiteado en el repo.
*Causa*: conveniencia sin pensar en consecuencias.
*Arreglo*: usar `${ANTHROPIC_API_KEY}` en el compose y definirla en el `.env` del servidor (que está en `.gitignore`). Nunca commitear valores de secretos.

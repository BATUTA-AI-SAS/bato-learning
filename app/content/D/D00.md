## Hilo conductor

En 00-18 aprendiste Git y configuraste tu clave SSH. En C03 terminaste la app multitenant con SQLAlchemy. El siguiente paso es desplegar esa app en un servidor real — D01 (Docker), D02 (Compose), D03 (Hetzner + Traefik). El problema: todos esos módulos asumen que puedes entrar al servidor, leer sus logs, reiniciar servicios y resolver problemas cuando algo falla. Sin ese vocabulario operativo, el primer outage te deja ciego. D00 llena ese hueco. No es teoría de Linux — es el conjunto mínimo de comandos que vas a usar en producción semana a semana.

## Idea central

Un servidor Ubuntu 24.04 LTS es una máquina que ejecuta procesos, gestiona archivos con permisos, y expone servicios protegidos por un firewall. **SSH** es la puerta de entrada: una conexión cifrada al shell del servidor. **systemd** es el init system: el proceso 1 que arranca y supervisa todos los servicios. **journalctl** es el lector de logs de systemd. **ufw** es el firewall. **cron** programa tareas. Dominar estas cinco herramientas cubre el 90% de lo que necesitas para operar el agente de ACME en producción.

## Por qué importa

D03 lanza al lector a Hetzner CX22 de €4/mes con Docker y Traefik. Si la app no responde a las 2 a.m., el diagnóstico empieza en el servidor, no en el código. Sin saber ejecutar `journalctl -u app -f` o `systemctl status`, el lector no tiene información para actuar. El Capstone E06 requiere que el agente sea operable en producción: eso incluye poder reiniciarlo, leer sus trazas y proteger el puerto SSH.

El caso **F-OPS-02 (gestión de inventario ACME)** ilustra el riesgo: el agente de forecast falló silenciosamente porque el proceso padre no tenía `Restart=on-failure` en su unit file. Sin systemd configurado correctamente, el proceso murió y nadie se enteró hasta el día siguiente.

> [!cuidado]
> En Ubuntu 24.04 LTS, el servicio SSH usa **socket activation**: el proceso escucha en `ssh.socket`, no directamente en `sshd.service`. Cuando cambies la configuración de SSH, reinicia `ssh.socket`, no `sshd`. Y nunca ejecutes `ufw enable` sin haber asegurado que el puerto 22 está permitido — o te cortas el acceso al servidor.

## Cómo funciona por dentro

### SSH: la puerta cifrada al servidor

SSH (Secure Shell) establece un canal cifrado entre tu máquina y el servidor. La autenticación por clave pública/privada elimina las contraseñas — más segura y sin escritura manual en cada conexión.

```bash
# Generar clave ed25519 (moderna, compacta, segura)
ssh-keygen -t ed25519 -C "ops@acme.com"
# Guarda en ~/.ssh/id_ed25519 (privada) y ~/.ssh/id_ed25519.pub (pública)

# Copiar la clave pública al servidor (desde tu máquina)
ssh-copy-id -i ~/.ssh/id_ed25519.pub usuario@IP_DEL_SERVIDOR

# Conectarse
ssh usuario@IP_DEL_SERVIDOR

# Primera conexión: verás un fingerprint. Confirma con 'yes'.
# Las siguientes conexiones no preguntan.
```

Configura un alias en `~/.ssh/config` para no escribir la IP cada vez:

```
Host acme-prod
    HostName 65.21.100.42
    User ubuntu
    IdentityFile ~/.ssh/id_ed25519
```

Después: `ssh acme-prod` — sin IP, sin flags.

> [!cuidado]
> **Nunca actives autenticación por contraseña en SSH de producción.** En Ubuntu 24.04, `/etc/ssh/sshd_config` debe tener `PasswordAuthentication no`. Si Hetzner te entrega el servidor con contraseña, copia tu clave primero, luego desactiva contraseñas.

### systemd: el init system

`systemd` es el proceso 1 de Ubuntu — arranca con el kernel y supervisa todos los demás procesos. Cada servicio es una **unit**: un archivo de configuración en `/etc/systemd/system/`.

Comandos esenciales:

```bash
systemctl status docker          # estado del servicio
systemctl start docker           # arrancar
systemctl stop docker            # detener
systemctl restart docker         # detener + arrancar
systemctl reload docker          # recargar config sin reiniciar (si el servicio lo soporta)
systemctl enable docker          # activar al inicio del sistema
systemctl disable docker         # desactivar al inicio
systemctl daemon-reload          # recargar todos los unit files (después de editar uno)
```

Un **unit file** mínimo para la app:

```ini
# /etc/systemd/system/audit-agent.service
[Unit]
Description=ACME Audit Agent
After=network.target docker.service
Requires=docker.service

[Service]
Type=simple
User=ubuntu
WorkingDirectory=/srv/audit-agent
ExecStart=/usr/bin/docker compose up
ExecStop=/usr/bin/docker compose down
Restart=on-failure
RestartSec=10s
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

Después de crear o editar el archivo:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now audit-agent
```

La directiva `Restart=on-failure` hace que systemd reinicie el servicio si el proceso termina con código de error. Sin esto, si el contenedor muere, muere para siempre hasta que alguien lo reinicie manualmente.

### journalctl: leer logs de systemd

Todos los servicios que tienen `StandardOutput=journal` escriben ahí. journalctl es el lector.

```bash
journalctl -u audit-agent           # todos los logs del servicio
journalctl -u audit-agent -f        # seguir en tiempo real (como tail -f)
journalctl -u audit-agent -n 100    # últimas 100 líneas
journalctl -u audit-agent --since "1h ago"       # última hora
journalctl -u audit-agent --since "2026-05-01"   # desde fecha
journalctl -u audit-agent -p err    # solo errores (err, warning, info, debug)
journalctl --disk-usage              # cuánto espacio ocupan los logs
journalctl --vacuum-time=7d         # borrar logs de más de 7 días
```

> [!nota]
> Docker tiene su propio sistema de logs. Para un contenedor que NO está gestionado por systemd, usa `docker logs nombre-contenedor -f`. El módulo D04 (observabilidad) integra ambos con Phoenix y logs estructurados.

### Procesos: ver qué está corriendo

```bash
ps aux                        # todos los procesos
ps aux | grep audit           # filtrar por nombre
top                           # vista dinámica (q para salir)
htop                          # versión mejorada (sudo apt install htop)

# Matar un proceso
kill 12345                    # envía SIGTERM (petición de terminar con gracia)
kill -9 12345                 # envía SIGKILL (fuerza bruta; el proceso no puede ignorarlo)
pkill -f "uvicorn app.main"   # matar por nombre de comando

# Correr en segundo plano
comando &                     # ampersand: lanza en background y devuelve el PID
nohup comando &               # mantiene el proceso vivo aunque cierres la terminal
disown %1                     # desvincula un job background de la terminal actual
```

> [!cuidado]
> `kill -9` no es la primera opción. El proceso no puede hacer cleanup (cerrar archivos, flush de buffers, liberar locks). Usa siempre `kill` primero y espera 5-10 segundos. Si el proceso no termina, entonces `kill -9`.

### Permisos: `chmod`, `chown`, sudo

Todo archivo en Linux tiene propietario (`user`), grupo (`group`) y permisos para tres categorías: propietario, grupo, otros.

```bash
ls -la app/                    # ver permisos
# drwxr-xr-x 2 ubuntu ubuntu  4096 may 16 10:22 app/
# ^ d = directorio; rwx = user; r-x = group; r-x = otros

# chmod con números (octal)
chmod 644 config.json          # rw-r--r-- (propietario lee+escribe; otros solo leen)
chmod 755 scripts/backup.sh    # rwxr-xr-x (propietario ejecuta; otros también ejecutan)
chmod 600 ~/.ssh/id_ed25519    # rw------- (solo el propietario; SSH lo exige)

# chmod simbólico
chmod +x scripts/backup.sh     # añadir bit de ejecución
chmod u-w config.json          # quitar escritura al propietario (u=user, g=group, o=others)

# cambiar propietario
chown ubuntu:ubuntu /srv/app   # propietario ubuntu, grupo ubuntu
chown -R ubuntu:ubuntu /srv/   # recursivo

# sudo: ejecutar como root
sudo systemctl restart docker  # comando que requiere permisos de root
sudo -i                        # abrir shell como root (evitar en prod; usar sudo por comando)
```

Los valores de `chmod` más usados en producción:

| Valor | Significado | Para qué |
|-------|-------------|----------|
| `600` | `rw-------` | Claves SSH, archivos `.env` |
| `644` | `rw-r--r--` | Archivos de configuración, logs |
| `755` | `rwxr-xr-x` | Directorios, scripts ejecutables |
| `700` | `rwx------` | Directorio `~/.ssh/` |

### Firewall ufw: proteger los puertos

**ufw** (Uncomplicated Firewall) es la interfaz de alto nivel para `iptables` en Ubuntu.

```bash
sudo ufw status                # estado actual
sudo ufw allow 22              # permitir SSH (¡antes de enable!)
sudo ufw allow 80              # HTTP
sudo ufw allow 443             # HTTPS
sudo ufw allow 9000            # Phoenix (solo en desarrollo; cierra en prod)
sudo ufw default deny incoming # bloquear todo lo no permitido (explicito)
sudo ufw default allow outgoing # el servidor puede conectarse afuera
sudo ufw enable                # activar (confirma con y)
sudo ufw reload                # aplicar cambios
sudo ufw delete allow 9000     # eliminar una regla
```

> [!cuidado]
> El orden importa: ejecuta `sudo ufw allow 22` **antes** de `sudo ufw enable`. Si activas el firewall sin el puerto 22 abierto, pierdes acceso SSH al servidor y necesitas acceder por la consola de Hetzner Cloud (o contactar soporte). En Hetzner Cloud, siempre verifica que el Security Group también tenga el puerto 22 habilitado.

### Cron: tareas programadas

`cron` ejecuta comandos en horario fijo. La sintaxis del crontab tiene cinco campos de tiempo más el comando:

```
┌─ minuto    (0-59)
│ ┌─ hora    (0-23)
│ │ ┌─ día   (1-31)
│ │ │ ┌─ mes (1-12)
│ │ │ │ ┌─ día de semana (0-7, donde 0 y 7 = domingo)
│ │ │ │ │
* * * * * /ruta/al/comando
```

```bash
crontab -e          # editar el crontab del usuario actual
crontab -l          # ver el crontab actual

# Ejemplos
0 23 * * * /srv/scripts/backup.sh          # cada día a las 23:00
*/15 * * * * /srv/scripts/healthcheck.sh   # cada 15 minutos
0 0 1 * * /srv/scripts/monthly-report.sh  # el día 1 de cada mes a medianoche
```

Problemas frecuentes con cron:

```bash
# PATH limitado: cron no tiene tu PATH habitual
# Añade al inicio del crontab:
PATH=/usr/local/bin:/usr/bin:/bin:/home/ubuntu/.local/bin

# Output que desaparece: redirige stderr y stdout a un log
0 23 * * * /srv/scripts/backup.sh >> /var/log/backup.log 2>&1

# Cron no lee las variables de .env automáticamente
# Debes exportarlas explícitamente o cargarlas en el script
```

> [!nota]
> En sistemas modernos con systemd, los **systemd timers** son preferibles a cron: tienen logs en journalctl, pueden depender de otros servicios, y soportan calendarios más expresivos. Cron sigue siendo válido para scripts simples. La diferencia práctica: si necesitas saber si el cron falló, usa systemd timer — tendrás el log en `journalctl` sin configuración extra.

### Filesystem básico: dónde vive cada cosa

El Filesystem Hierarchy Standard (FHS) define la estructura de directorios de Linux:

| Directorio | Qué contiene |
|-----------|--------------|
| `/etc/` | Configuración del sistema y de servicios |
| `/var/log/` | Logs del sistema (aunque systemd usa journald) |
| `/srv/` | Datos de servicios del servidor (aquí va tu app) |
| `/home/ubuntu/` | Home del usuario `ubuntu` |
| `/tmp/` | Archivos temporales (se borran al reiniciar) |
| `/usr/local/bin/` | Binarios instalados manualmente |

```bash
df -h             # espacio en disco por partición
df -h /srv        # espacio de la partición donde está la app
du -sh /srv/app   # cuánto ocupa el directorio app
du -sh /srv/*     # cuánto ocupa cada subdirectorio de /srv
```

## Ejemplo conducido

Escenario real: recibes una alerta de que el agente de auditoría de ACME no responde. Diagnóstico paso a paso:

```bash
# 1. Conectarse al servidor
ssh acme-prod

# 2. Verificar el estado del servicio
systemctl status audit-agent
# ● audit-agent.service - ACME Audit Agent
#    Loaded: loaded (/etc/systemd/system/audit-agent.service; enabled)
#    Active: failed (Result: exit-code) since...

# 3. Leer los últimos logs del servicio
journalctl -u audit-agent -n 50
# may 16 02:14:22 server audit-agent[1842]: Error: no space left on device

# 4. Verificar espacio en disco
df -h
# /dev/sda1  20G  19.8G  0 / (¡disco lleno!)

du -sh /var/log/*  # encontrar qué ocupa espacio

# 5. Limpiar logs viejos
sudo journalctl --vacuum-time=3d

# 6. Reiniciar el servicio
sudo systemctl restart audit-agent

# 7. Verificar que levantó
systemctl status audit-agent
journalctl -u audit-agent -f  # seguir en tiempo real

# 8. Verificar que el puerto responde
curl -s http://localhost:8080/health
```

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Ejecutar `systemctl restart app` | determinístico | Resultado predecible: el proceso se detiene y reinicia |
| Leer `journalctl -u app -n 100` | determinístico | Devuelve los logs exactos almacenados en journald |
| Aplicar reglas `ufw allow 22` | determinístico | Regla firewall: o permite o no permite; no hay grado de ambigüedad |
| Diagnosticar por qué el agente falla | agéntico | Requiere interpretar logs, correlacionar síntomas, conocer el contexto del negocio |
| Decidir si abrir un puerto extra | agéntico | Implica evaluar riesgos de seguridad, necesidades del servicio, contexto del cliente |
| Configurar un crontab | determinístico | Sintaxis cerrada; el horario es exacto |

## Errores típicos

**1. `ufw enable` sin abrir el puerto 22 primero.**
*Síntoma*: el siguiente intento de `ssh acme-prod` da «Connection refused» o timeout. Acceso perdido.
*Causa*: el firewall bloqueó SSH porque `allow 22` no se ejecutó antes de `enable`.
*Arreglo*: acceder por la consola VNC de Hetzner Cloud → ejecutar `sudo ufw allow 22` → `sudo ufw reload`. Preventivo: siempre verifica con `sudo ufw status` antes de `enable`.

**2. Servicio que falla sin `Restart=on-failure`.**
*Síntoma*: el agente estaba funcionando ayer; hoy no responde; `systemctl status` muestra «inactive (dead)».
*Causa*: el proceso terminó con código de error (OOM, excepción no capturada) y systemd no lo reinició.
*Arreglo*: añadir `Restart=on-failure` y `RestartSec=10s` al unit file + `systemctl daemon-reload` + `systemctl restart`.

**3. Cron que no corre porque `PATH` no incluye el binario.**
*Síntoma*: el backup corre manualmente pero no cuando lo llama cron. El log muestra «command not found».
*Causa*: cron ejecuta con un PATH mínimo que no incluye `/usr/local/bin` ni `/home/ubuntu/.local/bin`.
*Arreglo*: añadir `PATH=/usr/local/bin:/usr/bin:/bin` al inicio del crontab, o usar rutas absolutas en el comando.

**4. Permisos 777 en archivos de configuración.**
*Síntoma*: «funciona» pero cualquier proceso puede leer y escribir los secretos.
*Causa*: «le di permisos para que no diera error».
*Arreglo*: `.env` debe ser `600` (solo el propietario lee). Scripts ejecutables: `755`. Directorios: `755`.

**5. `kill -9` como primera opción.**
*Síntoma*: el proceso termina pero deja locks, archivos corruptos o transacciones abiertas en la DB.
*Causa*: el lector quiere que el proceso termine «ya» sin esperar.
*Arreglo*: `kill PID` (SIGTERM) → espera 10s → si sigue: `kill -9 PID`. Para servicios gestionados por systemd, usa siempre `systemctl stop`.

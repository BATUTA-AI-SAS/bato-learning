## Hilo conductor

En A05 organizaste funciones en módulos y paquetes. Ahora agrupas datos y
comportamiento en clases. Python no obliga a usar clases; las usas cuando
el estado y el comportamiento van siempre juntos y necesitas múltiples
instancias. En A07 estas clases aparecerán en funciones async. En C01
las mismas clases serán modelos SQLAlchemy con columnas y relaciones.

## Idea central

Una **clase** en Python es una plantilla que produce objetos. Cada objeto
tiene su propio estado (`self.amount`, `self.vendor_id`) y comparte
los métodos de la clase. `__init__` inicializa el estado; `self` es la
referencia al objeto actual — no es magia, es el primer argumento que Python
pasa automáticamente a cada método de instancia.

Un **`@dataclass`** es una clase donde Python genera `__init__`, `__repr__`
y `__eq__` automáticamente a partir de las anotaciones de campo. Úsalo cuando
el objeto es principalmente datos estructurados. Un **`Protocol`** declara
qué métodos debe tener un objeto sin importar de dónde hereda — es la forma
Python de tipado estructural («duck typing» con type hints).

## Por qué importa

El caso F-CMP-01 (comparación de cotizaciones) modela cada cotización como
un objeto `Quote` con sus propios atributos y métodos de comparación. Si todo
vive en dicts, el código que accede a `q["unit_price"]` falla silenciosamente
si alguien escribe `q["price"]` — no hay contrato. Con una clase, el editor
autocompleta, `mypy` detecta el error, y el `__repr__` imprime algo útil en
los logs.

## Cómo funciona por dentro

### `__init__` y `self`

```python
class Invoice:
    def __init__(self, invoice_id: str, amount: float, vendor_id: str) -> None:
        self.invoice_id = invoice_id
        self.amount = amount
        self.vendor_id = vendor_id
```

`self.invoice_id = invoice_id` crea un **atributo de instancia**: vive en el
objeto, no en la clase. Cada `Invoice` tiene su propio `invoice_id`.

Un **atributo de clase** vive en la clase y es compartido por todas las
instancias:

```python
class Invoice:
    DEFAULT_CURRENCY: str = "EUR"   # atributo de clase
    ...
```

Si modificas un atributo de clase a través de la instancia (`inv.DEFAULT_CURRENCY
= "USD"`), Python crea un atributo de instancia que _sombrea_ el de clase. El
atributo de clase no cambia. Este comportamiento sorprende a gente que viene
de otros lenguajes.

### `@dataclass` — cuándo y cómo

```python
from dataclasses import dataclass, field
from decimal import Decimal

@dataclass
class Vendor:
    vendor_id: str
    name: str
    approved: bool = True
    invoice_count: int = field(default=0, repr=False)
```

`@dataclass` genera `__init__`, `__repr__` y `__eq__`. Con `field(default=0,
repr=False)` controlas el valor por defecto y si aparece en el `repr`.

Para listas y dicts como default, usa `field(default_factory=list)` —
exactamente para evitar el anti-patrón de default mutable que viste en A04.

`@dataclass(frozen=True)` genera objetos inmutables (genera `__hash__` y
bloquea asignación post-creación). Útil para keys de dict y elementos de set.

### `@property` — acceso calculado sin getter explícito

```python
@dataclass
class Invoice:
    amount_net: Decimal
    tax_rate: float

    @property
    def amount_gross(self) -> Decimal:
        """Monto bruto incluyendo IVA."""
        return self.amount_net * Decimal(str(1 + self.tax_rate))
```

El llamador escribe `inv.amount_gross` como si fuera un campo, pero Python
llama la función. El setter es opcional:

```python
    @amount_gross.setter
    def amount_gross(self, value: Decimal) -> None:
        self.amount_net = value / Decimal(str(1 + self.tax_rate))
```

### `Protocol` — contratos sin herencia

```python
from typing import Protocol

class VendorRepository(Protocol):
    def get_by_id(self, vendor_id: str) -> Vendor | None: ...
    def list_approved(self) -> list[Vendor]: ...
```

Cualquier clase que tenga esos dos métodos con esas firmas cumple el
`Protocol`, sin importar de dónde hereda. Esto es **tipado estructural**:
`mypy` valida en tiempo de análisis; en runtime Python no verifica nada.

El uso práctico: el servicio declara `VendorRepository` como tipo del
parámetro, el test pasa un `FakeVendorRepository` sin necesidad de mocking
mágico.

### Composición sobre herencia

La herencia entre clases de dominio lleva a jerarquías frágiles. Si `Manager`
hereda de `Employee` y el día de mañana un `Manager` puede ser también
`Contractor`, tienes un problema de diamante.

La alternativa: composición. Un `Invoice` tiene un `Vendor`, no *es* un
`Vendor`.

```python
@dataclass
class Invoice:
    invoice_id: str
    amount: Decimal
    vendor: Vendor          # composición
```

En este curso, la única jerarquía de herencia que se usa deliberadamente es
la de excepciones (A04) — y eso es correcto porque la herencia de excepciones
tiene semántica bien definida en Python.

### El modelo real: `app/models/content.py`

```python
# fragmento real (simplificado)
class Module(Base):
    __tablename__ = "module"
    id: Mapped[int] = mapped_column(primary_key=True)
    slug: Mapped[str] = mapped_column(String(128), unique=True)
    title: Mapped[str] = mapped_column(String(256))
    exercises: Mapped[list[Exercise]] = relationship(
        back_populates="module", cascade="all, delete-orphan"
    )
```

`Mapped[int]` es la versión SQLAlchemy 2.x de la anotación de columna. El
`Base` que hereda `Module` es la única herencia necesaria — y la provee el
framework, no la diseñas tú. Dentro del dominio de negocio, `Module` no
hereda de nada propio.

### Cuándo usar qué

| Situación | Herramienta |
|---|---|
| Objeto principalmente datos, sin lógica compleja | `@dataclass` |
| Objeto con lógica de negocio y estado que cambia | clase normal con `__init__` |
| Objeto inmutable que va en sets o dict keys | `@dataclass(frozen=True)` |
| Declarar un contrato para inyección de dependencias | `Protocol` |
| Modelar una tabla ORM | clase que hereda de `Base` (C01) |
| Validar datos de entrada de una API | `pydantic.BaseModel` (B02) |
| Nunca | dict sin schema cuando el shape importa para negocio |

## Ejemplo conducido

```python
from dataclasses import dataclass, field
from decimal import Decimal

@dataclass
class Vendor:
    vendor_id: str
    name: str
    approved: bool = True

@dataclass
class Quote:
    quote_id: str
    vendor: Vendor
    unit_price: Decimal
    quantity: int

    @property
    def total(self) -> Decimal:
        """Precio total de la cotización."""
        return self.unit_price * self.quantity

    def is_cheaper_than(self, other: "Quote") -> bool:
        """Compara por total; devuelve True si self es más barata."""
        return self.total < other.total

v_acme = Vendor("V-01", "ACME Manufacturing")
v_globex = Vendor("V-02", "Globex Logistics")

q1 = Quote("Q-001", v_acme, Decimal("120.00"), 50)
q2 = Quote("Q-002", v_globex, Decimal("115.00"), 50)

assert q1.total == Decimal("6000.00")
assert q2.is_cheaper_than(q1)
```

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|---|---|---|
| Calcular `quote.total` como `unit_price * quantity` | determinístico | Regla aritmética cerrada. |
| Comparar cotizaciones por precio total | determinístico | Regla de comparación definida. |
| Elegir la «mejor» cotización considerando historial del proveedor, calidad y riesgo | agéntico | Requiere contexto que no cabe en regla: relaciones, calidad pasada, urgencia. |
| Generar un memo de recomendación al CFO | agéntico | Salida es texto para humano; exige redacción contextual. |

## Errores típicos

**1. Atributo de clase mutable compartido entre instancias.**
*Síntoma*: modificar `Invoice.findings` en una instancia afecta a todas.
*Causa*: se declaró `findings: list = []` a nivel de clase.
*Arreglo*: usa `field(default_factory=list)` en `@dataclass`, o inicializa
en `__init__` con `self.findings = []`.

**2. `@property` con side effect.**
*Síntoma*: leer `invoice.amount_gross` dispara una query a la DB.
*Causa*: se puso lógica IO dentro del `@property`.
*Arreglo*: el `@property` solo computa desde el estado del objeto. IO va en
métodos explícitos que el llamador decide cuándo invocar.

**3. Herencia de dominio cuando bastaría composición.**
*Síntoma*: `PaidInvoice` hereda de `Invoice`; cuando aparece `PartiallyPaidInvoice`
el árbol se vuelve imposible.
*Causa*: se modeló estado como tipo.
*Arreglo*: campo `status: str` (o un enum) dentro de `Invoice`, no subclases.

**4. Usar dict cuando el shape tiene nombre.**
*Síntoma*: `q["unit_price"]` lanza `KeyError`; `q.get("uniit_price")` devuelve
`None` silenciosamente (typo).
*Causa*: se usó dict para un objeto con campos conocidos.
*Arreglo*: `@dataclass` o `Protocol` según si el objeto produce o consume datos.

**5. `__eq__` roto en dataclass con campos mutable.**
*Síntoma*: dos `Quote` con los mismos datos no son iguales.
*Causa*: se añadió un campo `list` sin `field(compare=False)`.
*Arreglo*: usa `field(compare=False)` para campos que no deben entrar en la
comparación, o usa `@dataclass(eq=True)` (que es el default) con cuidado.

## Caso de uso real

**F-CMP-01 — Comparación de cotizaciones.**
ACME Manufacturing recibe cotizaciones de 3 a 8 proveedores para cada
orden de compra. La clase `Quote` encapsula `unit_price`, `lead_time_days`,
`vendor: Vendor`. Los métodos determinísticos comparan precio y plazo; el
agente interviene cuando hay que ponderar historial de calidad, riesgo de
proveedor nuevo, o restricciones de presupuesto que no caben en una regla.

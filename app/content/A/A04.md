## Hilo conductor

En A03 construiste listas y dicts para modelar datos planos. Ahora empaquetas
comportamiento en funciones con contratos explícitos: type hints que el editor
y las herramientas leen, y excepciones que hablan en el idioma del dominio.
En A05 esas funciones vivirán en módulos separados; en B02 algunas se
convertirán en endpoints FastAPI. Desde ya, diseña cada función como si fuera
una API pública: firma precisa, un solo trabajo, errores nombrados.

## Idea central

Una **función** en Python es un objeto de primera clase: se puede pasar como
argumento, devolver desde otra función, almacenar en un dict. Su **firma** —
nombre + parámetros + tipo de retorno — es el contrato que el llamador acepta.
Los **type hints** (PEP 484+, Python 3.12) no son comentarios: los lee `mypy`,
`pyright`, Ruff, y el autocompletado de tu editor. Una **excepción propia**
(`class InvoiceMissingVendorError(ValueError)`) hace que el stack trace hable
del negocio, no de `KeyError: 'vendor_id'`.

_Excel: un PEP es como una propuesta formal al equipo de Microsoft para cambiar cómo funciona Excel — cuando la aprueban, se convierte en una nueva característica. PEP 484 propuso los type hints; hoy son parte oficial del lenguaje. Los type hints funcionan como la validación de tipo de celda en Excel ("esta columna solo acepta números enteros"): el editor avisa si pasas un string donde se espera un int._

## Por qué importa

Sin type hints, el agente de auditoría de ACME puede pasar un `float` donde se
esperaba `Decimal` y nadie lo detecta hasta producción. Sin excepciones propias,
el router de FastAPI captura `Exception` y devuelve 500 genérico en lugar de
422 con mensaje legible para el cliente.

El caso F-CTA-01 (clasificación contable) organiza su lógica exactamente así:
una función por tipo de regla contable, firma tipada, excepción específica si
la regla no aplica. El equipo puede testear cada regla de forma aislada sin
levantar la app entera.

## Cómo funciona por dentro

### Parámetros: posicionales, keyword y defaults

```python
def validate_invoice(
    amount: float,
    vendor_id: str,
    *,
    period: str = "2026-04",
    strict: bool = False,
) -> None: ...
```

El `*` marca la frontera: todo lo que viene después es **keyword-only**. El
llamador no puede pasar `"2026-04"` posicionalmente; tiene que escribir
`period="2026-04"`. Eso hace el código de llamada más legible y evita errores
de orden.

`*args` captura posicionales extras como tupla; `**kwargs` captura keyword
extras como dict. Úsalos cuando el número de argumentos es genuinamente
variable — no como escape hatch para no pensar la firma.

### El anti-patrón del default mutable

```python
# MALO: la misma lista es compartida entre todas las llamadas
def add_finding(msg: str, findings: list[str] = []) -> list[str]:
    findings.append(msg)
    return findings

# BUENO
def add_finding(msg: str, findings: list[str] | None = None) -> list[str]:
    if findings is None:
        findings = []
    findings.append(msg)
    return findings
```

El objeto `[]` en un default se crea **una sola vez** al definir la función,
no en cada llamada. Dos llamadas distintas comparten la misma lista. Este bug
aparece exactamente en sistemas que acumulan findings entre auditorías de
distintos tenants.

### Type hints en Python 3.12+

Python 3.12 usa la sintaxis de `X | Y` (unión) y `list[T]`/`dict[K, V]`
directamente. No uses `Optional[X]` ni `List[X]` — eso es Python ≤ 3.8.

```python
from decimal import Decimal

def compute_tax(amount: Decimal, rate: float) -> Decimal:
    """Calcula impuesto aplicando rate sobre amount."""
    return amount * Decimal(str(rate))
```

El `from __future__ import annotations` (usado en la app con Python 3.12) hace
las anotaciones lazy — no se evalúan en tiempo de importación. Es útil para
referencias hacia adelante.

### Excepciones: jerarquía y diseño

```python
class AuditError(Exception):
    """Raíz de errores del dominio de auditoría."""

class InvoiceMissingVendorError(AuditError):
    """La factura no tiene vendor_id."""
    def __init__(self, invoice_id: str) -> None:
        super().__init__(f"invoice {invoice_id} has no vendor_id")
        self.invoice_id = invoice_id
```

Herencia de `Exception` → clase raíz del dominio → clases concretas. El
router captura `AuditError` genérico para 422; la capa de servicio puede
capturar `InvoiceMissingVendorError` específico para manejar el caso concreto.

_Excel: la jerarquía de excepciones es como los niveles de alerta en Excel — primero el error genérico `#VALUE!`, luego el específico `#DIV/0!`. En Python, defines tu propio árbol: `AuditError` (genérico) → `InvoiceMissingVendorError` (específico). Puedes capturar el genérico para todos los errores de auditoría, o el específico para manejar solo el caso de vendor faltante._

### try / except / else / finally

```python
def parse_amount(raw: str) -> Decimal:
    try:
        return Decimal(raw.replace(",", "."))
    except Exception as exc:
        raise ValueError(f"invalid amount: {raw!r}") from exc
```

`from exc` encadena la excepción original. El `else` se ejecuta si el `try`
no lanzó; `finally` siempre. Nunca uses `except Exception: pass` — estás
silenciando errores que necesitas ver.

### Una función, un trabajo

Una función de 80 líneas que parsea, valida, guarda y envía email es
imposible de testear y de reutilizar. El refactor canónico: identifica los
verbos y parte.

## Ejemplo conducido

```python
from decimal import Decimal

class InvoiceError(ValueError):
    """Base para errores de factura."""

class VendorMissingError(InvoiceError):
    def __init__(self, invoice_id: str) -> None:
        super().__init__(f"invoice {invoice_id} missing vendor")
        self.invoice_id = invoice_id

def parse_amount(raw: str) -> Decimal:
    """Convierte string a Decimal; lanza ValueError si el formato es inválido."""
    try:
        return Decimal(raw.replace(",", "."))
    except Exception as exc:
        raise ValueError(f"cannot parse amount: {raw!r}") from exc

def validate_invoice(invoice_id: str, vendor_id: str | None, amount_raw: str) -> Decimal:
    """Valida una factura y devuelve el monto parseado."""
    if not vendor_id:
        raise VendorMissingError(invoice_id)
    return parse_amount(amount_raw)
```

Tres funciones, tres responsabilidades. `validate_invoice` se testea con un
mock de `parse_amount`; `parse_amount` tiene sus propios tests de formato.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|---|---|---|
| Validar que `vendor_id` no sea `None` | determinístico | Regla cerrada; la factura o tiene vendor o no. |
| Parsear monto de string con separadores locales | determinístico | Algoritmo fijo; los casos de error son enumerables. |
| Clasificar contablemente una factura con descripción ambigua | agéntico | La descripción varía por proveedor; no hay regla cerrada. |
| Decidir si un monto fuera de rango es error o caso especial | agéntico | Exige contexto del cliente ACME; cambia período a período. |

## Errores típicos

**1. Default mutable.**
*Síntoma*: una lista o dict en el default de un parámetro acumula valores entre
llamadas distintas.
*Causa*: el objeto mutable se crea una sola vez al definir la función.
*Arreglo*: usa `None` como default y crea el objeto dentro de la función.

**2. `except Exception: pass` (o `except: pass`).**
*Síntoma*: el código falla silenciosamente; logs vacíos; bug imposible de rastrear.
*Causa*: se capturan *todos* los errores incluyendo los inesperados.
*Arreglo*: captura solo la excepción que sabes manejar; re-lanza el resto.

**3. Función de 80 líneas con 5 responsabilidades.**
*Síntoma*: el test requiere un fixture enorme; un cambio de UI rompe la lógica.
*Causa*: no se separó parseo, validación, persistencia y notificación.
*Arreglo*: identifica los verbos, crea una función por verbo, compón.

**4. `Optional[X]` en lugar de `X | None`.**
*Síntoma*: `mypy` acepta el código pero el editor sugiere el estilo viejo.
*Causa*: `Optional` es Python ≤ 3.8; en 3.12+ es redundante.
*Arreglo*: `vendor_id: str | None`.

**5. `*args`/`**kwargs` como primer recurso.**
*Síntoma*: el lector de la firma no sabe qué recibe la función.
*Causa*: se usó variadic para evitar pensar la firma.
*Arreglo*: define los parámetros explícitos; usa variadic solo cuando el
número es genuinamente desconocido (decoradores, wrappers genéricos).

## Caso de uso real

**F-CTA-01 — Clasificación contable.**
Una función por tipo de regla (`classify_by_vendor`, `classify_by_amount_range`,
`classify_by_description_keyword`). Cada función lanza `ClassificationError` si
la regla no aplica. El orquestador las llama en orden y cae al agente solo si
todas fallan. El agente es el último recurso, no el primero.

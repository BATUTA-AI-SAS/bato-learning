## Hilo conductor

En A01 viste que Python compila tu archivo y lo ejecuta. Lo que ejecuta son operaciones sobre **objetos** — y en Python, todo es un objeto. A02 explica qué tipos de objetos existen, cuáles se pueden modificar en su lugar y cuáles no, y qué sucede cuando dos variables apuntan al mismo objeto. Ese modelo mental es obligatorio para A03 (donde los bucles crean y modifican listas) y para A04 (donde las funciones reciben objetos como parámetros y ese paso tiene consecuencias).

## Idea central

_Excel: en una hoja, la celda B3 que contiene `=A3` no copia el número — apunta a la celda A3. En Python, `b = a` hace exactamente eso: `b` apunta al mismo objeto que `a`. La función `id()` devuelve la "dirección de celda" interna donde Python guarda ese objeto._

En Python, una variable no *es* un valor: es una **referencia** a un objeto en memoria. El objeto tiene un tipo, un valor y un **id** (dirección en memoria). Cuando asignas `b = a`, no copias el valor: copias la referencia. Si el objeto es **mutable** (`list`, `dict`, `set`, instancias de clase), modificarlo a través de `b` también modifica lo que ves por `a`. Si es **inmutable** (`int`, `float`, `str`, `tuple`, `frozenset`, `bytes`), ninguna operación lo modifica en sitio — lo que parece «modificar» crea un objeto nuevo.

## Por qué importa

El bug más silencioso en servicios Python es el **aliasing no intencional**: dos partes del código comparten la misma lista o dict sin saberlo, y una de ellas la muta mientras la otra la lee.

En el sistema de auditoría de ACME Manufacturing, el servicio que procesa facturas recibe `invoices: list[dict]` y pasa ese objeto a tres funciones: `flag_duplicates`, `normalize_vendors`, y `compute_totals`. Si `normalize_vendors` muta la lista — eliminando facturas inválidas — `compute_totals` verá una lista diferente a la que recibió el servicio. El reporte final será incorrecto y el bug no levantará ningún error.

El caso **F-CTA-02 (auditoría de transacciones contables)** documenta exactamente este escenario: un pipeline que muta el dataset de entrada en una función intermedia produce totales inconsistentes que aparecen en el reporte mensual del CFO. La corrección siempre pasa por entender qué es mutabilidad y cuándo copiar.

> [!cuidado]
> El error más común en funciones que reciben listas como argumento por defecto: `def process(items=[])`. Python evalúa el valor por defecto **una sola vez** al definir la función. La misma lista se reutiliza en cada llamada sin argumento. Verás el error típico #2 de este módulo.

## Cómo funciona por dentro

### El modelo de objetos de CPython

CPython implementa cada objeto con una estructura en C que contiene, mínimo, tres campos:

- **ob_refcnt**: contador de referencias (cuántas variables apuntan a este objeto).
- **ob_type**: puntero al tipo del objeto (`int`, `list`, `str`…).
- **ob_val**: el valor en sí (representación interna).

Cuando asignas `invoice_ids = [1001, 1002, 1003]`, Python crea un objeto `list` en el heap con `ob_refcnt = 1`. Si después haces `pending = invoice_ids`, no crea una lista nueva: incrementa `ob_refcnt` a 2 y copia la referencia. Ahora dos nombres apuntan al mismo objeto.

```
invoice_ids ──┐
               └──▶  [list object id=0x7f3a] → [1001, 1002, 1003]
pending     ──┘
```

La función `id()` devuelve el identificador del objeto (su dirección en memoria en CPython). `is` compara ids; `==` compara valores.

_Excel: `is` es como preguntar "¿son exactamente la misma celda?" mientras que `==` es "¿tienen el mismo valor?". Dos celdas distintas pueden mostrar 100 (`==` True) pero no son la misma celda (`is` False)._

```python
invoice_ids = [1001, 1002, 1003]
pending = invoice_ids

print(id(invoice_ids) == id(pending))  # True: mismo objeto
print(invoice_ids is pending)          # True: misma referencia
print(invoice_ids == pending)          # True: mismo valor (también)
```

### Inmutabilidad real vs reasignación

«Inmutable» no significa que la variable no pueda apuntar a otro objeto. Significa que el **objeto** no puede cambiar. Un `str` no puede añadir caracteres; un `int` no puede cambiar su valor numérico. Cuando haces `total += 100`, Python crea un objeto `int` nuevo con el valor sumado y reasigna el nombre `total` a ese nuevo objeto. El objeto original queda con `ob_refcnt` decrementado y eventualmente es recolectado por el GC.

```python
amount: int = 4820
original_id = id(amount)
amount += 100
print(id(amount) == original_id)  # False: objeto nuevo
```

Un `tuple` es inmutable: no puedes añadir ni modificar sus elementos. Pero si uno de sus elementos *es* mutable, ese elemento interno sí puede cambiar:

```python
invoice: tuple[str, list[str]] = ("ACME-001", ["linea_1", "linea_2"])
invoice[1].append("linea_3")  # funciona: list interna es mutable
# invoice[1] = ["otra_lista"]  # TypeError: tuple no permite reasignación de elemento
```

### `is` vs `==` en la práctica

| Comparación | Qué comprueba | Cuándo usar |
|-------------|---------------|-------------|
| `a is b` | Mismo objeto en memoria (`id(a) == id(b)`) | Para comparar con `None`, `True`, `False`; para verificar que no se copió innecesariamente |
| `a == b` | Valor igual según `__eq__` del tipo | Para comparar contenidos: dos listas con los mismos elementos |
| `a is not b` | Distinto objeto | Para forzar que se hizo una copia real |

> [!cuidado]
> CPython cachea los enteros de -5 a 256. `a = 100; b = 100; a is b` devuelve `True` por ese caché, aunque `a` y `b` se asignaron por separado. No concluyas que todos los ints comparten identidad: fuera de ese rango, no.

### Cómo copiar sin compartir

| Necesitas | Cómo |
|-----------|------|
| Copia superficial de lista | `copy = original[:]` o `list(original)` |
| Copia superficial de dict | `copy = dict(original)` o `original.copy()` |
| Copia profunda (nested mutable) | `import copy; copy.deepcopy(original)` |
| Evitar mutación en función | Recibe `Sequence` o `tuple`, documenta que no muta |

La copia *superficial* copia el contenedor pero comparte los objetos internos. Si tienes `list[dict]`, copiar la lista con `[:]` crea una lista nueva pero los dicts internos siguen siendo los mismos objetos. Para un dataset de facturas donde cada factura es un dict, necesitas `copy.deepcopy`.

### Tipos del sistema completo

| Tipo | Mutable | Hashable | Notas |
|------|---------|----------|-------|
| `int` | No | Sí | Cacheo -5..256 en CPython |
| `float` | No | Sí | `0.1 + 0.2 != 0.3` — nunca compares con `==` |
| `str` | No | Sí | `+` crea un string nuevo |
| `bool` | No | Sí | `True is 1` devuelve `False`; `True == 1` devuelve `True` |
| `None` | No | Sí | Singleton; siempre compara con `is None` |
| `tuple` | No | Sí si sus elementos son hashables | Permite elementos mutables internos |
| `list` | **Sí** | No | No se puede usar como clave de dict |
| `dict` | **Sí** | No | Desde Python 3.7, mantiene orden de inserción |
| `set` | **Sí** | No | `frozenset` es la versión inmutable y hashable |
| `bytes` | No | Sí | `bytearray` es la versión mutable |

## Ejemplo conducido

El siguiente código modela cómo el bug de aliasing aparece en código de servicio real:

```python
from copy import deepcopy

# dataset de facturas del período 2026-04 de ACME
invoices: list[dict] = [
    {"id": "INV-001", "vendor": "acme_global", "amount": 4820.0, "valid": True},
    {"id": "INV-002", "vendor": "acme_global", "amount": 0.0,    "valid": False},
    {"id": "INV-003", "vendor": "andina_co",   "amount": 1500.0, "valid": True},
]

def filter_invalid(data: list[dict]) -> list[dict]:
    """Elimina facturas con monto cero. BUG: muta la lista original."""
    for item in data[:]:           # itera copia, pero borra del original
        if item["amount"] == 0.0:
            data.remove(item)      # ← muta 'data', que es el mismo objeto
    return data

def compute_total(data: list[dict]) -> float:
    """Suma montos de facturas."""
    return sum(item["amount"] for item in data)

# el bug: filter_invalid y compute_total reciben el mismo objeto
filtered = filter_invalid(invoices)
total    = compute_total(invoices)  # invoices ya tiene 2 elementos, no 3

print(f"facturas en 'invoices': {len(invoices)}")   # 2 — perdimos INV-002
print(f"facturas en 'filtered': {len(filtered)}")   # 2
print(f"total calculado: {total}")                  # 6320.0 (correcto aquí, pero...)
# si compute_total se llamara antes de filter_invalid, total sería 6320.0
# si se llamara después de filtrar con la lista original, debería ser igual
# el problema: el conteo de facturas procesadas está mal

# la corrección: no mutar el argumento
def filter_invalid_safe(data: list[dict]) -> list[dict]:
    """Devuelve lista nueva sin facturas inválidas."""
    return [item for item in data if item["amount"] != 0.0]
```

El ejemplo muestra que el bug es silencioso: `total` devuelve el número correcto, pero `invoices` queda corrupto para cualquier código que lo use después.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Copiar una lista (`list[:]`) | determinístico | Operación pura; mismo objeto → mismo resultado |
| Comparar igualdad de facturas por monto y fecha | determinístico | Regla cerrada: `a["amount"] == b["amount"]` |
| Detectar si dos facturas son duplicados reales o timing | agéntico | Requiere contexto: misma empresa, fechas próximas, descripciones similares — sin regla exacta |
| Decidir si una factura con `amount=0` es error o nota de crédito | agéntico | El ERP de ACME puede emitir ambas con el mismo formato; sin regla que lo distinga siempre |

## Errores típicos

**1. Aliasing no intencional.**
*Síntoma*: modificar `lista_b` modifica `lista_a` aunque no lo intentaste.
*Causa*: `lista_b = lista_a` copia la referencia, no el objeto.
*Arreglo*: `lista_b = lista_a[:]` para copia superficial, `copy.deepcopy(lista_a)` para copia profunda.

**2. Argumento mutable por defecto.**
*Síntoma*: la función acumula estado entre llamadas; la segunda llamada ve los resultados de la primera.
*Causa*:
```python
def add_finding(finding: str, results: list[str] = []) -> list[str]:
    results.append(finding)
    return results
# primera llamada: ['duplicado']
# segunda llamada: ['duplicado', 'monto_fuera_de_rango'] — lista compartida
```
*Arreglo*: usar `None` como default y crear la lista dentro:
```python
def add_finding(finding: str, results: list[str] | None = None) -> list[str]:
    if results is None:
        results = []
    results.append(finding)
    return results
```

**3. Comparar floats con `==`.**
*Síntoma*: `0.1 + 0.2 == 0.3` devuelve `False`.
*Causa*: representación en punto flotante IEEE 754; `0.1 + 0.2` es `0.30000000000000004`.

_Excel: también usa IEEE 754 internamente. Si en Excel sumas 0.1 diez veces y comparas con 1, en ocasiones el resultado no es exactamente 1 — la diferencia es invisible en pantalla pero existe en memoria. En Python es el mismo fenómeno, solo que más visible._
*Arreglo*: para montos financieros, usa `decimal.Decimal` o compara con tolerancia (`abs(a - b) < 1e-9`). El módulo C01 introduce `Decimal` cuando lleguemos a modelos ORM.

**4. Comparar con `==` en lugar de `is None`.**
*Síntoma*: en clases con `__eq__` custom, `obj == None` puede devolver `True` aunque `obj` no sea `None`.
*Causa*: `__eq__` es arbitrario. `is None` comprueba identidad con el singleton.
*Arreglo*: siempre `if value is None:` o `if value is not None:`. Nunca `== None`.

**5. Asumir que `tuple` es «totalmente» inmutable.**
*Síntoma*: una función recibe un `tuple[str, list[str]]` y el caller ve su lista interna modificada.
*Causa*: la tuple no permite reemplazar sus elementos, pero los elementos mutables se pueden modificar desde dentro.
*Arreglo*: si necesitas inmutabilidad profunda, usa `tuple[str, tuple[str, ...]]` o `frozenset`.

## Idea central

**`uv`** instala todo lo que tu carpeta de proyecto necesita, en su propia caja aislada. Con tres comandos tienes un proyecto limpio, reproducible, listo para compartir:

```bash
uv init acme-audit          # crea la carpeta con la configuración inicial
uv add anthropic            # añade la librería al proyecto
uv run python audit.py      # ejecuta tu script dentro de la caja
```

Internamente Python compila tu archivo a un formato intermedio antes de ejecutarlo — consulta *Para curiosos* más abajo si quieres ver ese proceso en detalle.

> [!nota]
> Si has visto los términos `pip`, `virtualenv` o `pip-tools` en tutoriales externos, aquí va una referencia rápida:
>
> | Herramienta | Para qué servía | Con `uv` |
> |---|---|---|
> | `pip` | Descargar e instalar librerías de PyPI | Reemplazado por `uv add` |
> | `virtualenv` | Crear una caja aislada para las librerías de un proyecto | Reemplazado por el `.venv/` que crea `uv` automáticamente |
> | `pip-tools` | Generar un archivo de versiones exactas a partir de una lista de dependencias | Reemplazado por `uv.lock` |
>
> No necesitas usar ninguno de ellos directamente — `uv` los reemplaza a todos.

**`pyproject.toml`** es la tarjeta de receta: lista qué librerías necesita tu proyecto y con qué versión mínima.

**`uv.lock`** es el recibo con las versiones exactas; garantiza que cualquier máquina instale exactamente lo mismo que tú tienes hoy.

<details>
<summary>Para curiosos: qué pasa entre tu .py y la ejecución — bytecode y CPython (opcional)</summary>

El ejecutable `python3` que instalaste en 00-08 es **CPython**: la implementación de referencia del lenguaje Python, escrita en C.

CPython no ejecuta el `.py` directamente. Lo convierte a **bytecode** — una representación compacta de instrucciones para la máquina virtual de Python — y luego ejecuta ese bytecode. El archivo `.pyc` resultante se guarda en `__pycache__/` para no repetir la compilación en la próxima ejecución.

El proceso tiene cuatro pasos internos: tokenización, parseo (genera un árbol de sintaxis abstracta, AST), compilación a bytecode, y ejecución. Los detalles de cada paso se cubren en la sección "Cómo funciona por dentro" más abajo.

</details>

## Puente desde Excel

Cuando guardas un libro `.xlsx` y lo vuelves a abrir, Excel **lee el archivo, lo interpreta y reconstruye las celdas en memoria**. Python hace lo mismo con un `.py`: lee el archivo, lo interpreta, y construye en memoria las variables, funciones y módulos. La diferencia: en Excel ves el resultado en celdas; en Python, escribes `print` para verlo.

| Excel | Python |
|-------|--------|
| Abrir `.xlsx` | `python audit.py` |
| Recálculo automático | Tu script corre línea a línea |
| Función `=BUSCARV` | Función `def buscar(lista, valor)` |
| Macros (.xlsm) | Scripts (.py) |
| Add-ins instalados | Paquetes en `pyproject.toml` |
| Versión de Excel (2019 vs 365) | Versión de Python (3.11 vs 3.12) |

Lo que `uv` es para Python, sería "instalador automático de Excel + cada add-in en la versión correcta, configurado igual en cada PC de tu equipo". Sin `uv`, cada quien instala como puede y los archivos se rompen en otra máquina.

## Por qué importa

Cuando el agente de auditoría mensual de ACME Manufacturing falla en el servidor del cliente pero funciona en tu máquina, la causa más frecuente no es el código del agente: es una discrepancia de entorno. Una versión de `anthropic` distinta cambia el comportamiento del streaming. Una versión de `pydantic` distinta rechaza un campo que antes toleraba. Sin un lockfile, cada `pip install` es una lotería.

El caso de uso **F-FIN-01 (conciliación bancaria)** ilustra esto en producción: el script de ingestión de Siigo corre mensualmente en el servidor del cliente. Si el `requirements.txt` no está pinado y alguien actualiza la máquina, el parseo del CSV puede cambiar silenciosamente. Un `uv.lock` commiteado elimina esa clase de incidente.

> [!cuidado]
> Si ves `pip install` en cualquier ejemplo de este curso, es un error. El comando correcto es siempre `uv add <paquete>` para añadir una dependencia, o `uv sync` para instalar lo que ya está en `pyproject.toml`.

## Cómo funciona por dentro

<details>
<summary>Para curiosos: el pipeline completo de CPython — AST, bytecode y __pycache__ (opcional)</summary>

Cuando ejecutas `uv run python audit.py`, el flujo tiene cuatro etapas:

**1. Tokenización y parseo.** CPython lee el texto y lo convierte a un árbol de sintaxis abstracta (AST). Este paso detecta `SyntaxError` — antes de ejecutar ni una línea.

**2. Compilación a bytecode.** El compilador recorre el AST y produce instrucciones de la máquina virtual. Puedes inspeccionarlas con `dis`:

```python
import dis

def compute_total(amounts: list[float]) -> float:
    return sum(amounts)

dis.dis(compute_total)
```

Las instrucciones (`RESUME`, `LOAD_GLOBAL`, `CALL`, `RETURN_VALUE`) son bytecode, no ensamblador real.

**3. Caché en `__pycache__/`.** El bytecode compilado se guarda como `audit.cpython-312.pyc`. En la próxima ejecución, si el `.py` no cambió, Python carga el `.pyc` directamente.

**4. Ejecución en la VM.** El intérprete ejecuta las instrucciones en un bucle. `LOAD_GLOBAL` empuja un nombre del namespace global. `CALL` desapila los argumentos, llama la función y empuja el resultado.

</details>

### `python audit.py` vs `python -m audit`

La diferencia importa cuando tu código está organizado en paquetes:

| Modo | Qué hace | `__name__` | `sys.path[0]` |
|------|----------|------------|---------------|
| `python audit.py` | Trata `audit.py` como path absoluto | `"__main__"` | Directorio del archivo |
| `python -m audit` | Busca `audit` como módulo en `sys.path` | `"__main__"` | Directorio actual |
| `python -m app.main` | Busca el paquete `app`, luego `main` | `"__main__"` | Directorio actual |

Cuando el agente está en `app/main.py` y haces `python app/main.py`, los imports relativos (`from .routers import chat`) fallan porque Python no sabe que `app` es un paquete. Con `python -m app.main`, sí funciona.

### `uv` y el modelo de entorno

`uv` gestiona **tres piezas**:

1. **`pyproject.toml`**: declara las dependencias. Es el contrato — lo que tu código requiere.
2. **`uv.lock`**: fija las versiones exactas. Es la garantía de reproducibilidad.
3. **`.venv/`**: el entorno virtual. Nunca lo commitas; se regenera con `uv sync`.

El comando `uv add anthropic` hace todo en una: actualiza `pyproject.toml`, resuelve versiones y actualiza `uv.lock`. El `pip install` tradicional no toca ningún archivo de proyecto.

<details>
<summary>Para profundizar: qué hay dentro de uv.lock y cómo funciona la resolución (opcional)</summary>

`uv.lock` contiene el snapshot exacto de cada dependencia y subdependencia con su hash de wheel. El algoritmo de resolución es SAT (satisfiability): dado el grafo de restricciones de versiones, encuentra el conjunto único que satisface todos los rangos. El resultado es determinístico: el mismo `pyproject.toml` siempre produce el mismo `uv.lock`. Los hashes de wheel garantizan que lo que instalas es bit a bit idéntico a lo que cualquier otro miembro del equipo instala — sin depender de que el servidor de PyPI tenga el mismo archivo.

</details>

> [!nota]
> `uv` también gestiona la versión de Python. El archivo `.python-version` en la raíz del repo declara `3.12`. Si en tu máquina tienes `3.11`, `uv` descarga e instala `3.12` automáticamente. Sin config extra.

## Ejemplo conducido

Este es el `pyproject.toml` real de la app `bato-learning` (versión simplificada para el ejemplo):

```toml
[project]
name = "bato-learning"
version = "0.1.0"
requires-python = ">=3.12"
dependencies = [
    "fastapi>=0.115",
    "anthropic>=0.40",
    "sqlalchemy[asyncio]>=2.0.36",
]

[tool.uv]
dev-dependencies = [
    "pytest>=8.3",
    "ruff>=0.7",
]
```

Para arrancar el proyecto desde cero:

```bash
uv sync                       # instala dependencias de pyproject.toml
uv run uvicorn app.main:app   # corre la app dentro del entorno
uv run pytest                 # corre tests dentro del entorno
uv add httpx                  # añade httpx, actualiza uv.lock
```

<details>
<summary>Para curiosos: inspeccionar el bytecode con dis (opcional)</summary>

Ejecuta esto en la REPL (`uv run python`):

```python
import dis

def flag_overdue(days_since_issue: int) -> bool:
    """Marca factura como vencida si supera 30 días."""
    return days_since_issue > 30

dis.dis(flag_overdue)
# RESUME          0
# LOAD_FAST       0 (days_since_issue)
# LOAD_CONST      1 (30)
# COMPARE_OP      4 (>)
# RETURN_VALUE
```

Python traduce `days_since_issue > 30` a cuatro instrucciones de bytecode. El número no importa para escribir código; lo que importa es que el proceso de compilación existe y puede fallar (con `SyntaxError`) antes de ejecutar una sola línea de tu lógica.

</details>

<details>
<summary>Para profundizar: determinístico vs agéntico en la gestión de entorno (opcional)</summary>

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Compilar `.py` a bytecode | determinístico | Mismo código → mismo bytecode siempre |
| Resolver el grafo de dependencias en `uv lock` | determinístico | Algoritmo SAT puro; mismo `pyproject.toml` → mismo `uv.lock` |
| Decidir si un `requirements.txt` legado es compatible con el proyecto | agéntico | Requiere leer contexto, documentación de breaking changes, historial de incidentes |
| Decidir si hacer `uv add --upgrade-package anthropic` en producción | agéntico | Implica evaluar changelogs, riesgos operativos, cobertura de tests del cliente |

La gestión de entorno es casi toda determinística. El único tramo agéntico real es la *toma de decisión sobre upgrades*, que exige contexto del negocio.

</details>

## Errores típicos

**1. Mezclar `pip install` con `uv`.**
*Síntoma*: `ModuleNotFoundError` al hacer `uv run python -c "import anthropic"` tras `pip install anthropic`.
*Arreglo*: `uv add anthropic` — siempre. `pip install` instala en el Python del sistema, no en el `.venv` del proyecto.

**2. Confundir `python audit.py` con `python -m audit`.**
*Síntoma*: `ImportError: attempted relative import with no known parent package`.
*Arreglo*: usa `python -m <paquete>.<módulo>` cuando hay imports relativos.

**3. Commitear `.venv/` o ignorar `uv.lock`.**
*Síntoma*: el CI instala versiones distintas a las que tienes local.
*Arreglo*: `.venv/` debe estar en `.gitignore`; `uv.lock` debe estar commiteado. No al revés.

<details>
<summary>Para profundizar: el error del __pycache__ con versión antigua (opcional)</summary>

**`__pycache__` con bytecode de una versión anterior de Python.**
*Síntoma*: el código cambia pero el comportamiento no.
*Causa*: si el sistema de archivos tiene resolución de tiempo gruesa (algunos Docker layers), Python puede usar el `.pyc` obsoleto.
*Arreglo*: `find . -name "*.pyc" -delete`. Raro en práctica; más común en CI con cachés agresivos.

</details>

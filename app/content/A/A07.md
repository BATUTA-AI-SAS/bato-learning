## Hilo conductor

En A06 definiste clases con estado y comportamiento. Ahora esas clases y
funciones se convierten en **coroutinas** que ceden el control mientras
esperan I/O. FastAPI (B02) y SQLAlchemy async (C01) asumen que tu código es
async; si llamas una función bloqueante dentro, congelas toda la app. Este
módulo te enseña a distinguir cuándo async ayuda, cuándo no, y cuáles son
los errores que nadie menciona hasta que el servicio se cae.

En A05 viste que `app/repos/modules.py` tiene `async def list_all(...)`.
Ahora entiendes por qué.

## Idea central

Una **coroutina** (`async def f()`) es una función que puede pausarse en
puntos `await` y ceder el control al **event loop**. El event loop es un
bucle que mantiene una cola de coroutinas listas para continuar; cuando una
espera I/O, el loop avanza otra. Un solo hilo ejecuta todo esto de forma
concurrente (no paralela). `asyncio.gather` lanza varias coroutinas a la vez
y espera a que todas terminen — si el I/O es la operación dominante, esto es
equivalente a ejecutarlas en paralelo.

## Por qué importa

El caso F-OPS-01 (planeación de producción para ACME) tiene que obtener datos
de tres fuentes distintas al inicio del workflow: inventario actual, órdenes
pendientes, y precios de materia prima. Si las tres llamadas son secuenciales
y cada una tarda 400 ms, el total es 1.2 s. Con `gather`, el total es ~400 ms.
El agente espera esos datos antes de calcular el plan — optimizar aquí tiene
impacto directo en la latencia del usuario.

Pero `asyncio` no ayuda si las operaciones son **CPU-bound** (calcular,
comprimir, procesar imágenes). En ese caso, el event loop sigue bloqueado.
El curso enseña la distinción para que no uses async como dogma.

## Cómo funciona por dentro

### La máquina de estados debajo de `async def`

```python
async def fetch_vendor(vendor_id: str) -> dict:
    await asyncio.sleep(0.1)   # simula I/O
    return {"id": vendor_id, "name": "ACME"}
```

`async def fetch_vendor` crea una función que, al llamarse, devuelve un
objeto **coroutine** — no ejecuta nada todavía. El objeto coroutine se
ejecuta cuando el event loop lo consume, normalmente vía `await` o `gather`.

```python
# esto NO ejecuta fetch_vendor:
coro = fetch_vendor("V-01")     # objeto coroutine

# esto SÍ ejecuta:
result = await fetch_vendor("V-01")
```

Si llamas `fetch_vendor("V-01")` sin `await` y no usas el objeto, Python
emite `RuntimeWarning: coroutine 'fetch_vendor' was never awaited`. Es un bug
silencioso: la función simplemente no corrió.

### `asyncio.gather` — I/O concurrente

```python
import asyncio

async def fetch_inventory(tenant: str) -> list[dict]:
    await asyncio.sleep(0.4)   # simula 400ms
    return [{"item": "acero", "qty": 500}]

async def fetch_orders(tenant: str) -> list[dict]:
    await asyncio.sleep(0.4)
    return [{"order_id": "ORD-1", "qty": 200}]

async def fetch_prices(tenant: str) -> dict:
    await asyncio.sleep(0.4)
    return {"acero": 1200.0}

async def collect_production_inputs(tenant: str) -> tuple:
    inventory, orders, prices = await asyncio.gather(
        fetch_inventory(tenant),
        fetch_orders(tenant),
        fetch_prices(tenant),
    )
    return inventory, orders, prices
```

Las tres coroutinas corren «en paralelo» dentro del mismo hilo. El loop las
intercala en sus puntos `await`. El tiempo total es ~400 ms, no 1200 ms.

### Blocking calls: el error que congela todo

```python
import requests   # librería SYNC

async def get_rate(currency: str) -> float:
    # MALO: requests.get bloquea el hilo entero
    r = requests.get(f"https://api.example.com/rate/{currency}")
    return r.json()["rate"]
```

`requests.get` es síncrono. Dentro de `async def`, el event loop no puede
avanzar mientras esta llamada espera. Con 100 requests concurrentes, los otros
99 se congela hasta que este termine. La alternativa es `httpx` con
`await client.get(...)`, o `aiohttp`.

Para código bloqueante que no tiene alternativa async (librerías legacy, cálculo
pesado), usa `asyncio.to_thread`:

```python
import asyncio

async def compute_forecast_async(data: list[dict]) -> dict:
    """Corre cálculo pesado en un thread para no bloquear el loop."""
    return await asyncio.to_thread(_compute_forecast, data)

def _compute_forecast(data: list[dict]) -> dict:
    # cálculo sincrónico intensivo
    ...
```

### aiosqlite: honestidad técnica

`aiosqlite` (el driver que usa esta app para SQLite) **no es async real**:
lanza un hilo de fondo por cada conexión y comunica via `asyncio.Queue`.
El API parece async (`async with session.begin()`, `await session.execute()`),
pero internamente hay un thread. Lo decimos honesto porque:

1. El límite de concurrencia lo pone el número de hilos, no el event loop.
2. En producción con PostgreSQL + `asyncpg` sí hay async real (sin hilo extra).
3. El patrón de código es idéntico — lo que aprendes aquí sirve igual en C03.

### `async with` y `async for`

Cuando el recurso a gestionar (sesión de DB, cliente HTTP) necesita apertura y
cierre async, usas `async with`:

```python
async with session.begin():
    result = await session.execute(stmt)
```

Cuando iteras sobre un stream async (eventos SSE, líneas de un archivo grande):

```python
async for chunk in stream.text_stream:
    yield StreamChunk(text=chunk)
```

Ese `async for` exacto está en `app/services/anthropic_client.py` — la función
`stream_chat` que alimenta el chat tutor de esta misma app.

### Cuándo NO usar async

| Caso | Por qué no async |
|---|---|
| Script de análisis batch offline | No hay concurrencia que ganar; complica el código. |
| Función puramente CPU-bound (ML inference, compresión) | async no ayuda; usa `ProcessPoolExecutor`. |
| CLI sencilla con una sola llamada a la API | `asyncio.run(main())` es el único overhead. No vale el coste cognitivo si la CLI no tiene concurrencia. |
| Función que llama solo funciones sync internas | `async def` sin ningún `await` es código sincrónico con overhead de coroutine. Mypy advierte. |

## Ejemplo conducido

```python
import asyncio
from dataclasses import dataclass

@dataclass
class ProductionInput:
    inventory: list[dict]
    pending_orders: list[dict]
    material_prices: dict[str, float]

async def _fetch(name: str, delay: float, value: object) -> object:
    """Simula llamada a API externa con delay."""
    await asyncio.sleep(delay)
    return value

async def collect_inputs(tenant: str) -> ProductionInput:
    inventory, orders, prices = await asyncio.gather(
        _fetch("inventory", 0.3, [{"item": "acero", "qty": 500}]),
        _fetch("orders",    0.2, [{"order_id": "ORD-1", "qty": 200}]),
        _fetch("prices",    0.4, {"acero": 1200.0}),
    )
    return ProductionInput(inventory, orders, prices)

result = asyncio.run(collect_inputs("acme"))
print(result.material_prices)   # {'acero': 1200.0}
```

El tiempo total es ~0.4 s (el delay mayor), no 0.9 s.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|---|---|---|
| Paralelizar fetch de inventario, órdenes y precios con `gather` | determinístico | La estructura del gather y el orden de los resultados son fijos. |
| Manejar timeout de una coroutina con `asyncio.wait_for` | determinístico | La regla de timeout es exacta: N segundos. |
| Decidir qué fuentes de datos consultar para un tenant específico | agéntico | Depende de la configuración del cliente, que varía y no cabe en regla fija. |
| Interpretar un resultado ambiguo de una API de inventario (campo `"qty"` puede ser unidades o kg según cliente) | agéntico | Sin esquema normalizado por cliente, requiere razonamiento contextual. |

## Errores típicos

**1. Llamar función sync bloqueante dentro de `async def`.**
*Síntoma*: la app se congela intermitentemente; la latencia sube bajo carga.
*Causa*: `requests.get`, `time.sleep`, operación CPU pesada dentro de async.
*Arreglo*: usa `httpx` async para HTTP, `asyncio.sleep` para esperas,
`asyncio.to_thread` para código sync que no puedes cambiar.

**2. Olvidar `await`.**
*Síntoma*: la función devuelve un objeto `coroutine` en lugar del resultado
esperado; Python emite `RuntimeWarning: coroutine 'X' was never awaited`.
*Causa*: se llamó `result = fetch_vendor("V-01")` sin `await`.
*Arreglo*: `result = await fetch_vendor("V-01")`.

**3. `asyncio.run()` dentro de una función async.**
*Síntoma*: `RuntimeError: This event loop is already running`.
*Causa*: se intentó crear un nuevo loop dentro de uno ya corriendo.
*Arreglo*: usa `await` directamente. `asyncio.run()` solo va en el punto de
entrada del programa (o en tests con `@pytest.mark.asyncio`).

**4. `asyncio.gather` sin manejar excepciones.**
*Síntoma*: si una de las coroutinas lanza, las demás pueden quedar sin
cancelar y `gather` relanza solo la primera excepción.
*Arreglo*: usa `return_exceptions=True` para capturar todas, o estructura
el manejo de error explícitamente.

**5. Asumir que `async def` = paralelo real.**
*Síntoma*: código CPU-bound en async no mejora la latencia.
*Causa*: confundir concurrencia (async) con paralelismo (multiprocessing).
*Arreglo*: `asyncio.to_thread` para bloqueante, `ProcessPoolExecutor` para CPU.

## Caso de uso real

**F-OPS-01 — Planeación de producción.**
ACME Manufacturing consulta tres fuentes antes de que el agente calcule el
plan: inventario del ERP, órdenes confirmadas del CRM, y precios de materia
prima de una API externa. Con `asyncio.gather` las tres llamadas corren
en paralelo. El agente recibe todos los datos en el tiempo de la llamada más
lenta (~400 ms) en lugar de la suma (~1.2 s). El tramo de planeación en sí
(qué producir, cuánto, cuándo) es agéntico cuando el modelo necesita priorizar
con restricciones cambiantes.

## Hilo conductor

En A02 aprendiste que los contenedores mutables — `list`, `dict`, `set` — pueden modificarse en sitio y que asignar uno a otro nombre crea una referencia compartida, no una copia. A03 enseña las herramientas principales para *construir* y *recorrer* esos contenedores: `if`, `for`, `while`, y las comprensiones. Las comprensiones no son azúcar sintáctico trivial: crean objetos nuevos, lo que evita el aliasing de A02 por construcción. A03 prepara directamente a A04, donde las funciones recibirán y devolverán los contenedores que aquí construyes.

## Idea central

Python tiene tres estructuras de control de flujo: `if/elif/else` (condicional), `for` (iteración sobre iterable), y `while` (iteración con condición). Sobre ellas se construyen las **comprensiones**: expresiones que crean un contenedor nuevo (`list`, `dict`, `set`) o un iterador perezoso (generador) en una sola línea. Una comprensión `[expr for x in iterable if condición]` es equivalente a un `for` con `append`, pero más rápida (el bytecode es distinto y más eficiente) y siempre devuelve un objeto nuevo.

## Por qué importa

La diferencia entre filtrar una lista con un bucle que la muta y una comprensión que crea una nueva es exactamente el bug de A02 aplicado al control de flujo. Toda agregación, filtrado y transformación de datos en un pipeline de auditoría se escribe con estas herramientas.

El caso **F-CTA-02 (auditoría de transacciones contables)** tiene un pipeline de tres pasos: (1) filtrar transacciones por período, (2) agrupar por cuenta contable, (3) identificar cuentas con saldo fuera de rango. Los tres pasos se implementan con comprensiones y un `while` para el reintento cuando la fuente de datos es inestable. Sin comprensiones, el código triplica de longitud y multiplica los lugares donde puede aparecer un bug de mutación.

> [!cuidado]
> Una comprensión que no cabe en una línea sin scroll horizontal es demasiado compleja. Si tienes un `if` anidado dentro del `for` y otro `if` externo como filtro, usa un bucle `for` explícito con una función interna nombrada. La regla: si tardas más de dos segundos en leer la comprensión, usa el bucle.

## Cómo funciona por dentro

### `if/elif/else`

Python evalúa las condiciones de arriba hacia abajo y ejecuta el primer bloque cuya condición sea verdadera. No hay `switch/case` clásico, pero desde Python 3.10 existe `match/case` — que A03 menciona pero no profundiza, porque el caso de uso principal (pattern matching sobre objetos) no aparece hasta A06.

La **truthiness** de un objeto determina si se considera `True` en un contexto booleano:

| Valor | Truthiness |
|-------|-----------|
| `0`, `0.0`, `""`, `[]`, `{}`, `set()`, `None` | Falsy |
| Cualquier número no cero, string no vacío, contenedor no vacío | Truthy |
| Objetos custom sin `__bool__` ni `__len__` | Truthy |

```python
pending_invoices: list[dict] = []

# antipatrón: comparar con len
if len(pending_invoices) == 0:
    print("sin facturas")

# idiomático
if not pending_invoices:
    print("sin facturas")
```

### `for` y la iteración en Python

`for x in iterable` no requiere índice. El objeto iterable implementa `__iter__` que devuelve un iterador, y el `for` llama `__next__` hasta que recibe `StopIteration`. Esto funciona con listas, strings, dicts (itera claves), archivos (itera líneas), y cualquier objeto con `__iter__`.

_Excel: un iterador es como la función `DESREF` avanzando fila a fila: cada llamada a `__next__` entrega la siguiente celda sin necesitar tener toda la columna cargada. A diferencia de un rango fijo `A1:A100`, el iterador no sabe cuántos elementos hay — solo sabe dar el siguiente._

```python
invoices: list[dict] = [
    {"id": "INV-001", "amount": 4820.0},
    {"id": "INV-002", "amount": 1500.0},
]

# itera dicts — obtiene claves
for key in invoices[0]:
    print(key)  # "id", luego "amount"

# itera items
for key, value in invoices[0].items():
    print(f"{key}: {value}")
```

Cuando necesitas índice y valor: `enumerate(iterable)` — nunca `range(len(iterable))`.

```python
for i, invoice in enumerate(invoices, start=1):
    print(f"#{i} — {invoice['id']}")
```

`break` sale del bucle más interno inmediatamente. `continue` salta a la siguiente iteración. En ambos casos, el bloque `else` del `for` se ejecuta solo si el bucle terminó **sin** `break`.

```python
# buscar el primer proveedor nuevo en una lista de facturas
NEW_VENDORS: set[str] = {"andina_co", "expreslog_mx"}

for invoice in invoices:
    if invoice["vendor"] in NEW_VENDORS:
        first_new = invoice
        break
else:
    first_new = None  # no se encontró ninguno
```

### `while`

Útil cuando el número de iteraciones no se conoce de antemano: reintentos, polling de un estado, lectura de stream.

```python
MAX_RETRIES: int = 3
attempts: int = 0
fetched: bool = False

while attempts < MAX_RETRIES and not fetched:
    attempts += 1
    # simula fetch con probabilidad de fallo
    fetched = attempt_fetch_from_erp(tenant="acme", period="2026-04")

if not fetched:
    raise RuntimeError(f"ERP no respondió tras {MAX_RETRIES} intentos")
```

> [!cuidado]
> Un `while True` sin condición de escape en el cuerpo es un bucle infinito. Siempre tiene un `break` o una condición de salida explícita, y un límite de iteraciones como guardia.

### Comprensiones: la forma canónica

Una **comprensión de lista** crea una lista nueva, nunca muta la original:

```python
# bucle equivalente (más verboso, mismo resultado)
valid: list[dict] = []
for inv in invoices:
    if inv["amount"] > 0:
        valid.append(inv)

# comprensión (idiomática, crea lista nueva)
valid: list[dict] = [inv for inv in invoices if inv["amount"] > 0]
```

Una **comprensión de dict** crea un dict nuevo a partir de pares `clave: valor`:

```python
# índice de facturas por id
invoice_by_id: dict[str, dict] = {inv["id"]: inv for inv in invoices}
# uso: invoice_by_id["INV-001"]["amount"]
```

Una **comprensión de set** crea un conjunto (sin duplicados):

```python
# vendors únicos en el dataset
vendors: set[str] = {inv["vendor"] for inv in invoices}
```

Una **expresión generadora** es como una comprensión de lista pero perezosa: no crea la lista en memoria, calcula cada elemento bajo demanda. Útil cuando solo necesitas el resultado de una función de agregación.

_Excel: una generadora es como una fórmula que calcula el valor de una celda solo cuando esa celda es visible en pantalla. En cambio, una comprensión de lista es como calcular toda la columna auxiliar de golpe aunque solo vayas a usar el total final. La evaluación lazy ahorra memoria cuando el dataset es grande._

```python
# sin generadora: crea lista intermedia de floats
total_with_list = sum([inv["amount"] for inv in invoices])

# con generadora: no crea lista intermedia
total_with_gen = sum(inv["amount"] for inv in invoices)
```

Para datasets pequeños (< 10 000 elementos), la diferencia de memoria es despreciable. Para datasets grandes o streaming, la generadora es la única opción correcta.

### Cuándo NO usar comprensión

| Situación | Usar |
|-----------|------|
| Transformación simple con filtro opcional | comprensión |
| Dos o más operaciones sobre el mismo elemento | bucle `for` + función auxiliar |
| Efecto lateral (escribir a DB, loguear) | bucle `for` explícito |
| Lógica condicional con más de un `if` anidado | bucle `for` + función auxiliar |
| Número de iteraciones no determinado a priori | `while` |

La última regla en la tabla es la más ignorada: una comprensión con `if/else` dentro del cuerpo *y* un `if` al final como filtro es legal en Python pero ilegible en revisión de código.

```python
# esto compila y funciona, pero es un antipatrón
tags = [
    "duplicado" if inv["amount"] == prev_amount else "nuevo"
    for inv in invoices
    if inv["vendor"] != "interno"
]

# versión legible: extrae la lógica
def classify_invoice(inv: dict, prev_amount: float) -> str:
    if inv["amount"] == prev_amount:
        return "duplicado"
    return "nuevo"

tags = [
    classify_invoice(inv, prev_amount)
    for inv in invoices
    if inv["vendor"] != "interno"
]
```

## Ejemplo conducido

Pipeline completo de auditoría de período para ACME:

```python
from datetime import date

invoices: list[dict] = [
    {"id": "INV-001", "vendor": "acme_global",  "amount": 4820.0, "date": "2026-04-05"},
    {"id": "INV-002", "vendor": "andina_co",    "amount": 0.0,    "date": "2026-04-12"},
    {"id": "INV-003", "vendor": "acme_global",  "amount": 4820.0, "date": "2026-04-19"},
    {"id": "INV-004", "vendor": "expreslog_mx", "amount": 3100.0, "date": "2026-04-22"},
]

# paso 1: filtrar válidas (monto > 0)
valid: list[dict] = [inv for inv in invoices if inv["amount"] > 0.0]

# paso 2: agrupar montos por vendor
totals_by_vendor: dict[str, float] = {}
for inv in valid:
    vendor = inv["vendor"]
    totals_by_vendor[vendor] = totals_by_vendor.get(vendor, 0.0) + inv["amount"]

# paso 3: detectar duplicados (mismo vendor, mismo monto)
seen: set[tuple[str, float]] = set()
duplicates: list[dict] = []
for inv in valid:
    key = (inv["vendor"], inv["amount"])
    if key in seen:
        duplicates.append(inv)
    else:
        seen.add(key)

print(f"válidas: {len(valid)}")
print(f"totales: {totals_by_vendor}")
print(f"duplicados: {[d['id'] for d in duplicates]}")
```

Salida esperada:
```
válidas: 3
totales: {'acme_global': 9640.0, 'expreslog_mx': 3100.0}
duplicados: ['INV-003']
```

Nota el paso 3: usa un `set` de tuplas como índice de «ya visto». La detección de duplicados exactos es determinística — misma regla, mismo resultado siempre. La detección de duplicados «aproximados» (mismo vendor, montos similares pero no idénticos) sería un tramo agéntico.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Filtrar facturas por monto `> 0` | determinístico | Regla numérica cerrada |
| Agrupar totales por `vendor` (string exacto) | determinístico | Igualdad de string; no hay ambigüedad |
| Detectar duplicado exacto (mismo vendor + monto) | determinístico | Regla cerrada; `set` de tuplas |
| Detectar duplicado aproximado («ACME GLOBAL» vs «acme_global») | agéntico | Variabilidad de formato; sin regla que cubra todos los casos |
| Clasificar si un duplicado es error o timing (dos pagos legítimos) | agéntico | Requiere contexto del negocio del cliente |
| Generar el texto del hallazgo en el reporte | agéntico | Salida para humano, requiere redacción contextual |

## Errores típicos

**1. Comprensión anidada ilegible.**
*Síntoma*: una comprensión con dos `for` y un `if` que nadie entiende en revisión.
*Causa*: intentar hacer demasiado en una sola expresión.
*Arreglo*: extrae la lógica interna a una función con nombre. La comprensión queda como `[f(x) for x in data if condición]`.

**2. Mutar la lista que estás iterando.**
*Síntoma*: el bucle se salta elementos o se comporta de forma errática.
*Causa*:
```python
for inv in invoices:
    if inv["amount"] == 0.0:
        invoices.remove(inv)  # muta la lista mientras la itera
```
*Arreglo*: nunca mutes la lista que iteras. Itera sobre una copia (`invoices[:]`) o construye una lista nueva con comprensión.

**3. Usar `range(len(lista))` cuando necesitas índice y valor.**
*Síntoma*: `for i in range(len(invoices)): inv = invoices[i]`.
*Causa*: hábito de otros lenguajes.
*Arreglo*: `for i, inv in enumerate(invoices):`. Más idiomático, menos propenso a errores de off-by-one.

**4. Confundir comprensión de lista con expresión generadora en contextos de mutación.**
*Síntoma*: `gen = (x for x in invoices)` y luego añadirle elementos — no funciona; los generadores son de un solo uso.
*Causa*: confundir el tipo `generator` (iterable exhausible) con `list` (reiniciable, indexable).
*Arreglo*: si necesitas iterar más de una vez o acceder por índice, usa comprensión de lista `[...]` no generador `(...)`.

**5. `while` sin guardia de iteraciones máximas.**
*Síntoma*: bucle infinito cuando el ERP no responde o la condición de salida tiene un bug.
*Causa*: `while not success:` sin límite.
*Arreglo*: añade `attempts` con un máximo y levanta excepción si se supera. Todo bucle `while` en producción tiene timeout.

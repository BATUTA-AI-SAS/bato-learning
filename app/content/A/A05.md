vivir en archivos separados y necesitan encontrarse entre sí. Esto es lo que
hace el sistema de imports de Python. Al terminar este módulo, puedes leer la

En A06 usarás esa estructura para entender por qué las clases van en `models/`.
En B04 la arquitectura de capas cobra su forma definitiva.

## Idea central

Un **módulo** es cualquier archivo `.py`. Un **paquete** es una carpeta con un
`__init__.py` dentro. Cuando escribes `from app.services import content_loader`,
Python busca `app/` (paquete), dentro `services/` (subpaquete), dentro
`content_loader.py` (módulo). Ejecuta ese archivo una vez y cachea el resultado
en `sys.modules`. La segunda vez que alguien importa el mismo módulo, Python
devuelve el objeto cacheado sin volver a ejecutar nada.

_Excel: `sys.path` es la lista de carpetas donde Python busca módulos, como la lista de rutas de red donde Excel busca libros vinculados. Si el libro no está en ninguna de esas rutas, aparece un error de referencia. `sys.modules` es el caché de módulos ya importados — como la memoria de Excel que evita reabrir un libro que ya está abierto. Un namespace es como una pestaña de Excel: las variables dentro de un módulo viven en su propio espacio y no colisionan con las de otro módulo aunque se llamen igual._

## Por qué importa

Sin entender imports no puedes leer ninguna capa de un servicio real. Los
errores más frecuentes al empezar a construir agentes son `ModuleNotFoundError`
(buscas en el lugar equivocado), `ImportError` (el nombre no existe en el
módulo), y el ciclo de imports (A importa B, B importa A, Python se atasca).

El caso F-FIN-01 (conciliación bancaria) estructura su lógica en un paquete
`reconciliation/` con módulos `parser.py`, `matcher.py`, `reporter.py`. Sin
saber cómo organizar eso, el código queda como un script de 400 líneas
imposible de testear.

## Cómo funciona por dentro

### Un módulo es un archivo

```text
audit.py        → módulo `audit`
services/       → paquete `services` (si tiene __init__.py)
  __init__.py
  content_loader.py  → módulo `services.content_loader`
```

Cuando ejecutas `import audit`, Python busca `audit.py` en cada directorio de
`sys.path`. `sys.path` arranca con el directorio del script que lanzaste (o el
CWD si usas `-m`), seguido de las rutas de la instalación de Python.

### `__init__.py`: qué hace y qué no hace

`__init__.py` le dice a Python que la carpeta es un paquete importable. Puede
estar vacío — su presencia es lo que importa. Pero también puede reexportar
símbolos para simplificar el import del consumidor:

```python
# app/models/__init__.py  (código real de esta app)
from .chat import ChatMessage, ChatSession        # noqa: F401
from .content import Exercise, Module, Quiz       # noqa: F401
from .progress import ExerciseAttempt, ModuleVisit  # noqa: F401
from .user import User                            # noqa: F401
```

Con esto, cualquier capa puede escribir `from app.models import Module` en
lugar de `from app.models.content import Module`. El `__init__.py` actúa como
fachada del paquete.

### Import absoluto vs relativo

**Absoluto**: `from app.services.content_loader import load_all`. Funciona
desde cualquier punto; es el estilo preferido y el que usa toda esta app.

**Relativo**: `from ..db import Base`. El `..` sube un nivel. Se usa dentro de
un paquete para referirse a sibling modules sin hardcodear el nombre del
paquete raíz. Si mueves el paquete, los imports relativos siguen funcionando;
los absolutos se rompen.

En `app/models/content.py` (código real):

```python
from ..db import Base   # sube a app/, entra a db.py
```

El `from __future__ import annotations` que ves en todos los archivos de la
app hace las anotaciones de tipo lazy — no ejecuta la expresión del tipo en
tiempo de import. Evita problemas de forward references.

### `python -m app` vs `python app/main.py`

`python app/main.py` añade `app/` al frente de `sys.path`. Si `main.py` hace
`from .routers import chat`, el import relativo falla — Python no sabe que
`app/` es un paquete porque lo trata como directorio de scripts, no como
paquete.

`python -m app` (o `uvicorn app.main:app`) añade el **directorio padre** de
`app/` al frente de `sys.path`. Ahora `app` es un paquete importable y todos
los imports relativos funcionan.

```
bato-learning/       ← esto entra en sys.path con python -m
  app/
    __init__.py
    main.py
```

### `__name__ == "__main__"`

Cuando ejecutas `python audit.py` directamente, la variable especial
`__name__` del módulo vale `"__main__"`. Cuando lo importas desde otro módulo,
vale `"audit"`. Por eso el idioma canónico:

```python
if __name__ == "__main__":
    import asyncio
    asyncio.run(main())
```

Si no usas este guard y pones código de ejecución en el nivel del módulo, ese
código corre al hacer `import audit`, lo que raramente quieres.

### Ciclos de import y cómo romperlos

Un ciclo ocurre cuando A importa B y B importa A. Python detecta el ciclo pero
lo maneja de forma parcial — el módulo en construcción existe en `sys.modules`
pero sin todos sus atributos aún. El resultado es `ImportError: cannot import
name 'X' from partially initialized module 'Y'`.

```text
# Ciclo clásico en capas:
routers/chat.py  → from app.services.chat import ChatService
services/chat.py → from app.routers.chat import some_helper  ← ERROR
```

La solución no es mover el import al interior de la función (workaround frágil).
La solución es **dependency injection**: el router recibe `ChatService` como
parámetro (o vía `Depends` de FastAPI), no lo importa directamente. El servicio
nunca importa el router.

### src-layout: por qué el repo usa raíz directa

Muchos proyectos Python modernos usan un layout con una carpeta `src/`:

```
src/
  mypackage/
    __init__.py
```

Esta app usa raíz directa (`app/` en la raíz del repo). Funciona porque
`uvicorn app.main:app` y `uv run` manejan el `sys.path` correctamente.
El src-layout tiene ventajas (evita imports accidentales del directorio de
trabajo), pero añade fricción en el setup. Si ves un proyecto con `src/`,
ahora sabes qué hace y por qué.

### La estructura real de `app/`

```text
app/
  __init__.py            ← "bato-learning 0.1.0"
  main.py                ← punto de entrada ASGI; monta routers
  db.py                  ← sesión AsyncSession + Base declarativa
  settings.py            ← configuración vía pydantic-settings
  models/
    __init__.py          ← reexporta todos los modelos (fachada)
    content.py           ← Module, Exercise, Quiz, QuizOption
    progress.py          ← ModuleVisit, ExerciseAttempt, QuizAnswer
    chat.py              ← ChatSession, ChatMessage
    user.py              ← User
  routers/
    __init__.py          ← solo docstring: "HTTP entry points"
    pages.py, modules.py, exercises.py, quizzes.py, chat.py, progress.py
  services/
    __init__.py          ← solo docstring: "Business logic"
    content_loader.py, exercise_runner.py, anthropic_client.py, ...
  repos/
    __init__.py          ← solo docstring: "Data-access layer"
    modules.py, chat.py, progress.py
```

La regla de dependencias es estricta y va en una sola dirección:

```
routers → services → repos → models
```

`routers` nunca importa desde `repos` directamente. `models` nunca importa
desde `services`. Si ves ese patrón en código de producción, es una deuda
técnica que eventualmente produce un ciclo de import.

## Ejemplo conducido

```python
# vendor_check/
#   __init__.py
#   rules.py
#   reporter.py

# rules.py
def is_known_vendor(vendor_id: str, approved: set[str]) -> bool:
    """Devuelve True si vendor_id está en la lista aprobada."""
    return vendor_id in approved

# reporter.py
from .rules import is_known_vendor     # import relativo dentro del paquete

def build_report(invoices: list[dict], approved: set[str]) -> list[str]:
    """Devuelve lista de vendor_ids desconocidos."""
    return [
        inv["vendor_id"]
        for inv in invoices
        if not is_known_vendor(inv["vendor_id"], approved)
    ]

# __init__.py — fachada
from .reporter import build_report     # el consumidor solo necesita esto
```

El consumidor escribe `from vendor_check import build_report`. No necesita
saber que `build_report` vive en `reporter.py` — el `__init__.py` lo expone.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|---|---|---|
| Resolver qué archivo corresponde a `from app.services import content_loader` | determinístico | El algoritmo de búsqueda en `sys.path` es fijo y determinístico. |
| Validar que el paquete no tenga ciclos de import | determinístico | Es análisis estático; herramientas como `pydeps` lo hacen sin LLM. |
| Decidir qué módulos pertenecen a qué capa en un nuevo proyecto | agéntico | Depende del dominio, del equipo, de convenciones que cambian. El LLM puede sugerir; el arquitecto decide. |
| Elegir si usar import absoluto o relativo en una base de código nueva | agéntico | Hay criterios, pero el trade-off cambia según estructura del repo y preferencias del equipo. |

## Errores típicos

**1. `ModuleNotFoundError: No module named 'app'`.**
*Síntoma*: ejecutas `python app/main.py` y el primer import falla.
*Causa*: `python app/main.py` añade `app/` a `sys.path`, no la raíz del repo.
Los imports absolutos `from app.xxx` no funcionan desde ahí.
*Arreglo*: usa `uvicorn app.main:app` o `python -m app.main` desde la raíz.

**2. `ImportError: cannot import name 'X' from partially initialized module 'Y'`.**
*Síntoma*: el error aparece en imports que parecen correctos, normalmente al
arrancar la app.
*Causa*: ciclo de import. Dos módulos se importan mutuamente.
*Arreglo*: mueve el código compartido a un tercer módulo sin dependencias, o
usa inyección de dependencias para romper el ciclo.

**3. Código de ejecución en nivel de módulo sin `if __name__ == "__main__"`.**
*Síntoma*: importar el módulo tiene efectos secundarios (abre conexiones,
imprime, ejecuta queries).
*Causa*: se puso código ejecutable fuera de funciones y sin el guard.
*Arreglo*: todo código que no debe ejecutarse al importar va dentro de
funciones o bajo el guard.

**4. `from app.models.content import Module` cuando `__init__.py` ya lo reexporta.**
*Síntoma*: import muy largo; si `content.py` se renombra, hay que cambiar
imports en toda la app.
*Causa*: se ignora la fachada del paquete.
*Arreglo*: `from app.models import Module` — el `__init__.py` actúa de
intermediario.

**5. `from .utils import helper` en un script que se ejecuta directo.**
*Síntoma*: `ImportError: attempted relative import in non-package`.
*Causa*: los imports relativos solo funcionan cuando el archivo es parte de un
paquete importado, no cuando es el script de entrada.
*Arreglo*: usa import absoluto en el punto de entrada, o usa `python -m`.

## Para profundizar

> [!profundizar]
> `app/main.py` muestra el patrón de wiring completo: importa routers con
> `from .routers import chat, exercises, health, pages, progress, quizzes`
> (import relativo desde el paquete `app`), y los monta con
> `app.include_router(chat.router)`. Eso cierra el círculo: el router es un
> objeto definido en `routers/chat.py` y montado en `main.py`.
>
> Compara con `app/models/__init__.py`: reexporta todos los modelos para que
> cualquier capa pueda hacer `from app.models import Module` sin saber en qué
> archivo vive la clase.

## Caso de uso real

**F-FIN-01 — Conciliación bancaria.**
El paquete `reconciliation/` tiene `parser.py` (carga extracto), `matcher.py`
(match exacto + fuzzy), `reporter.py` (genera hallazgos). Cada módulo importa
solo del que está debajo en la jerarquía de dependencias. El runner de Temporal
importa solo la función de entrada del paquete vía `__init__.py`. Cuando el
equipo añade un banco nuevo, toca `parser.py` sin modificar `matcher.py` ni
`reporter.py`.

## Hilo conductor

La REPL es útil para explorar, pero cada vez que cierras la sesión pierdes todo lo que escribiste. Un archivo `.py` guarda el código de forma permanente: puedes ejecutarlo mañana, pasarle parámetros, programarlo para que corra a las 3 AM, y compartirlo con el equipo de ACME. Este módulo crea tu primer script real y lo ejecuta con `uv run python`.

## Puente desde Excel

Un libro Excel `.xlsx` guarda fórmulas y datos que se recalculan cada vez que lo abres. Un script `.py` guarda código que se ejecuta de arriba abajo cada vez que lo llamas desde la terminal. Las diferencias importantes: el script no tiene interfaz visual, el resultado va a la terminal (no a celdas), y puedes ejecutarlo sin abrir nada, incluso desde otro programa o un servidor remoto.

## Idea central

Un **script** es un archivo de texto con extensión `.py` que Python ejecuta línea a línea, de arriba abajo, una vez. No hay celdas ni recálculo automático: el script termina cuando llega a la última línea (o cuando encuentra un error). La función `print()` es la que envía texto a la terminal — sin ella, ningún valor se muestra. Los **comentarios** empiezan con `#`; Python los ignora completamente.

## Por qué importa

Un script puede hacer cosas que la REPL no puede:

- **Reutilizable**: lo guardas, lo ejecutas cuantas veces quieras con los mismos resultados.
- **Automatizable**: un servidor puede ejecutarlo cada noche sin intervención humana.
- **Versionable**: puedes ver qué cambió entre la versión de ayer y la de hoy con Git.
- **Compartible**: mandas el archivo al equipo y ellos obtienen los mismos resultados.

La REPL es para probar una expresión antes de escribirla en el script. El script es donde vive el código real.

Cada script de auditoría de ACME que aparece en Tracks A–E es un archivo `.py`. Los agentes que automatizan la reconciliación bancaria son archivos `.py`. El hábito de «esto va en un archivo» versus «esto lo pruebo en la REPL» es fundamental desde ahora.

## Cómo funciona por dentro

### Qué pasa cuando ejecutas `uv run python audit.py`

1. `uv` activa el entorno virtual del proyecto (`.venv/`).
2. `uv` llama al ejecutable `python` de ese entorno con el archivo `audit.py` como argumento.
3. Python lee el archivo de texto completo y lo ejecuta instrucción por instrucción.
4. Cada `print()` que encuentra envía texto a **stdout** (la salida estándar), que la terminal muestra en pantalla.
5. Al terminar la última línea, Python sale con código 0 (éxito).

Si hay un error en cualquier punto, Python imprime un **traceback** — las líneas del error en rojo — y sale con código 1. El traceback se lee de abajo hacia arriba: la última línea es el error; las anteriores son el camino que tomó el código hasta llegar a él.

### `print()`: la única forma de ver resultados en un script

En la REPL, la última expresión de cada línea se imprime automáticamente. En un script, no. Si no usas `print()`, nada aparece en pantalla aunque el cálculo se haya hecho correctamente:

```python
# en un script .py: esto NO imprime nada
amount = 4820.0
total = amount * 1.21

# esto SÍ imprime
print(total)
```

`print()` acepta varios argumentos separados por coma; los imprime separados por un espacio:

```python
print("Factura", "INV-001", "total:", 5832.2)
# → Factura INV-001 total: 5832.2
```

Con f-string es más legible y te da control de formato:

```python
print(f"Factura INV-001 total: {total:.2f} EUR")
# → Factura INV-001 total: 5832.20 EUR
```

### Comentarios con `#`

Un comentario empieza con `#` y llega hasta el final de la línea. Python lo ignora por completo: no se ejecuta, no consume tiempo, no produce errores.

```python
# calcular IVA sobre el monto base de ACME
tax = base_amount * 0.21   # tipo estándar español
```

Los comentarios son para ti y para el equipo. Úsalos para explicar el *por qué*, no el *qué*. «`# multiplica por 0.21`» no añade información; «`# tipo IVA general vigente 2026`» sí.

### stdout vs stderr

Python tiene dos canales de salida:

- **stdout**: la salida normal del programa. Lo que `print()` envía por defecto. El terminal lo muestra en blanco.
- **stderr**: los mensajes de error. Los tracebacks van aquí. El terminal los muestra en rojo.

Por ahora ambos aparecen mezclados en la terminal. La separación importará cuando redirigías la salida a un archivo o a otro programa (eso llega en módulos posteriores).

### Ejecutar el script: dos formas

```bash
# forma 1: con uv (recomendada en este curso)
uv run python audit.py

# forma 2: activando el entorno primero y luego directamente
source .venv/bin/activate    # macOS/Linux
python audit.py
```

La primera forma es preferible porque `uv run` activa y desactiva el entorno automáticamente. No tienes que recordar activar el entorno cada vez que abres una nueva terminal.

> [!cuidado]
> Ejecuta `uv run python audit.py` desde la carpeta que contiene `audit.py`. Si ejecutas desde otra carpeta, Python no encuentra el archivo y lanza `FileNotFoundError`. Comprueba con `pwd` dónde estás y con `ls` que el archivo existe.

## Ejemplo conducido

<!-- INLINE_QUIZ: print-required -->

Crea un directorio `acme-audit`, entra en él, y crea el archivo `audit.py`:

```python
# audit.py — primer script de auditoría ACME
# período: 2026-04

# datos de la factura
invoice_id = "INV-001"
vendor = "ACME Manufacturing"
base_amount = 4820.0
tax_rate = 0.21
period = "2026-04"

# cálculos
tax = base_amount * tax_rate
total = base_amount + tax

# construir la línea de reporte
status = "PENDIENTE"
report_line = (
    f"[{period}] {invoice_id} | {vendor} | "
    f"base {base_amount:.2f} EUR | IVA {tax:.2f} EUR | "
    f"total {total:.2f} EUR | {status}"
)

print("=== Auditoría ACME Manufacturing ===")
print(report_line)
print(f"Factura procesada correctamente.")
```

Ejecuta desde la terminal:

```bash
uv run python audit.py
```

Salida:

```
=== Auditoría ACME Manufacturing ===
[2026-04] INV-001 | ACME Manufacturing | base 4820.00 EUR | IVA 1012.20 EUR | total 5832.20 EUR | PENDIENTE
Factura procesada correctamente.
```

El script hace siempre lo mismo: calcula, construye el texto, y lo imprime. Sin REPL, sin interacción, sin celdas. Solo código que entra por arriba y resultado que sale por abajo.

## Errores típicos

**1. Ejecutar desde la carpeta equivocada.**
*Síntoma*: `python: can't open file 'audit.py': No such file or directory`.
*Causa*: estás en una carpeta distinta a la que tiene `audit.py`.
*Arreglo*: ejecuta `pwd` para ver dónde estás y `cd` para ir a la carpeta correcta. Luego `ls` para confirmar que `audit.py` está ahí.

**2. Olvidar `print()` y no ver nada.**
*Síntoma*: el script ejecuta sin error pero la terminal no muestra nada.
*Causa*: calculaste el valor pero no lo imprimiste.
*Arreglo*: añade `print(variable)` para cada valor que quieras ver en pantalla.

**3. Indentación con tabs mezclada con espacios.**
*Síntoma*: `TabError: inconsistent use of tabs and spaces in indentation`.
*Causa*: Python exige consistencia. Si mezclas tabs y espacios, falla. VS Code usa espacios por defecto.
*Arreglo*: configura VS Code para que use 4 espacios como indentación. En 00-10 (condicionales) la indentación será obligatoria; mejor adquirir el hábito ahora.

**4. Nombre del archivo con espacios.**
*Síntoma*: `bash: audit run.py: command not found` o error similar.
*Causa*: `audit run.py` tiene un espacio. La terminal interpreta `run.py` como un argumento separado.
*Arreglo*: usa `snake_case` sin espacios: `audit_run.py`. Los nombres de archivos Python nunca llevan espacios.

## Salida esperada

Al terminar este módulo puedes:

- Crear un archivo `.py` con VS Code y escribir código Python válido.
- Ejecutarlo con `uv run python nombre_archivo.py` desde la terminal.
- Usar `print()` para mostrar valores en la terminal.
- Usar comentarios `#` para anotar el código.
- Leer un traceback básico de abajo hacia arriba para localizar el error.

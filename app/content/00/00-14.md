## Hilo conductor

En 00-13 definiste funciones propias con `def` y `return`. Ahora esas funciones van a recibir datos del mundo real: montos escritos como texto, claves que no siempre existen, divisores que a veces son cero. Cuando algo falla, Python lanza una **excepción** y detiene el programa salvo que la captures. `try/except` es la herramienta para eso. En 00-15 leerás archivos reales donde los datos imperfectos son la norma.

## Puente desde Excel

En Excel tienes `=IFERROR(A1/B1, 0)`: si la división falla (por ejemplo, B1 es cero), devuelves `0` en lugar de `#DIV/0!`. En Python la misma idea se escribe:

```python
try:
    ratio = numerator / denominator
except ZeroDivisionError:
    ratio = 0.0
```

La diferencia importante: Excel envuelve el error en silencio. Python te obliga a nombrar qué tipo de error esperas. Eso es una ventaja: si hay un error que no esperabas, Python lo dice en voz alta en lugar de devolverte un cero engañoso.

## Idea central

Una **excepción** es una señal de error que Python lanza cuando no puede ejecutar una instrucción. Si nadie la captura, el programa se detiene y muestra un **traceback** (la ruta de dónde vino el error). `try` marca el bloque de código que puede fallar; `except TipoDeError` define qué hacer si ese error específico ocurre. El programa continúa normalmente después del bloque.

## Por qué importa

Los datos que llegan del mundo real nunca son perfectos. Una factura puede tener el monto escrito como `"4.820,50"` en lugar de `4820.50`. Una clave puede faltar en un diccionario. Un proveedor nuevo puede tener un nombre que rompe tu regex. Si no capturas esos errores, tu script se detiene a mitad y deja el trabajo a medias.

Capturar errores de forma selectiva —solo los que esperas, con un valor de reemplazo claro— es lo que separa un script frágil de una función robusta.

## Cómo funciona por dentro

### El traceback: léelo de abajo hacia arriba

Cuando Python lanza una excepción sin capturar, muestra algo así en la terminal:

```
Traceback (most recent call last):
  File "audit.py", line 12, in process_invoices
    ratio = total / count
  File "audit.py", line 12, in process_invoices
ZeroDivisionError: division by zero
```

La última línea es el error y su tipo. Las líneas anteriores son el camino que tomó el código hasta llegar al error. Lees de abajo hacia arriba: primero el error, luego dónde ocurrió, luego quién llamó a ese código.

### Los tres errores más comunes en datos de negocio

**`ValueError`**: el tipo es correcto pero el valor no tiene sentido para la operación.

```python
int("4820.50")   # → ValueError: invalid literal for int() with base 10
float("cuatro mil")  # → ValueError
```

**`KeyError`**: accediste a una clave que no existe en un diccionario.

```python
invoice = {"vendor": "ACME", "amount": 4820.0}
invoice["paid"]  # → KeyError: 'paid'
```

**`ZeroDivisionError`**: divisor es cero.

```python
4820.0 / 0  # → ZeroDivisionError: float division by zero
```

### La estructura `try/except`

```python
try:
    # código que puede fallar
    result = int(raw_text)
except ValueError:
    # qué hacer si el tipo de error esperado ocurre
    result = None
```

El bloque `except` solo se ejecuta si ocurre el error nombrado. Si el `try` funciona sin problemas, el `except` se salta. Después de cualquiera de los dos casos, el programa continúa con la siguiente línea.

### Capturar un error específico, no todos

```python
# MAL: captura cualquier error, incluidos errores de programación
try:
    result = int(raw_text)
except Exception:
    result = None

# BIEN: captura solo lo que esperas
try:
    result = int(raw_text)
except ValueError:
    result = None
```

`except Exception:` (o peor, `except:`) silencia errores que no deberías silenciar: errores de sintaxis en tu propio código, problemas de memoria, errores de programación que te indican que algo está mal. Capturas solo el error que esperas porque es el único que sabes cómo manejar.

> [!nota]
> Python tiene una jerarquía de excepciones: `ValueError` hereda de `Exception`, que hereda de `BaseException`. En módulos avanzados (A08) aprenderás a crear tus propias excepciones y a capturar clases padre. Por ahora, captura siempre el tipo más específico posible.

### Una función que devuelve `None` cuando falla

El patrón más útil para Track 0 es una función que intenta convertir un valor y devuelve `None` si no puede, en lugar de lanzar un error:

```python
def parse_amount(text: str) -> float | None:
    """Convierte texto a float. Devuelve None si el texto no es un número válido."""
    try:
        return float(text)
    except ValueError:
        return None
```

Llamadas:

```python
parse_amount("4820.50")   # → 4820.5
parse_amount("4.820,50")  # → None  (coma decimal: no es float válido en Python)
parse_amount("")           # → None
parse_amount("cuatro mil") # → None
```

El llamador puede comprobar si el resultado es `None` con `if amount is not None:` antes de usarlo. El error no detiene el programa; el valor inválido queda marcado como `None` para tratamiento posterior.

## Ejemplo conducido

<!-- INLINE_QUIZ: traceback-reading -->

ACME recibe un lote de facturas con montos en texto. Algunas están bien formadas, otras no. La función `parse_amount` limpia el lote:

```python
def parse_amount(text: str) -> float | None:
    """Devuelve float si text es un número válido, None si no."""
    try:
        return float(text)
    except ValueError:
        return None


raw_amounts = ["4820.50", "N/A", "1500.0", "", "3100.75"]
parsed = []

for raw in raw_amounts:
    amount = parse_amount(raw)
    if amount is not None:
        parsed.append(amount)

# total solo de los que pudieron parsearse
total = sum(parsed)
print(f"Total válido: {total:.2f} EUR")
# → Total válido: 9421.25 EUR
```

El script no se detiene cuando encuentra `"N/A"` o `""`. Los salta y sigue.

## Errores típicos

**1. `except:` sin tipo.**
*Síntoma*: el programa sigue pero errores reales de tu código (como `NameError` por un typo) desaparecen sin explicación.
*Causa*: `except:` captura todo, incluyendo errores que son bugs tuyos.
*Arreglo*: siempre nombra el tipo: `except ValueError:`, `except KeyError:`.

**2. Retornar `0` en lugar de `None` cuando el parse falla.**
*Síntoma*: un monto inválido se suma como `0` y distorsiona los totales.
*Causa*: `0` parece inofensivo pero es un valor que contamina sumas y promedios.
*Arreglo*: usa `None` como sentinel de «no sé el valor» y filtra antes de calcular.

**3. Capturar un error que no lanzas.**
*Síntoma*: el `except` nunca se ejecuta aunque hay datos malos; el programa igual falla más arriba.
*Causa*: pusiste `except ValueError` pero la función en realidad lanza `TypeError` (tipos incompatibles) o `KeyError`.
*Arreglo*: lee el traceback completo para saber el tipo exacto del error, luego cámbialo en `except`.

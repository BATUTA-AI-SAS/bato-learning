## Hilo conductor

No hay módulo previo: este es el primer paso del curso. Si llevas años construyendo reportes en Excel, ya sabes programar — solo que con una interfaz distinta. Este módulo nombra eso de forma explícita, traza la línea entre lo que Excel resuelve bien y lo que no puede resolver, y deja listo el terreno para `terminal-and-shell` (00-02), donde verás por primera vez cómo se ejecuta código fuera de una hoja de cálculo.

## Puente desde Excel

Cuando escribes `=SUMA(A1:A10)` en una celda, estás dando una instrucción precisa a Excel: «suma los valores de estas diez celdas y muestra el resultado aquí». Eso es exactamente lo que hace una línea de Python: dar una instrucción precisa a un programa para que procese datos y produzca un resultado. La diferencia no está en la idea — está en dónde vive la instrucción, quién la ejecuta y qué puede hacer con el resultado después.

## Idea central

**Programar** es escribir instrucciones que una máquina ejecuta en secuencia para transformar datos de entrada en datos de salida. Una **fórmula Excel** es código: la escribe el usuario, la evalúa Excel, y el resultado aparece en otra celda. Una **línea Python** es también código: la escribe el usuario, la evalúa el intérprete de Python, y el resultado puede usarse en la siguiente instrucción sin necesidad de celdas. La diferencia operativa es que en Python el resultado vive en memoria con un nombre que tú eliges — no en una coordenada de cuadrícula.

## Por qué importa

Excel resuelve bien el análisis interactivo de datos que caben en una hoja y que un humano va a revisar a mano. Falla en tres escenarios concretos que aparecen constantemente en empresas medianas:

1. **Volumen.** Excel se atasca con más de ~500 000 filas. Un archivo de facturas de ACME Manufacturing para el período `"2026-04"` con 2 millones de registros de movimientos no cabe en una hoja. En Python, esos mismos datos se procesan en segundos con las herramientas adecuadas.

2. **Reproducibilidad.** En Excel, el historial de cambios no existe de forma confiable. Si alguien edita una fórmula el martes y el reporte del miércoles da un número distinto, rastrear qué cambió es casi imposible. En Python, el código vive en un archivo de texto que se puede versionar: cada cambio queda registrado, firmado y revertible.

3. **Integración.** Excel no puede hacer una petición a una API externa, leer una base de datos de 40 tablas o enviar un correo con el resultado de un análisis. Python puede hacer todo eso desde el mismo script.

## Cómo funciona por dentro

### Qué pasa cuando Excel evalúa una fórmula

Cuando presionas Enter en una celda con `=SUMA(A1:A10)`, Excel hace lo siguiente:

1. Lee el texto de la fórmula.
2. Lo parsea: identifica la función `SUMA` y el rango `A1:A10`.
3. Lee los valores de las celdas A1 a A10.
4. Ejecuta la suma.
5. Guarda el resultado en la celda donde está la fórmula y lo muestra.

El resultado solo vive en esa celda. Si quieres usarlo en otra fórmula, referencias esa celda por su coordenada (por ejemplo, `B11`).

### Qué pasa cuando Python evalúa una instrucción

Cuando el intérprete de Python lee `total = 4820.0 + 1500.0`, hace algo análogo:

1. Lee el texto de la instrucción.
2. Lo parsea: identifica la operación `+`, los dos valores y el nombre `total`.
3. Ejecuta la suma.
4. Guarda el resultado en memoria con el nombre `total`.

La diferencia clave: el resultado no vive en una celda — vive en memoria con el nombre `total`. La siguiente instrucción puede usar `total` directamente, sin coordenadas.

### El nombre es más poderoso que la coordenada

En Excel, si mueves la fórmula de B11 a C15, todas las referencias que apuntaban a B11 se rompen o hay que actualizarlas. En Python, el nombre `total` siempre encuentra el valor, sin importar el orden en que se declararon las instrucciones — mientras el nombre exista en el mismo ámbito.

### Programar es encadenar instrucciones

La potencia de programar no está en una sola instrucción sino en encadenar muchas. En Excel esto se expresa anidando funciones: `=SI(A1>0, SI(A1>1000, "alto", "normal"), "cero")`. Cuatro niveles de anidación son casi ilegibles. En Python, las mismas condiciones se expresan en líneas separadas con sangría, una debajo de la otra, y se leen de arriba abajo como prosa.

## Ejemplo conducido

<!-- INLINE_QUIZ: three-limits-of-excel -->

La siguiente tabla muestra la misma operación expresada en Excel y en Python. No es necesario entender la sintaxis Python ahora — el objetivo es ver que la estructura es equivalente.

**Escenario**: calcular el total de tres facturas de ACME y determinar si supera el umbral de revisión (10 000 EUR).

En Excel, en tres celdas separadas:
```
A1: 4820.0
A2: 1500.0
A3: 3980.0
B1: =SUMA(A1:A3)          → 10300.0
C1: =SI(B1>10000,"revisar","ok")  → "revisar"
```

En Python, en tres instrucciones seguidas:
```python
amount_1 = 4820.0
amount_2 = 1500.0
amount_3 = 3980.0
total = amount_1 + amount_2 + amount_3    # 10300.0
status = "revisar" if total > 10000 else "ok"  # "revisar"
```

La lógica es idéntica. La diferencia: en Python puedes ejecutar este código sobre 2 millones de facturas en un bucle, sin tocar la hoja.

## Tabla Excel ↔ Python

| Concepto Excel | Equivalente Python | Diferencia práctica |
|---|---|---|
| Fórmula en celda (`=SUMA(...)`) | Instrucción en archivo `.py` | Python encadena resultados con nombres, no con coordenadas |
| Celda con resultado | Variable (nombre que apunta a un valor en memoria) | El nombre no depende de la posición en el archivo |
| Excel evalúa al recalcular | Python evalúa al ejecutar el archivo o línea | Python no recalcula solo: tú lo ejecutas cuando quieres |
| Función nativa (`SUMA`, `SI`, `BUSCARV`) | Función de Python (`sum`, `if/else`, dict lookup) | Misma idea, sintaxis distinta |
| Libro con hojas múltiples | Módulo (archivo `.py`) con funciones | Un módulo puede importar a otro, un libro no puede ejecutar a otro |
| Límite de filas (~1 M) | Sin límite práctico (RAM es el límite) | Python procesa conjuntos que no caben en memoria con técnicas de streaming |

## Errores típicos del que viene de Excel

**1. Buscar la celda donde ver el resultado.**
*Síntoma*: el lector escribe código Python y espera que aparezca algo en algún lugar de la pantalla automáticamente.
*Causa*: en Excel el resultado siempre aparece en la celda. En Python, un resultado solo se muestra si explícitamente se imprime (`print(total)`) o si se trabaja en la REPL interactiva (que se ve en 00-05).
*Arreglo*: añadir `print(total)` cuando se quiera ver el valor. Es deliberado, no un olvido del lenguaje.

**2. Pensar que cambiar el código se «recalcula» solo.**
*Síntoma*: el lector edita el archivo `.py`, espera que el resultado cambie sin hacer nada más.
*Causa*: Excel recalcula automáticamente al cambiar un valor. Python no recalcula: hay que ejecutar el archivo de nuevo.
*Arreglo*: guardar el archivo y volver a ejecutarlo con el comando de la terminal (se ve en 00-02).

**3. Usar coordenadas mentales («la variable de arriba», «la variable de la derecha»).**
*Síntoma*: el lector nombra variables `a1`, `b2`, `c3` imitando las coordenadas de Excel.
*Causa*: hábito de 10 años referenciando por posición.
*Arreglo*: los nombres en Python describen el contenido, no la posición. `invoice_amount` es mejor que `a1`. El código se leerá en 6 meses y los nombres deben ser autoexplicativos.

## Hilo conductor

En 00-17 configuraste tu entorno: terminal, Python, `uv`, VS Code. Tienes todo para escribir código. El problema: hoy tu versionado es `informe_v3_final_FINAL2.xlsx`. Cuando algo se rompe, no sabes qué cambió ni cuándo. Cuando alguien más toca el mismo archivo, uno pisa al otro. **Git** resuelve eso. Es el sistema de control de versiones que usa absolutamente todo lo que vas a construir en este curso. A01 ya asume que tienes un repositorio funcionando. Este módulo te lo da.

## Idea central

Un **repositorio** es una carpeta con su historia completa. Cada **commit** es una foto del estado de la carpeta en un momento dado: tiene fecha, autor, mensaje, y un enlace al commit anterior. Una **branch** (rama) es una línea paralela de commits — puedes experimentar sin tocar la rama principal. **Push** sincroniza tus commits al servidor remoto; **pull** trae los commits del servidor a tu máquina. Un **conflict** ocurre cuando dos personas cambian la misma línea: Git detiene el merge y pide que decidas tú.

## Por qué importa

Sin Git no hay CI/CD (D05), no hay deploy controlado (D09), no hay colaboración real con ningún cliente. Cada módulo del track D asume que tu código vive en un repositorio y que los cambios entran por un PR con CI verde. D03 requiere `git clone` en el servidor. D05 dispara deploys desde un push. Si llegas a esos módulos sin dominar Git, vas a estar ciego.

El caso **F-OPS-02 (gestión de inventario de ACME)** ilustra el riesgo operativo: un cambio de configuración sin versionar en el servidor de producción rompió el agente durante 4 horas. Con Git y un pipeline, ese cambio entra por un PR con revisión — o no entra.

> [!cuidado]
> Nunca commites credenciales, API keys ni archivos `.env` al repositorio. Una vez que un secreto entra al historial de Git, se considera comprometido aunque lo borres en un commit posterior. El repo tiene `.gitignore` configurado; respétalo.

## Cómo funciona por dentro

### El repositorio: carpeta + historia

Cuando ejecutas `git init` en una carpeta, Git crea un subdirectorio `.git/`. Ahí vive todo: la historia de commits, los branches, la configuración del remote. La carpeta en sí (lo que ves con `ls`) es el **working directory** — el estado actual. La historia en `.git/` es inmutable salvo que la manipules a propósito.

```
mi-proyecto/
├── .git/          ← la historia (no toques esto manualmente)
├── app/
│   └── main.py
└── pyproject.toml
```

### El ciclo staging → commit

Git tiene tres zonas:

| Zona | Qué es | Comando |
|------|--------|---------|
| **Working directory** | Los archivos en tu carpeta, con cambios sin confirmar | — |
| **Staging area** (índice) | Los cambios que vas a incluir en el próximo commit | `git add archivo.py` |
| **Repository** | El historial de commits ya guardados | `git commit -m "mensaje"` |

El staging existe por diseño: te permite hacer un commit de solo parte de los cambios. Si editaste `main.py` y `config.py` pero solo `main.py` está listo, haces `git add main.py` y commitas solo eso.

```bash
git status          # qué cambió, qué está en staging
git add main.py     # manda main.py al staging
git add .           # manda TODO lo modificado al staging (úsalo con criterio)
git commit -m "feat: agregar endpoint de auditoría"
git log --oneline   # ver historia condensada
```

### Branches: líneas paralelas

La rama por defecto se llama `main`. Una branch nueva es solo un puntero a un commit — no duplica archivos. Crear una branch es gratis en tiempo y disco.

```bash
git switch -c feature/audit-export   # crea y cambia a la nueva rama
# ... haces cambios ...
git switch main                       # vuelves a main
git merge feature/audit-export        # trae los cambios de la rama
```

> [!nota]
> `git switch` y `git restore` son los comandos modernos (Git 2.23+). El antiguo `git checkout` hacía las dos cosas mezcladas. En este curso usamos los modernos: `git switch` para ramas, `git restore` para descartar cambios en archivos.

### Remote: sincronizar con GitHub

Un **remote** es la versión del repositorio en un servidor (GitHub, GitLab, etc.). El remote por defecto se llama `origin`.

```bash
git remote add origin git@github.com:acme/audit-agent.git
git push origin main          # sube tus commits a GitHub
git pull                      # trae commits del remote y hace merge
git fetch                     # trae commits del remote SIN hacer merge
```

### Conflicts: cuando dos tocan la misma línea

Un conflicto ocurre cuando dos branches modifican la misma línea de un archivo y tratas de fusionarlas. Git no adivina cuál versión es correcta: te lo muestra y espera tu decisión.

```
<<<<<<< HEAD
    return total * 1.19   # tu versión (rama actual)
=======
    return total * 1.21   # versión de la otra rama
>>>>>>> feature/iva-colombia
```

Editas el archivo manualmente, dejas la versión correcta, eliminas los marcadores `<<<<<<<`, `=======`, `>>>>>>>`, y luego:

```bash
git add main.py
git commit -m "fix: resolver conflict de tasa IVA"
```

### `.gitignore`: lo que no se versiona

El archivo `.gitignore` en la raíz del repo lista los patrones de archivos que Git ignora. Siempre deben estar ahí:

```
.env
.venv/
__pycache__/
*.pyc
data/*.db
```

> [!cuidado]
> Si añades un archivo a `.gitignore` **después** de haberlo commiteado, Git no lo olvidará automáticamente. Tienes que ejecutar `git rm --cached archivo` para quitarlo del tracking sin borrarlo de tu disco.

### Conventional Commits: mensajes que informan

Un mensaje de commit tiene que explicar **qué** y **por qué**, no «cambios» o «fix». El estándar **Conventional Commits** define el formato:

```
<tipo>[scope opcional]: <descripción corta>

[cuerpo opcional]
```

Tipos principales:

| Tipo | Cuándo usarlo |
|------|---------------|
| `feat` | Nueva funcionalidad |
| `fix` | Corrección de bug |
| `refactor` | Reorganización sin cambio de comportamiento |
| `test` | Añadir o modificar tests |
| `docs` | Solo documentación |
| `chore` | Tareas de mantenimiento (deps, config) |

Ejemplos reales:

```
feat(audit): agregar exportación PDF del reporte mensual
fix: corregir cálculo de IVA para facturas en USD
chore: actualizar anthropic a 0.42.0
```

Con este formato, `git log --oneline` se convierte en un registro legible de decisiones, no de movimientos de archivos.

### GitHub: SSH key para autenticación

Para hacer `git push` a GitHub sin ingresar contraseña en cada operación, configuras autenticación por SSH:

```bash
# Genera el par de claves (ed25519 es el algoritmo moderno)
ssh-keygen -t ed25519 -C "tu@email.com"
# Acepta la ruta por defecto (~/.ssh/id_ed25519)
# Pon una passphrase si el equipo puede ser accedido por otros

# Muestra la clave pública para copiarla a GitHub
cat ~/.ssh/id_ed25519.pub
```

En GitHub: Settings → SSH and GPG keys → New SSH key → pega el contenido de `id_ed25519.pub`.

Luego prueba:
```bash
ssh -T git@github.com
# Hi acme-dev! You've successfully authenticated...
```

> [!nota]
> Este mismo par de claves lo usas en D00 para acceder al servidor Hetzner por SSH. Generas las claves una vez; las reutilizas.

## Ejemplo conducido

<!-- INLINE_QUIZ: staging-vs-committed -->

Flujo completo desde cero para el proyecto de auditoría de ACME:

```bash
# 1. Crear el repo local
git init audit-agent
cd audit-agent

# 2. Primer commit
echo "# Agente de auditoría ACME" > README.md
git add README.md
git commit -m "chore: init repositorio"

# 3. Trabajar en una feature en rama separada
git switch -c feat/invoice-parser
# ... editas app/parsers/invoice.py ...
git add app/parsers/invoice.py
git commit -m "feat(parser): extraer campos de factura PDF"

# 4. Volver a main y fusionar
git switch main
git merge feat/invoice-parser

# 5. Ver la historia
git log --oneline
# a3f92d1 feat(parser): extraer campos de factura PDF
# 1b2e445 chore: init repositorio

# 6. Subir a GitHub
git remote add origin git@github.com:acme/audit-agent.git
git push -u origin main
```

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| `git add`, `git commit`, `git push` | determinístico | El resultado es exactamente lo que el usuario especificó — sin variación |
| Resolver un merge conflict | determinístico | Hay dos versiones concretas; el humano elige una regla de negocio, no el LLM |
| Escribir el mensaje de commit | agéntico | Puede apoyarse en el LLM para generar el borrador, pero el humano lo revisa y aprueba |
| Decidir si un PR está listo para merge | agéntico | Requiere leer el contexto del cambio, el estado del sistema, riesgos |
| CI/CD: ejecutar tests tras cada push | determinístico | Pipeline declarativo; mismo código → mismo resultado siempre |

## Errores típicos

**1. Commit gigante mensual.**
*Síntoma*: `git log` muestra un commit con 47 archivos cambiados y mensaje «actualizaciones de mayo».
*Causa*: el lector guardó todo para el final, como guardaba el Excel al final del día.
*Arreglo*: commit cada vez que un cambio lógico está completo. Si añades un endpoint, ese es un commit. Si arreglas un bug, ese es otro.

**2. `git add .` sin revisar qué hay.**
*Síntoma*: el `.env` con `ANTHROPIC_API_KEY` entra al historial.
*Causa*: `git add .` añade todo, incluido lo que no debería ir.
*Arreglo*: antes de `git add .`, ejecuta siempre `git status` para ver qué cambió. Mantén `.gitignore` actualizado.

**3. Trabajar directo en `main` en lugar de usar branches.**
*Síntoma*: cada commit va directo a `main`; si rompe algo, lo rompe en producción.
*Causa*: el hábito de editar en el archivo «principal» sin copia de seguridad.
*Arreglo*: todo cambio nace en una rama. `main` solo recibe merges desde ramas revisadas.

**4. Ignorar el conflict y hacer `git commit` con los marcadores dentro.**
*Síntoma*: el archivo en producción tiene `<<<<<<< HEAD` literalmente en el código.
*Causa*: el lector hizo `git add` sin resolver el conflicto.
*Arreglo*: después de editar el conflicto, verifica con `git diff` que no quede ningún marcador antes de hacer `git add`.

**5. Usar `git checkout` mezclando cambio de rama y descarte de archivo.**
*Síntoma*: confusión frecuente entre «cambié de rama» y «deshice mis cambios».
*Causa*: `git checkout` hace ambas cosas dependiendo del argumento.
*Arreglo*: usa `git switch nombre-rama` para cambiar de rama, y `git restore archivo.py` para descartar cambios en un archivo.

## JSON — el formato de datos entre programas

**JSON** (JavaScript Object Notation) es el formato más usado para que dos programas intercambien datos. Una API de facturas, el SDK de Anthropic, los webhooks de Slack: todos usan JSON.

### Sintaxis en 30 segundos

```json
{
  "id": "INV-001",
  "vendor": "ACME",
  "amount": 4820.50,
  "paid": false,
  "tags": ["urgente", "2026-04"],
  "contact": null
}
```

Reglas que lo diferencian de un dict Python:

| Regla | JSON | Python dict |
|---|---|---|
| Claves | Siempre entre comillas dobles `"clave"` | Pueden ser cualquier hashable |
| Booleanos | `true` / `false` (minúscula) | `True` / `False` (mayúscula) |
| Nulo | `null` | `None` |
| Comentarios | **No admite** | Sí (`#`) |
| Coma al final | **No admite** trailing comma | Python sí la admite en listas/dicts |

### Leer y escribir JSON con Python

```python
import json

# texto JSON → dict Python
raw = '{"id": "INV-001", "amount": 4820.50, "paid": false}'
invoice = json.loads(raw)
print(invoice["amount"])   # → 4820.5
print(invoice["paid"])     # → False  (Python bool, no string)

# dict Python → texto JSON
data = {"vendor": "Globex", "amount": 1500.0, "paid": True}
text = json.dumps(data)
print(text)
# → {"vendor": "Globex", "amount": 1500.0, "paid": true}

# con formato legible
print(json.dumps(data, indent=2))
```

`json.loads` (load string) convierte texto JSON a objeto Python. `json.dumps` (dump string) hace lo contrario.

> [!cuidado]
> JSON solo puede representar strings, números, booleanos, null, arrays y objetos. Tipos Python como `datetime`, `set`, o clases personalizadas no se pueden serializar a JSON directamente. Cuando la API devuelve una fecha, la devuelve como string `"2026-04-30"` — tú la conviertes después.

## YAML — JSON legible para humanos, para configuración

**YAML** (YAML Ain't Markup Language) es el formato de configuración de Docker Compose, GitHub Actions, y el propio frontmatter de los módulos de este curso (lo que hay entre `---` al principio de cada archivo `.md`).

### Sintaxis en 30 segundos

```yaml
# esto es un comentario válido en YAML
invoice:
  id: INV-001
  vendor: ACME
  amount: 4820.50
  paid: false
  tags:
    - urgente
    - "2026-04"
  contact: null
```

Las mismas reglas que JSON, pero con diferencias:

| Característica | YAML | JSON |
|---|---|---|
| Sintaxis | Indentación, sin llaves ni corchetes | Llaves `{}`, corchetes `[]` |
| Comentarios | `# comentario` | No |
| Strings | Sin comillas en la mayoría de casos | Siempre comillas dobles |
| Indentación | **Obligatoria y significativa** | Opcional, solo visual |

### Leer YAML con Python

```python
# requiere: uv add pyyaml
import yaml

raw = """
vendor: ACME
amount: 4820.50
paid: false
tags:
  - urgente
  - "2026-04"
"""

data = yaml.safe_load(raw)
print(data["vendor"])   # → ACME
print(data["paid"])     # → False
print(data["tags"])     # → ['urgente', '2026-04']
```

`yaml.safe_load` es la función correcta para cargar YAML de fuentes externas. Existe también `yaml.load` pero requiere un argumento extra de seguridad — usa `safe_load` por defecto.

> [!nota]
> `pyyaml` es librería externa: `uv add pyyaml`. En los ejercicios de este módulo usamos JSON (librería estándar) para no requerir instalación. Cuando en Track A o B encuentres un archivo `.yaml`, ya sabes cómo leerlo.

> [!cuidado]
> La indentación en YAML es con **espacios**, nunca con tabulaciones. Si mezclas espacios y tabulaciones, el parser lanza un error. Los editores modernos (VS Code) convierten Tab a espacios automáticamente.

## HTTP — el protocolo de la web

**HTTP** (HyperText Transfer Protocol) es el idioma con el que tu navegador habla con los servidores, y también el que tus scripts usarán para llamar APIs externas (Anthropic, Slack, tu propio ERP).

### Un request HTTP en una imagen

```
GET /clientes/42 HTTP/1.1
Host: api.acme.com
Authorization: Bearer mi-token-secreto
Accept: application/json
```

Cuatro partes:
1. **Verbo** (`GET`): qué quieres hacer.
2. **Ruta** (`/clientes/42`): sobre qué recurso.
3. **Headers**: metadatos (quién eres, qué aceptas).
4. **Body**: datos adicionales (solo en `POST`, `PUT`, `PATCH`).

### Los verbos principales

| Verbo | Para qué | Ejemplo |
|---|---|---|
| `GET` | Obtener un recurso | `GET /facturas/INV-001` |
| `POST` | Crear un recurso nuevo | `POST /facturas` con body JSON |
| `PUT` | Reemplazar un recurso completo | `PUT /facturas/INV-001` con body JSON |
| `PATCH` | Modificar parcialmente | `PATCH /facturas/INV-001` con solo los campos que cambian |
| `DELETE` | Eliminar | `DELETE /facturas/INV-001` |

### La respuesta

```
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 87

{"id": "INV-001", "vendor": "ACME", "amount": 4820.50, "paid": true}
```

El **status code** (el número) comunica si fue bien o mal:

| Rango | Significado | Ejemplos |
|---|---|---|
| `2xx` | Éxito | `200 OK`, `201 Created`, `204 No Content` |
| `4xx` | Error del **cliente** (tu request está mal) | `400 Bad Request`, `401 Unauthorized`, `404 Not Found`, `422 Unprocessable Entity` |
| `5xx` | Error del **servidor** (el servidor falló) | `500 Internal Server Error`, `503 Service Unavailable` |

La distinción `4xx` vs `5xx` es la más práctica: `4xx` significa que debes cambiar algo en tu request; `5xx` significa que el servidor tiene un problema y puedes reintentar después.

### Llamar una API con Python

```python
# requiere: uv add httpx
import httpx

response = httpx.get(
    "https://api.acme.com/facturas/INV-001",
    headers={"Authorization": "Bearer mi-token"}
)

print(response.status_code)   # → 200
data = response.json()         # convierte el body JSON a dict
print(data["vendor"])          # → ACME
```

> [!nota]
> `httpx` es la librería que este curso usa para HTTP en scripts. El módulo B01 lo explica en detalle junto con `curl` (la herramienta de terminal para hacer requests). Por ahora solo necesitas saber que HTTP tiene verbos, rutas, headers, y status codes.

Parsear un payload JSON que llegaría de un webhook de ACME:

```python
import json

# payload que llega de un sistema externo (como string)
payload = """
{
  "event": "invoice.created",
  "invoice": {
    "id": "INV-042",
    "vendor": "Globex",
    "amount": 3100.75,
    "paid": false,
    "period": "2026-04"
  }
}
"""

data = json.loads(payload)

event_type = data["event"]
invoice = data["invoice"]

if not invoice["paid"]:
    print(f"Factura pendiente: {invoice['id']} — {invoice['amount']} EUR")
    print(f"Proveedor: {invoice['vendor']}, período: {invoice['period']}")
```

Salida:

```
Factura pendiente: INV-042 — 3100.75 EUR
Proveedor: Globex, período: 2026-04
```

## Errores típicos

**1. Comillas simples en JSON.**
*Síntoma*: `json.loads(...)` lanza `JSONDecodeError`.
*Causa*: JSON exige comillas dobles. `{'id': 'INV-001'}` es válido en Python pero inválido en JSON.
*Arreglo*: en JSON, siempre `"id"`, nunca `'id'`.

**2. Indentación con tabulaciones en YAML.**
*Síntoma*: `yaml.safe_load` lanza `TabError` o `ScannerError`.
*Causa*: YAML rechaza tabulaciones. Solo admite espacios.
*Arreglo*: configura tu editor para insertar espacios al pulsar Tab (VS Code lo hace por defecto para archivos `.yaml`).

**3. Confundir `4xx` con `5xx` al depurar.**
*Síntoma*: buscas el bug en el servidor cuando el problema está en tu request.
*Causa*: `422 Unprocessable Entity` o `400 Bad Request` significan que tu datos o tu URL están mal, no que el servidor falló.
*Arreglo*: lee el body de la respuesta de error — las APIs bien diseñadas incluyen un mensaje de error en JSON que dice exactamente qué campo está mal.

## Salida esperada

Al terminar, puedes:

- Reconocer JSON por sus llaves y comillas dobles, y distinguirlo de un dict Python por `true`/`false`/`null`.
- Usar `json.loads(texto)` para convertir JSON a dict Python y `json.dumps(dict)` para el camino inverso.
- Reconocer YAML por su indentación y saber que lo verás en archivos de configuración como `compose.yaml` y frontmatter de `.md`.
- Leer un request HTTP e identificar verbo, ruta, headers, y body.
- Interpretar un status code: `2xx` es éxito, `4xx` es error tuyo, `5xx` es error del servidor.

## TOML — el formato de configuración de proyectos Python

**TOML** (Tom's Obvious Minimal Language) es el formato que Python usa para el archivo de configuración del proyecto: `pyproject.toml`. Lo verás en A01 y en todos los proyectos del curso.

### Sintaxis en 30 segundos

```toml
# esto es un comentario válido en TOML
[project]
name = "acme-audit"
version = "0.1.0"
requires-python = ">=3.12"
dependencies = [
    "anthropic>=0.40",
    "fastapi>=0.115",
]

[tool.uv]
dev-dependencies = [
    "pytest>=8.3",
]
```

Lo que diferencia a TOML de JSON y YAML:

| Característica | TOML | JSON | YAML |
|---|---|---|---|
| Uso principal | Configuración de proyectos | Intercambio de datos | Configuración de sistemas |
| Secciones | `[nombre-seccion]` | Objetos anidados `{}` | Indentación |
| Comentarios | `# comentario` | No | `# comentario` |
| Strings | Comillas simples o dobles | Solo dobles | Sin comillas en la mayoría de casos |
| Listas | Corchetes `[]`, una por línea | Corchetes `[]` | Guiones `-` con indentación |

### Por qué importa en este curso

`pyproject.toml` tiene dos secciones que usarás siempre:

- **`[project].dependencies`**: las librerías que tu código necesita en producción (`anthropic`, `fastapi`).
- **`[tool.uv].dev-dependencies`**: las herramientas que solo usas al desarrollar (`pytest`, `ruff`). No se instalan en el servidor del cliente.

Cuando en A01 veas `requires-python = ">=3.12"`, ya sabes que esa línea es TOML: sección `[project]`, clave `requires-python`, valor string `">=3.12"`.

> [!nota]
> No necesitas escribir TOML a mano: `uv init` genera el `pyproject.toml` inicial y `uv add` actualiza las dependencias automáticamente. Pero leerlo sí importa — cuando el agente falla en producción, la primera pista suele estar en ese archivo.

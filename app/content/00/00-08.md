## Hilo conductor

Hasta ahora has ejecutado Python en el navegador (Pyodide). Eso funciona para aprender. Para construir scripts reales, automatizar procesos de ACME, y seguir Tracks A–E, necesitas Python instalado en tu máquina. Este módulo cubre **cómo** y **con qué herramienta**. La respuesta a ambas preguntas es `uv`.

## Puente desde Excel

Cuando instalas Excel, instalas el programa que ejecuta libros `.xlsx`. Cuando instalas Python, instalas el programa que ejecuta archivos `.py`. Pero Python tiene una complicación extra: hay múltiples versiones (3.11, 3.12, 3.13) y miles de librerías externas que tu proyecto puede necesitar. `uv` es el instalador que resuelve todo eso: instala la versión correcta de Python, las librerías que tu proyecto declara, y garantiza que otro equipo obtenga exactamente el mismo resultado.

## Idea central

**Python** (el lenguaje) y **`python3`** (el ejecutable) son cosas distintas: el lenguaje es la especificación; el ejecutable es el programa que corre en tu sistema operativo. **`uv`** es una herramienta de Astral (2024) que reemplaza a `pip`, `virtualenv`, `pyenv`, y `pip-tools` en un solo comando. Instalar Python con `uv` garantiza que todos los que trabajen en el proyecto usen la misma versión. En este curso, `pip install` es un error siempre que aparezca.

## Por qué importa

Un script de auditoría de ACME que funciona en tu laptop puede fallar en el servidor del cliente si tienen versiones distintas de Python o de las librerías. Eso no es un bug del código: es un bug del entorno. El `uv.lock` que genera `uv` registra la versión exacta de cada librería con su hash de verificación. Cuando el equipo de ACME ejecuta `uv sync` en un servidor nuevo, obtiene exactamente el mismo entorno que tú tienes. Sin sorpresas.

Este módulo es el puente crítico: A01 asume que tienes `uv` instalado, Python 3.12 disponible vía `uv`, y sabes ejecutar `uv run python archivo.py`. Sin este paso, A01 y todos los tracks técnicos quedan bloqueados.

> [!cuidado]
> El curso nunca usa `pip install`. Si ves ese comando en un tutorial externo, es código antiguo. El comando correcto en este curso es siempre `uv add <paquete>` para añadir una dependencia, o `uv sync` para instalar las dependencias ya declaradas.

## Cómo funciona por dentro

### Qué es PyPI

**PyPI** (Python Package Index, `pypi.org`) es el registro público de librerías Python. Tiene más de 500 000 paquetes. Cuando tu proyecto necesita `anthropic`, `fastapi`, o `pandas`, esas librerías se descargan de PyPI. `uv` usa PyPI por defecto.

En Excel, el equivalente sería el marketplace de complementos de Office. Pero en Python es gratis y cualquiera puede publicar.

### Qué hace `uv` exactamente

`uv` gestiona cuatro cosas:

1. **Versiones de Python**: descarga e instala la versión que le pidas (`3.12`, `3.13`, etc.) sin tocar el Python del sistema operativo. Cada proyecto puede usar una versión diferente.

2. **Entorno virtual (`.venv/`)**: crea un directorio aislado para las librerías de tu proyecto. Dos proyectos con librerías incompatibles no interfieren entre sí.

3. **`pyproject.toml`**: el archivo de configuración del proyecto. Declara el nombre, la versión de Python requerida, y las dependencias con rangos de versión (`anthropic>=0.40`). Es el archivo que commiteas al repositorio.

4. **`uv.lock`**: el snapshot exacto de cada librería y subdependencia, con hashes de verificación. Garantiza reproducibilidad. También se commitea.

El directorio `.venv/` **no** se commitea (está en `.gitignore`). Se regenera en segundos con `uv sync`.

### Instalación de `uv`

**macOS y Linux** (una línea en la terminal):

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

**Windows** (PowerShell):

```powershell
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

Después de instalar, cierra y vuelve a abrir la terminal. Verifica:

```bash
uv --version
```

Debes ver algo como `uv 0.7.x (...)`. Si aparece «comando no encontrado», el instalador no añadió `uv` al PATH. En macOS/Linux, añade `source $HOME/.local/bin/env` a tu `~/.bashrc` o `~/.zshrc` y recarga la terminal.

> [!nota]
> En Windows, el curso asume **WSL2 con Ubuntu**. Si todavía no lo tienes, instálalo antes de continuar: `wsl --install` en PowerShell y sigue las instrucciones. Todos los comandos de esta sección y del curso funcionan en el shell de Ubuntu dentro de WSL2.

### Instalar Python 3.12 con `uv`

```bash
uv python install 3.12
```

`uv` descarga el intérprete oficial de Python 3.12 y lo guarda en su directorio interno. No modifica el Python del sistema. Verifica:

```bash
uv run python --version
```

Debe mostrar `Python 3.12.x`. Si ves una versión distinta, `uv` está usando la del sistema; ejecuta `uv python pin 3.12` para anclarla en el directorio actual.

### Los cuatro comandos que usarás en el curso

```bash
# crear un nuevo proyecto
uv init nombre-del-proyecto

# añadir una dependencia (actualiza pyproject.toml y uv.lock)
uv add anthropic

# instalar todas las dependencias declaradas en pyproject.toml
uv sync

# ejecutar un script dentro del entorno del proyecto
uv run python audit.py
```

`uv init` crea la estructura mínima: `pyproject.toml`, `README.md`, y un `.python-version` con la versión de Python del sistema. `uv run` activa el entorno virtual del proyecto y ejecuta el comando dentro de él, sin que tengas que activar el entorno manualmente.

### Y ¿qué es `pip`?

`pip` es la herramienta original para instalar librerías en Python. Existe desde 2011 y la encontrarás en casi todos los tutoriales de internet. El problema es que `pip install` instala la librería en el entorno activo del sistema, sin registrar nada en un archivo del proyecto, sin verificar que las versiones sean reproducibles, y sin crear la caja aislada que protege un proyecto de otro.

`uv` hace lo mismo que `pip` pero más rápido y de forma reproducible: descarga la librería de PyPI, la instala en la caja del proyecto, y además actualiza `pyproject.toml` y `uv.lock` automáticamente.

La regla práctica para este curso: cuando veas `pip install anthropic` en un tutorial externo, tradúcelo mentalmente a `uv add anthropic`. El resultado final (la librería instalada) es el mismo; la diferencia está en que `uv` deja rastro en el proyecto.

### Qué NO entra en este módulo

- `pyproject.toml` en detalle: lo cubre A01.
- `uv.lock` en detalle: lo cubre A01.
- Entornos virtuales por dentro: lo cubre A01.
- `uv add` con dependencias de desarrollo y opcionales: lo cubre A01.
- `pip install` no aparece en este curso. Si lo ves, es un error.

## Ejemplo conducido

<!-- INLINE_QUIZ: uv-lock-purpose -->

Este es el flujo completo desde cero hasta tener un proyecto funcionando:

```bash
# 1. verificar que uv está instalado
uv --version

# 2. instalar Python 3.12 si no lo tienes
uv python install 3.12

# 3. crear el proyecto de auditoría de ACME
uv init acme-audit
cd acme-audit

# 4. verificar qué creó uv
ls
# pyproject.toml  README.md  .python-version  hello.py

# 5. ejecutar el script que viene por defecto
uv run python hello.py
# Hello from acme-audit!
```

En el directorio `acme-audit/` tienes ahora un entorno Python 3.12 aislado, listo para añadir dependencias con `uv add`.

## Errores típicos

**1. `uv: command not found` después de instalar.**
*Síntoma*: la terminal no reconoce `uv`.
*Causa*: el instalador añadió `uv` a una ruta que la shell actual no conoce.
*Arreglo*: cierra y vuelve a abrir la terminal. Si persiste, en macOS/Linux ejecuta `source $HOME/.local/bin/env` y añade esa línea a `~/.bashrc` o `~/.zshrc`.

**2. `uv run python --version` muestra la versión del sistema, no 3.12.**
*Síntoma*: `Python 3.11.x` en lugar de `Python 3.12.x`.
*Causa*: `uv` usa la versión del sistema si no hay una `.python-version` en el directorio.
*Arreglo*: ejecuta `uv python pin 3.12` en el directorio del proyecto para crear el archivo `.python-version`.

**3. Usar `pip install` en lugar de `uv add`.**
*Síntoma*: la dependencia se instala pero no aparece en `pyproject.toml` ni en `uv.lock`.
*Causa*: `pip install` instala en el entorno pero no actualiza los archivos del proyecto.
*Arreglo*: desinstala con `pip uninstall` y vuelve a instalar con `uv add`. El equipo que haga `uv sync` no verá esa dependencia si no está en `pyproject.toml`.

**4. Commitear el directorio `.venv/`.**
*Síntoma*: el repositorio crece cientos de MB con binarios.
*Causa*: el `.gitignore` no excluye `.venv/`.
*Arreglo*: añade `.venv/` a `.gitignore`. El `.venv/` se regenera con `uv sync` en cualquier máquina.

## Hilo conductor

En B02 viste cómo FastAPI valida y despacha requests. En A05 aprendiste que los imports tienen dirección. Este módulo une ambas ideas: la dirección de los imports define la arquitectura. Cuando termines, cualquier feature de la app tendrá exactamente cuatro archivos, y sabrás cuál hace qué. Eso prepara C01, donde cada `repo` recibe su `AsyncSession`.

---

## Idea central

La regla es una sola: **las dependencias fluyen hacia adentro**.

```
routers → services → repos → models
```

`routers` importa de `services`. `services` importa de `repos`. `repos` importa de `models`. `models` no importa de nadie de la app. Ninguna capa salta hacia fuera. Eso es todo. El resto — nombres de archivos, firmas de funciones, inyección de sesión — son consecuencias de esa regla.

## Por qué importa

Sin separación de capas, el router termina siendo un monolito: valida el body, consulta la DB directamente, mezcla reglas de negocio con serialización. Cuando cambias el schema de la DB, tocas el router. Cuando cambias la respuesta HTTP, tocas las queries. Cada cambio propaga por todo el archivo. Una feature nueva en producción se convierte en una operación de arqueología.

El caso real: en BATUTA el agente de conciliación (F-CTA-04) inicia un proceso que involucra ingestión de extractos, queries de match y generación de reportes. Si todo eso vive en el router, el día que cambias el proveedor de datos bancarios reescribes el endpoint entero. Con capas, cambias el repo y el servicio sabe que recibe `list[Transaction]` — no sabe de dónde vienen.

> B04 introduce el **patrón** de capas sobre la app existente. C01 profundiza en la capa de modelos (ORM). C02 cubre cómo el schema de los modelos viaja a la DB vía migraciones.

## Cómo funciona por dentro

### La regla operativa

Una capa nunca importa desde una capa más externa. Formalmente:

| Capa | Responsabilidad | Importa de | No importa de |
|------|----------------|------------|---------------|
| `routers/` | Parsing HTTP, validación Pydantic, formatar respuesta | `services/`, `deps.py`, `templating` | `repos/`, `models/` directamente |
| `services/` | Lógica de negocio, orquestar repos, decidir qué hacer | `repos/`, `models/` | `routers/`, FastAPI, `Request` |
| `repos/` | Queries SQL concretas, una operación por función | `models/` | `services/`, `routers/` |
| `models/` | Definición de tablas ORM | `db.py` (`Base`) | nada de la app |

### Por qué no «todo en el router»

El router recibe y responde HTTP. Mezclar queries ahí hace que el router conozca dos responsabilidades distintas: el protocolo y la persistencia. Si cambias de SQLite a Postgres (C03), tienes que buscar por todo el archivo. Si quieres testear la lógica de negocio, tienes que levantar un servidor HTTP.

El **servicio** es la capa que sabe *qué hacer* con los datos. No sabe si llega por HTTP, por CLI o por un workflow Temporal. Eso hace que el mismo servicio sirva para el router y para los scripts de seed.

El **repo** sabe *cómo obtener* datos de la DB. Recibe una `AsyncSession` por parámetro — no la crea. Eso permite que el llamador (el servicio, o el router vía `Depends`) controle el ciclo de vida de la sesión y pueda hacer rollback si algo falla.

### Cómo se ve en la app real

`app/routers/pages.py` es el ejemplo canónico. Lee el módulo del repo, registra la visita del usuario, pasa los datos a Jinja2. No tiene ninguna lógica de negocio propia; por eso no pasa por un servicio intermedio — la lógica es trivial (leer + renderizar). Pero sí llama a dos repos: `modules_repo.get_by_slug` y `progress_repo.touch_visit`. Si la lógica fuera más compleja (validar prereqs, calcular score, decidir si desbloquear), iría a un servicio.

`app/repos/progress.py` define `add_attempt`, `touch_visit`, `progress_summary`. Cada función hace una cosa. Recibe `AsyncSession`. No sabe que existe FastAPI.

### Cuándo se rompe la regla (y por qué)

Hay casos donde el router llama directo al repo sin pasar por un servicio. `app/routers/progress.py` es el ejemplo: llama directamente a `progress_repo.get_or_create_user` y `progress_repo.progress_summary`. Esto es aceptable cuando no hay lógica de negocio que orquestar, y añadir un servicio sería una capa vacía. La regla no es «siempre cuatro archivos»; es «las capas no saltan hacia fuera». Si el router llama al repo directamente pero nunca al revés, la dirección sigue siendo correcta.

El anti-patrón que sí rompe la arquitectura: el repo importa del router (para leer headers HTTP) o el modelo importa del repo (para ejecutar queries desde el propio `__init__`). Eso es un ciclo de importación disfrazado.

## Ejemplo conducido

Router monolítico que hace todo:

```python
# ANTES: router que mezcla HTTP + lógica + DB (no hacer esto)
@router.get("/api/progress")
async def get_progress(session: SessionDep):
    user = await session.get(User, 1)
    if user is None:
        user = User(id=1, name="tú")
        session.add(user)
        await session.commit()
    total = (await session.execute(
        select(func.count()).select_from(Module)
    )).scalar_one()
    return {"user": user.name, "modules_total": total}
```

Separado en capas:

```python
# models/user.py — define la tabla, nada más
class User(Base):
    __tablename__ = "user"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(64), server_default="tú")

# repos/progress.py — queries concretas, recibe session
async def get_or_create_user(session: AsyncSession, name: str = "tú") -> User:
    user = await session.get(User, 1)
    if user is None:
        user = User(id=1, name=name)
        session.add(user)
        await session.commit()
    return user

# routers/progress.py — parsea HTTP, llama repo, devuelve JSON
@router.get("/api/progress")
async def get_progress(session: SessionDep) -> dict:
    user = await progress_repo.get_or_create_user(session)
    summary = await progress_repo.progress_summary(session, user.id)
    visited = sorted(await progress_repo.visited_module_ids(session, user.id))
    return {"summary": summary, "visited_module_ids": visited}
```

El router pasó de 12 líneas de lógica mezclada a 4 líneas de glue. Los repos son testeables sin HTTP. Los modelos son leíbles sin queries.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Parsear y validar el body de la request | determinístico | Pydantic aplica reglas cerradas. |
| Decidir qué query ejecutar dado el estado del usuario | determinístico | Regla de negocio codificada en el servicio. |
| Clasificar si un hallazgo de auditoría es error o timing | agéntico | Requiere contexto del cliente; no hay regla cerrada. |
| Formatear la respuesta JSON | determinístico | Serialización mecánica. |
| Decidir si escalar un finding a revisión humana | agéntico | El umbral depende del contexto; no es configurable de forma cerrada. |

En la arquitectura de capas, los tramos agénticos viven en el servicio (que llama al SDK de Anthropic). El repo nunca invoca el LLM. El router nunca decide si algo es anomalía.

## Errores típicos

**1. El repo crea su propia sesión.**
*Síntoma*: la función del repo hace `async with get_session() as session:` internamente.
*Causa*: facilidad inmediata — funciona en aislamiento. Pero el llamador pierde control del ciclo de vida. No puedes hacer rollback desde el servicio.
*Arreglo*: el repo siempre recibe `session: AsyncSession` por parámetro. El llamador (router vía `Depends`, servicio, script) abre y cierra la sesión.

**2. El servicio importa `Request` de FastAPI.**
*Síntoma*: `from fastapi import Request` en `app/services/`.
*Causa*: el desarrollador quiso leer un header desde el servicio.
*Arreglo*: extrae el valor del header en el router y pásalo como parámetro al servicio. El servicio recibe un `str`, no un objeto HTTP.

**3. El modelo define una función que ejecuta queries.**
*Síntoma*: un método `Invoice.get_unpaid(session)` en `app/models/`.
*Causa*: patrones del ORM antiguo (active record). SQLAlchemy 2.x con `DeclarativeBase` no invita a esto, pero es posible.
*Arreglo*: esa query va en `repos/invoices.py` como función libre. El modelo define columnas y relaciones; nada más.

**4. El router llama a dos servicios distintos que comparten una transacción.**
*Síntoma*: el router hace `await service_a.do_x(session)` y `await service_b.do_y(session)` sin envolver ambos en `session.begin()`.
*Causa*: se asume que cada función commitea por sí sola.
*Arreglo*: si dos operaciones deben ser atómicas, se envuelven en un servicio que las orqueste bajo una sola transacción, o el router abre `async with session.begin():` explícito.

## Caso de uso real

**F-CX-02 — Resumen automático de llamadas de soporte.**

El flujo: el router `POST /api/calls/{call_id}/summarize` recibe el ID de la llamada. Un servicio `call_service.summarize(call_id, tenant)` orquesta tres repos: `call_repo.get_transcript(session, call_id)` trae el texto, `nlp_repo.get_entities(session, call_id)` trae entidades extraídas, y finalmente el servicio construye el prompt y llama al SDK de Anthropic. El resultado lo persiste `call_repo.save_summary(session, call_id, summary_md)`. El router devuelve 202 con el `summary_id`.

Cada capa hace su parte. Si mañana cambias el proveedor de NLP (de un modelo local a un servicio externo), cambias `nlp_repo` y el servicio sigue recibiendo `list[Entity]`. El router no sabe que hubo un cambio de proveedor.

## Salida esperada

Al terminar este módulo puedes:

## 1. Hilo conductor

En A01 viste cómo Python ejecuta un archivo. Ahora vas a ver lo que pasa antes de que tu código siquiera arranque: el viaje de una URL desde el teclado hasta el servidor y de vuelta. Todo lo que construyes en B02 en adelante vive dentro de este ciclo. B02 entra de lleno en FastAPI; aquí solo ves el protocolo.

## 2. Idea central

Antes de que HTTP ocurra, hay tres pasos previos que conviene nombrar desde ya: **DNS** traduce el nombre de dominio a una IP, como una agenda telefónica que convierte "acme.com" en un número. **TCP** abre el canal verificado entre tu computador y el servidor, como marcar el número y esperar que descuelguen. **TLS** cifra esa línea para que nadie en el camino pueda leer los datos, como hablar en un idioma secreto acordado al inicio de la llamada. Solo después de esos tres pasos empieza HTTP. En desarrollo local (`localhost`) no hay DNS ni TLS — son pasos que aparecen en producción.

**HTTP** (HyperText Transfer Protocol) es un protocolo de texto sobre TCP: el cliente envía un mensaje estructurado (verbo + ruta + headers + body), el servidor responde con otro (status code + headers + body). Sin estado: cada request es independiente; si necesitas recordar al usuario entre requests, lo haces tú con cookies o tokens. La imagen mental correcta es una conversación de fax, no una llamada telefónica.

## Puente desde Excel

Imagina que en Excel tienes una función `=BUSCAR_PRECIO("ACME-001")` que va a otro libro, busca el precio y lo trae. El otro libro es el **servidor**; tu fórmula es el **cliente**. HTTP es el protocolo que se hablan, como si Excel pasara una hoja al otro libro diciendo "necesito el precio de ACME-001" y el otro respondiera "12.50".

| Excel | Web/HTTP |
|-------|----------|
| Tu hoja con `=BUSCAR_PRECIO("ACME-001")` | Cliente (navegador, app móvil, otro servidor) |
| El otro libro que tiene los precios | Servidor (la API) |
| El nombre del producto | Request (verbo + URL + body) |
| El número devuelto | Response (status + headers + body) |
| Error #N/A | 404 Not Found |
| Error si el otro libro está cerrado | 503 Service Unavailable |

HTTP es 100% determinístico: misma pregunta → misma estructura de respuesta. La parte agéntica entra cuando el servidor del otro lado es un agente IA.

## 3. Por qué importa

Sin entender HTTP no puedes leer un error de producción. Cuando ACME tiene un webhook de Siigo que falla, el log dice `422 Unprocessable Entity` con un body JSON. Si no distingues un 422 de un 400, o un error de cliente de uno de servidor, pierdes 30 minutos buscando en el lugar equivocado.

Esto conecta directamente con la ficha F-CX-01 (triage de tickets vía webhook entrante): el agente recibe un payload HTTP, lo valida, y decide si escala o resuelve. Si el payload llega malformado, el endpoint debe responder 400, no 500 — y el agente debe manejarlo.

## 4. Cómo funciona por dentro

### El camino completo de una URL

Cuando escribes `http://localhost:8080/health` y presionas Enter:

1. **DNS** — el navegador resuelve el nombre a una IP. Para `localhost`, apunta a `127.0.0.1` sin red. En prod, un resolver consulta la jerarquía DNS hasta llegar al registro A o AAAA del dominio.

2. **TCP handshake** — cliente envía `SYN`, servidor responde `SYN-ACK`, cliente confirma con `ACK`. Solo después de los tres pasos el canal está abierto y se pueden enviar bytes.

3. **TLS handshake** (solo en HTTPS) — se negocia la versión (TLS 1.3 en 2026), se intercambian certificados, se acuerda una clave simétrica. Todo esto antes de que el HTTP ocurra. En desarrollo local se usa HTTP plano; en prod (módulo D03) Traefik gestiona TLS automático.

4. **HTTP request** — el cliente envía:

```
GET /health HTTP/1.1
Host: localhost:8080
Accept: application/json
```

5. **HTTP response** — el servidor responde:

```
HTTP/1.1 200 OK
content-type: application/json
content-length: 15

{"status": "ok"}
```

6. **TCP cierre** — si el header `Connection: keep-alive` está activo (el default en HTTP/1.1), la conexión se reutiliza para la siguiente request. Esto importa para el rendimiento.

### Métodos HTTP y su semántica

| Método | Semántica | Body en request | Idempotente |
|--------|-----------|-----------------|-------------|
| GET | Obtener recurso | No | Sí |
| POST | Crear o procesar | Sí | No |
| PUT | Reemplazar recurso completo | Sí | Sí |
| PATCH | Modificar parcialmente | Sí | No necesariamente |
| DELETE | Eliminar recurso | No (opcional) | Sí |

**Idempotente** significa que hacer la misma operación N veces produce el mismo resultado que hacerla una sola vez. Importa cuando un cliente reintenta por timeout.

### Status codes que importan

| Rango | Significado | Ejemplos concretos |
|-------|-------------|-------------------|
| 2xx | Éxito | 200 OK, 201 Created, 204 No Content |
| 3xx | Redirección | 301 Moved Permanently, 302 Found |
| 4xx | Error del **cliente** | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 422 Unprocessable Entity |
| 5xx | Error del **servidor** | 500 Internal Server Error, 503 Service Unavailable |

La distinción 4xx vs 5xx no es cosmética: 4xx implica que el cliente puede corregir la request; 5xx implica que el problema está en el servidor y el cliente debe reintentar después.

> [!cuidado]
> 401 y 403 no son sinónimos. **401** significa «no sé quién eres» (falta autenticación). **403** significa «sé quién eres, pero no puedes». Confundirlos produce bugs de seguridad difíciles de depurar.

### Headers que importan

- **`Content-Type`**: describe el formato del body. `application/json` para JSON, `text/html; charset=utf-8` para HTML. Si el servidor espera JSON y recibes `text/plain`, el parser falla antes de que tu código llegue.
- **`Authorization`**: transporta credenciales. Dos formas comunes: `Bearer <jwt_token>` (stateless) y `Basic <base64(user:pass)>` (legacy). En bato-learning no tenemos auth por ahora; aparece en E04.
- **`Cache-Control`**: instruye a caches intermedios. `no-store` prohíbe cacheado. `max-age=3600` autoriza cachear 1 hora. Ignorar esto en una API pública puede devolver datos obsoletos durante horas.
- **`Cookie`** / **`Set-Cookie`**: el servidor deposita un valor con `Set-Cookie`; el navegador lo devuelve en todas las requests siguientes con `Cookie`. La base de las sesiones web clásicas.
- **`HX-Request`**: header que HTMX 2.x añade automáticamente a sus requests. FastAPI lo usa para distinguir si una request viene del frontend HTMX o de un cliente externo (ver B03).

### Cookies vs JWT: cuándo usar cada uno

| Criterio | Cookie de sesión | JWT (Bearer token) |
|----------|------------------|--------------------|
| Estado en servidor | Sí (session store) | No (stateless) |
| Revocación | Inmediata (borras la sesión) | Hasta que expire el token |
| Multi-dominio | Complicado (SameSite) | Fácil (header Authorization) |
| Uso típico | Apps web tradicionales | APIs consumidas por móvil/SPA |

bato-learning usa una cookie de sesión simple (sin auth completa). Los agentes productivos que construirás en Track E usan JWT para llamadas entre servicios.

### Query params, path params y body

```
GET /api/modules?track=B&page=1           # query params: filtros opcionales
GET /api/modules/http-and-the-browser     # path param: identificador del recurso
POST /api/exercises/42/backend            # path param + body JSON con el código
```

Regla práctica: el path identifica **qué** recurso. El query param filtra o pagina. El body lleva datos que cambiarían el estado (creación, actualización).

### SSE y WebSocket: una línea por ahora

**SSE** (Server-Sent Events): stream unidireccional servidor→cliente sobre HTTP/1.1. El servidor deja la conexión abierta y empuja eventos con `data: ...\n\n`. El chat tutor de bato-learning usa esto (ver B06). **WebSocket**: canal bidireccional full-duplex. Necesario para juegos en tiempo real o colaboración simultánea; overkill para chat unidireccional.

## 5. Ejemplo conducido

La app ya corre. Ejecuta estos comandos y observa cada parte de la respuesta:

```bash
# request GET simple, muestra headers completos
curl -v http://localhost:8080/health

# POST con body JSON — simula un intento de ejercicio
curl -X POST http://localhost:8080/api/exercises/1/pyodide \
  -H "Content-Type: application/json" \
  -d '{"code": "print(1)", "passed": true, "output": "1", "error": "", "duration_ms": 12}'

# forzar un 404: slug inexistente
curl -i http://localhost:8080/m/slug-que-no-existe
```

Con `-v` verás el TCP handshake (para HTTPS) y los headers completos de request y response. Con `-i` ves solo los headers de respuesta sin el verbose del handshake.

```bash
# leer solo el status code
curl -o /dev/null -s -w "%{http_code}\n" http://localhost:8080/health
# → 200
curl -o /dev/null -s -w "%{http_code}\n" http://localhost:8080/m/no-existe
# → 404
```

## 8. Caso de uso real

**F-CX-01 — Triage de tickets vía webhook**: cuando Clínica San Rafael recibe un ticket de soporte vía webhook de Zendesk, el sistema valida el `Content-Type: application/json` y el `X-Zendesk-Webhook-Id` en headers antes de procesar. Un 401 en el webhook significa que la firma HMAC no matchea (el cliente envió datos sin firmar). Un 422 significa que el payload llegó pero le falta el campo `ticket.subject`. El agente solo puede actuar si el servidor devuelve 200 — conocer qué status devolver en cada caso es la diferencia entre un webhook robusto y uno que silencia errores.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Parsear los headers de la request y validar `Content-Type` | Determinístico | Regla cerrada: `application/json` o falla. |
| Devolver el status code correcto según el resultado | Determinístico | Tabla fija: 200/201/400/404/422/500. |
| Verificar firma HMAC del webhook | Determinístico | Operación criptográfica. |
| Clasificar el tono de un ticket de soporte (urgente/normal) | Agéntico | El texto libre no tiene regla cerrada de clasificación. |
| Redactar una respuesta inicial al cliente | Agéntico | Salida para humano, depende de contexto del ticket. |

## 10. Errores típicos

**1. Confundir 401 con 403.**
*Síntoma*: el cliente recibe 401 con token válido pero sin permisos suficientes.
*Causa*: el endpoint devuelve 401 en todas las fallas de auth, sin distinguir «falta token» de «token sin permiso».
*Arreglo*: `401` cuando no hay `Authorization` header o el token no se puede verificar; `403` cuando el token es válido pero el rol no tiene permiso.

**2. POST que no envía `Content-Type: application/json`.**
*Síntoma*: FastAPI devuelve `422 Unprocessable Entity` con `body → value is not a valid dict`.
*Causa*: `curl` por default usa `Content-Type: application/x-www-form-urlencoded`. FastAPI espera JSON.
*Arreglo*: añadir `-H "Content-Type: application/json"` a la llamada curl.

**3. Cachear una respuesta que no debe cachearse.**
*Síntoma*: el cliente recibe datos obsoletos durante horas; refrescar la página no ayuda.
*Causa*: el servidor no envía `Cache-Control: no-store` en endpoints que devuelven datos del usuario.
*Arreglo*: añadir el header explícitamente en el router de FastAPI o como middleware.

**4. Tratar 5xx como error del cliente.**
*Síntoma*: el código de integración lanza un error no recuperable cuando el servidor está caído.
*Causa*: el handler de errores no distingue 4xx (no reintentar) de 5xx (reintentar con backoff).
*Arreglo*: política de reintentos solo para 5xx y 429 (rate limit); 4xx se propaga como error del caller.

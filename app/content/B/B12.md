## 1. Hilo conductor

En B08 definiste el contrato de una API REST con sus rutas, verbos y schemas. En B11 viste que los clientes reintentan cuando el servidor falla. Este módulo responde la pregunta que aparece en producción: ¿qué pasa cuando necesitas cambiar ese contrato? Cambiar una API sin versionar rompe a todos los clientes existentes. B13 (testing) conecta directamente: los tests de regresión de versiones anteriores son la única forma de detectar si rompiste algo.

---

## 2. Idea central

**Versionado de API** es la práctica de mantener múltiples versiones estables del contrato de una API al mismo tiempo, para que los clientes existentes no se rompan cuando evoluciona. El mecanismo más común y explícito es el prefijo de versión en el path: `/api/v1/invoices` vs `/api/v2/invoices`.

Imagen mental: versionar una API es como publicar una nueva edición de un libro de contabilidad. La edición 2 tiene nuevas secciones y algunas tablas reorganizadas. Los contadores que usan la edición 1 siguen trabajando — sus fórmulas apuntan a la edición 1. Solo migran a la 2 cuando quieren las nuevas funcionalidades.

## 3. Por qué importa

ACME Manufacturing tiene un agente de auditoría que llama a `POST /api/v1/audits`. Si el equipo de plataforma cambia el schema del body de `v1` (por ejemplo, renombra `period` a `audit_period`), el agente deja de funcionar sin aviso. La solución no es «los clientes que migren» — es mantener `v1` funcionando mientras `v2` existe, con un plan de deprecación explícito.

El costo de no versionar es alto: un cambio de schema rompe agentes en producción, webhook receivers, integraciones con ERPs externos — cualquier caller que no controles. El costo de versionar también existe: mantener `v1` y `v2` en paralelo duplica el código de tests y puede crear inconsistencias si la DB cambia bajo ambas versiones.

---

## 4. Cómo funciona por dentro

### Breaking vs non-breaking changes

Un **breaking change** rompe a los clientes existentes si no actualizan su código:

- Eliminar un campo del response (`amount` deja de existir)
- Renombrar un campo (`period` → `audit_period`)
- Cambiar el tipo de un campo (`amount: float` → `amount: str`)
- Cambiar el método HTTP de un endpoint (`POST` → `PUT`)
- Eliminar un endpoint
- Cambiar la semántica de un status code

Un **non-breaking change** es retrocompatible — los clientes existentes siguen funcionando:

- Añadir un campo opcional al response (los clientes que no lo usan lo ignoran)
- Añadir un campo opcional al request con un default value
- Añadir un nuevo endpoint
- Añadir un nuevo status code que el cliente no esperaba (si trata 2xx como éxito genérico)

Regla práctica: **expandir es seguro; contraer o renombrar es breaking**.

### Versionado en path

La forma más explícita y recomendada: el número de versión va en el prefijo del path.

```python
# app/routers/invoices_v1.py
from fastapi import APIRouter

router_v1 = APIRouter(prefix="/api/v1/invoices", tags=["invoices-v1"])

@router_v1.post("")
async def create_invoice_v1(body: InvoiceV1Body) -> InvoiceV1Response:
    ...

# app/routers/invoices_v2.py
router_v2 = APIRouter(prefix="/api/v2/invoices", tags=["invoices-v2"])

@router_v2.post("")
async def create_invoice_v2(body: InvoiceV2Body) -> InvoiceV2Response:
    ...

# app/main.py
app.include_router(router_v1)
app.include_router(router_v2)
```

Las dos versiones coexisten. `/api/v1/invoices` sigue funcionando para ACME Manufacturing. `/api/v2/invoices` está disponible para nuevos clientes o para los que migraron voluntariamente.

### Estrategia expand-contract

Para migrar clientes de `v1` a `v2` sin downtime:

1. **Expand** (non-breaking): añade el campo nuevo (`audit_period`) manteniendo el viejo (`period`) en la misma versión. Los clientes pueden empezar a usar `audit_period` mientras migran.
2. **Contract** (en la nueva versión): elimina el campo viejo (`period`) en `v2`. Solo cuando todos los clientes migraron.

```
v1:  { "period": "2026-04", "audit_period": "2026-04" }  ← transitorio
v2:  { "audit_period": "2026-04" }                       ← limpio
```

### Deprecation headers

Cuando una versión va a ser eliminada, el servidor la anuncia en los headers:

```
HTTP/1.1 200 OK
Deprecation: true
Sunset: Sat, 31 Dec 2026 23:59:59 GMT
Link: </api/v2/invoices>; rel="successor-version"
```

- `Deprecation: true` — este endpoint está deprecado.
- `Sunset` — fecha en que dejará de funcionar (RFC 8594).
- `Link` — dónde está la versión nueva.

Los clientes que procesan estos headers pueden avisar automáticamente al equipo antes de que el endpoint desaparezca.

### El coste de soportar versiones antiguas

Mantener `v1` en producción tiene un coste real:

- **Tests**: debes testear `v1` y `v2` en cada deploy. Un cambio en la lógica de negocio puede afectar a las dos versiones de formas distintas.
- **Migraciones de DB**: si añades un campo nuevo en `v2` que no existe en `v1`, ¿qué devuelve `v1` cuando ese campo existe en el objeto? Hay que manejar los dos casos.
- **Bugs**: un bug arreglado en `v2` puede no estar arreglado en `v1` si el código está duplicado.
- **Deuda cognitiva**: el equipo tiene que recordar las diferencias entre versiones.

Por eso el plan de deprecación es obligatorio desde el momento en que publicas `v2`. No hay «v1 se queda para siempre».

### Versionado en bato-learning

La app tiene un solo número de versión en su API interna (sin `/v1/` explícito por ahora — es una app de aprendizaje single-tenant). Pero el principio aplica: los módulos del frontend (JavaScript de HTMX, el runner de Pyodide) consumen endpoints como `POST /api/exercises/{id}/pyodide`. Si ese contrato cambia, hay que versionar o migrar los clientes con cuidado.

## 5. Ejemplo conducido

Dos versiones de un endpoint de facturas coexistiendo:

```python
from pydantic import BaseModel
from fastapi import APIRouter

class InvoiceBodyV1(BaseModel):
    vendor_id: str
    amount: float
    period: str  # "YYYY-MM"

class InvoiceBodyV2(BaseModel):
    vendor_id: str
    amount: float
    audit_period: str   # renombrado de period
    currency: str = "EUR"  # campo nuevo con default

router_v1 = APIRouter(prefix="/api/v1")
router_v2 = APIRouter(prefix="/api/v2")

@router_v1.post("/invoices")
async def create_v1(body: InvoiceBodyV1) -> dict:
    # v1 sigue funcionando — los clientes no se rompen
    return {"id": "inv-001", "period": body.period, "amount": body.amount}

@router_v2.post("/invoices")
async def create_v2(body: InvoiceBodyV2) -> dict:
    # v2 tiene el campo renombrado y el nuevo campo currency
    return {
        "id": "inv-001",
        "audit_period": body.audit_period,
        "amount": body.amount,
        "currency": body.currency,
    }
```

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Enrutar `POST /api/v1/invoices` al handler de v1 | Determinístico | El prefix matching es código puro — mismo path, mismo handler. |
| Detectar si un campo es breaking o non-breaking | Determinístico | La definición es binaria: eliminar/renombrar es breaking; añadir con default no lo es. |
| Parsear el header `Sunset` y calcular días hasta deprecación | Determinístico | Aritmética de fechas con regla fija. |
| Decidir cuándo deprecar v1 y qué timeline dar a los clientes | Agéntico / estratégico | Depende de cuántos clientes hay, su velocidad de migración, el SLA del contrato. Sin regla única. |

## 10. Errores típicos

**1. Hacer un breaking change sin versionar.**
*Síntoma*: el agente de ACME empieza a fallar con `422 Unprocessable Entity` después de un deploy.
*Causa*: se renombró un campo requerido en el request body sin crear una nueva versión.
*Arreglo*: cualquier breaking change requiere un nuevo prefijo de versión (`/api/v2/`). La versión antigua sigue en pie hasta el sunset.

**2. Deprecar sin anunciar el sunset.**
*Síntoma*: los clientes descubren que el endpoint desapareció el día que lo eliminaste.
*Causa*: no se publicó un timeline de deprecación ni se añadieron los headers `Deprecation` y `Sunset`.
*Arreglo*: añadir los headers al menos 90 días antes del sunset. Comunicar activamente a los consumidores conocidos.

**3. Copiar el código de v1 a v2 en lugar de usar transformación.**
*Síntoma*: un bug arreglado en v2 sigue en v1. Las dos versiones divergen.
*Causa*: el handler de v2 es un copy-paste del de v1 con pequeñas modificaciones.
*Arreglo*: compartir la lógica de negocio en el servicio. Los handlers de v1 y v2 solo transforman el request/response — la lógica interna es la misma función.

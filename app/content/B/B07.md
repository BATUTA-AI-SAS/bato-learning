## 1. Hilo conductor

En A05 aprendiste que el código se organiza en capas con dirección de dependencia. Ahora vas a entender qué ocurre en el extremo más externo de esa cadena: la interfaz que el humano ve y toca. Sin este modelo mental, B03 (Jinja + HTMX) y B06 (SSE + chat) son recetas que copias sin entender. Con él, puedes diseñar cualquier UI y justificar por qué.

## 2. Idea central

**UI (User Interface)** es cualquier capa que transforma datos en una representación perceptible por un humano y que acepta acciones de vuelta. No es solo visual: una CLI es una UI, un archivo CSV que abre Excel es una UI, una respuesta JSON que lee un desarrollador en Postman es una UI. Lo que las define no es el formato sino la dirección del flujo: **datos → representación → acción del usuario → datos**.

Imagen mental: una hoja de cálculo Excel. La celda A1 contiene la fórmula `=SUM(B1:B10)`. Excel calcula el valor (datos), lo renderiza como número en la celda (representación), el usuario edita B3 (acción), Excel recalcula y actualiza A1 (nueva representación). Eso es el ciclo de render completo.

## Puente desde Excel

Una hoja de Excel es una UI: tiene datos (celdas), representación visual (colores, formatos), y eventos (cuando cambias B3, todo lo dependiente se recalcula). Una UI web es exactamente eso, pero el render lo hace el navegador.

| Excel | UI web |
|-------|--------|
| Celdas con datos | DOM en memoria del navegador |
| Fórmula `=A1+B1` | Plantilla Jinja2 que muestra `{{ total }}` |
| Recálculo cuando cambia A1 | Re-render cuando llega nueva request HTTP |
| Botón con macro | Botón con `hx-post` (HTMX) |

</details>

## 3. Por qué importa

Una API devuelve JSON para otro servicio o agente. Una UI muestra datos para un humano. Confundirlas es construir el backend correcto para el usuario equivocado.

En F-FIN-01: el parseo y matching corre sin humano (sin UI). El reporte de hallazgos para el CFO de ACME sí es UI — puede ser un PDF, una pantalla web o un email. La decisión es quién lo lee y dónde vive, no qué tecnología es más moderna.

## 4. Cómo funciona por dentro

### El ciclo de render

Todo sistema UI repite el mismo ciclo:

1. **Estado** — los datos que describen qué se muestra (una lista de facturas, el usuario activo, si un panel está abierto).
2. **Render** — convertir ese estado en una representación (HTML, SVG, texto en terminal, celdas en Excel).
3. **Evento** — el usuario hace algo (click, tecla, scroll, submit de formulario).
4. **Update** — el evento modifica el estado y dispara un nuevo render.

La diferencia entre paradigmas de UI es **quién ejecuta cada paso y dónde**.

### El DOM y el navegador

_Excel analogy: cuando abres un libro Excel, Excel carga el archivo y construye en memoria un modelo de objetos — tienes `Workbook`, `Worksheet`, `Range`, `Cell`. Puedes acceder a esos objetos con VBA o Python y modificarlos sin tocar el archivo en disco. El DOM es exactamente eso, pero para HTML: cuando el navegador recibe el HTML, construye en memoria un árbol de objetos (`Document`, `Element`, nodos de texto). JavaScript y HTMX modifican ese árbol; el navegador pinta lo que el árbol dice, no lo que dice el archivo original._

Cuando el navegador recibe HTML, construye el **DOM**: un árbol de nodos en memoria. El navegador usa el DOM para pintar píxeles. Cuando HTMX (o JavaScript) modifica un nodo del DOM, el navegador recalcula solo la parte afectada y repinta.

El DOM es el puente entre el markup que el servidor produce y los píxeles que el usuario ve.

<details>
<summary>Para profundizar: tipos de nodo del DOM — Element, HTMLElement, NodeType (opcional)</summary>

El DOM distingue varios tipos de nodo: `Element` es cualquier elemento HTML (`<div>`, `<p>`, `<form>`). `HTMLElement` es la subclase específica para HTML (vs SVG, por ejemplo). Cada nodo tiene un `nodeType`: `1` para elementos, `3` para texto, `8` para comentarios. En la práctica, casi siempre operas con `Element` via `querySelector` sin necesitar conocer el tipo exacto. Esta distinción importa cuando escribes JavaScript que recorre el árbol manualmente — no cuando usas HTMX.

</details>

### Server-rendered vs SPA

| Criterio | Server-rendered (Jinja2 + HTMX) | SPA (React, Vue, Svelte) |
|----------|----------------------------------|--------------------------|
| Dónde ocurre el render | En el servidor (Python) | En el navegador (JS) |
| Qué viaja por la red | HTML listo para mostrar | JSON + bundle JS |
| Inspección con View Source | Muestra el HTML real | Muestra `<div id="app"></div>` vacío |
| Curva de aprendizaje | Baja (Python + HTML) | Alta (build tool, bundler, JSX/TS) |
| Cuándo usarlo | Formularios, dashboards, contenido que cambia poco | Estado local complejo, editor colaborativo |

**bato-learning es server-rendered.** Cero build pipeline, inspeccionable con View Source. El lector puede leer el HTML que llega y mapear la plantilla Jinja2 al output — en un SPA, esa trazabilidad desaparece.

> [!nota]
> HTMX no convierte una app server-rendered en una SPA. HTMX permite que fragmentos de HTML reemplacen partes de la página sin recargar todo — pero el render sigue ocurriendo en el servidor.

<details>
<summary>Para profundizar: bundle sizes, tiempo de carga y cuándo una SPA se justifica (opcional)</summary>

Una SPA descarga el bundle de JavaScript antes de mostrar nada — en React + TypeScript, un bundle de producción mínimo ronda los 150-300 KB gzip. Esto introduce latencia en la primera carga (Time to Interactive). En proyectos grandes con interacciones complejas (editor de texto colaborativo, diseño gráfico, juego), ese costo se justifica. En dashboards y formularios de backoffice, casi nunca.

Los frameworks modernos (Next.js, SvelteKit, Nuxt) mitigan esto con SSR (render inicial en el servidor) y code splitting. Pero añaden un build pipeline (Vite/webpack, TypeScript, ESLint) que tiene un costo de mantenimiento real. Para equipos pequeños y herramientas internas, server-rendered con HTMX cubre el 90% de los casos sin ese overhead.

**Web Components** (Custom Elements + Shadow DOM) son un estándar del navegador para encapsular UI sin framework. Funcionan en cualquier contexto pero carecen del ecosistema de React o Vue. No los usamos en este curso; mencionarlos para que no confundas «framework» con «SPA».

</details>

Para un agente IA en una PYME LATAM, la UI más común no es una web — es un **email**, un **PDF** o un **mensaje en WhatsApp**. El agente produce texto estructurado; la UI es la bandeja de entrada del humano. No sobrediseñes.

<details>
<summary>Para profundizar: el render cycle completo en bato-learning paso a paso (opcional)</summary>

En bato-learning:

1. El navegador hace `GET /m/what-is-a-ui`.
2. FastAPI llama al router, que llama al servicio, que llama al repo para obtener el módulo.
3. El router llama a `templates.TemplateResponse("module.html", {...})`.
4. Jinja2 evalúa la plantilla con el contexto Python y genera el HTML.
5. El HTML viaja por HTTP al navegador.
6. El navegador construye el DOM y pinta.
7. El usuario envía código en el editor: HTMX hace `POST /api/exercises/{id}/pyodide` con el código.
8. El servidor responde con un fragmento HTML (`<div id="result">...</div>`).
9. HTMX reemplaza el nodo target con ese fragmento. El DOM se actualiza parcialmente.

El paso 7-9 es el corazón de HTMX: una request HTTP que devuelve HTML, no JSON.

</details>

## 5. Ejemplo conducido

```python
from jinja2 import Environment

env = Environment()
template = env.from_string("{{ vendor }}: {{ amount }} EUR")
output = template.render(vendor="Proveedor A", amount=4820)
print(output)  # Proveedor A: 4820 EUR
```

Estado → `template.render` → representación. Cuando el estado cambia, el render produce una salida distinta. Sin framework, sin magia.

<details>
<summary>Para profundizar: ejemplo completo con lista de facturas (opcional)</summary>

```python
from jinja2 import Environment

env = Environment()
template = env.from_string(
    "{% for inv in invoices %}"
    "{{ inv.vendor }}: {{ inv.amount }} EUR\n"
    "{% endfor %}"
)

invoices = [
    {"vendor": "Proveedor A", "amount": 4820},
    {"vendor": "Proveedor B", "amount": 1200},
]

output = template.render(invoices=invoices)
print(output)
# Proveedor A: 4820 EUR
# Proveedor B: 1200 EUR
```

</details>

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Renderizar una plantilla Jinja2 con datos de DB | Determinístico | Mismos datos → mismo HTML siempre. Regla cerrada. |
| Decidir qué datos mostrar en el reporte de auditoría | Determinístico | Los KPIs y campos del reporte son reglas del negocio codificadas. |
| Seleccionar qué fragmento HTMX reemplaza en el DOM | Determinístico | El atributo `hx-target` en el HTML fija el destino. |
| Redactar el resumen ejecutivo del hallazgo para el CFO | Agéntico | Requiere contexto del cliente, tono, prioridades — sin regla cerrada. |
| Decidir si mostrar una alerta roja o amarilla dado el monto | Determinístico | Umbral numérico fijo (codificado en la plantilla o en el servicio). |

</details>

## 10. Errores típicos

**1. Confundir «la API devuelve JSON» con «la UI está hecha».**
*Síntoma*: construyes `/api/invoices` y crees que terminaste la feature.
*Arreglo*: pregunta «¿quién consume esto?». Si es un humano, falta la pantalla o el email.

**2. Usar SPA donde server-rendered es suficiente.**
*Síntoma*: el proyecto tarda semanas en configurar Vite, TypeScript y el bundler.
*Arreglo*: empieza con server-rendered + HTMX. Migra a SPA solo si tienes estado local complejo que no puede vivir en el servidor.

**3. Modificar el DOM desde Python (imposible).**
*Síntoma*: intentas hacer `document.querySelector(...)` desde Python.
*Arreglo*: Python corre en el servidor; el DOM existe en el navegador. El puente es HTTP + Jinja2 + HTMX.

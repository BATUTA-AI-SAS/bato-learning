## Hilo conductor

En B03 viste cómo el servidor renderiza HTML con Jinja2 y HTMX actualiza fragmentos sin recargar la página. Este módulo examina la pieza más peculiar de la app: el código Python del lector no viaja al servidor para ejecutarse — corre en el propio navegador, en un intérprete real compilado a WebAssembly. Entender ese flujo explica por qué los ejercicios responden en ~50ms y por qué algunos ejercicios requieren `runner: backend`. B06 tomará las mismas herramientas del DOM y las conectará al SSE del chat.

## Idea central

**Pyodide** (SHARED §1.1) es CPython compilado a WebAssembly. No es una reimplementación de Python ni un transpilador — es el mismo intérprete que corres en tu terminal, empaquetado para el navegador. Cuando el lector pulsa «Ejecutar», el JavaScript de la app llama a `pyodide.runPython(code)` directamente, sin hacer ningún request HTTP.

**CodeMirror 6** es el editor de código en el navegador. En la app actual es un `<textarea>` básico con syntax highlighting mínimo (la versión completa de CodeMirror 6 se describe en ARCHITECTURE.md §2.3 como el objetivo; lo que hay ahora funciona). El editor captura lo que el lector escribe; el runner lo toma y lo ejecuta.

## Por qué importa

Sin Pyodide, cada ejecución de ejercicio haría un POST al servidor, ejecutaría el código en un subprocess de Python, y devolvería el output. Eso tiene dos problemas: latencia de red (100-500ms) y carga en el servidor proporcional al número de lectores. Con Pyodide, el servidor no hace nada cuando el lector ejecuta un ejercicio — todo corre en el navegador del lector.

El límite es claro: Pyodide no puede importar paquetes que tienen código nativo (C extensions) no compiladas para WebAssembly. SQLAlchemy, asyncpg, pydantic 2.x en modo core — nada de eso funciona en Pyodide. Para ejercicios que requieren esas librerías, el `runner` del ejercicio es `backend` y el código sí viaja al servidor.

> [!nota]
> La versión de Pyodide en `_layout.html` es `v0.27.0`. El objetivo de producción del curso es `0.29.4` (Python 3.13, Emscripten 4.x) según ARCHITECTURE.md §2.3. Los ejercicios de este módulo son compatibles con ambas versiones.

## Cómo funciona por dentro

### El flujo de ejecución paso a paso

Cuando el lector pulsa «Ejecutar» en `app/static/js/pyodide_runner.js`:

1. `runBtn.addEventListener("click")` lee `editor.value` (el código del lector).
2. Llama a `runPyodide(userCode, testCode)`.
3. `runPyodide` espera a que `pyodideReady` (la promesa de inicialización) se resuelva.
4. Inyecta `__USER_CODE` y `__TEST_CODE` en los globals de Pyodide.
5. Ejecuta un bloque Python que:
   - Redirige `sys.stdout` a un `io.StringIO()` llamado `__cap`.
   - Ejecuta el código del usuario en un namespace aislado `__ns`.
   - Si hay test code, lo ejecuta en el mismo `__ns`.
   - Restaura `sys.stdout`.
6. Devuelve `{ output: __captured, error: __err }`.
7. El JS actualiza el `<pre class="output">` con el resultado.
8. Si el runner es `pyodide`, llama a `reportAttempt` (POST al servidor) para persistir el intento.

### El namespace aislado (`__ns`)

```python
__ns = {'__name__': '__main__'}
exec(__USER_CODE, __ns)
if __TEST_CODE:
    exec(__TEST_CODE, __ns)
```

`exec` recibe un diccionario opcional como namespace global. Al usar `__ns`, el código del usuario no contamina el namespace de Pyodide ni el de otros ejercicios. Las variables que el código del usuario define quedan en `__ns`, y el código de test puede inspeccionarlas porque corre en el mismo `__ns`.

Por eso los tests de los ejercicios de este curso pueden hacer `assert factura["pagada"] is True` — `factura` existe en `__ns` porque el código del usuario la definió ahí.

### La captura de stdout (`__cap`)

```python
import sys, io
__cap = io.StringIO()
sys.stdout = __cap
# ... código del usuario
sys.stdout = __old
__captured = __cap.getvalue()
```

`sys.stdout` es el objeto al que `print()` escribe. Reemplazarlo temporalmente con un `StringIO` captura todo lo que el código del usuario imprime, sin mostrarlo en la consola del navegador. El test puede leer `__cap.getvalue()` para verificar el output exacto.

### Qué detecta Pyodide y qué no

| Tipo de error | Qué ve el lector |
|---------------|-----------------|
| `SyntaxError` (código roto) | Capturado en el `except Exception as e`; aparece en el `<pre>` |
| `AssertionError` de un test | El mensaje del assert aparece en `__err` |
| `NameError` (variable no definida) | Idem — capturado |
| Import de paquete no disponible | `ModuleNotFoundError` capturado |
| Bucle infinito | El tab del navegador se congela — **Pyodide no tiene timeout** |

El bucle infinito es el caso problemático. La app no lo resuelve en esta versión; el lector tiene que recargar el tab. En producción se usa un Web Worker para poder terminar la ejecución con `worker.terminate()`.

### El runner `backend`

Los ejercicios con `runner: backend` en su YAML usan el endpoint `POST /api/exercises/{id}/backend`. El código viaja al servidor, se ejecuta en un subprocess Python real (no Pyodide), y el resultado vuelve como JSON. El router `exercises.py` llama a `run_backend_exercise` del servicio. Esto permite ejercicios con SQLAlchemy, asyncio real, etc.

### Cómo añadir un ejercicio nuevo

Los ejercicios viven en `app/content/{TRACK}/{EXT_ID}.exercises.yaml`. El `content_loader.py` los carga al arrancar. Al arrancar la app (o ejecutar `make seed`), `load_all` recorre los `.md` y sus `.exercises.yaml` y hace upsert en la DB.

Para añadir un ejercicio a B05:

1. Edita `app/content/B/B05.exercises.yaml`.
2. Añade una entrada con `slug`, `title`, `runner: pyodide`, `prompt`, `hint`, `starter`, `test`, `solution`.
3. Reinicia la app o ejecuta `uv run python -m scripts.seed`.
4. El ejercicio aparece en `/m/codemirror-and-pyodide`.

> [!profundizar]
> Lee [`app/static/js/pyodide_runner.js`](../../app/static/js/pyodide_runner.js) función `runPyodide`. Ahí verás el bloque Python que se inyecta como string — todo el mecanismo de `__cap`, `__ns`, `__err` y la restauración de `sys.stdout`. Compara con el código de ejercicio en `app/content/A/A01.exercises.yaml` para ver cómo el test usa `__cap.getvalue()`.

## Ejemplo conducido

Traza del ejercicio más simple — «haz que imprima `hola mundo`»:

```python
# Lo que el lector escribe:
print("hola mundo")

# El test que el runner ejecuta (del YAML):
out = __cap.getvalue().strip()
assert out == "hola mundo", f"esperaba 'hola mundo', obtuve {out!r}"
print("ok")
```

Secuencia en Pyodide:
1. El lector pulsa «Ejecutar».
2. `exec('print("hola mundo")', __ns)` → escribe `"hola mundo\n"` en `__cap`.
3. `exec(test_code, __ns)` → `__cap.getvalue().strip()` devuelve `"hola mundo"`.
4. La aserción pasa. `print("ok")` añade `"ok\n"` a `__cap`.
5. `__err` queda en `None`.
6. El output en pantalla: `"hola mundo\nok"`.

Si el lector escribe `print("Hola mundo")` (mayúscula), la aserción falla con `AssertionError: esperaba 'hola mundo', obtuve 'Hola mundo'` — que aparece en el `<pre class="output err">`.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Ejecutar el código del usuario con `exec` | determinístico | Dado el mismo código, el mismo output. Pyodide es determinístico. |
| Capturar stdout con `StringIO` | determinístico | Redirección mecánica. |
| Evaluar si el test pasa | determinístico | La aserción tiene resultado binario. |
| Detectar si el ejercicio está «bien hecho» más allá del test | agéntico | «La solución es elegante» o «podría mejorarse» requiere juicio. El tutor puede opinar; el test solo pasa o falla. |
| Sugerir el siguiente ejercicio según el historial de intentos | agéntico | Depende del patrón de errores del lector — no hay regla cerrada. |

## Errores típicos

**1. El ejercicio tiene un `print` de depuración en el starter code.**
*Síntoma*: el test falla porque `__cap.getvalue()` contiene texto extra además de la salida esperada.
*Causa*: el starter tiene `print("debugging")` que el lector no borró.
*Arreglo*: los tests usan `.strip()` y comprobaciones parciales (`in`), no igualdad exacta del output completo, a menos que el ejercicio sea explícitamente de output.

**2. El lector importa una librería no disponible en Pyodide.**
*Síntoma*: `ModuleNotFoundError: No module named 'sqlalchemy'`.
*Causa*: Pyodide no tiene la librería compilada para WebAssembly.
*Arreglo*: el ejercicio debe tener `runner: backend` si requiere librerías con C extensions. El `notes_for_subagent` del módulo indica qué runner usar.

**3. El test modifica variables de `__ns` que el código del usuario ya definió.**
*Síntoma*: el test parece pasar aunque el código del usuario esté incompleto.
*Causa*: el test redefinió la variable que iba a comprobar.
*Arreglo*: los tests solo leen de `__ns`, no redefinen variables. Usan `assert variable == expected`, no `variable = expected; assert ...`.

**4. Bucle infinito en el código del usuario.**
*Síntoma*: el tab del navegador se congela.
*Causa*: Pyodide corre en el hilo principal; un bucle infinito bloquea el event loop del navegador.
*Arreglo*: en producción, mover Pyodide a un Web Worker y usar `worker.terminate()` después de un timeout. En esta versión del curso, el lector recarga el tab.

## Caso de uso real

El sistema de ejercicios de `bato-learning` es el caso de uso. Cada vez que BATUTA necesita enseñar a un cliente cómo usar una herramienta (p.ej., cómo formatear un CSV para importación a Siigo), puede publicar un ejercicio Pyodide donde el cliente práctica la transformación en el navegador, sin instalar nada, sin dar acceso al servidor de BATUTA.

El patrón es exportable: cualquier aplicación que necesite un «playground» Python interactivo puede usar el mismo mecanismo — `pyodide_runner.js`, un `<textarea>`, y un bloque de test en el YAML del módulo.

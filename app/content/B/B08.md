## 1. Hilo conductor

En B01 aprendiste el protocolo HTTP: verbo, ruta, headers, body, status code. En B07 viste que una API puede ser la UI de un desarrollador. Este módulo une ambas ideas: una **API** es un contrato de endpoints que sigue reglas precisas — y hay cuatro estilos distintos, cada uno con trade-offs. Cuando termines, podrás leer la documentación de cualquier API externa (Siigo, Anthropic, Belvo) sin quedarte atascado en la terminología.

---

## 2. Idea central

**API (Application Programming Interface)** es el contrato que un sistema expone para que otro lo consuma sin necesitar acceso a su código interno. En este curso «API» significa «API HTTP», normalmente JSON. La anatomía básica es siempre la misma: un **verbo** (qué hacer), una **ruta** (sobre qué recurso), **headers** (metadatos: autenticación, formato, versionado), **body** (datos del request), y una **respuesta** con status code + headers + body.

Imagen mental: es una barra de fórmulas de Excel, pero para sistemas. `=SUMA(B1:B10)` es una invocación con nombre de función y argumentos. `POST /api/invoices` con un JSON es lo mismo: nombre del contrato (`POST /api/invoices`) y argumentos (el body JSON).

## 3. Por qué importa

Sin entender la anatomía de una API, cualquier integración es un proceso de trial-and-error ciego. ACME Manufacturing conecta su agente de auditoría a Siigo (ERP), Belvo (banco), y Anthropic (LLM). Cada uno expone una API distinta. Si no sabes leer un schema OpenAPI, un error de autenticación 401 vs un error de rate limit 429, o la diferencia entre REST y webhook, cada integración te cuesta días en lugar de horas.

La ficha F-CMP-03 (matching de facturas con órdenes de compra) depende de tres APIs: la del ERP, la del proveedor, y la de Anthropic. Entender el contrato de cada una es lo que separa un agente robusto de uno que se rompe con la primera anomalía.

## 4. Cómo funciona por dentro

### REST: el estilo dominante

**REST (Representational State Transfer)** no es un protocolo — es un conjunto de convenciones sobre cómo usar HTTP para exponer recursos. Las que importan en la práctica:

1. **Los recursos son sustantivos, no verbos.** `GET /invoices` (bien). `GET /getInvoices` (mal).
2. **El método HTTP define la acción.** `GET` lee, `POST` crea, `PUT`/`PATCH` actualiza, `DELETE` elimina.
3. **Sin estado en el servidor entre requests.** Cada request debe ser autocontenida con sus credentials y parámetros.
4. **Los status codes tienen semántica.** No respondas `200 OK` con un body de error.

Anatomía de un request REST real:

```
POST /api/v1/invoices
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...
Content-Type: application/json
Idempotency-Key: acme-inv-2026-04-0042

{
  "vendor_id": "vnd-001",
  "amount": 4820.00,
  "period": "2026-04",
  "tenant": "acme"
}
```

Y la respuesta:

```
HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/v1/invoices/inv-0042

{
  "id": "inv-0042",
  "vendor_id": "vnd-001",
  "amount": 4820.00,
  "status": "pending",
  "created_at": "2026-04-15T09:30:00Z"
}
```

### OpenAPI y el `/docs` de FastAPI

**OpenAPI 3.1** es el estándar para describir una API REST como un documento JSON o YAML. Define cada endpoint: qué acepta, qué devuelve, qué errores puede lanzar. FastAPI genera este documento automáticamente desde las firmas de tus funciones y los modelos Pydantic. La UI interactiva en `/docs` (Swagger UI) y `/redoc` la genera del mismo documento.

> [!profundizar]
> Abre [`http://localhost:8080/docs`](http://localhost:8080/docs) en tu browser con la app corriendo. Cada endpoint de bato-learning aparece ahí con su schema. Busca `POST /api/exercises/{exercise_id}/pyodide` — verás el body esperado y la respuesta documentados automáticamente.

### JSON sobre HTTP: el formato de facto

**JSON (JavaScript Object Notation)** es el formato de serialización dominante en APIs REST modernas. Es texto plano legible, jerárquico, y con tipos básicos: string, number, boolean, null, array, object. Sus limitaciones: no tiene tipos de fecha nativos (se usa ISO 8601 como string), no soporta comentarios, y es verbose comparado con formatos binarios.

Cuando FastAPI devuelve un `dict` de Python, lo serializa a JSON automáticamente. Cuando recibes JSON del exterior, lo parseas con `json.loads()` o con un modelo Pydantic directamente.

### GraphQL: un endpoint, queries flexibles

**GraphQL** es un lenguaje de queries donde el cliente especifica exactamente qué campos quiere. Hay un solo endpoint (típicamente `POST /graphql`) y el cliente envía una query en el body:

```graphql
query {
  invoices(period: "2026-04", tenant: "acme") {
    id
    amount
    vendor { name }
  }
}
```

El servidor devuelve solo los campos pedidos. Ventaja: el cliente no recibe datos que no necesita (útil en móvil con ancho de banda limitado). Desventaja: N+1 queries en el servidor, autorización field-level compleja, curva de aprendizaje más alta.

**Cuándo usar**: proyectos donde múltiples clientes (web, móvil, terceros) consumen la misma API y tienen necesidades de datos muy distintas. Para un agente IA consumiendo una API interna, REST es casi siempre suficiente.

### gRPC: binario, tipado, RPC

**gRPC** (Google Remote Procedure Call) usa **Protocol Buffers** como formato de serialización binario y HTTP/2 como transporte. El cliente llama funciones como si fueran locales; el framework serializa los argumentos a binario, los envía, y deserializa la respuesta.

```protobuf
service AuditService {
  rpc RunAudit (AuditRequest) returns (AuditResult);
}
```

Ventajas: payloads 3-10× más pequeños que JSON, fuertemente tipado con schema obligatorio, streaming bidireccional nativo, latencia menor en LAN. Desventajas: no legible por humanos (necesitas herramientas), más complejo de depurar que REST/JSON, la mayoría de herramientas HTTP no lo entienden sin adaptadores.

**Cuándo usar**: comunicación entre microservicios internos con volumen alto y latencia crítica. Entre un agente IA y una API externa de un proveedor, es raro — la mayoría usa REST.

### Webhooks: la API invertida

Un **webhook** es una API al revés. En lugar de que tu código llame a una API externa para pedir datos, el sistema externo llama a un endpoint tuyo cuando ocurre un evento.

| Dirección | Nombre | Ejemplo |
|-----------|--------|---------|
| Tu código → API externa | REST (pull) | `GET /invoices?period=2026-04` |
| API externa → tu endpoint | Webhook (push) | Siigo llama `POST /webhooks/siigo/invoice-created` con el payload |

Siigo, Stripe, GitHub, y la mayoría de ERPs y plataformas de pago exponen webhooks. Cuando se crea una factura en Siigo, Siigo hace un `POST` a la URL que tú registraste, con el payload de la factura. Tu servidor procesa el evento en tiempo real, sin polling.

**Riesgos de los webhooks**:
- Si tu servidor está caído cuando llega el evento, el evento se pierde (salvo que el proveedor tenga retry).
- El proveedor puede enviar el mismo evento más de una vez (retry automático tras timeout). Tu endpoint **debe ser idempotente** (ver B11).
- El payload puede ser falso — cualquiera puede hacer `POST` a tu URL. Debes verificar la firma HMAC en el header (ver B10).

### Decision tree: ¿qué protocolo uso?

```
¿El cliente necesita datos en tiempo real sin polling?
├─ Sí, bidireccional → WebSocket
├─ Sí, solo servidor→cliente → SSE (ver B06)
└─ No → ¿quién inicia la comunicación?
   ├─ Mi código llama a un sistema externo
   │  ├─ Múltiples clientes con necesidades muy distintas → GraphQL
   │  ├─ Comunicación interna con latencia crítica → gRPC
   │  └─ Caso general → REST + JSON
   └─ El sistema externo me notifica de eventos → Webhook
```

## 5. Ejemplo conducido

Una simulación del ciclo request-response REST con Python puro (sin httpx, para que corra en Pyodide):

```python
import json

def simulate_api_call(
    method: str,
    path: str,
    body: dict | None = None,
) -> dict:
    """Simula el ciclo de una API REST sin red real."""
    # router ficticio: diccionario de (método, path) → handler
    routes: dict[tuple[str, str], dict] = {
        ("GET", "/api/v1/invoices"): {
            "status": 200,
            "body": [{"id": "inv-001", "amount": 4820.0, "status": "pending"}],
        },
        ("POST", "/api/v1/invoices"): {
            "status": 201,
            "body": {"id": "inv-002", **body} if body else {},
        },
        ("GET", "/api/v1/invoices/inv-999"): {
            "status": 404,
            "body": {"detail": "not found"},
        },
    }
    key = (method.upper(), path)
    result = routes.get(key, {"status": 405, "body": {"detail": "method not allowed"}})
    return result

# GET → lista de facturas
resp = simulate_api_call("GET", "/api/v1/invoices")
print(f"GET /invoices → {resp['status']}: {json.dumps(resp['body'])}")

# POST → crear factura
new_invoice = {"vendor_id": "vnd-001", "amount": 1200.0, "tenant": "acme"}
resp = simulate_api_call("POST", "/api/v1/invoices", body=new_invoice)
print(f"POST /invoices → {resp['status']}: {json.dumps(resp['body'])}")

# GET con id inexistente → 404
resp = simulate_api_call("GET", "/api/v1/invoices/inv-999")
print(f"GET /invoices/inv-999 → {resp['status']}: {json.dumps(resp['body'])}")
```

## 8. Caso de uso real

**F-CMP-03 — Three-way match automatizado**: ACME Manufacturing recibe facturas de proveedores vía webhook de Siigo. El agente hace `GET /api/purchase-orders/{po_id}` a la API interna para traer la OC, y `GET /api/receipts?vendor={vendor_id}&period={period}` para traer los albaranes. El matching determinístico (monto + fecha + referencia) es código Python puro. El matching fuzzy (descripción ambigua) llama a Anthropic via REST. Tres APIs distintas en un solo flujo — todas siguen la misma anatomía.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Parsear el body JSON de la request con Pydantic | Determinístico | Regla cerrada: si falta un campo requerido, falla con 422. |
| Seleccionar el endpoint REST según el método HTTP y el path | Determinístico | Router fijo — la tabla de rutas no cambia en runtime. |
| Verificar la firma HMAC de un webhook entrante | Determinístico | Operación criptográfica. |
| Clasificar el tipo de API correcto para una integración nueva | Agéntico | Depende de requisitos del cliente, latencia, volumen, madurez del proveedor — sin regla única. |
| Decidir el status code correcto para un error de negocio específico | Determinístico | La tabla de status codes es estándar; el mapeado a errores de negocio se codifica. |

## 10. Errores típicos

**1. Diseñar rutas con verbos en lugar de sustantivos.**
*Síntoma*: `POST /runAudit`, `GET /getInvoices`, `POST /deleteUser`.
*Causa*: pensar en acciones en lugar de recursos.
*Arreglo*: `POST /audits` (crea una ejecución de auditoría), `GET /invoices`, `DELETE /users/{id}`. El verbo lo pone el método HTTP.

**2. Devolver `200 OK` cuando hay un error.**
*Síntoma*: el cliente recibe `HTTP 200` con `{"error": "not found"}` en el body.
*Causa*: el servidor no usa los status codes estándar — pone todo en el body.
*Arreglo*: usar `404` para not found, `422` para validación, `500` para error interno. El cliente no debería tener que parsear el body para saber si la request fue exitosa.

**3. Confundir webhook con polling.**
*Síntoma*: el agente hace `GET /invoices` cada 30 segundos para detectar nuevas facturas.
*Causa*: no saber que Siigo (u otro proveedor) expone webhooks. El polling es ineficiente y costoso.
*Arreglo*: registrar un webhook en el proveedor. El agente recibe el evento en tiempo real en lugar de consultar repetidamente.

**4. No validar la firma del webhook.**
*Síntoma*: cualquiera puede enviar un payload falso al endpoint de webhook y el agente lo procesa.
*Causa*: el endpoint no verifica el header `X-Signature` o `X-Hub-Signature-256`.
*Arreglo*: siempre verificar HMAC antes de procesar (ver B10).

**5. Asumir que la API del proveedor es idempotente.**
*Síntoma*: al reintentar un `POST /invoices` tras un timeout, se crean dos facturas duplicadas.
*Causa*: el cliente no envía `Idempotency-Key` y el proveedor no deduplica.
*Arreglo*: enviar siempre una `Idempotency-Key` en requests que modifican estado (ver B11).

## 1. Hilo conductor

En B01 viste el ciclo HTTP desde fuera: curl, headers, status codes. Ahora entras al servidor. FastAPI convierte funciones Python en endpoints HTTP: el decorador registra la ruta, Pydantic valida el body antes de que tu código lo toque, `Depends` inyecta dependencias sin acoplamiento. Al terminar este módulo puedes leer y escribir cualquier router de la app. C01 reutiliza directamente el patrón de `Depends` que construyes aquí para inyectar la sesión de base de datos.

## 2. Idea central

**FastAPI** es un framework que convierte funciones Python en endpoints HTTP usando decoradores y type hints. No hay magia: `@router.post("/exercises/{exercise_id}/backend")` registra la función en un `APIRouter`; cuando llega un request a esa ruta, FastAPI valida el body con Pydantic, inyecta las dependencias declaradas, llama tu función, y serializa el return a JSON. El decorador no ejecuta la función; la registra. La ejecución ocurre en el event loop cuando llega el request.

## 3. Por qué importa

Sin entender el ciclo interno de FastAPI, cometes tres errores recurrentes: dejas validación en el router cuando debería estar en Pydantic, acoplas el handler a la DB directamente en lugar de usar `Depends`, y mezclas código async con llamadas bloqueantes. En producción el resultado es una app que falla silenciosamente bajo carga.

Esto conecta con la ficha F-OPS-02 (forecast de demanda expuesto como API interna): el endpoint que expone el forecast recibe un body con el período y las SKUs, lo valida con Pydantic, llama al servicio de cálculo mediante una dependencia inyectada, y devuelve el resultado serializado. Si el ciclo no está claro, el endpoint se convierte en una función de 80 líneas que nadie puede testear.

## 4. Cómo funciona por dentro

### Ciclo de un request en FastAPI

```
HTTP request
    │
    ▼
uvicorn/ASGI recibe bytes → construye scope ASGI
    │
    ▼
FastAPI router: busca la ruta que matchea (método + path pattern)
    │
    ▼
Extrae path params, query params, headers (type-checking con Pydantic)
    │
    ▼
Parsea body JSON → instancia el modelo Pydantic (valida tipos, required, constraints)
    │
    ▼
Resuelve el grafo de Depends: ejecuta cada función dependencia en orden topológico
    │
    ▼
Llama tu función handler con todos los argumentos resueltos
    │
    ▼
Serializa el return: si es un dict o un BaseModel → JSON; si es Response → tal cual
    │
    ▼
HTTP response
```

Si Pydantic falla en la validación, FastAPI devuelve `422 Unprocessable Entity` automáticamente con el detalle exacto del campo inválido. Tu código nunca llega a ejecutarse.

### Routers y `include_router`

En bato-learning, cada archivo de `app/routers/` crea su propio `APIRouter`. El `app/main.py` los monta:

```python
from .routers import exercises, health, pages

app = FastAPI(title="bato-learning", lifespan=lifespan)
app.include_router(health.router)
app.include_router(exercises.router)
app.include_router(pages.router)
```

`APIRouter(prefix="/api/exercises", tags=["exercises"])` en `exercises.py` aplica el prefijo a todos los endpoints del router y los agrupa en `/docs`. Si olvidas `include_router`, la ruta no existe y obtienes 404.

### Pydantic v2: validación de body

_Pydantic es una librería que toma un diccionario de datos crudos (como el body JSON de una request) y lo convierte en un objeto Python con tipos verificados. Excel analogy: es como la validación de datos de una celda — si la columna dice "solo enteros" y alguien escribe "abc", Excel rechaza la entrada; Pydantic hace lo mismo automáticamente antes de que tu código vea el dato._

_Un decorador es una función que envuelve a otra función para añadirle comportamiento sin modificarla. La sintaxis `@router.post("/ruta")` encima de una función es un decorador: registra esa función como manejador de esa ruta. Excel analogy: como aplicar un formato condicional a una celda — la celda sigue siendo la misma, pero ahora tiene comportamiento adicional._

Pydantic v2 es **distinto** de v1. Los cambios que afectan el código del curso:

| Pydantic v1 | Pydantic v2 |
|-------------|-------------|
| `class Config: orm_mode = True` | `model_config = ConfigDict(from_attributes=True)` |
| `@validator("campo")` | `@field_validator("campo")` |
| `BaseModel.__fields__` | `BaseModel.model_fields` |
| `model.dict()` | `model.model_dump()` |

En `app/routers/exercises.py` los modelos son:

```python
class PyodideAttempt(BaseModel):
    code: str
    passed: bool
    output: str = ""
    error: str = ""
    duration_ms: int = 0

class BackendAttempt(BaseModel):
    code: str
```

FastAPI detecta que el parámetro `body: PyodideAttempt` es un subtipo de `BaseModel` y lo lee del body JSON. Si el campo `code` falta o no es string, el handler nunca se llama.

> [!cuidado]
> Pydantic v2 **no** importa `Optional` de typing para campos opcionales: usa `str | None = None`. Si copias código v1 con `Optional[str]`, funciona, pero es estilo antiguo. Dentro del curso siempre usamos la forma 3.12+.

### Path params, query params, body: cómo FastAPI los distingue

FastAPI usa el nombre del parámetro y si aparece en la plantilla de ruta para decidir de dónde leerlo:

```python
@router.get("/m/{slug}")
async def module_page(
    slug: str,          # path param — está en "{slug}"
    debug: bool = False, # query param — no está en la ruta, tiene default
    session: SessionDep  # dependencia — es Annotated[AsyncSession, Depends(...)]
) -> HTMLResponse:
    ...
```

Si un parámetro **no** tiene tipo `BaseModel`, **no** está en la ruta, y **no** es `Depends` → FastAPI lo busca como query param. Si lo tiene como `BaseModel` → lo lee del body.

### Dependency injection con `Depends`

`SessionDep` en `app/db.py` es el ejemplo canónico:

```python
async def get_session() -> AsyncIterator[AsyncSession]:
    async with SessionFactory() as session:
        yield session

SessionDep = Annotated[AsyncSession, Depends(get_session)]
```

`Annotated[T, Depends(f)]` es la forma moderna (PEP 593): el tipo es `T`, el metadata es la dependencia. FastAPI ejecuta `get_session()` una vez por request, pasa la sesión al handler, y ejecuta el `finally` implícito del `async with` al terminar (commit o rollback si falla).

Ventaja de este patrón: en tests puedes hacer `app.dependency_overrides[get_session] = mock_session` sin modificar el handler. El handler no sabe de dónde viene su `session`.

### HTTPException y errores

```python
from fastapi import HTTPException

if mod is None:
    raise HTTPException(status_code=404, detail="módulo no encontrado")
```

FastAPI captura `HTTPException` y la convierte en un response JSON:
```json
{"detail": "módulo no encontrado"}
```
con el status code que pasaste. Para errores de validación Pydantic, FastAPI genera el body automáticamente (no necesitas capturarlos).

Para errores inesperados (excepciones no controladas), FastAPI devuelve 500 con el traceback **solo en modo debug**. En producción, el body de un 500 no revela el error — registra el traceback en logs (D04).

### Async vs sync handlers

FastAPI puede manejar handlers tanto `async def` como `def`:

- `async def`: el handler corre en el event loop de uvicorn. Correcto para I/O async (DB, HTTP cliente, SDK Anthropic).
- `def` (sync): FastAPI lo ejecuta en un thread pool para no bloquear el event loop.

**No mezcles**: si tu handler es `async def` y llama `requests.get(...)` (sync, bloqueante), estás bloqueando el event loop completo. Usa `httpx.AsyncClient` o `asyncio.to_thread`.

### `/docs` y OpenAPI automático

FastAPI genera `/docs` (Swagger UI) y `/redoc` a partir de los type hints y los docstrings. Los `tags` del router agrupan los endpoints. No hay nada que configurar; si los tipos son correctos, la documentación es correcta.

## 5. Ejemplo conducido

Reproduce el patrón completo de `exercises.py` en un router nuevo, paso a paso:

```python
# app/routers/forecast.py — router hipotético para F-OPS-02
from __future__ import annotations

from fastapi import APIRouter, HTTPException
from pydantic import BaseModel, field_validator

from ..db import SessionDep

router = APIRouter(prefix="/api/forecast", tags=["forecast"])


class ForecastRequest(BaseModel):
    tenant: str
    period: str       # "YYYY-MM"
    sku_ids: list[str]

    @field_validator("period")
    @classmethod
    def period_format(cls, v: str) -> str:
        if len(v) != 7 or v[4] != "-":
            raise ValueError("period must be YYYY-MM")
        return v


class ForecastResult(BaseModel):
    tenant: str
    period: str
    forecasts: dict[str, float]   # sku_id → unidades estimadas


@router.post("/demand", response_model=ForecastResult)
async def forecast_demand(
    body: ForecastRequest, session: SessionDep
) -> ForecastResult:
    if body.tenant != "acme":
        raise HTTPException(status_code=403, detail="tenant not allowed")
    # aquí iría la llamada al servicio real
    return ForecastResult(
        tenant=body.tenant,
        period=body.period,
        forecasts={sku: 0.0 for sku in body.sku_ids},
    )
```

Luego en `main.py`:
```python
from .routers import forecast
app.include_router(forecast.router)
```

Prueba con curl:
```bash
curl -X POST http://localhost:8080/api/forecast/demand \
  -H "Content-Type: application/json" \
  -d '{"tenant": "acme", "period": "2026-04", "sku_ids": ["SKU-001", "SKU-002"]}'
```

## 8. Caso de uso real

**F-OPS-02 — Forecast de demanda como API interna**: Globex Logistics expone el forecast de demanda semanal como un endpoint FastAPI interno. Los equipos de almacén consumen el endpoint desde sus scripts Python. El body incluye `tenant`, `warehouse_id`, y `horizon_weeks`. Pydantic valida que `horizon_weeks` esté en rango [1, 13]; si no, el cliente recibe 422 automáticamente sin que el handler se ejecute. La sesión de DB se inyecta con `Depends` para que el servicio de forecast pueda leer el historial sin conocer cómo se conecta la BD.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Validar el body con Pydantic (tipos, required, constraints) | Determinístico | Regla cerrada; si el campo no es `str`, Pydantic lo rechaza. |
| Rutear el request al handler correcto por método + path | Determinístico | El router es una tabla de lookup; no hay razonamiento. |
| Calcular el forecast estadístico (media móvil, ARIMA) | Determinístico | Algoritmo cerrado sobre datos históricos. |
| Interpretar una solicitud ambigua ("el mes que viene" sin fecha) | Agéntico | «El mes que viene» depende de contexto que no cabe en regla. |
| Generar la narrativa del forecast para el gerente de bodega | Agéntico | Salida para humano; calidad depende de contexto del negocio. |

## 10. Errores típicos

**1. Olvidar `include_router` en `main.py`.**
*Síntoma*: el endpoint no aparece en `/docs` y todas las requests devuelven 404.
*Causa*: el router se declaró pero nunca se montó.
*Arreglo*: añadir `app.include_router(mi_router.router)` en `app/main.py`.

**2. Handler `async def` con llamada bloqueante.**
*Síntoma*: la app responde bien con un usuario; bajo carga, las respuestas se vuelven lentas en cascada.
*Causa*: `requests.get(...)` dentro de `async def` bloquea el event loop.
*Arreglo*: usar `httpx.AsyncClient` o `asyncio.to_thread(requests.get, url)`.

**3. Usar Pydantic v1 syntax en un proyecto v2.**
*Síntoma*: `pydantic.errors.PydanticUserError: `orm_mode` has been renamed to `from_attributes``.
*Causa*: copiar código de tutoriales antiguos con `class Config: orm_mode = True`.
*Arreglo*: usar `model_config = ConfigDict(from_attributes=True)`.

**4. No capturar la excepción del `Depends` en tests.**
*Síntoma*: el test llama al endpoint y falla con un error de conexión a DB.
*Causa*: no se sobrescribió la dependencia de sesión en el test client.
*Arreglo*: `app.dependency_overrides[get_session] = lambda: mock_session` antes de los tests; limpiarlo después.

**5. Devolver un dict en lugar del modelo Pydantic cuando hay `response_model`.**
*Síntoma*: FastAPI valida la respuesta y lanza 500 en producción si el dict tiene campos extra.
*Causa*: el handler devuelve `{"campo": valor, "campo_extra": ...}` y `response_model` no incluye `campo_extra`.
*Arreglo*: devolver siempre la instancia del modelo declarado en `response_model`, o usar `response_model_exclude_unset=True`.

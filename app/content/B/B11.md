## 1. Hilo conductor

En B09 viste que un webhook puede llegar dos veces. En B10 aprendiste que un agente llama APIs con credenciales. Este módulo responde la pregunta que surge inmediatamente: ¿qué pasa cuando la red falla y hay que reintentar? Si no diseñas la respuesta correctamente, reintentar crea duplicados, saturar el servidor provoca bloqueos, y el sistema entra en cascada de fallos. E05 (Temporal) conecta directamente con esto: las `heartbeats` de Temporal y las `retry_policy` asumen que tus actividades son idempotentes.

---

## 2. Idea central

**Idempotencia**: una operación es idempotente si ejecutarla N veces produce el mismo resultado que ejecutarla una vez. `GET /invoices` es idempotente. `POST /invoices` no lo es por defecto — llamarla dos veces crea dos facturas. La **idempotency key** convierte un `POST` en idempotente: el servidor deduplica usando la key.

**Rate limit**: el servidor rechaza requests que exceden un umbral de frecuencia. Responde `429 Too Many Requests` con un header `Retry-After`. El cliente debe esperar antes de reintentar.

**Retry seguro**: reintentar inmediatamente ante un error 5xx suele empeorar el problema (el servidor ya está saturado). El backoff exponencial con jitter añade una espera creciente y aleatoria entre reintentos.

Imagen mental: una impresora de cheques. Si el papel se atasca a mitad de impresión, no imprimes el cheque de nuevo sin confirmar si el primero salió o no — eso podría duplicar el pago.

## 3. Por qué importa

Sin idempotencia, un timeout de red en `POST /invoices` crea facturas duplicadas. Sin manejo de rate limit, el agente recibe una cascada de `429` y se bloquea. Sin backoff, reintentar agresivamente convierte un problema temporal del servidor en un colapso permanente.

ACME Manufacturing tiene un agente que crea órdenes de compra en el ERP cada vez que detecta un hallazgo de inventario. Si la red falla entre el `POST` del agente y el `201` del ERP, ¿el agente debe reintentar? ¿Seguro que la orden no se creó ya? Sin idempotency key, no hay respuesta segura.

## 4. Cómo funciona por dentro

### Idempotency key

La **idempotency key** es un identificador único que el cliente incluye en el header de la request:

```
POST /api/invoices
Idempotency-Key: acme-inv-2026-04-0042
Content-Type: application/json
```

El servidor guarda la key en una tabla. Si llega el mismo `Idempotency-Key` por segunda vez, devuelve la respuesta cacheada del primer request — sin ejecutar el efecto lateral de nuevo.

```python
# Implementación mínima del servidor
async def ensure_idempotent(
    idempotency_key: str | None,
    session: AsyncSession,
) -> dict | None:
    """Devuelve la respuesta cacheada si la key ya fue procesada."""
    if idempotency_key is None:
        return None  # no se requiere idempotencia para este endpoint
    cached = await idempotency_repo.get(session, key=idempotency_key)
    return cached  # None si es la primera vez

async def save_idempotent_result(
    idempotency_key: str,
    result: dict,
    session: AsyncSession,
) -> None:
    await idempotency_repo.save(session, key=idempotency_key, result=result)
```

La idempotency key tiene dos partes fundamentales:

1. **Debe ser única por intento de negocio** — no por request HTTP. Si el cliente quiere crear la factura INV-042 de ACME para el período 2026-04, la key es `"acme:inv-042:2026-04"`. Si falla y reintenta, usa la **misma key** — eso es lo que permite la deduplicación.

2. **Debe incluir el tenant** — sin `tenant_id`, una key colisiona entre tenants distintos. La key `"inv-042"` de ACME y la key `"inv-042"` de Globex son operaciones distintas.

```python
def make_idempotency_key(tenant: str, resource: str, resource_id: str) -> str:
    return f"{tenant}:{resource}:{resource_id}"

# Ejemplos:
# "acme:invoice:2026-04-0042"
# "globex:audit:2026-04"
# "initech:purchase-order:po-0099"
```

### Rate limiting: sliding window

El servidor lleva un contador de requests por cliente en una ventana de tiempo. El algoritmo más común es **sliding window**:

```
Límite: 100 requests por 60 segundos por API key

Timestamps de requests recientes de acme-key:
[T-55s, T-45s, T-30s, T-10s, T-5s, T-2s, T-0s]  ← 7 requests en la ventana
                                                      ↑ nueva request → T

Si el número de timestamps en la ventana [T-60s, T] < 100: allow
Si >= 100: devuelve 429 con Retry-After: 60 - (T - timestamps[0])
```

El header `Retry-After` en la respuesta `429` indica cuántos segundos debe esperar el cliente:

```
HTTP/1.1 429 Too Many Requests
Retry-After: 30
Content-Type: application/json

{"detail": "rate limit exceeded", "retry_after": 30}
```

### Retry con backoff exponencial y jitter

El **backoff exponencial** aumenta la espera entre reintentos: intento 1 espera 1s, intento 2 espera 2s, intento 3 espera 4s, intento 4 espera 8s...

El **jitter** añade aleatoriedad para evitar el «thundering herd»: si 100 clientes reciben el mismo error al mismo tiempo y todos esperan exactamente el mismo tiempo, todos reintentarán al mismo tiempo de nuevo — el servidor se satura otra vez.

```python
import asyncio
import random

async def retry_with_backoff(
    fn,
    *args,
    max_attempts: int = 4,
    base_delay: float = 1.0,
    max_delay: float = 30.0,
    **kwargs,
):
    """Reintenta fn con backoff exponencial + jitter."""
    for attempt in range(max_attempts):
        try:
            return await fn(*args, **kwargs)
        except RateLimitError as e:
            if attempt == max_attempts - 1:
                raise
            # si el servidor da Retry-After, usarlo; sino backoff propio
            wait = e.retry_after if hasattr(e, "retry_after") else (
                min(base_delay * (2 ** attempt), max_delay)
            )
            jitter = random.uniform(0, wait * 0.1)  # ±10% de jitter
            await asyncio.sleep(wait + jitter)
        except ServerError:
            if attempt == max_attempts - 1:
                raise
            wait = min(base_delay * (2 ** attempt), max_delay)
            jitter = random.uniform(0, wait * 0.1)
            await asyncio.sleep(wait + jitter)
```

**Qué errores reintentar y cuáles no**:

| Status | Reintentar | Por qué |
|--------|-----------|---------|
| 429 | Sí, con Retry-After | Rate limit temporal |
| 500, 502, 503, 504 | Sí, con backoff | Error del servidor temporal |
| 401 | No | Credenciales incorrectas — reintentar no ayuda |
| 403 | No | Sin permiso — reintentar no ayuda |
| 404 | No | El recurso no existe — reintentar no ayuda |
| 422 | No | Error de validación del cliente — el body es incorrecto |

> [!cuidado]
> Nunca reintentes un 4xx (excepto 429). Un `POST` a un endpoint con body inválido que devuelve `422` no se va a corregir solo con más reintentos. El reintento de 4xx es el segundo error más común después de no reintentar nunca.

### Conexión con Temporal (E05)

Los workflows de Temporal tienen `RetryPolicy` configurable por actividad. Temporal mismo hace el backoff exponencial — el código de la actividad solo necesita lanzar la excepción adecuada. Lo que Temporal **no hace** es garantizar idempotencia: si tu actividad `create_invoice` no es idempotente y Temporal la reintenta tras un fallo, crea dos facturas. La idempotency key es **tu responsabilidad**, no del framework.

```python
# api/app/activities/agent.py
@activity.defn
async def create_purchase_order(tenant: str, po_data: dict) -> dict:
    # La idempotency_key garantiza deduplicación si Temporal reintenta
    idempotency_key = make_idempotency_key(tenant, "purchase-order", po_data["id"])
    return await erp_client.create_po(po_data, idempotency_key=idempotency_key)
```

## 5. Ejemplo conducido

```python
import random

def compute_backoff(attempt: int, base: float = 1.0, cap: float = 30.0) -> float:
    """Backoff exponencial con jitter ±10%."""
    delay = min(base * (2 ** attempt), cap)
    jitter = random.uniform(-delay * 0.1, delay * 0.1)
    return max(0.0, delay + jitter)

# intento 0: ~1s, intento 1: ~2s, intento 2: ~4s, intento 3: ~8s
for i in range(5):
    d = compute_backoff(i)
    print(f"intento {i}: espera {d:.2f}s")
```

## 8. Caso de uso real

**F-CMP-01 — Órdenes de compra automáticas**: el agente de ACME detecta un punto de reorden (ROP) alcanzado y crea una OC en el ERP. Si el ERP tarda 15 segundos en responder y la conexión se corta a los 10s, el agente no sabe si la OC se creó. Con idempotency key `"acme:po:sku-047:2026-04-15"`, puede reintentar con seguridad — si la OC ya existe, el ERP devuelve la misma respuesta sin crear duplicado. Sin idempotency key, ACME recibe dos órdenes de compra para el mismo SKU el mismo día.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Generar la idempotency key a partir de tenant + recurso + id | Determinístico | Función pura — mismos inputs, misma key siempre. |
| Verificar si una idempotency key ya fue procesada en DB | Determinístico | Lookup por clave primaria — resultado binario. |
| Calcular el backoff exponencial para el intento N | Determinístico | Fórmula fija: `base * 2^N`. El jitter es pseudoaleatorio, no agéntico. |
| Decidir si una operación fallida debe reintentarse | Determinístico | Tabla de status codes: 4xx no se reintenta, 5xx y 429 sí. |
| Decidir cuántos reintentos son aceptables en un workflow de negocio | Agéntico / diseño | Depende del SLA del cliente, el costo de la operación, y el contexto del proveedor. No hay una regla única. |

## 10. Errores típicos

**1. Usar una idempotency key que cambia en cada reintento.**
*Síntoma*: se crean N órdenes de compra duplicadas (una por cada intento).
*Causa*: la key se genera con `uuid.uuid4()` en cada llamada — nunca es la misma.
*Arreglo*: generar la key **antes** del primer intento y reutilizarla en todos los reintentos. `key = make_idempotency_key(tenant, "purchase-order", po_id)` — el `po_id` es el identificador de negocio, no el ID de la request HTTP.

**2. Reintentar un 401 en bucle.**
*Síntoma*: el agente llena los logs con intentos fallidos durante horas hasta que alguien lo detiene manualmente.
*Causa*: la política de reintentos no distingue entre errores temporales (5xx, 429) y errores permanentes (4xx).
*Arreglo*: la función de retry debe propagar inmediatamente cualquier `HTTPError` con status 4xx (excepto 429).

**3. Backoff sin jitter — thundering herd.**
*Síntoma*: el servidor se sobrecarga exactamente cada 4 segundos (el backoff de todos los clientes).
*Causa*: todos los clientes que fallaron al mismo tiempo usan exactamente el mismo `base * 2^N` sin variación.
*Arreglo*: añadir jitter: `delay = base * 2^N + random.uniform(0, base)`. La aleatoriedad desincroniza los reintentos.

**4. No usar `Retry-After` del servidor cuando está disponible.**
*Síntoma*: el cliente espera demasiado (backoff propio) cuando el servidor ya indica exactamente cuándo puede reintentar.
*Causa*: el código de retry ignora el header `Retry-After` en la respuesta 429.
*Arreglo*: siempre parsear `Retry-After` del header 429 y usarlo en lugar del backoff propio cuando está presente.

## 1. Hilo conductor

En B08 aprendiste la anatomía de una API. En B06 viste cómo el chat tutor usa SSE para enviar tokens en tiempo real. Este módulo conecta ambas ideas con el problema concreto: ¿cómo se integra un agente IA con el mundo exterior? Hay tres patrones canónicos — pull, webhook, streaming — y cada uno tiene un caso de uso diferente. Track E (E01, E02, E03) profundizará en el SDK de Anthropic y LangGraph; aquí el énfasis es el contrato de integración, no el LLM en sí.

---

## 2. Idea central

Un agente IA no vive en el vacío. Necesita recibir inputs del entorno y devolver outputs. La forma en que el entorno y el agente se comunican define el patrón de integración:

- **Pull**: el entorno llama al agente y espera la respuesta sincrónica o asincrónicamente.
- **Webhook**: el entorno notifica al agente cuando ocurre un evento; el agente procesa y puede llamar de vuelta.
- **Streaming**: el agente emite resultados token a token mientras los genera, sin esperar a tener la respuesta completa.

Imagen mental: el agente es un consultor externo. Pull = tú le llamas, él responde. Webhook = te llama cuando tiene novedad. Streaming = habla mientras piensa, sin esperar a preparar un discurso completo.

## 3. Por qué importa

El patrón de integración determina la experiencia del usuario y los requisitos de infraestructura. Si eliges pull síncrono para un agente que tarda 30 segundos, el cliente recibe un timeout. Si usas webhook cuando el cliente necesita la respuesta inmediata, el flujo de UX se rompe. Si implementas streaming cuando la respuesta es un JSON estructurado de 50 bytes, añades complejidad innecesaria.

ACME Manufacturing tiene tres necesidades distintas: auditoria mensual (pull asíncrono), alerta de factura duplicada entrante (webhook), y asistente de análisis interactivo (streaming). Tres casos, tres patrones.

## 4. Cómo funciona por dentro

### Patrón 1 — Pull (síncrono y asíncrono)

El cliente llama al agente vía `POST` y espera la respuesta. Hay dos variantes:

**Síncrono** (útil para respuestas cortas, < 10s): el cliente hace `POST /agent/audit` y espera el `200 OK` con el resultado. FastAPI mantiene la conexión abierta durante el procesamiento.

```python
# app/routers/agent.py — endpoint pull síncrono
from fastapi import APIRouter
from pydantic import BaseModel

router = APIRouter(prefix="/api/agent", tags=["agent"])

class AuditRequest(BaseModel):
    tenant: str
    period: str  # "YYYY-MM"

class AuditResult(BaseModel):
    audit_id: str
    findings: list[dict]
    summary_md: str

@router.post("/audit", response_model=AuditResult)
async def run_audit(req: AuditRequest) -> AuditResult:
    # llama al agente — puede tardar varios segundos
    result = await audit_service.run(tenant=req.tenant, period=req.period)
    return result
```

**Asíncrono** (para procesos largos, > 10s): el cliente hace `POST /agent/audits` y recibe un `202 Accepted` con un `job_id`. Luego hace `GET /agent/audits/{job_id}` para consultar el estado hasta que sea `"done"`.

```
POST /api/agent/audits
→ 202 {"job_id": "job-acme-2026-04", "status": "queued"}

GET /api/agent/audits/job-acme-2026-04
→ 200 {"job_id": "job-acme-2026-04", "status": "running", "progress": 0.4}

GET /api/agent/audits/job-acme-2026-04
→ 200 {"job_id": "job-acme-2026-04", "status": "done", "result": {...}}
```

Este patrón lo usa la app demo en `api/app/routers/temporal_demo.py`: el workflow Temporal corre en background; el cliente consulta el estado periódicamente.

### Patrón 2 — Webhook (event-driven)

El sistema externo (Siigo, Stripe, GitHub) registra una URL del agente y hace `POST` cuando ocurre un evento. El agente debe:

1. Responder `200 OK` inmediatamente (en < 5 segundos, o el proveedor marca el webhook como fallido).
2. Encolar el procesamiento pesado para después (Temporal, Celery, o simplemente `asyncio.create_task`).

```python
@router.post("/webhooks/siigo/invoice")
async def receive_siigo_invoice(
    req: Request,
    background_tasks: BackgroundTasks,
) -> dict:
    # 1. verificar firma HMAC antes de procesar (ver B10)
    body = await req.body()
    _verify_hmac(body, req.headers.get("X-Siigo-Signature", ""))

    # 2. parsear payload
    payload = await req.json()

    # 3. encolar para procesamiento asíncrono — responder inmediatamente
    background_tasks.add_task(process_invoice_event, payload)
    return {"received": True}
```

> [!cuidado]
> Nunca proceses el evento en el mismo request del webhook si puede tardar más de 5 segundos. El proveedor asume timeout y reintenta — recibirás el mismo evento dos veces. Encola y procesa en background.

### Patrón 3 — Streaming SSE

El agente genera tokens o resultados parciales mientras procesa. El cliente recibe cada fragmento en tiempo real. Ya viste la implementación en B06 con el chat tutor. Aquí el patrón se aplica a un agente de análisis:

```python
from fastapi.responses import StreamingResponse

@router.post("/agent/analyze/stream")
async def analyze_stream(req: AuditRequest) -> StreamingResponse:
    async def gen():
        async for chunk in audit_agent.stream(
            tenant=req.tenant, period=req.period
        ):
            # chunk puede ser un fragmento de texto o un hallazgo estructurado
            if chunk.kind == "finding":
                yield _sse("finding", chunk.data)
            elif chunk.kind == "token":
                yield _sse("delta", {"text": chunk.text})
        yield _sse("done", {"audit_id": chunk.audit_id})

    return StreamingResponse(gen(), media_type="text/event-stream")
```

### Comparativa de los tres patrones

| Criterio | Pull síncrono | Pull asíncrono | Webhook | Streaming SSE |
|----------|---------------|----------------|---------|---------------|
| Quién inicia | Cliente | Cliente | Sistema externo | Cliente |
| Latencia de respuesta | Inmediata | Polling | En tiempo del evento | Token a token |
| Casos de uso | Clasificación rápida, queries | Auditoría larga, batch | Alertas, integraciones ERP | Chat, análisis en vivo |
| Complejidad | Baja | Media (polling o websocket para notificar) | Media (verificación HMAC, idempotency) | Media (parser SSE en cliente) |
| Riesgo principal | Timeout si el agente tarda | Race condition en polling | Evento duplicado (ver B11) | Conexión cortada a mitad |

### El flujo determinístico vs agéntico en la integración

La llamada a la API del agente es 100% determinística: un `POST` con un JSON fijo a una URL fija produce el mismo flujo de código. Lo que **no es determinístico** es lo que el agente decide hacer con los datos dentro del llamado: qué findings considera importantes, cómo redacta el resumen, si el monto es sospechoso. Esa distinción es fundamental: el contrato de la API es determinístico; la semántica de la respuesta es agéntica.

### Referencia en la app real

El chat tutor de bato-learning implementa el patrón streaming completo:

- `app/routers/chat.py` → endpoint `POST /api/chat` que devuelve `StreamingResponse`.
- `app/services/anthropic_client.py` → `stream_chat()` que envuelve `client.messages.stream()`.
- `app/static/js/chat.js` → cliente que consume el stream con `fetch + ReadableStream`.

La `api/app/routers/temporal_demo.py` implementa el patrón pull asíncrono con Temporal como motor de ejecución.

## 5. Ejemplo conducido

Simulación de los tres patrones sin red, en Python puro:

```python
import asyncio
from typing import AsyncIterator

# Patrón 1: Pull síncrono
async def pull_agent(tenant: str, period: str) -> dict:
    # en producción: llama al SDK de Anthropic, LangGraph, etc.
    await asyncio.sleep(0.01)  # simula latencia del agente
    return {
        "tenant": tenant,
        "period": period,
        "findings": [{"type": "duplicate", "amount": 4820.0}],
    }

# Patrón 2: Webhook handler (responde inmediatamente, encola)
async def webhook_handler(payload: dict, enqueue) -> dict:
    enqueue(payload)  # encolar — no procesar aquí
    return {"received": True}

# Patrón 3: Streaming
async def stream_agent(tenant: str) -> AsyncIterator[dict]:
    tokens = ["Analizando", " facturas", " de ", tenant, "...", " Hallazgo: ", "duplicado"]
    for token in tokens:
        await asyncio.sleep(0.005)
        yield {"kind": "token", "text": token}
    yield {"kind": "done", "findings": 1}

async def main():
    # Pull
    result = await pull_agent("acme", "2026-04")
    print(f"Pull: {result['findings']}")

    # Streaming
    print("Stream: ", end="")
    async for chunk in stream_agent("acme"):
        if chunk["kind"] == "token":
            print(chunk["text"], end="", flush=True)
    print()

asyncio.run(main())
```

## 8. Caso de uso real

**F-OPS-02 — Auditoría de inventario**: Globex Logistics tiene tres flujos de integración del agente en un solo proceso:

1. **Pull asíncrono**: el scheduler semanal hace `POST /agent/inventory-audit` y recibe `job_id`. El workflow Temporal ejecuta el agente en background durante 5-10 minutos.
2. **Webhook**: cuando el WMS (Drivin) registra un movimiento de stock inesperado, notifica al endpoint del agente vía webhook. El agente genera una alerta inmediata.
3. **Streaming**: el operador de bodega puede preguntar al asistente «¿por qué hay discrepancia en el SKU-047?» y ve el análisis construirse en tiempo real.

Tres patrones, un solo agente, tres contextos de uso distintos.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Parsear el body del `POST /agent/audit` con Pydantic | Determinístico | Validación de tipos con regla cerrada. |
| Verificar la firma HMAC del webhook entrante | Determinístico | Operación criptográfica — mismo input, mismo output. |
| Enrutar el event type del webhook al handler correcto | Determinístico | Tabla fija de tipos de evento → handler. |
| Decidir si un hallazgo en la auditoría es crítico o informativo | Agéntico | Depende del contexto del cliente, historial, y criterio del negocio. |
| Redactar el resumen del reporte de auditoría | Agéntico | Salida para humano; el tono y la selección de información requieren juicio. |
| Calcular el `retry_after` cuando el servidor está saturado | Determinístico | Fórmula con backoff exponencial — regla cerrada (ver B11). |

## 10. Errores típicos

**1. Procesar el evento del webhook en el mismo request handler.**
*Síntoma*: el proveedor marca el webhook como fallido y lo reintenta; el agente procesa el mismo evento dos veces.
*Causa*: el handler tarda > 5s y el proveedor asume timeout.
*Arreglo*: responder `200 OK` inmediatamente, encolar con `BackgroundTasks` o Temporal, procesar en background.

**2. Usar pull síncrono para un agente que tarda > 10 segundos.**
*Síntoma*: el cliente recibe `504 Gateway Timeout` de Nginx o el load balancer.
*Causa*: el timeout de red es menor que el tiempo de ejecución del agente.
*Arreglo*: pasar al patrón pull asíncrono: `202 Accepted` + polling o notificación por webhook de vuelta al cliente.

**3. No manejar la desconexión del cliente en streaming.**
*Síntoma*: el servidor sigue generando tokens aunque el cliente cerró la conexión (usuario cerró el tab).
*Causa*: el generador SSE no detecta que el cliente se desconectó.
*Arreglo*: envolver el `yield` en un try/except que detecta `asyncio.CancelledError` o `GeneratorExit` y cancela el streaming del LLM.

**4. Asumir que el webhook llegará exactamente una vez.**
*Síntoma*: la factura se procesa y guarda dos veces en la DB.
*Causa*: el proveedor reintentó el webhook tras un timeout aparente.
*Arreglo*: implementar idempotency key en el handler del webhook (ver B11).

## 2. Idea central

Un **test de API** verifica que el contrato de un endpoint se cumple: dado este input, el servidor devuelve este output con este status code. No testea la implementación interna — testea el comportamiento observable desde el exterior. `pytest` es el framework, `httpx.AsyncClient` es el cliente HTTP para tests async, y las **fixtures** son la forma de crear y limpiar el estado de prueba (DB en memoria, mocks de dependencias).

Imagen mental: testear un API es como QA en una fábrica. No abres la máquina para ver si el engranaje está bien colocado — pones la materia prima en la entrada y verificas que la salida es la esperada.

## 3. Por qué importa

Sin tests, refactorizar es una operación de fe. ACME Manufacturing tiene un agente que llama al endpoint `POST /api/audits`. Cuando el equipo de plataforma refactoriza la capa de servicios (Track B04), sin un test que verifique que el endpoint sigue respondiendo `201 Created` con el schema correcto, no hay garantía de que el agente siga funcionando.

La cobertura (coverage %) no es la métrica. Una suite con 95% de coverage que solo testea el camino feliz no detecta un `403` que debería ser `404`. Lo que importa es que los **comportamientos del contrato** (casos felices, errores esperados, edge cases de negocio) tengan tests.

## 4. Cómo funciona por dentro

### El cliente de test async: `httpx.AsyncClient`

FastAPI recomienda `httpx.AsyncClient` con el transport `ASGITransport` para tests. No levanta un servidor real — invoca el ASGI app directamente en el mismo proceso:

```python
import pytest
import pytest_asyncio
from httpx import AsyncClient, ASGITransport
from app.main import app

@pytest_asyncio.fixture
async def client() -> AsyncClient:
    async with AsyncClient(
        transport=ASGITransport(app=app),
        base_url="http://test",
    ) as client:
        yield client
```

El `yield` garantiza que el cliente se cierra después del test (teardown). Sin `async with`, el cliente queda abierto y el test puede filtrar conexiones.

### Fixtures de DB en memoria

Los tests no deben tocar la DB de desarrollo. La solución: una fixture que crea una DB SQLite en memoria para cada test y la descarta al terminar:

```python
import pytest_asyncio
from sqlalchemy.ext.asyncio import AsyncSession, create_async_engine, async_sessionmaker
from app.models import Base
from app.deps import get_session

@pytest_asyncio.fixture
async def db_session() -> AsyncSession:
    engine = create_async_engine("sqlite+aiosqlite:///:memory:")
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
    factory = async_sessionmaker(engine, expire_on_commit=False)
    async with factory() as session:
        yield session
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.drop_all)
    await engine.dispose()

@pytest_asyncio.fixture
async def client(db_session: AsyncSession) -> AsyncClient:
    # Override the get_session dependency para usar la DB de test
    app.dependency_overrides[get_session] = lambda: db_session
    async with AsyncClient(
        transport=ASGITransport(app=app),
        base_url="http://test",
    ) as client:
        yield client
    app.dependency_overrides.clear()
```

La clave es `app.dependency_overrides` — FastAPI permite sustituir cualquier `Depends` en tiempo de test. Aquí reemplazamos `get_session` por una lambda que devuelve la sesión de test.

### Mockear el SDK de Anthropic

El chat tutor llama a Anthropic. En tests, no queremos hacer llamadas reales (cuestan dinero y son lentas). La solución es mockear `stream_chat` con `unittest.mock.AsyncMock`:

```python
from unittest.mock import patch, AsyncMock
from app.services.anthropic_client import StreamChunk

@pytest.fixture
def mock_anthropic():
    async def fake_stream(**kwargs):
        yield StreamChunk(text="Respuesta de prueba del tutor.")
        yield StreamChunk(done=True, usage={"input_tokens": 10, "output_tokens": 5}, cost_usd=0.0)

    with patch("app.services.anthropic_client.stream_chat", side_effect=fake_stream) as mock:
        yield mock
```

El `side_effect` es un generador async que devuelve chunks predecibles. Los tests del chat pueden verificar que el endpoint emite los eventos SSE correctos sin llamar a Anthropic.

### Estructura de un test de API

```python
@pytest.mark.asyncio
async def test_create_invoice_returns_201(client: AsyncClient):
    payload = {
        "vendor_id": "vnd-001",
        "amount": 4820.0,
        "period": "2026-04",
        "tenant": "acme",
    }
    response = await client.post(
        "/api/v1/invoices",
        json=payload,
        headers={"Authorization": "Bearer acme-key-abc123"},
    )
    assert response.status_code == 201
    body = response.json()
    assert body["vendor_id"] == "vnd-001"
    assert body["status"] == "pending"
    assert "id" in body

@pytest.mark.asyncio
async def test_create_invoice_without_auth_returns_401(client: AsyncClient):
    response = await client.post("/api/v1/invoices", json={"amount": 100.0})
    assert response.status_code == 401

@pytest.mark.asyncio
async def test_create_invoice_missing_vendor_id_returns_422(client: AsyncClient):
    response = await client.post(
        "/api/v1/invoices",
        json={"amount": 100.0, "period": "2026-04"},
        headers={"Authorization": "Bearer acme-key-abc123"},
    )
    assert response.status_code == 422
```

### Unit vs integration vs E2E

| Tipo | Qué testea | Velocidad | DB | Red |
|------|-----------|-----------|-----|-----|
| Unit | Una función o clase aislada | Muy rápido | No | No |
| Integration | Varias capas juntas (endpoint + service + DB) | Rápido | En memoria | No |
| E2E | El sistema completo desde el navegador | Lento | Real o staging | Real |

Los tests de B13 son **integration tests**: testean el endpoint, el servicio y el repo juntos, con una DB en memoria. Es el nivel de abstracción correcto para verificar el contrato de la API sin la fragilidad de E2E.

### Coverage: métrica, no objetivo

```
pytest --cov=app --cov-report=term-missing
```

El coverage reporta qué líneas se ejecutaron durante los tests. Un 100% de coverage con tests que solo verifican `assert response.status_code == 200` es inútil. Lo que importa:

- ¿El test verifica el **schema** de la respuesta, no solo el status code?
- ¿Hay tests para los **paths de error** (401, 403, 422, 404)?
- ¿Hay tests para los **edge cases de negocio** (monto negativo, período inválido, tenant inexistente)?

Una suite con 70% de coverage que cubre los contratos críticos es mejor que 95% que solo testea el camino feliz.

## 5. Ejemplo conducido

Un test completo con fixture de DB en memoria y override de dependencia:

```python
import pytest
import pytest_asyncio
from httpx import AsyncClient, ASGITransport

# fixture mínima para la app de ejemplo
@pytest_asyncio.fixture
async def client():
    from fastapi import FastAPI
    from pydantic import BaseModel

    mini_app = FastAPI()

    class Item(BaseModel):
        name: str
        value: float

    @mini_app.post("/items", status_code=201)
    async def create_item(item: Item) -> dict:
        return {"id": "item-001", "name": item.name, "value": item.value}

    async with AsyncClient(
        transport=ASGITransport(app=mini_app),
        base_url="http://test",
    ) as c:
        yield c

@pytest.mark.asyncio
async def test_create_item(client: AsyncClient):
    resp = await client.post("/items", json={"name": "Factura ACME", "value": 4820.0})
    assert resp.status_code == 201
    assert resp.json()["name"] == "Factura ACME"

@pytest.mark.asyncio
async def test_create_item_missing_field(client: AsyncClient):
    resp = await client.post("/items", json={"value": 100.0})  # falta "name"
    assert resp.status_code == 422
```

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Verificar que el endpoint devuelve el schema correcto | Determinístico | El test tiene una expectativa fija — mismo input, misma verificación. |
| Crear la fixture de DB con `create_all` y limpiarla con `drop_all` | Determinístico | Orden fijo de operaciones; mismo resultado siempre. |
| Decidir qué casos de test cubrir para un endpoint nuevo | Agéntico / criterio | Depende del contexto del negocio, los riesgos del endpoint, y el tiempo disponible. No hay una lista universal. |
| Mockear el SDK de Anthropic en tests de chat | Determinístico | `patch` intercepta siempre la misma función con el mismo mock. |

## 10. Errores típicos

**1. No limpiar el override de dependencias después del test.**
*Síntoma*: tests pasan en aislamiento pero fallan en suite completa por orden de ejecución.
*Causa*: `app.dependency_overrides` no se limpia en el teardown de la fixture. El override de un test contamina los siguientes.
*Arreglo*: siempre `app.dependency_overrides.clear()` en el teardown. La fixture con `yield` garantiza que el teardown ocurre aunque el test falle.

**2. Testear solo el camino feliz (`200 OK`).**
*Síntoma*: la suite tiene 80% de coverage pero no detecta que el endpoint devuelve `500` cuando el `tenant` no existe.
*Causa*: los tests solo verifican `assert response.status_code == 200`.
*Arreglo*: para cada endpoint, añadir al menos: el test del caso feliz, el test del error de validación (422), el test de auth (401/403), y el edge case de negocio más probable (404, 409).

**3. Usar la DB real en tests.**
*Síntoma*: los tests que corren bien en local fallan en CI porque el estado de la DB local está «contaminado».
*Causa*: los tests no usan fixture de DB en memoria — tocan la `data/app.db` real.
*Arreglo*: siempre `sqlite+aiosqlite:///:memory:` en fixtures de test. La DB en memoria se crea y destruye por test — sin estado compartido entre tests.

**4. Interpretar coverage % como calidad.**
*Síntoma*: el equipo celebra 95% de coverage pero el agente falla en producción con un caso no cubierto.
*Causa*: los tests ejecutan el código pero no verifican el comportamiento (solo `assert response is not None`).
*Arreglo*: la métrica útil no es coverage sino «¿cada test tiene al menos una aserción sobre el comportamiento observable?». Un test sin `assert` es documentación, no un test.

## 1. Hilo conductor

En B02 construiste endpoints que devuelven JSON. Esos endpoints existen; el
problema es que ningún humano navega un JSON. B03 conecta el servidor al
navegador con HTML renderizado en el servidor: Jinja2 produce el HTML completo
en el primer request; HTMX 2 reemplaza fragmentos concretos cuando el usuario
interactúa, sin compilar nada y sin ningún framework JS. B04 profundiza en la
separación de capas que hace que los routers que sirven HTML sean tan limpios
como los que sirven JSON.

## 2. Idea central

**Server-rendered** significa que el servidor envía HTML listo para mostrar, no
datos crudos que el cliente tiene que convertir en DOM. **Jinja2** es el motor
de plantillas de la app: recibe un `dict` de contexto y produce una cadena de
texto HTML. **HTMX 2** es una librería de 14 KB que añade atributos HTML
(`hx-get`, `hx-post`, `hx-target`, `hx-swap`) para hacer requests HTTP y
colocar la respuesta directamente en el DOM — sin escribir JavaScript. El
resultado es una UI completamente funcional donde el servidor sigue siendo la
única fuente de verdad sobre el estado.

## 3. Por qué importa

Una SPA React para un dashboard interno de soporte tiene tres costos ocultos:
un build pipeline, un estado cliente que puede desincronizarse del servidor, y
un bundle que el navegador descarga antes de mostrar nada útil. Para casos de
uso como la base de conocimiento de CX de F-CX-03 — donde un agente de soporte
revisa sugerencias y las acepta o rechaza — ese costo no se justifica: la
interacción es `click → POST → replace row`, que HTMX resuelve con dos
atributos.

Server-rendered también premia el SEO: los buscadores ven HTML completo en el
primer request, no un `<div id="root"></div>` vacío que espera JavaScript.

Si el proyecto requiere estado cliente profundo — paginación virtual de
100 000 filas, un gráfico D3 con zoom y filtro, un editor de código — entonces
HTMX no es la herramienta adecuada. En esos casos, una isla de JavaScript
vanilla (como `app/static/js/chat.js`) o una librería específica es lo
correcto. HTMX y vanilla JS coexisten bien: elige cada herramienta para lo que
hace bien.

## 4. Cómo funciona por dentro

### Jinja2: el modelo mental

Jinja2 trabaja con un `Environment` que sabe dónde buscar plantillas (el
directorio `app/templates/`). Cuando el router llama a
`templates.TemplateResponse("module.html", {"request": req, "module": m})`,
Jinja2 carga `module.html`, evalúa cada expresión `{{ }}` y directiva `{% %}`,
y devuelve una cadena UTF-8.

El mecanismo de **herencia** es la pieza central. `_layout.html` define la
estructura base con `{% block main %}{% endblock %}`. Una plantilla hija
empieza con `{% extends "_layout.html" %}` y llena cada bloque sin repetir
el `<head>`, la barra lateral, ni los scripts comunes.

```
_layout.html          module.html
┌─────────────┐       ┌──────────────────────────────┐
│ <head>      │  ←──  │ {% extends "_layout.html" %}  │
│ <sidebar>   │       │ {% block title %}...{% endblock %} │
│ {% block    │       │ {% block main %}             │
│   main %}   │       │   <article>{{ module... }}   │
│ {% endblock %}│     │ {% endblock %}               │
│ <chat>      │       └──────────────────────────────┘
└─────────────┘
```

Los **partials** son plantillas sin herencia que se incluyen con
`{% include "partials/exercise.html" %}`. No heredan; solo se renderizan en el
contexto donde se incluyen. Esto es lo que permite el loop
`{% for ex in module.exercises %} {% include "partials/exercise.html" %} {% endfor %}`
en `module.html`: cada iteración pasa la variable `ex` disponible en el
contexto local.

### Filtros de Jinja2 en la app

La app registra tres filtros personalizados:

- `| md` — convierte Markdown a HTML (usado como `{{ module.body_md | md | safe }}`).
- `| track_label` — convierte `"B"` en la etiqueta legible del track.
- `| dept_label` — para fichas F, convierte el código de departamento.

El filtro `| safe` le dice a Jinja2 que no escape el HTML producido. Se usa
siempre después de `| md` porque el HTML ya viene validado del servidor.

### HTMX 2: el modelo mental

HTMX 2 intercepta eventos del navegador (click, change, submit, timers) y
ejecuta un request HTTP usando los atributos del elemento. La respuesta debe
ser HTML — no JSON. HTMX la inserta en el DOM según `hx-target` y `hx-swap`.

```
elemento con hx-get="/api/suggestions"
    │
    ▼ evento trigger (click por defecto)
HTMX hace GET /api/suggestions
    │
    ▼ servidor devuelve <ul>…</ul>
HTMX encuentra el elemento indicado en hx-target
    │
    ▼ aplica hx-swap (innerHTML por defecto)
DOM actualizado sin reload de página
```

Los atributos esenciales:

| Atributo | Qué hace |
|----------|----------|
| `hx-get="/ruta"` | GET al endpoint cuando se dispara el trigger |
| `hx-post="/ruta"` | POST con el form serializado |
| `hx-target="#elemento"` | selector CSS del contenedor a actualizar |
| `hx-swap="outerHTML"` | reemplaza el elemento completo (no solo su interior) |
| `hx-swap="innerHTML"` | reemplaza solo el contenido interior (default) |
| `hx-trigger="every 5s"` | dispara el request cada 5 segundos |
| `hx-trigger="change"` | dispara al cambiar el valor del elemento |

En HTMX 2.x, `hx-on:` reemplaza al viejo `hx-on=`. Para capturar el evento
`htmx:afterRequest` en el propio elemento: `hx-on:htmx:after-request="..."`.

### El header HX-Request

HTMX añade `HX-Request: true` a cada request que hace. El servidor puede
usarlo para devolver solo el fragmento que HTMX necesita, en lugar de la
página completa:

```python
from fastapi import Request
from fastapi.responses import HTMLResponse

@router.get("/suggestions/{ticket_id}", response_class=HTMLResponse)
async def get_suggestions(ticket_id: int, request: Request):
    suggestions = await suggestion_service.get(ticket_id)
    # si es HTMX, devuelve solo el partial
    if request.headers.get("HX-Request"):
        return templates.TemplateResponse(
            "partials/suggestion_list.html",
            {"request": request, "suggestions": suggestions},
        )
    # si es navegación directa, devuelve la página completa
    return templates.TemplateResponse(
        "dashboard.html",
        {"request": request, "suggestions": suggestions},
    )
```

### Render del template es determinístico

El template no decide nada. Dado el mismo contexto (`module`, `exercises`,
`quizzes`), produce el mismo HTML byte a byte. El LLM no participa en el
render. La decisión de qué datos van en el contexto ocurre en el servicio
(determinístico) o en el agente (agéntico); el template solo los muestra.

## 5. Ejemplo conducido

Dashboard de sugerencias para el agente de soporte de F-CX-03. Tres piezas:
el partial, el endpoint, y el botón de aceptar.

**Partial `partials/suggestion_row.html`** — renderiza una sugerencia:

```html
<tr id="suggestion-{{ s.id }}">
  <td>{{ s.title }}</td>
  <td>{{ s.score | round(2) }}</td>
  <td>
    <button
      hx-post="/api/suggestions/{{ s.id }}/accept"
      hx-target="#suggestion-{{ s.id }}"
      hx-swap="outerHTML"
    >Aceptar</button>
  </td>
</tr>
```

**Endpoint en `app/routers/suggestions.py`**:

```python
@router.post("/api/suggestions/{suggestion_id}/accept",
             response_class=HTMLResponse)
async def accept_suggestion(
    suggestion_id: int,
    request: Request,
    session: SessionDep,
) -> HTMLResponse:
    suggestion = await suggestion_service.accept(session, suggestion_id)
    return templates.TemplateResponse(
        "partials/suggestion_row_accepted.html",
        {"request": request, "s": suggestion},
    )
```

Al hacer click, HTMX envía POST, el servidor ejecuta la acepción y devuelve el
partial `suggestion_row_accepted.html` (misma `<tr>` pero con badge «aceptado»).
HTMX reemplaza el `<tr>` entero porque `hx-swap="outerHTML"`. No hay una línea
de JavaScript.

**Polling de lista de tickets abiertos**:

```html
<div
  hx-get="/api/tickets/open"
  hx-trigger="every 30s"
  hx-target="#ticket-list"
  hx-swap="innerHTML"
>
  <!-- cargado inicialmente por Jinja2; HTMX lo refresca -->
</div>
```

## 6. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Renderizar `module.html` con el contexto recibido | determinístico | Mismo contexto → mismo HTML, siempre. |
| Incluir `partials/exercise.html` en un loop | determinístico | El orden lo dicta `module.exercises`, no el modelo. |
| Decidir qué sugerencias de KB mostrar al agente de soporte | agéntico | El ranking usa embeddings + contexto del ticket; no hay regla cerrada. |
| Marcar una sugerencia como aceptada cuando el usuario hace click | determinístico | El estado cambia por acción explícita del humano, no por inferencia. |
| Refresco periódico de la lista de tickets con `hx-trigger="every 30s"` | determinístico | Es un GET sin decisión; el servidor devuelve lo que la DB tiene. |

## 7. Errores típicos

**1. Usar `hx-swap="innerHTML"` cuando se necesita `outerHTML`.**
*Síntoma*: el atributo `hx-post` del elemento reemplazado desaparece y el
segundo click no hace nada.
*Causa*: `innerHTML` reemplaza el contenido interior del target; si el target
es el propio elemento con los atributos HTMX, esos atributos se pierden.
*Arreglo*: usa `hx-swap="outerHTML"` cuando el elemento mismo debe cambiar
(e.g., una fila de tabla que pasa de estado «pendiente» a «aceptado»).

**2. El servidor devuelve JSON donde HTMX espera HTML.**
*Síntoma*: HTMX inserta `{"id": 1, "status": "accepted"}` como texto plano en
el DOM.
*Causa*: el endpoint tiene `response_class=JSONResponse` (o no lo declara) y
devuelve un dict.
*Arreglo*: cambia el endpoint a `response_class=HTMLResponse` y devuelve
`TemplateResponse`.

**3. Olvidar `| safe` después de `| md`.**
*Síntoma*: el Markdown se muestra como texto literal lleno de `&lt;p&gt;`.
*Causa*: Jinja2 escapa el HTML por defecto para prevenir XSS.
*Arreglo*: `{{ contenido | md | safe }}`. Solo usa `safe` con HTML que el
servidor produce; nunca con input de usuario sin sanitizar.

**4. Heredar de `_layout.html` en un partial.**
*Síntoma*: la página muestra `<html>` anidado en el DOM; los estilos se rompen.
*Causa*: un partial usa `{% extends "_layout.html" %}` en lugar de solo
definir el fragmento HTML.
*Arreglo*: los partials no extienden nada. Solo las plantillas completas
(`module.html`, `progress.html`) usan `extends`.

**5. Usar HTMX para interacciones con estado cliente profundo.**
*Síntoma*: el código termina con `hx-trigger` en cascada, out-of-band swaps
en 5 elementos distintos, y una lógica de estado que no se puede entender sin
depurar el DOM.
*Causa*: se está usando HTMX como sustituto de React para una interfaz que
necesita estado local complejo.
*Arreglo*: para esa pantalla, escribe una isla JS vanilla (ver el patrón de
`app/static/js/chat.js`) o usa una librería específica. HTMX no es un
framework de estado; es un mecanismo de intercambio de fragmentos.

## 8. Caso de uso real

**F-CX-03 — Generación y mantenimiento de base de conocimiento.**

El dashboard de soporte de TiendaBox muestra una columna con sugerencias de
artículos KB relevantes para el ticket abierto. El agente acepta o descarta
con un click. Cada acción es un `hx-post` que devuelve el partial actualizado
(`outerHTML`); el agente no recarga la página. La lista de tickets abiertos
se refresca cada 30 segundos con `hx-trigger="every 30s"`. El ranking de
sugerencias lo genera el agente (tramo agéntico); el render de la lista y el
cambio de estado al aceptar son tramos determinísticos manejados por el
servidor y el template.

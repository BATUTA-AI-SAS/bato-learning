## Hilo conductor

En B04 aprendiste que cada capa tiene una responsabilidad. En A07 viste cómo `async/await` permite esperar I/O sin bloquear. Este módulo une ambas ideas en el caso más concreto de la app: el chat tutor. El servidor hace streaming de tokens de Anthropic al navegador token a token, sin recargar la página ni abrir un WebSocket permanente. Eso es Server-Sent Events (SSE). E01 profundizará en el SDK de Anthropic; aquí el énfasis es el protocolo de transporte y la arquitectura de las capas del chat.

---

## Idea central

**SSE (Server-Sent Events)** es un protocolo HTTP donde el servidor mantiene la conexión abierta y envía fragmentos de texto con un formato específico. El cliente — navegador o `fetch` — recibe esos fragmentos uno a uno. La conexión es **unidireccional**: solo el servidor envía. El cliente envió su pregunta en un POST, y ahora escucha.

Imagen mental: el servidor es una radio que emite en frecuencia continua. El cliente sintoniza y recibe lo que va llegando. No puede responder por el mismo canal; para eso haría otro POST.

## Por qué importa

El LLM no tiene la respuesta lista de antemano — la genera token a token. Si el servidor esperara a tener la respuesta completa antes de enviarla, el usuario esperaría 10-20 segundos mirando una pantalla en blanco. Con SSE, el primer token llega en ~300ms y el usuario ve la respuesta construirse.

WebSocket resolvería el mismo problema pero agrega complejidad: handshake de upgrade, reconexión con backoff, manejo de mensajes bidireccionales. Para el patrón de chat — el usuario envía un mensaje, el servidor responde — SSE es suficiente y más simple. No hay nada que el cliente necesite decirle al servidor mientras recibe la respuesta.

El caso real (F-CX-02): el sistema de resumen de llamadas muestra el resumen generándose en tiempo real. El operador ve los puntos clave aparecer uno a uno, puede pulsar «detener» si es suficiente, y el sistema guarda lo que se generó hasta ese punto.

## Cómo funciona por dentro

### El protocolo SSE

El `Content-Type` de la respuesta es `text/event-stream`. Cada evento tiene este formato:

```
event: delta\n
data: {"text": "Para"}\n
\n
event: delta\n
data: {"text": " calcular"}\n
\n
event: done\n
data: {"usage": {...}, "cost_usd": 0.000042}\n
\n
```

Cada bloque termina con doble salto de línea (`\n\n`). El navegador los lee en `EventSource` o via `fetch + ReadableStream`. La app usa el segundo — `chat.js` lee la respuesta como stream de bytes y parsea los bloques manualmente.

### El endpoint en FastAPI

`app/routers/chat.py` define `POST /api/chat`. Recibe un `ChatBody` (Pydantic), carga el módulo y la sesión de chat, persiste el mensaje del usuario, construye el payload del system prompt y el historial, y devuelve un `StreamingResponse` cuyo generador hace streaming del LLM:

```python
async def gen():
    text_acc = ""
    async for chunk in stream_chat(system_blocks=system_blocks, messages=messages):
        if chunk.text:
            text_acc += chunk.text
            yield _sse("delta", {"text": chunk.text})
        if chunk.done:
            yield _sse("done", {"usage": chunk.usage, "cost_usd": chunk.cost_usd})
    # persiste el turno del asistente después de que el stream termina
    await chat_repo.append_message(session, ..., content_md=text_acc, ...)
```

La función `_sse(event, payload)` formatea el bloque SSE:

```python
def _sse(event: str, payload: dict) -> bytes:
    return f"event: {event}\ndata: {json.dumps(payload, ensure_ascii=False)}\n\n".encode()
```

### El servicio de streaming: `stream_chat`

`app/services/anthropic_client.py` define `stream_chat`, un generador async que envuelve el SDK de Anthropic. Usa `client.messages.stream(...)` y emite `StreamChunk` con el texto parcial. Al final emite un chunk con `done=True` y el objeto `usage` del SDK, que incluye `cache_read_input_tokens` y `cache_creation_input_tokens` para calcular el costo.

El servicio no sabe que existe FastAPI ni SSE. Recibe `system_blocks` y `messages`, devuelve chunks. El router convierte esos chunks a eventos SSE.

### El contexto del prompt: prompt caching

`app/services/context_builder.py` construye el payload del system prompt en dos bloques:

1. **Bloque estático** — las reglas del tutor + el body del módulo. Marcado con `cache_control: {"type": "ephemeral", "ttl": "1h"}`. Anthropic lo almacena hasta 1 hora. Si el mismo lector manda 10 preguntas sobre el mismo módulo, solo el primero paga el costo de escritura de cache; los 9 siguientes pagan el costo de lectura de cache (10× más barato).

2. **Mensajes dinámicos** — historial de chat + intentos recientes + código del editor. Sin cache. Cambia en cada turno.

Esta separación es la razón por la que el system prompt se parte en `system_blocks: list[dict]` en lugar de un string único.

### El cliente JavaScript: `chat.js`

`app/static/js/chat.js` usa `fetch` en lugar de `EventSource` nativo del navegador. Motivo: `EventSource` solo hace GET y no permite enviar headers de autenticación. Como el chat necesita un POST (para enviar el mensaje), usa `fetch` con streaming:

```javascript
const resp = await fetch("/api/chat", { method: "POST", body: JSON.stringify({...}) });
const reader = resp.body.getReader();
while (true) {
    const { value, done } = await reader.read();
    if (done) break;
    // parsea bloques SSE del buffer
}
```

### Cómo las capas del chat se mapean a archivos

| Capa | Archivo | Hace |
|------|---------|------|
| Router | `app/routers/chat.py` | Valida `ChatBody`, carga módulo y sesión, orquesta, devuelve `StreamingResponse` |
| Service | `app/services/anthropic_client.py` | Llama al SDK de Anthropic en modo stream, emite `StreamChunk` |
| Service | `app/services/context_builder.py` | Construye `system_blocks` y `messages` con el historial y contexto dinámico |
| Repo | `app/repos/chat.py` | Persiste sesiones y mensajes en la DB |
| Repo | `app/repos/progress.py` | Trae intentos recientes para el contexto dinámico |
| Model | `app/models/chat.py` | Define `ChatSession` y `ChatMessage` |

El router llama a dos servicios (no es lo habitual, pero aquí tiene sentido: `context_builder` es un servicio de preparación, `anthropic_client` es un servicio de ejecución). El patrón de capas se mantiene: ningún servicio importa del router.

> [!profundizar]
> Lee [`app/services/context_builder.py`](../../app/services/context_builder.py) función `_system_blocks`. Ahí verás cómo el `cache_control` con `ttl` explícito se aplica a los dos bloques estáticos. La clave `"ttl"` es obligatoria desde marzo 2026 — el SDK ya no usa el default silencioso (ver `_team/ARCHITECTURE.md §2.4`).

## Ejemplo conducido

Un endpoint SSE mínimo en FastAPI que streamea texto carácter a carácter (sin LLM, solo para entender el protocolo):

```python
import asyncio
from fastapi import APIRouter
from fastapi.responses import StreamingResponse

router = APIRouter()

async def _emit_chars(text: str):
    for char in text:
        yield f"event: delta\ndata: {char}\n\n".encode()
        await asyncio.sleep(0.05)
    yield b"event: done\ndata: {}\n\n"

@router.get("/api/demo/stream")
async def stream_demo():
    return StreamingResponse(
        _emit_chars("Hola, soy el tutor de bato-learning."),
        media_type="text/event-stream",
    )
```

El cliente recibe un evento `delta` con cada carácter, y un evento `done` al terminar. Sin WebSocket, sin `sse-starlette` — solo HTTP con `Transfer-Encoding: chunked`.

> [!nota]
> La app real usa `StreamingResponse` de Starlette directamente (sin `sse-starlette`). `sse-starlette` es una utilidad que facilita el manejo de reconexión del cliente; en esta app no es necesaria porque el cliente (el JavaScript de chat.js) maneja el stream con `fetch`.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Parsear el body del POST (`ChatBody`) | determinístico | Pydantic valida tipos y campos. |
| Construir el historial de mensajes para el prompt | determinístico | Toma los últimos N mensajes en orden — regla cerrada. |
| Seleccionar los bloques a marcar con `cache_control` | determinístico | Regla fija: los bloques estáticos se cachean, los dinámicos no. |
| Responder a la pregunta del lector | agéntico | El modelo decide qué responder según el contexto; no hay regla cerrada. |
| Decidir si el lector necesita refuerzo de un módulo previo | agéntico | El tutor infiere del historial de errores — juicio contextual. |
| Calcular el costo en USD de la respuesta | determinístico | Fórmula fija con los tokens del `usage` del SDK. |

## Errores típicos

**1. El generador de SSE no hace `yield` — devuelve una lista.**
*Síntoma*: el cliente recibe todo el contenido de golpe al final, no en streaming.
*Causa*: usar `return [chunk1, chunk2]` en lugar de `yield chunk` en la función generadora.
*Arreglo*: la función que pasa a `StreamingResponse` debe ser un generador async (`async def ... yield`). Si devuelves una lista, Starlette la itera, pero Python la materializa entera antes de empezar a iterar.

**2. Persistir el mensaje del asistente dentro del generador antes de que termine.**
*Síntoma*: si el cliente corta la conexión a mitad del stream, el mensaje queda incompleto en la DB pero marcado como si hubiera terminado.
*Causa*: llamar a `chat_repo.append_message` dentro del loop del generador.
*Arreglo*: acumula en `text_acc` durante el stream. Persiste al final, después del último `yield`.

**3. La sesión SQLAlchemy se cierra antes de que el generador termine.**
*Síntoma*: `MissingGreenlet` o `AsyncSession` ya cerrada al intentar persistir al final del stream.
*Causa*: la `SessionDep` cierra la sesión al salir del scope del endpoint, pero el generador todavía no ha terminado.
*Arreglo*: en la app actual, `StreamingResponse` mantiene vivo el generador y la sesión mientras se itera. Si usas un middleware que cierra la sesión early, debes pasar la sesión explícitamente al generador y manejar su ciclo de vida.

**4. Olvidar `ensure_ascii=False` en el `json.dumps` del evento SSE.**
*Síntoma*: el texto en español llega con secuencias de escape (`é` en lugar de `é`).
*Causa*: `json.dumps` por defecto escapa caracteres no-ASCII.
*Arreglo*: siempre `json.dumps(payload, ensure_ascii=False)` cuando el contenido puede tener texto en español.

## Caso de uso real

**F-CX-02 — Resumen automático de llamadas de soporte.**

El flujo completo con SSE: el agente de Clínica San Rafael ingiere la transcripción de una llamada de soporte, la procesa con el SDK de Anthropic, y el operador ve el resumen construirse en tiempo real en su pantalla. El endpoint `POST /api/calls/{call_id}/summarize/stream` devuelve un `StreamingResponse` que emite eventos `delta` con cada fragmento del resumen. El operador puede detener el stream si el resumen es suficiente; el sistema persiste lo generado hasta ese punto.

Las capas: el router valida y despacha; el servicio `call_service.stream_summary` llama al SDK; el repo `call_repo` guarda la transcripción y persiste el resumen final. El frontend es idéntico al chat tutor: `fetch` + `ReadableStream` + actualización del DOM fragmento a fragmento.

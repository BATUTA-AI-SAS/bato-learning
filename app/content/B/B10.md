## 2. Idea central

**Autenticación** (AuthN) responde «¿quién eres?» — verifica la identidad del caller. **Autorización** (AuthZ) responde «¿qué puedes hacer?» — verifica si esa identidad tiene permiso para la operación solicitada. Son distintas y ocurren en ese orden: primero saber quién, luego decidir si puede.

Imagen mental: la autenticación es el guardia de seguridad que revisa tu cédula. La autorización es el sistema de puertas que abre o no según tu nivel de acceso. Puedes pasar la autenticación (cédula válida) y fallar la autorización (tu acceso no llega a ese piso).

## 4. Cómo funciona por dentro

### API keys: el mecanismo más simple

Una **API key** es un string secreto que el servidor usa para identificar al caller. El cliente la envía en el header `Authorization: Bearer <key>`. El servidor la busca en su registro y determina a qué tenant pertenece y qué permisos tiene.

```python
# app/deps.py — dependency de autenticación con API key
from fastapi import Header, HTTPException, Depends
from ..repos import api_keys as api_key_repo

async def require_api_key(
    authorization: str = Header(...),
    session: AsyncSession = Depends(get_session),
) -> dict:
    if not authorization.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="Bearer token required")
    key = authorization.removeprefix("Bearer ")
    tenant_info = await api_key_repo.get_by_key(session, key)
    if tenant_info is None:
        raise HTTPException(status_code=401, detail="invalid API key")
    return tenant_info  # {"tenant": "acme", "scopes": ["read:invoices", "write:audits"]}
```

El endpoint protegido usa la dependency:

```python
@router.post("/api/audits")
async def create_audit(
    req: AuditRequest,
    caller: dict = Depends(require_api_key),
) -> AuditResult:
    # caller["tenant"] garantiza que el agente solo accede a datos de su tenant
    if req.tenant != caller["tenant"]:
        raise HTTPException(status_code=403, detail="tenant mismatch")
    ...
```

**Cuándo usar API keys**: comunicación machine-to-machine sin usuario humano. El agente de ACME llama a la API de auditoría con su API key. La key se rota periódicamente (90 días máximo). **Nunca** en un frontend — el usuario puede verla en el código fuente.

**Scopes**: las API keys modernas tienen scopes (`"read:invoices"`, `"write:audits"`). Es el principio de mínimo privilegio: la key del agente solo puede lo que necesita, no todo.

### JWT: stateless y autocontenido

**JWT (JSON Web Token)** es un token firmado criptográficamente que contiene claims (datos) sobre el usuario o el cliente. Tiene tres partes separadas por puntos: `header.payload.signature`.

El payload es un JSON base64 con claims estándar:

```json
{
  "sub": "user-42",
  "tenant": "acme",
  "scopes": ["read:invoices"],
  "iat": 1746000000,
  "exp": 1746003600
}
```

- `sub` (subject): quién es.
- `exp` (expiry): cuándo expira. **Obligatorio**. Un JWT sin `exp` es válido para siempre — es un bug de seguridad.
- `iat` (issued at): cuándo se emitió.
- Claims personalizados: `tenant`, `scopes`, cualquier dato que el servidor necesita para autorizar.

La firma usa HMAC (shared secret) o RSA/ECDSA (keypair). El servidor puede verificar el JWT sin consultar una DB: tiene la clave y verifica la firma. Eso es «stateless».

```python
import jwt  # PyJWT

def verify_jwt(token: str, secret: str) -> dict:
    try:
        return jwt.decode(token, secret, algorithms=["HS256"])
    except jwt.ExpiredSignatureError:
        raise HTTPException(status_code=401, detail="token expired")
    except jwt.InvalidTokenError:
        raise HTTPException(status_code=401, detail="invalid token")
```

**Cuándo usar JWT**: cuando el servidor no quiere guardar estado de sesión. APIs consumidas por múltiples clientes (web, móvil, otros servicios). La revocación es el punto débil: un JWT no se puede invalidar antes de su `exp` sin una lista negra (lo que niega la ventaja stateless). Por eso los JWT de acceso deben tener `exp` corto (15-60 minutos).

> [!cuidado]
> El payload del JWT está encodado en base64, **no cifrado**. Cualquiera puede decodificarlo y leer los claims. No pongas PII (nombre, email, número de documento) en el payload. Solo identificadores y scopes.

### OAuth2: delegación de autorización

**OAuth2** es un framework de autorización (no autenticación) para que un usuario dé permiso a una aplicación para acceder a sus recursos en otro sistema. Los dos flows más relevantes:

**Authorization Code** (usuario humano): el usuario hace login en el proveedor (Google, GitHub, Salesforce), aprueba el acceso, y el proveedor devuelve un code que tu app intercambia por un access token. Tu app nunca ve la contraseña del usuario.

```
1. Tu app redirige al usuario a: https://proveedor.com/oauth/authorize?
     client_id=mi_app&scope=read:facturas&redirect_uri=https://mi-app.com/callback

2. El usuario hace login y aprueba el acceso.

3. El proveedor redirige a:
     https://mi-app.com/callback?code=abc123

4. Tu backend intercambia el code por un access token:
     POST https://proveedor.com/oauth/token
     { "grant_type": "authorization_code", "code": "abc123", ... }
     → { "access_token": "eyJ...", "expires_in": 3600 }
```

**Client Credentials** (machine-to-machine): no hay usuario humano. El cliente presenta su `client_id` y `client_secret` y recibe un access token. Es el OAuth2 para agentes y servicios.

```
POST /oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&
client_id=acme-agent&
client_secret=secret-muy-largo&
scope=read:invoices write:audits
```

**Cuándo usar OAuth2**: cuando necesitas delegar autorización desde un sistema tercero — banco, ERP SaaS, plataforma de pago. El agente de ACME usa OAuth2 Client Credentials para obtener el token de Belvo (agregador bancario) antes de traer transacciones.

### Autenticación en FastAPI: dependency injection

FastAPI usa `Depends` para reutilizar lógica de autenticación. Una sola función verifica el token; todos los endpoints que la usan la declaran en su firma:

```python
# La dependency
async def get_current_tenant(
    authorization: str = Header(...),
) -> str:
    token = authorization.removeprefix("Bearer ")
    claims = verify_jwt(token, settings.jwt_secret)
    return claims["tenant"]

# Los endpoints la usan
@router.get("/api/invoices")
async def list_invoices(
    tenant: str = Depends(get_current_tenant),
    session: AsyncSession = Depends(get_session),
) -> list[Invoice]:
    return await invoice_repo.list_by_tenant(session, tenant)
```

Si la verificación falla, `get_current_tenant` lanza una `HTTPException(401)` y FastAPI la propaga automáticamente — el endpoint ni se ejecuta.

### Autenticación para el agente

Para un agente IA llamando APIs externas:

| Sistema externo | Mecanismo típico | Dónde vive la key |
|----------------|------------------|-------------------|
| Anthropic API | API key (`ANTHROPIC_API_KEY`) | Variable de entorno, nunca en código |
| Siigo / Alegra (LATAM ERP) | API key o OAuth2 Client Credentials | `settings.py` desde env vars |
| Belvo / Plaid (banco) | OAuth2 Client Credentials | `settings.py` desde env vars |
| Propia API interna | API key con scopes o JWT | `settings.py` desde env vars |

La regla es absoluta: **secretos como variables de entorno, nunca en código o en el repo**.

## 5. Ejemplo conducido

Una dependency de FastAPI que verifica API keys con scopes, sin DB (en memoria):

```python
from fastapi import Header, HTTPException, Depends
from typing import Annotated

# Registro en memoria (en producción: tabla api_keys en DB)
_API_KEYS: dict[str, dict] = {
    "acme-key-abc123": {"tenant": "acme", "scopes": {"read:invoices", "write:audits"}},
    "globex-key-xyz789": {"tenant": "globex", "scopes": {"read:invoices"}},
}

async def require_scope(scope: str):
    """Retorna una dependency que verifica que el caller tiene el scope requerido."""
    async def _check(authorization: Annotated[str, Header()]) -> dict:
        if not authorization.startswith("Bearer "):
            raise HTTPException(401, detail="Bearer token required")
        key = authorization.removeprefix("Bearer ")
        info = _API_KEYS.get(key)
        if info is None:
            raise HTTPException(401, detail="invalid API key")
        if scope not in info["scopes"]:
            raise HTTPException(403, detail=f"scope {scope!r} required")
        return info
    return _check

# Uso en un endpoint
@router.post("/api/audits")
async def create_audit(
    req: AuditRequest,
    caller: dict = Depends(require_scope("write:audits")),
) -> dict:
    return {"tenant": caller["tenant"], "audit_id": "aud-001"}
```

## 8. Caso de uso real

**F-FIN-01 — Conciliación bancaria**: el agente de Initech necesita tres tokens distintos: API key de Anthropic (para el LLM), OAuth2 access token de Belvo (para traer transacciones del banco), y JWT de sesión de la propia API de Initech. Los tres expiran en momentos distintos. El agente debe renovar el token de Belvo automáticamente cuando llega a la mitad de su `expires_in` (token rotation proactiva). Si falla la renovación, el agente retrocede a modo offline — usa los datos cacheados del día anterior y añade un `caveat` al reporte indicando que los datos pueden estar desactualizados.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Verificar la firma del JWT con la clave secreta | Determinístico | Operación criptográfica — mismo token + misma clave = mismo resultado siempre. |
| Comprobar que el claim `exp` no expiró | Determinístico | Comparación `exp > now()` — regla cerrada. |
| Decidir si conceder acceso basándose en el `tenant` y el `scope` | Determinístico | Tabla de permisos fija en código. |
| Seleccionar el mecanismo de auth correcto para una integración nueva | Agéntico | Depende del proveedor, el contexto de uso, los requisitos de seguridad — sin regla única. |
| Detectar si un token ha sido comprometido (anomalía de uso) | Agéntico | Requiere análisis de patrones — volumen inusual, geo-anomalía, uso fuera de horario. Sin regla cerrada. |

## 10. Errores típicos

**1. Poner la API key en el código fuente o en el repositorio.**
*Síntoma*: la API key aparece en el historial de git aunque la borres del archivo — sigue en los commits anteriores.
*Causa*: usar `API_KEY = "acme-key-abc123"` directamente en el código.
*Arreglo*: siempre `os.environ["API_KEY"]` o `settings.api_key` cargado desde `.env` (que está en `.gitignore`).

**2. JWT sin `exp` — válido para siempre.**
*Síntoma*: un token robado sigue funcionando meses después del incidente.
*Causa*: no incluir el claim `exp` al generar el JWT.
*Arreglo*: siempre fijar `exp = iat + 3600` (1 hora máximo para tokens de acceso). Para API keys de larga duración, usar rotación programada cada 90 días.

**3. Confundir 401 con 403.**
*Síntoma*: el endpoint devuelve `401` cuando el token es válido pero el rol no tiene permiso.
*Causa*: mezclar autenticación y autorización en el mismo bloque de código.
*Arreglo*: `401` cuando no hay token o el token no se puede verificar (AuthN falla). `403` cuando el token es válido pero el scope o el rol no tiene permiso (AuthZ falla).

**4. Guardar el access token de OAuth2 sin verificar `expires_in`.**
*Síntoma*: el agente falla con `401` en mitad de un workflow largo porque el token expiró.
*Causa*: guardar el token y usarlo hasta que falle, en lugar de renovarlo proactivamente.
*Arreglo*: guardar `expires_at = now() + expires_in - 60s` (margen de 60s). Antes de cada llamada verificar si `expires_at < now()` y renovar.

## Hilo conductor

En E01 construiste el agente con el SDK de Anthropic directamente. H04 mostró que el costo de LLM domina la factura a partir de Etapa 2. H07 cierra el último gap operativo: qué pasa cuando Anthropic está caído, cuando sus precios suben, o cuando un cliente de un país con restricciones regulatorias necesita que los datos no salgan de AWS. El gateway de modelos resuelve los tres escenarios.

## Idea central

Un **gateway de modelos** es una capa de abstracción entre tu agente y los proveedores de LLM. Cuando el agente llama a `llm.complete(messages)`, no llama a Anthropic directamente: llama al gateway, que decide qué proveedor usar según la política configurada (primario, fallback, cost routing). El gateway en 2026 tiene dos sabores: **LiteLLM** (auto-hospedado, open source, cero dependencias externas) y **OpenRouter** (SaaS, pay-per-use, catálogo de 100+ modelos).

## Por qué importa

Anthropic tuvo incidentes de disponibilidad documentados en 2024 y 2025. No son frecuentes —el uptime histórico es > 99,5%— pero cuando ocurren durante una auditoría nocturna o un onboarding crítico, el impacto es total: el agente no responde. Sin failover, la respuesta es «esperar a que Anthropic se recupere». Con failover, es una transparencia de 30 segundos.

El segundo argumento es económico: el costo por millón de tokens cambia. Un modelo de OpenAI o Bedrock puede ser más barato para ciertos tasks (clasificación, FAQ cortos) sin perder calidad. La ficha **F-FIN-03** (cierre mensual) documenta que el 40% de las llamadas del agente en esa ficha son clasificaciones simples que Haiku o `gpt-4o-mini` manejan con la misma precisión a la mitad del costo.

## Cómo funciona por dentro

### Patrón 1: primario + secundario con switch automático

El patrón más simple: Anthropic como primario, OpenAI como secundario. Si la llamada a Anthropic falla con un error transitorio (5xx, timeout), reintentar con el secundario.

```python
import anthropic
from openai import AsyncOpenAI
from tenacity import retry, stop_after_attempt, wait_exponential

PRIMARY_MODEL = "claude-sonnet-4-6"
FALLBACK_MODEL = "gpt-4o"  # o "gpt-4o-mini" para tasks simples

async def complete_with_fallback(messages: list[dict], *, tenant: str) -> str:
    """Intenta con Anthropic primero; si falla, usa OpenAI."""
    try:
        response = await _call_anthropic(messages)
        return response
    except (anthropic.APIStatusError, anthropic.APITimeoutError) as exc:
        if exc.status_code in (529, 503, 500):
            # Anthropic overloaded o caído → fallback
            return await _call_openai(messages, tenant=tenant)
        raise  # otros errores (auth, schema) no hacen fallback

async def _call_anthropic(messages: list[dict]) -> str:
    client = anthropic.AsyncAnthropic()
    response = await client.messages.create(
        model=PRIMARY_MODEL,
        max_tokens=2048,
        messages=messages,
    )
    return response.content[0].text

async def _call_openai(messages: list[dict], *, tenant: str) -> str:
    client = AsyncOpenAI()
    # OpenAI usa el formato messages directamente (compatible con Anthropic)
    # pero el role "system" debe estar fuera, no dentro de messages en Anthropic.
    # Aquí asumimos un adaptador ya normalizado.
    response = await client.chat.completions.create(
        model=FALLBACK_MODEL,
        messages=messages,
    )
    return response.choices[0].message.content
```

> [!cuidado]
> El formato de mensajes de Anthropic y OpenAI difiere sutilmente. El `system` en Anthropic es un parámetro separado; en OpenAI es un mensaje con `role: "system"`. Un gateway normaliza eso. Si haces el switch manual, normaliza antes de llamar al secundario.

### Patrón 2: LiteLLM como proxy unificado

LiteLLM provee una interfaz OpenAI-compatible para 100+ proveedores. Se despliega como un contenedor en tu Compose y tu agente llama siempre a `http://litellm:4000/v1/chat/completions`.

```yaml
# Añadir a compose.yaml (Etapa 3+)
services:
  litellm:
    image: ghcr.io/berriai/litellm:main-stable
    ports:
      - "4000:4000"
    environment:
      - ANTHROPIC_API_KEY=${ANTHROPIC_API_KEY}
      - OPENAI_API_KEY=${OPENAI_API_KEY}
    volumes:
      - ./infra/litellm/config.yaml:/app/config.yaml
    command: ["--config", "/app/config.yaml", "--port", "4000"]
```

```yaml
# infra/litellm/config.yaml
model_list:
  - model_name: primary
    litellm_params:
      model: anthropic/claude-sonnet-4-6
      api_key: "os.environ/ANTHROPIC_API_KEY"

  - model_name: fallback
    litellm_params:
      model: openai/gpt-4o
      api_key: "os.environ/OPENAI_API_KEY"

  - model_name: cheap
    litellm_params:
      model: anthropic/claude-haiku-4
      api_key: "os.environ/ANTHROPIC_API_KEY"

router_settings:
  routing_strategy: "fallback"
  fallbacks: [{"primary": ["fallback"]}]
  retry_policy:
    num_retries: 2
    retry_after: 0

litellm_settings:
  budget_duration: "1mo"
  max_budget: 500  # USD total por mes
```

Tu agente en Python llama a LiteLLM exactamente igual que a la API de OpenAI:

```python
from openai import AsyncOpenAI

# Apunta al gateway LiteLLM en lugar de a Anthropic directamente
gateway = AsyncOpenAI(
    base_url="http://litellm:4000/v1",
    api_key="any-string",  # LiteLLM gestiona las API keys reales
)

async def call_agent(messages: list[dict], *, tenant: str) -> str:
    response = await gateway.chat.completions.create(
        model="primary",  # LiteLLM resuelve a claude-sonnet-4-6
        messages=messages,
        extra_headers={"x-tenant-id": tenant},  # para logging
    )
    return response.choices[0].message.content
```

### Patrón 3: feature flag para switch manual

El caso más conservador para PYME: un feature flag en DB que permite cambiar el proveedor activo sin deploy. Si Anthropic anuncia mantenimiento o sube precios, el equipo activa el flag y el sistema empieza a usar el secundario.

```python
# app/services/llm_router.py
async def get_active_provider(tenant_id: str) -> str:
    """Lee el proveedor activo desde la configuración del sistema."""
    flag = await get_feature_flag("llm_provider_override")
    if flag and flag.value:
        return flag.value  # "anthropic" | "openai" | "bedrock"
    return "anthropic"  # default
```

### OpenRouter como alternativa a LiteLLM

OpenRouter es un SaaS pay-per-use que provee acceso a 200+ modelos con un único endpoint OpenAI-compatible. No tienes que gestionar múltiples API keys ni mantener un contenedor LiteLLM. El costo es un markup de ~10-20% sobre el precio del proveedor.

**Cuándo elegir OpenRouter vs LiteLLM**:
- **OpenRouter**: prototipado rápido, explorar modelos nuevos, equipo sin capacidad de operar servicios adicionales.
- **LiteLLM**: control total, sin markup, auto-hospedado, auditable, sin dependencia de un tercer SaaS.

Para BATUTA en producción: LiteLLM desde Etapa 2 si ya tienes Docker Compose. OpenRouter en Etapa 1 si quieres failover sin overhead operativo.

### Trade-offs del multi-modelo

| Consideración | Impacto | Mitigación |
|---------------|---------|-----------|
| El prompt caching no funciona cross-provider | Pierdes el ahorro de cache cuando hay fallback | Aceptable en failover emergencial (raro). En routing habitual: mantén caché por proveedor. |
| Cada modelo tiene particularidades de formato | El fallback puede producir outputs con formato diferente | Añadir un post-procesador de normalización de output. |
| Los precios y modelos cambian | La política de routing puede quedar desactualizada | Revisar trimestralmente. LiteLLM tiene costo-routing automático. |
| Regulaciones en LATAM (LGPD, Habeas Data) | Algunos clientes no pueden enviar datos a ciertos países | Verificar dónde procesa cada proveedor; Bedrock permite región AWS específica. |
| El comportamiento del modelo secundario difiere | Puede pasar golden set con A y fallar con B | Ejecutar golden set contra todos los modelos del pool. |

### Cuándo usar AWS Bedrock

Bedrock expone `claude-sonnet-4-6` y otros modelos de Anthropic con facturación por AWS. Ventajas para PYME LATAM:
- Los datos no salen de la región AWS que elijas (us-east-1, sa-east-1 para São Paulo).
- Si el cliente ya tiene AWS, se puede facturar en su cuenta AWS directamente.
- No hay dependencia de la API de Anthropic; Bedrock es el intermediario con su propio SLA.

Desventaja: los precios de Bedrock suelen ser ligeramente más altos que la API directa. Y el SDK de Anthropic es diferente para Bedrock (`AnthropicBedrock`). LiteLLM abstrae eso.

### Helicone como capa de observabilidad

Helicone es un proxy observabilidad-primero compatible con Anthropic y OpenAI. Registra cada llamada (latencia, tokens, costo), permite reproducir conversaciones, y tiene alertas de costo. Para BATUTA en Etapa 2+ es complementario a Phoenix: Phoenix para trazas de agente, Helicone para billing y análisis de llamadas LLM individuales.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Detectar un error 5xx de Anthropic y activar el fallback | determinístico | Regla cerrada: código de error → proveedor alternativo. |
| Normalizar el formato de mensajes Anthropic → OpenAI | determinístico | Transformación estructural con regla fija. |
| Decidir qué modelo usar para una tarea específica (Haiku vs Sonnet) | agéntico | Requiere evaluar complejidad de la tarea; no hay regla universal. |
| Decidir si activar el switch manual de proveedor | agéntico | Depende de la gravedad del incidente, SLA del cliente y alternativas disponibles. |
| Diseñar la política de routing cost-optimal | agéntico | Requiere analizar el historial de calls, el golden set y los precios vigentes. |

## Errores típicos

**1. Fallback sin normalizar el formato de mensajes.**
*Síntoma*: el fallback a OpenAI falla con un error de schema porque el `system` de Anthropic no es compatible.
*Causa*: se asumió que los formatos son idénticos.
*Arreglo*: un adaptador que convierte el array de `system_blocks` de Anthropic al formato `role: system` de OpenAI.

**2. Usar el mismo modelo para todo, del secundario también.**
*Síntoma*: el fallback a `gpt-4o` cuesta 3× lo que costaría usar `gpt-4o-mini` para clasificaciones.
*Causa*: el fallback no distingue el tipo de tarea.
*Arreglo*: la política de fallback puede ser `gpt-4o` para tasks críticos y `gpt-4o-mini` para clasificaciones. LiteLLM permite policies por modelo-name.

**3. No testear el fallback en staging.**
*Síntoma*: el primer incidente real revela que las API keys del secundario están expiradas.
*Causa*: el failover nunca se activó intencionalmente.
*Arreglo*: test mensual del fallback: desactivar manualmente Anthropic y verificar que el sistema opera correctamente con el secundario.

**4. Caching desactivado cuando el fallback está activo.**
*Síntoma*: el costo se dispara cuando hay un incidente de Anthropic porque todas las llamadas al secundario pagan tokens completos.
*Causa*: el cache de Anthropic no se transfiere al secundario.
*Arreglo*: es un trade-off conocido. Minimizarlo: mantener el TTL del cache de Anthropic alto (1h) para que los hits normales se beneficien; aceptar el costo extra durante el failover emergencial (que es raro).

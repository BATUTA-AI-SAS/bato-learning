## Hilo conductor

En D04 conectaste Phoenix y un ledger de costo que registra `tokens_in`, `tokens_out` y `cost_usd` por mensaje. El `anthropic_client.py` de la app calcula ese costo con `_estimate_cost`. H04 toma esos datos y construye el sistema FinOps completo: presupuesto por tenant, alertas en 80 % y hard stop en 100 %, tabla de técnicas de optimización con números reales. H05 usará estos números para el plan de escalado.

## Idea central

**FinOps** aplicado a agentes IA multitenant significa: saber exactamente cuánto cuesta cada tenant, en tiempo casi real, y actuar antes de que el costo supere el margen. El anti-patrón es «el costo total del mes fue X», que llega cuando ya no puedes actuar. El patrón correcto es «el tenant `globex` lleva el 78 % de su presupuesto a 15 días del mes; activa la alerta». El costo del servidor (Hetzner, ~5-10 EUR/mes) es ruido comparado con el costo de los tokens LLM en un servicio activo.

## Por qué importa

Un tenant que usa el agente de auditoría intensivamente puede generar 50-200 USD/mes en tokens. Si tu precio de venta es 150 USD/mes y el costo de tokens es 120 USD/mes, estás perdiendo margen sin saberlo. La ficha **F-FIN-03** (cierre mensual asistido) y **F-CTA-02** (auditoría interna) identifican el modelo de pricing basado en costo real como uno de los tres factores que diferencian un SaaS rentable de uno que «no entiende por qué no gana dinero».

Sin un presupuesto hard stop por tenant, un solo cliente con un script de automatización mal configurado puede agotar 10 veces su presupuesto en un fin de semana.

## Cómo funciona por dentro

### El ledger de costo: base del sistema FinOps

El `anthropic_client.py` ya calcula el costo por llamada con precios de `claude-sonnet-4-6`:

```python
_PRICE_PER_M = {
    "input": 3.00,      # USD por millón de tokens de entrada
    "output": 15.00,    # USD por millón de tokens de salida
    "cache_read": 0.30,  # USD por millón de tokens leídos de cache
    "cache_write": 3.75, # USD por millón de tokens escritos a cache
}
```

Ese costo se debe persistir por `tenant_id` y `period` (YYYY-MM) en una tabla de ledger:

```sql
-- Tabla de ledger de costo (en app/models/cost_ledger.py)
CREATE TABLE cost_ledger (
    id          INTEGER PRIMARY KEY,
    tenant_id   TEXT NOT NULL,
    period      TEXT NOT NULL,    -- "2026-05"
    tokens_in   INTEGER DEFAULT 0,
    tokens_out  INTEGER DEFAULT 0,
    cache_read  INTEGER DEFAULT 0,
    cache_write INTEGER DEFAULT 0,
    cost_usd    REAL DEFAULT 0.0,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE UNIQUE INDEX idx_cost_tenant_period ON cost_ledger(tenant_id, period);
```

### Presupuesto por tenant y alertas

Cada tenant tiene un `budget_usd_month` configurado. El sistema compara el acumulado contra ese presupuesto:

```python
BUDGET_ALERT_THRESHOLD = 0.80   # alerta al 80%
BUDGET_HARD_STOP = 1.00         # hard stop al 100%

def check_budget(tenant_id: str, period: str) -> dict:
    """Devuelve estado del presupuesto del tenant para el período."""
    usage = get_ledger(tenant_id, period)          # costo acumulado
    budget = get_tenant_budget(tenant_id)          # presupuesto configurado
    ratio = usage.cost_usd / budget if budget > 0 else 0.0
    return {
        "tenant_id": tenant_id,
        "period": period,
        "cost_usd": usage.cost_usd,
        "budget_usd": budget,
        "ratio": ratio,
        "status": (
            "over_budget" if ratio >= BUDGET_HARD_STOP
            else "warning" if ratio >= BUDGET_ALERT_THRESHOLD
            else "ok"
        ),
    }
```

El hard stop no bloquea el servicio; **degrada gracefully**: el agente responde con el modelo más barato (Haiku) o con una respuesta cacheada, y notifica al equipo. Bloquear el servicio completamente es un riesgo de SLA.

### Tabla de técnicas de optimización

| Técnica | Ahorro típico | Complejidad | Cuándo aplicar |
|---------|--------------|-------------|----------------|
| Prompt caching (TTL 1h) para system prompt + módulo | 60-80% de input tokens | Baja | Siempre; ya en `anthropic_client.py` |
| Truncar historial de sesión a N últimos turnos | 20-40% de input tokens | Baja | Cuando las sesiones superan 20 turnos |
| Batch API (50% descuento, async) | 50% de todos los tokens | Media | Para auditorías nocturnas, no chat en vivo |
| Switch a Haiku para clasificaciones simples | 80-90% vs Sonnet | Media | Cuando el task no requiere razonamiento complejo |
| Reutilizar output cacheado para misma query | 100% del costo | Alta | Para queries idénticas del mismo tenant |
| Compresión del contexto (resumir historial antiguo) | 30-50% de input tokens | Alta | Cuando el historial supera 100 turnos |
| Embeddings pre-calculados (no regenerar por chunk) | 70-90% del costo de embeddings | Media | Cuando se vectoriza el mismo documento varias veces |

> [!cuidado]
> **No optimices antes de medir.** El anti-patrón clásico es implementar toda la tabla antes de tener datos. El procedimiento correcto: medir 1 semana en producción, identificar el top-3 de consumo de tokens por tenant, atacar el ítem #1. Optimizar lo que no es el cuello de botella no mejora el margen.

### Costo por modelo (referencia 2026)

| Modelo | Input (USD/M) | Output (USD/M) | Cache read (USD/M) | Caso de uso |
|--------|-------------|----------------|-------------------|-------------|
| claude-haiku-4 | 0.80 | 4.00 | 0.08 | Clasificación, FAQ, responses cortos |
| claude-sonnet-4-6 | 3.00 | 15.00 | 0.30 | Agente de auditoría, análisis complejo |
| claude-opus-4-7 | 15.00 | 75.00 | 1.50 | Análisis profundo, diff complejo |

Para comparar: una auditoría mensual de ACME que procesa 200 facturas con `claude-sonnet-4-6` consume aproximadamente:
- Input: ~150.000 tokens × 0,003 USD = 0,45 USD
- Output: ~20.000 tokens × 0,015 USD = 0,30 USD
- Total por auditoría: ~0,75 USD

Con caching activo (60% de ahorro en input): ~0,57 USD por auditoría. A 4 auditorías/mes: ~2,28 USD/mes de LLM para ACME.

### Dashboard de costo por tenant

```python
# Consulta que alimenta el dashboard en /admin/finops (acceso solo para el lector)
async def finops_dashboard(period: str) -> list[dict]:
    rows = await session.execute(
        select(CostLedger)
        .where(CostLedger.period == period)
        .order_by(CostLedger.cost_usd.desc())
    )
    tenants = rows.scalars().all()
    return [
        {
            "tenant_id": t.tenant_id,
            "cost_usd": round(t.cost_usd, 4),
            "budget_usd": get_tenant_budget(t.tenant_id),
            "ratio": round(t.cost_usd / get_tenant_budget(t.tenant_id), 2),
            "status": check_budget(t.tenant_id, period)["status"],
        }
        for t in tenants
    ]
```

### Herramientas FinOps externas: cuándo y cuáles

Para un SaaS en Hetzner con <25 tenants, **no necesitas Vantage ni CloudZero**. Son herramientas para AWS/GCP con miles de recursos. Tu FinOps es el ledger en tu propia DB más BetterStack para las alertas.

A partir de ~100 tenants en infraestructura cloud (AWS/GCP) con múltiples servicios: CloudZero ofrece cost-per-tenant con integración a billing de cloud. Vantage es adecuado para equipos que quieren dashboards rápidos pero no multi-tenant granular. Ambos cuestan 10-25% del cloud bill; no los uses antes de que el cloud bill justifique el gasto.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Acumular tokens y calcular costo USD por ledger | determinístico | Aritmética con precios fijos; no requiere razonamiento. |
| Comparar acumulado vs presupuesto y emitir status | determinístico | Regla cerrada: ratio ≥ 0.80 = warning. |
| Elegir qué técnica de optimización aplicar primero | agéntico | Requiere analizar el patrón de uso del tenant; no hay regla universal. |
| Diseñar el presupuesto por tenant (cuánto asignar) | agéntico | Depende del valor del contrato, el historial de uso, el margen objetivo. |
| Decidir si degradar a Haiku o respuesta cacheada ante hard stop | agéntico | Depende del tipo de request y del SLA del cliente específico. |

## Errores típicos

**1. El costo se calcula en batch mensual, no en tiempo real.**
*Síntoma*: el equipo descubre a fin de mes que el tenant `globex` consumió 3× su presupuesto.
*Causa*: el ledger se actualiza solo cuando corre el job nocturno.
*Arreglo*: actualizar el ledger en cada llamada al LLM, en el mismo handler de streaming.

**2. Hard stop bloquea el servicio completamente.**
*Síntoma*: el cliente llama furioso porque su servicio dejó de funcionar a mitad del día.
*Causa*: el hard stop lanzaba una excepción sin degradar gracefully.
*Arreglo*: el hard stop activa el modo degradado (Haiku + respuestas cacheadas), no un 503.

**3. Presupuesto fijo para todos los tenants.**
*Síntoma*: el tenant pequeño (5 USD/mes de costo real) tiene el mismo límite que el grande (80 USD/mes).
*Causa*: el presupuesto no está configurado por tenant.
*Arreglo*: cada tenant tiene `budget_usd_month` en su registro. El default puede ser 50 USD, pero debe ser configurable.

**4. Optimizar antes de medir.**
*Síntoma*: se implementó compresión de contexto compleja antes de saber que el costo era bajo.
*Causa*: optimización prematura.
*Arreglo*: ejecutar el ledger una semana, ver el top-3 de costo, decidir.

## Hilo conductor

En D04 conectaste Phoenix para trazar cada llamada. H03 cierra el ciclo: ahora que el agente está en producción, ¿cómo sabes que sigue funcionando bien? No «¿el servidor responde?» —eso lo cubre H01—, sino «¿el agente produce la calidad que prometiste?»

## Idea central

El **drift** de un agente es la degradación de su comportamiento útil sin cambio aparente en el código. Sus fuentes son tres: (1) el modelo proveedor cambia silenciosamente (versión, fine-tune, temperatura por defecto); (2) los datos del cliente cambian (nuevos formatos de factura, nuevos proveedores); (3) el prompt envejece (lo que funcionaba con 10 clientes no funciona con 100). El sistema de detección son los **golden sets**: ejemplos versionados de input → output esperado que se ejecutan contra el agente en producción y registran la tasa de coincidencia.

## Por qué importa

Anthropic actualiza los modelos. Un `claude-sonnet-4-6` de hoy no es idéntico al de hace seis meses. Los cambios son silenciosos: el modelo no te avisa. El CFO de ACME tampoco te avisa cuando empieza a ver hallazgos raros en el reporte —simplemente deja de confiar en el agente y no renueva. La ficha **F-CTA-02** (auditoría interna) y **F-FIN-01** (conciliación bancaria) listan la «degradación silenciosa» como el riesgo operativo número uno para servicios de agentes IA en producción PYME.

La solución no es monitoreo de latencia ni de uptime: es monitoreo de calidad. Eso se mide contra ejemplos reales con respuesta esperada conocida.

## Cómo funciona por dentro

### Golden set: qué es y cómo se construye

Un **golden set** es una colección de 5–50 pares `(input, expected_output)` extraídos de conversaciones reales o construidos a mano. Cada par cubre un caso que el agente debe manejar correctamente. Los criterios de calidad del golden set:

- **Cobertura**: incluye casos normales, casos límite y casos que históricamente fallaron.
- **Versionado**: cada cambio al agente o al prompt genera una nueva ejecución contra el mismo golden set. Los resultados se almacenan con timestamp y versión del prompt.
- **Estabilidad**: los ejemplos se eligen para que la respuesta esperada no cambie si el agente funciona correctamente. No uses ejemplos que dependan de la fecha actual.
- **Tamaño mínimo**: 5 ejemplos es suficiente para empezar. 20-50 para un agente en producción con SLA.

```python
# Estructura de un golden set para el agente de auditoría de ACME
GOLDEN_SET_AUDIT_V3 = [
    {
        "id": "dup_exact",
        "input": {
            "tenant": "acme",
            "invoices": [
                {"id": "INV-001", "vendor": "Proveedor SA", "amount": 4820.0, "date": "2026-04-03"},
                {"id": "INV-002", "vendor": "Proveedor SA", "amount": 4820.0, "date": "2026-04-03"},
            ],
        },
        "expected": {
            "finding_type": "duplicate",
            "confidence": "high",
            "invoice_ids": ["INV-001", "INV-002"],
        },
    },
    {
        "id": "new_vendor",
        "input": {
            "tenant": "acme",
            "invoices": [
                {"id": "INV-010", "vendor": "Proveedor Nuevo SRL", "amount": 1200.0, "date": "2026-04-15"},
            ],
        },
        "expected": {
            "finding_type": "new_vendor",
            "confidence": "medium",
        },
    },
    # ... más casos
]
```

### Evaluación: tres niveles

**Nivel 1 — Comparación exacta** (determinístico, barato): cuando el output esperado es un valor discreto (`finding_type`, `confidence`, lista de IDs), se compara campo a campo. Falla si el agente produce un campo diferente. Útil para clasificaciones.

**Nivel 2 — LLM-as-judge** (agéntico, más caro): cuando el output es texto libre (el resumen del hallazgo, la redacción del reporte), se usa un LLM evaluador para juzgar si la respuesta del agente captura los elementos clave del expected. Se define una rúbrica explícita.

```python
JUDGE_PROMPT = """
Eres un evaluador de calidad de agente de auditoría financiera.

Rúbrica (evalúa cada punto: 0 = no cumple, 1 = cumple parcialmente, 2 = cumple):
1. El hallazgo identifica el tipo correcto ({expected_type}).
2. El hallazgo menciona las facturas involucradas (ids: {expected_ids}).
3. El hallazgo tiene un nivel de confianza explícito.
4. El hallazgo NO inventa datos que no están en el input.

Responde solo con JSON: {{"scores": [0|1|2, 0|1|2, 0|1|2, 0|1|2], "total": 0-8, "pass": true|false}}
El umbral de aprobación es total >= 6.

Input del agente: {agent_input}
Output del agente: {agent_output}
Output esperado: {expected_output}
"""
```

> [!cuidado]
> LLM-as-judge tiene sesgo hacia sus propias respuestas. Si evalúas `claude-sonnet-4-6` con `claude-sonnet-4-6`, el evaluador tiende a aprobar respuestas que el modelo produciría. Mitiga usando una versión distinta del modelo (Opus para evaluar Sonnet) o evaluadores humanos para un subconjunto.

**Nivel 3 — Human spot-check** (el más caro, el más preciso): mensualmente, revisar 5-10 casos reales recientes del cliente. No se puede automatizar completamente. Es la calibración que mantiene al LLM-as-judge honesto.

### A/B entre versiones del agente

Cuando cambias el prompt, el modelo, o una tool, no promuevas directamente a producción. El patrón es:

```
Rama A (actual, producción) → 80% del tráfico
Rama B (candidato, nueva versión) → 20% del tráfico
Métrica declarada antes de arrancar: tasa de aprobación en golden set ≥ 95%
Duración: 48-72h o 100 conversaciones (lo que llegue primero)
Decisión: si Rama B ≥ A en la métrica → promover. Si B < A → revertir.
```

La métrica se declara **antes** de arrancar el A/B. Declarar la métrica después, cuando ya ves los resultados, es p-hacking. Si el A/B no tiene una métrica pre-declarada, no es un experimento: es una suposición.

### Integración con Phoenix

Phoenix registra cada span de agente con input, output, tokens y latencia. Esos datos son la materia prima para reconstruir golden sets históricos: toma los spans de las últimas 4 semanas, selecciona los casos que el agente manejó correctamente (según feedback del cliente o revisión humana), y conviértelos en el nuevo golden set.

```python
# Consulta Phoenix para extraer spans del agente de auditoría de acme
# en las últimas 2 semanas como candidatos para golden set
from phoenix.client import Client as PhoenixClient

def extract_golden_candidates(tenant: str, days: int = 14) -> list[dict]:
    client = PhoenixClient()
    spans = client.get_spans(
        filter=f"attributes['tenant_id'] == '{tenant}' and name == 'audit_agent'",
        limit=100,
    )
    # Filtrar por latencia razonable (no timeouts) y output no vacío
    return [
        {"input": s.input, "output": s.output, "span_id": s.id}
        for s in spans
        if s.status == "OK" and s.output
    ]
```

### Frecuencia recomendada

| Evento | Acción |
|--------|--------|
| Cada cambio a prompt o modelo | Ejecutar golden set completo antes de promover |
| Cada semana en producción | Ejecutar golden set automáticamente en CI |
| Cada mes | A/B test si hay una versión candidata |
| Cada mes | Human spot-check de 5-10 casos reales |
| Cada trimestre | Revisar y actualizar el golden set con nuevos casos del cliente |

### Métricas de drift a monitorear

```
pass_rate          = casos_aprobados / total_casos  (umbral típico: ≥ 0.90)
regression_rate    = casos_que_antes_pasaban_y_ahora_fallan / total_casos
new_failure_rate   = casos_nuevos_que_fallan / total_casos
latency_p95        = percentil 95 de tiempo de respuesta (Phoenix)
cost_per_run       = costo USD por ejecución del golden set completo
```

Si `pass_rate` cae más de 5 puntos porcentuales entre dos semanas consecutivas: abrir investigación. Si `regression_rate > 0.10`: revertir el último cambio.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Ejecutar el golden set y comparar campos discretos | determinístico | Comparación exacta; no requiere interpretación. |
| Calcular pass_rate y regression_rate | determinístico | Aritmética sobre resultados. |
| Evaluar calidad de texto libre con LLM-as-judge | agéntico | El texto esperado tiene múltiples formas válidas; una regla no puede cubrirlas. |
| Decidir si un A/B merece promover o revertir | agéntico | Requiere juicio sobre contexto del cliente y riesgo de cambio. |
| Seleccionar qué spans de Phoenix pasan a ser golden set | agéntico | Requiere criterio de calidad que va más allá del status OK. |

## Errores típicos

**1. Golden set desactualizado.**
*Síntoma*: el agente pasa el golden set al 100% pero el cliente reporta errores con los nuevos formatos de factura que empezó a usar en enero.
*Causa*: el golden set no se actualiza cuando cambian los datos del cliente.
*Arreglo*: revisión trimestral del golden set con el cliente o con nuevos spans de Phoenix.

**2. LLM-as-judge sin rúbrica explícita.**
*Síntoma*: el evaluador aprueba respuestas incorrectas porque no tiene criterio claro.
*Causa*: el prompt del juez pide «evalúa si la respuesta es buena» sin definir «bueno».
*Arreglo*: rúbrica de 3-5 criterios específicos, cada uno puntuado independientemente.

**3. A/B sin métrica pre-declarada.**
*Síntoma*: el equipo «siente» que la Rama B es mejor y la promueve.
*Causa*: la métrica se elige después de ver los resultados.
*Arreglo*: la métrica y el umbral van en el commit que crea la Rama B, antes de arrancar el experimento.

**4. Drift detectado pero sin acción.**
*Síntoma*: el `pass_rate` bajó a 0.82 hace tres semanas; nadie hizo nada.
*Causa*: la detección no tiene umbral de alerta que dispare un ticket o notificación.
*Arreglo*: añadir a BetterStack o Grafana OnCall una alerta cuando `pass_rate < 0.90` en la ejecución semanal.

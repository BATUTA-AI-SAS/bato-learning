## Hilo conductor

En D03 configuraste el backup automático de SQLite con `litestream` o el script `backup.sh` en Hetzner. H06 cierra el ciclo que D03 dejó abierto: no basta con tener el backup — tienes que demostrar que puedes restaurarlo. El restore drill mensual es la prueba de que el SLA de recuperación que prometiste al cliente es real. H07 añade el escenario donde el propio proveedor de LLM falla y hay que cambiar de modelo en producción.

## Idea central

Un **restore drill** es un ejercicio programado donde tomas un backup real de producción, lo restauras en una infraestructura paralela, verificas la integridad de los datos y mides cuánto tardaste. Los dos indicadores que genera son **RTO** (Recovery Time Objective: cuánto tiempo tardas en volver a operar) y **RPO** (Recovery Point Objective: cuántos datos perderías si el incidente ocurre ahora). Sin el drill, RTO y RPO son estimaciones. Con el drill, son datos medidos.

## Por qué importa

D03 enseña a generar backups. Pero el 60-70% de los equipos que tienen backups automáticos nunca los han restaurado. El primer intento de restauración suele revelar: el backup no incluía los volumes de datos; el script de restauración tiene una ruta hardcodeada al servidor de producción; el backup tiene permisos de lectura solo para el usuario que lo generó. La ficha **F-CTA-02** (auditoría interna para ACME) declara que el criterio de «listo para producción» incluye explícitamente un restore drill exitoso con tiempo documentado.

Para PYME LATAM, el RTO razonable es **4 horas** y el RPO razonable es **1 hora**. Eso no requiere replicación en tiempo real ni HA compleja. Requiere tener el drill hecho y el runbook de restauración escrito.

## Cómo funciona por dentro

### RTO y RPO: definiciones operativas

**RPO (Recovery Point Objective)**: cuántos datos puedes perder. Si el backup se genera cada 1h y el incidente ocurre 55 minutos después del último backup, perderás 55 minutos de datos. RPO = 1h significa que eso es aceptable.

**RTO (Recovery Time Objective)**: cuánto tiempo puedes estar caído. RTO = 4h significa que desde el momento del incidente hasta que el servicio vuelve a funcionar no pueden pasar más de 4 horas.

Para BATUTA (SaaS PYME):
- **RPO = 1h**: backup cada hora con litestream o cron. Aceptable para un servicio de auditoría donde los datos del cliente son facturas del ERP (se pueden re-ingestar si el worst case ocurre).
- **RTO = 4h**: razonable para un equipo de 1-3 personas con el runbook de restauración. Si el cliente exige RTO < 1h, se necesita HA activo-pasivo con failover automático (Etapa 3+).

### Script de verificación de integridad

```python
import sqlite3
from datetime import datetime, timezone

def verify_restored_db(db_path: str, expected_tenants: list[str]) -> dict:
    """Verifica la integridad básica de una DB restaurada."""
    conn = sqlite3.connect(db_path)
    results = {}
    try:
        # Tabla de módulos
        count = conn.execute("SELECT COUNT(*) FROM modules").fetchone()[0]
        results["modules_count"] = count
        results["modules_ok"] = count > 0

        # Última visita registrada
        last_visit = conn.execute(
            "SELECT MAX(created_at) FROM module_visits"
        ).fetchone()[0]
        results["last_visit_ts"] = last_visit

        # Tenants con datos en chat
        tenants_in_db = {
            row[0]
            for row in conn.execute(
                "SELECT DISTINCT tenant_id FROM chat_messages"
            ).fetchall()
        }
        missing_tenants = set(expected_tenants) - tenants_in_db
        results["missing_tenants"] = list(missing_tenants)
        results["tenants_ok"] = len(missing_tenants) == 0

        results["overall_ok"] = (
            results["modules_ok"]
            and results["tenants_ok"]
        )
    finally:
        conn.close()
    return results
```

### Estrategias de backup por etapa

| Etapa | Estrategia | RPO alcanzable | Costo adicional |
|-------|-----------|----------------|-----------------|
| 1 (SQLite) | Cron cada 1h + `sqlite3 .backup` a volumen Hetzner | ~1h | €0 |
| 1 (SQLite) | Litestream → S3-compatible (Hetzner Object Storage) | ~5 min | ~€1-2/mes |
| 2 (Postgres) | `pg_dump` cada 1h → Hetzner Storage | ~1h | €0 |
| 2 (Postgres) | WAL archiving + PITR (Point-in-Time Recovery) | ~1 min | €3-5/mes |
| 3+ (Postgres) | Streaming replica + WAL archiving | Segundos | €10-20/mes |

Para PYME en Etapa 1-2: litestream a Hetzner Object Storage (€5.80/TB/mes, el primero 500 GB incluidos en algunos planes) es la mejor relación costo/RPO. Combinar con un drill mensual da la tranquilidad suficiente.

### Automatización del drill

Para automatizar el drill mensual como un workflow Temporal:

```python
@workflow.defn
class RestoreDrillWorkflow:
    """Drill mensual de restauración — workflow determinístico."""

    @workflow.run
    async def run(self, tenant_id: str, period: str) -> dict:
        # Paso 1: tomar el backup más reciente (activity — IO real)
        backup_path = await workflow.execute_activity(
            fetch_latest_backup,
            args=[tenant_id],
            start_to_close_timeout=timedelta(minutes=10),
        )
        # Paso 2: restaurar en VM temporal (activity — IO real)
        drill_db_path = await workflow.execute_activity(
            restore_backup_to_temp,
            args=[backup_path],
            start_to_close_timeout=timedelta(minutes=30),
        )
        # Paso 3: verificar integridad (activity — DB queries)
        result = await workflow.execute_activity(
            verify_restored_db_activity,
            args=[drill_db_path, ["acme", "globex"]],
            start_to_close_timeout=timedelta(minutes=5),
        )
        # Paso 4: destruir VM temporal (activity — IO real)
        await workflow.execute_activity(
            cleanup_temp_vm,
            args=[drill_db_path],
            start_to_close_timeout=timedelta(minutes=5),
        )
        return result
```

> [!nota]
> El drill automatizado no reemplaza el drill manual. El drill manual anual (donde una persona sigue el runbook sin apoyo de scripts) verifica que el runbook funciona si el sistema de automatización también falla.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Ejecutar el checksum del backup y comparar | determinístico | Comparación binaria; resultado es igual/diferente. |
| Ejecutar queries de sanity check y comparar counts | determinístico | Aritmética sobre datos; regla cerrada. |
| Medir el RTO real del drill y comparar con objetivo | determinístico | Tiempo medido vs número declarado. |
| Decidir si el resultado del drill requiere acción correctiva | agéntico | Una discrepancia pequeña en counts puede ser normal o crítica según contexto. |
| Diseñar el plan de mejora del RTO si supera el objetivo | agéntico | Depende de qué paso fue lento y del stack de la etapa actual. |

## Errores típicos

**1. El backup existe pero no se ha restaurado nunca.**
*Síntoma*: el primer intento de restauración falla con «ruta no encontrada» o «permisos incorrectos».
*Causa*: el script de backup nunca fue probado en restauración.
*Arreglo*: el primer drill, inmediatamente después de configurar el backup. No esperar a necesitarlo.

**2. El drill se hace en la misma máquina que el backup.**
*Síntoma*: el drill «funciona» pero si el servidor muere, el procedimiento no funciona.
*Causa*: el drill no simula el escenario real de pérdida del servidor.
*Arreglo*: siempre restaurar en una VM diferente, aunque sea temporal y de bajo costo.

**3. RTO y RPO declarados sin medirlos.**
*Síntoma*: el contrato dice «RTO 4h» pero el primer drill tarda 6 horas.
*Causa*: los números fueron estimados sin ejecutar el drill.
*Arreglo*: el RTO del contrato debe venir del drill medido, no de una estimación optimista.

**4. No registrar los resultados del drill.**
*Síntoma*: después de 6 meses, no sabes si el RTO mejoró o empeoró.
*Causa*: el drill se hace pero no se documenta.
*Arreglo*: la tabla de log de drills es obligatoria. Sin registro, el drill no cuenta.

## Hilo conductor

En D03 desplegaste la app en Hetzner con Traefik y TLS. En D04 añadiste trazas Phoenix y un ledger de costo. Pero ambos módulos asumen que, cuando algo falla, sabes qué hacer. H01 cierra ese hueco: define cómo actúas ante un incidente —qué comandos corres, en qué orden, con qué criterio escalar— antes de que el incidente suceda. H02 continúa con el postmortem de lo que el runbook no pudo resolver.

## Idea central

Un **runbook** es un documento ejecutable que convierte un síntoma en acción sin exigirle razonamiento al ingeniero de turno a las 3am. No es prosa de intenciones; es una lista de comandos con output esperado, ramificaciones concretas y un criterio de escalación medible. El **sistema on-call** es la capa organizativa: quién responde qué, en qué tiempo, con qué severidad, usando qué herramienta de alertas. Sin runbooks el on-call es heroísmo. Con runbooks es proceso.

## Por qué importa

El primer incidente de producción sin runbook produce dos daños simultáneos: el cliente pierde disponibilidad y el ingeniero de turno improvisa bajo presión, lo que alarga el tiempo de resolución (MTTR) y genera acciones incorrectas que escalan el incidente. La ficha **F-CX-04** (gestión de soporte multitenant) registra que el 60 % de los MTTR elevados en SaaS PYME vienen de no tener el comando correcto a mano, no de falta de conocimiento.

Sin severidades definidas, toda alerta es SEV1. Con severidades definidas, el equipo duerme ante un SEV3 y actúa en 15 minutos ante un SEV1. La diferencia no es técnica: es diseño del sistema.

## Cómo funciona por dentro

### Estructura canónica de runbook

Un runbook cubre exactamente un síntoma. No es "problemas con la app": es "agente devuelve 503 intermitente". La estructura de 6 bloques es fija:

```
1. SÍNTOMA         — qué alerta disparó o qué reportó el cliente
2. HIPÓTESIS       — causas ordenadas por probabilidad (las más frecuentes primero)
3. DIAGNÓSTICO     — comandos con output esperado para confirmar o descartar
4. ACCIÓN          — qué cambiar, con rollback explícito por cada acción
5. VERIFICACIÓN    — cómo confirmas que resolviste
6. ESCALACIÓN      — cuándo y a quién, con qué información
```

Cada bloque tiene comandos copiables. No tiene prosa entre los comandos salvo una línea de contexto. La razón: el ingeniero de turno copia, ejecuta, lee el output, sigue la rama correcta.

### Anti-patrón: runbook como documento de intenciones

```markdown
<!-- MAL: prosa que describe qué hacer sin mostrar cómo -->
Si el agente falla, verifica los logs y reinicia si es necesario.
Comprueba la conexión a la base de datos y el estado de los workers de Temporal.

<!-- BIEN: comandos copiables con output esperado -->
## Diagnóstico rápido (< 2 min)

# ¿El contenedor está corriendo?
docker ps --filter name=api --format "{{.Status}}"
# esperado: "Up N minutes (healthy)"
# si está "Exited": ir a → Bloque 4 reinicio de contenedor

# ¿Hay errores 5xx en los últimos 5 min?
docker logs api --since 5m 2>&1 | grep -c "ERROR\|500\|503"
# > 0: confirma hipótesis 1 → continuar diagnóstico
# = 0: descartar hipótesis 1 → saltar a hipótesis 2
```

### Severidades SEV1/SEV2/SEV3

| SEV | Definición | Tiempo objetivo de respuesta | Quién responde |
|-----|-----------|------------------------------|----------------|
| SEV1 | Servicio caído para todos los tenants | 15 min (24/7) | On-call primario |
| SEV2 | Servicio degradado o caído para ≥1 tenant | 30 min (horario) / 1h (fuera) | On-call primario |
| SEV3 | Error funcional sin impacto en disponibilidad | 4h (próximo día hábil) | Equipo de turno |

El criterio para escalar de SEV a SEV: si llevas 20 min sin progreso visible, escala. No es señal de debilidad; es parte del runbook.

### Herramientas: criterio de elección para PYME

**BetterStack** (antes Logtail + Uptime): plan gratuito hasta 5 monitores y 3 integraciones de alertas (email, Slack, SMS), pay-as-you-go después. Es la opción default para un equipo de 1-3 personas que empieza. Integra con Grafana.

**Grafana OnCall Cloud**: plan gratuito disponible (la versión OSS entró en modo mantenimiento en marzo 2025 y fue archivada en marzo 2026; usar la versión Cloud). On-call scheduling, escalation policies, integración nativa con Grafana Alerts.

**PagerDuty**: desde 21 USD/usuario/mes. Justificado solo cuando el equipo supera 5 personas on-call o el cliente exige SLA contractual con auditoría de incidentes. No es la elección para PYME.

**Opsgenie** (Atlassian): detenió ventas nuevas en junio 2025; cierre programado para abril 2027. No lo elijas para proyectos nuevos.

La decisión para BATUTA en etapa 1-2 (1-25 clientes): BetterStack free → BetterStack paid cuando la factura mensual lo justifique. Migrar a PagerDuty solo con un SLA firmado que exija respuesta auditada.

### Rotación on-call mínima viable

Para un equipo de 2 personas:

```
Semana A: Persona 1 primaria, Persona 2 backup (escalación a 20 min)
Semana B: rotación
Alertas fuera de horario: solo SEV1 y SEV2 despiertan
SEV3 fuera de horario: va a la cola del día hábil siguiente
```

Para un equipo de 1 persona (fundador solo):
- Define qué es SEV1 de forma muy conservadora: solo "cero tenants pueden entrar".
- Configura tiempo de silencio nocturno (23-7h local) para SEV3.
- El primer cliente debe conocer el SLA real: «disponibilidad 99 % con respuesta en 2h para incidentes críticos».

### Runbook mínimo para BATUTA: "agente devuelve 503 intermitente"

```markdown
# RB-001: agente devuelve 503 intermitente

## Síntoma
Alerta BetterStack: "HTTP 503 en /api/chat" — o cliente reporta "el chat no responde".

## Hipótesis (orden de probabilidad)
1. El contenedor `api` se reinició y está en health-check
2. Anthropic API devolvió 529 (overloaded) — ver H07 multi-model failover
3. La DB SQLite está bloqueada por un lock de backup (ver D03)
4. Traefik no encuentra el servicio (red de compose rota)

## Diagnóstico

# H1: estado del contenedor
docker ps --filter name=api --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
# Si "starting" o "unhealthy": esperar 30s y repetir. Si persiste → Acción H1.

# H2: errores del LLM en los últimos 10 min
docker logs api --since 10m 2>&1 | grep -E "529|overloaded|rate_limit" | wc -l
# > 0: ir a runbook H07-fallback

# H3: lock de backup
docker exec api flock --nonblock /data/app.db echo "libre" 2>&1
# "libre": OK. Error "lock": Acción H3.

# H4: red de compose
docker network inspect bato-learning_default | python3 -c \
  "import sys,json; nets=json.load(sys.stdin); print([c['Name'] for c in nets[0]['Containers'].values()])"
# Debe listar "api" y "phoenix"

## Acción H1: reinicio controlado
docker compose restart api
# Esperar 30s. Verificar con curl -sf http://localhost:8000/health | python3 -m json.tool

## Acción H3: liberar lock
# El proceso de backup se cuelga; matar y limpiar:
docker exec api pkill -f "sqlite3.*backup" || true
sleep 5
docker compose restart api

## Rollback general
docker compose down && docker compose up -d
# Si persiste: git stash; git checkout <último tag estable>; docker compose up -d

## Verificación
curl -sf https://{DOMAIN}/api/health | python3 -c "import sys,json; d=json.load(sys.stdin); print('OK' if d['status']=='ok' else 'FALLO')"

## Escalación
Si llevas 20 min sin resolver: notificar a {backup_person} con:
- El síntoma exacto
- Las hipótesis descartadas
- El último output de docker logs api --tail 50
```

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Ejecutar comandos de diagnóstico del runbook | determinístico | Secuencia fija; el output decide la rama. |
| Clasificar la severidad del incidente según tabla SEV | determinístico | Regla cerrada: tenant afectado + tiempo + impacto. |
| Decidir si escalar antes de cumplir los 20 min | agéntico | Requiere juicio de contexto (¿el cliente VIP está mirando?). |
| Redactar la comunicación al cliente durante el incidente | agéntico | Salida para humano; varía por gravedad y audiencia. |
| Decidir si el incidente merece postmortem (SEV3 frontera) | agéntico | El criterio SEV3 no cubre todos los casos límite. |

## Errores típicos

**1. Runbook escrito como prosa narrativa.**
*Síntoma*: el ingeniero de turno pasa 10 minutos leyendo para saber qué ejecutar.
*Causa*: el autor escribió lo que haría, no cómo hacerlo.
*Arreglo*: reescribir con comandos copiables. Cada párrafo de prosa → un bloque de código con comentario de una línea.

**2. Todas las alertas son SEV1.**
*Síntoma*: el equipo tiene alarm fatigue; ignora alertas.
*Causa*: no hay criterio de severidad definido.
*Arreglo*: aplicar la tabla SEV con criterio de impacto en tenants. Si "el log de errors tiene 1 warning" no es SEV1, no lo mandes a las 3am.

**3. Runbook sin rollback.**
*Síntoma*: una acción correctiva empeora el incidente.
*Causa*: el runbook solo dice qué hacer, no cómo deshacer.
*Arreglo*: cada acción tiene un rollback explícito dentro del mismo bloque. La línea `## Rollback` es obligatoria.

**4. Herramienta de alertas configurada pero nunca probada.**
*Síntoma*: el primer incidente real no genera notificación.
*Causa*: el webhook de BetterStack apunta a una URL vieja o el canal de Slack fue renombrado.
*Arreglo*: test de alerta cada 30 días. BetterStack tiene un botón "Send test alert" — presiónalo en el drill mensual.

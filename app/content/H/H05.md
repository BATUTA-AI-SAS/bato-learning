## Hilo conductor

En D03 desplegaste la app en Hetzner CX22 con Docker Compose. En D07 estudiaste estrategias de deploy blue-green y canary. En H04 calculaste el costo por tenant con el ledger. H05 junta todo eso y responde la pregunta que ningún tutorial responde: ¿cuándo pasar de un servidor a dos? ¿Cuándo separar la DB? ¿Cuándo necesitas Kubernetes? La respuesta tiene números.

## Idea central

El **escalado prematuro** es el error número uno de sistemas técnicamente correctos que quiebran operativamente. Sobre-engineerizas a 5 clientes y tienes un sistema de 1.500 USD/mes para una factura de 750 USD. El camino correcto es el contrario: empezar con la infraestructura mínima que puede crecer, medir el umbral de saturación, y escalar solo cuando lo tocas. Las **cuatro etapas** de este módulo no son teóricas: cada una tiene números reales de Hetzner 2026 y un criterio medible de «es hora de pasar a la siguiente».

## Por qué importa

Un SaaS PYME LATAM tiene márgenes ajustados. Si infraestructura + LLM consume más del 30-40% del revenue, el negocio no es viable. La ficha **F-CTA-02** y los datos de **F-FIN-03** muestran que los primeros 10 clientes deben poder operar con < 100 USD/mes de infraestructura, porque el precio de venta está entre 150-500 USD/mes y el costo de adquisición (CAC) es alto. El escalado tiene que crecer el margen, no destruirlo.

El módulo **G05 (saas-metrics-for-founders)** detalla cómo calcular si el unit economics es positivo; este módulo da los números de infraestructura que alimentan ese cálculo.

## Cómo funciona por dentro

### Las cuatro etapas con números reales

#### Etapa 1: 1-5 clientes (el MVP en producción)

**Objetivo**: validar que alguien paga, no que la infraestructura escala.

```
Infraestructura:
  - 1 servidor: Hetzner CX22 (2 vCPU, 4 GB RAM)
    Precio 2026: €4.49/mes (~$4.90/mes)
  - SQLite en volumen local (respaldado con litestream o backup manual)
  - Docker Compose: 1 contenedor api + 1 contenedor phoenix
  - Traefik con TLS automático (Let's Encrypt)

Límite antes de saturar:
  - ~20-50 requests/min sostenidos
  - ~5 sesiones de chat concurrentes
  - DB SQLite hasta ~1 GB sin problema notable

Costo mensual típico:
  - Servidor: €4.49 (~$4.90)
  - LLM (5 tenants × $2/mes): ~$10
  - Dominio + DNS: ~$2
  - BetterStack free: $0
  - Total: ~$17-25/mes

Señal de "pasar a Etapa 2":
  - CPU > 70% sostenida más de 30 min/día
  - Latencia p95 > 3s para requests normales
  - El backup de SQLite supera los 500 MB
  - Tienes 5+ tenants activos con uso diario
```

> [!cuidado]
> No saltes a Etapa 2 por miedo a saturar. Satura primero; mide. Un servidor CX22 con un proceso FastAPI asíncrono y Anthropic SDK maneja mucho más de lo que imaginas para un SaaS en etapa early.

#### Etapa 2: 5-25 clientes (separar compute y storage)

**Objetivo**: eliminar el único punto de fallo de la DB y dar headroom de compute.

```
Infraestructura:
  - 1 servidor: Hetzner CX32 (4 vCPU, 8 GB RAM)
    Precio 2026: €6.80/mes (~$7.40/mes)
  - Postgres managed: Neon free tier (0.5 GB storage, auto-pause) → Neon paid (~$19/mes)
    O Postgres en propio Hetzner CX22 separado (€4.49/mes)
  - Docker Compose: api + worker (si tienes Temporal) + redis (si usas caching de sesiones)
  - BetterStack free → paid cuando el número de monitores exige

Límite antes de saturar:
  - ~100-200 requests/min
  - ~20-30 sesiones concurrentes
  - Storage ilimitado con Postgres managed

Costo mensual típico (con Neon paid):
  - Compute: €6.80 (~$7.40)
  - Postgres (Neon paid): ~$19
  - LLM (20 tenants × $3/mes): ~$60
  - BetterStack (si paid): ~$24
  - Total: ~$110-130/mes

Costo mensual típico (Postgres en Hetzner):
  - Compute: €6.80 + €4.49 = €11.29 (~$12.30)
  - LLM: ~$60
  - BetterStack free: $0
  - Total: ~$75-90/mes

Señal de "pasar a Etapa 3":
  - CPU > 70% en el CX32 sostenida
  - DB query p95 > 500ms
  - Tienes 20+ tenants activos con uso diario
  - Los tenants grandes están correlacionados con spikes de tráfico
```

#### Etapa 3: 25-100 clientes (replicación horizontal de API)

**Objetivo**: escalar el compute sin tocar la DB todavía; añadir caching para reducir costo de tokens.

```
Infraestructura:
  - 2-3 servidores API: Hetzner CX32 × 2-3 = €13.60-€20.40/mes
  - Load balancer: Hetzner LB11 = €5.39/mes
  - Postgres con replica de lectura: Hetzner managed Postgres o 2 VMs
    (€15-30/mes según config)
  - Redis (Hetzner managed o VM): €5-10/mes
  - Worker dedicado para Temporal: CX22 = €4.49/mes

Límite antes de saturar:
  - ~500-1000 requests/min (con 3 nodos API)
  - Read-heavy workloads manejados por replica
  - Redis reduce 40-60% de llamadas al LLM con caching de sesiones

Costo mensual típico:
  - Compute (API×2 + worker + LB): ~€25-35 (~$27-38)
  - DB (Postgres + replica): ~€25-35
  - Redis: ~€8-12
  - LLM (80 tenants × $3/mes avg): ~$240
  - BetterStack paid: ~$24
  - Total: ~$325-550/mes

Señal de "pasar a Etapa 4":
  - Necesitas deploys sin downtime para 80+ tenants activos
  - Multi-región exigida por un contrato grande
  - El equipo tiene más de 3 personas en operaciones
  - Revenue > $15.000/mes (justifica el overhead operativo de K8s)
```

#### Etapa 4: 100+ clientes (orquestación y multi-región)

**Objetivo**: disponibilidad contractual, aislamiento de tenants grandes, escalado automático por demanda.

```
Infraestructura:
  - k3s (Kubernetes ligero) en Hetzner: 3+ nodos CX32
  - KEDA para auto-scaling de workers basado en queue depth
  - Hetzner Cloud con multi-region (Nuremberg + Falkenstein + Helsinki)
  - Postgres con streaming replication o CockroachDB para multi-región
  - CDN para assets estáticos (Cloudflare free tier)
  - Vector DB dedicada (Qdrant, Weaviate) si el agente usa RAG intensivo

Costo mensual típico:
  - Compute k3s (3+ nodos): €30-60/mes
  - DB cluster: €40-80/mes
  - Extras (CDN, monitoring, vector DB): €20-40/mes
  - LLM (100 tenants × $4/mes avg): ~$400
  - Herramientas SaaS (Grafana Cloud, Sentry, etc.): ~$50-100
  - Total: ~$540-980/mes de infra + ~$400 LLM = ~$940-1.380/mes

Señal de "ya estás aquí":
  - Revenue > $20.000/mes (40+ tenants a $500/mes avg)
  - Tienes SLAs contractuales con penalizaciones
  - El equipo tiene un SRE dedicado o un contrato de soporte 24/7
```

| Request latency p95 | > 3s | > 1.5s | > 800ms |
| DB storage | > 500 MB | > 20 GB | > 100 GB |
| Revenue mensual | > $1.500 | > $5.000 | > $15.000 |

### Reglas para no quemar dinero

**1. Empezar manual antes de automatizar.**
El auto-scaling (KEDA, Hetzner autoscaler) tiene sentido cuando tienes patrones de tráfico predecibles con datos de 3+ meses. Antes de eso, escalar manualmente (añadir un nodo en 5 minutos con Hetzner) es más barato y menos propenso a fallos.

**2. Métricas antes de optimizar.**
No añadas Redis porque «puede ayudar». Añádelo cuando los logs muestran que el 40%+ de las requests hacen queries DB repetitivas a los mismos datos. Sin ese dato, es gasto sin retorno.

**3. Vertical antes que horizontal.**
Subir de CX22 a CX32 (€2.31/mes más) resuelve 80% de los problemas de capacidad en Etapa 1→2. Escalar horizontalmente añade complejidad de sesiones distribuidas, coherencia de cache y balanceo. Solo hazlo cuando lo vertical ya no alcanza.

**4. Auto-scale solo con datos históricos.**
KEDA funciona bien cuando sabes que los lunes a las 9am hay 3× el tráfico normal. Sin ese patrón documentado, el autoscaler sobre-provisiona o sub-provisiona. Los primeros 6 meses: manual scaling con schedule predictivo.

**5. Reservas/commitments solo con 6+ meses de tracción.**
Hetzner no tiene reservas como AWS, pero si usas AWS/GCP para algún componente: reservar instancias antes de tener 6 meses de datos es apostar, no planificar.

**6. El costo del LLM escala con los tenants, no con el servidor.**
En Etapa 1, el costo del servidor (€4.49) es el 20-30% del costo total. En Etapa 3, el servidor es el 10-15% y el LLM es el 60-70%. La optimización de tokens (H04) tiene más impacto que la optimización de infraestructura a partir de Etapa 2.

### Ejemplo: plan de escalado 5 → 50 tenants en 12 meses

```
Mes 1-3 (5 tenants): Etapa 1. CX22 + SQLite. Costo ~$20/mes.
  Foco: validar que los tenants usan el agente. Medir tokens/tenant/mes.

Mes 4-5 (10 tenants): Seguir en Etapa 1. El CX22 aguanta.
  Foco: optimizar el top-3 de costo de tokens (H04). Implementar ledger.

Mes 6 (15 tenants): Migrar a Etapa 2. CX32 + Postgres (Neon paid).
  Costo: ~$110/mes. Revenue estimado: 15 × $200 = $3.000/mes. Margen > 90%.
  Trigger: CPU CX22 > 70% en horas pico.

Mes 7-9 (25 tenants): Etapa 2. Agregar Redis para caching de sesiones.
  Costo: ~$130/mes. Revenue: $5.000/mes. Infra/revenue: 2.6%.

Mes 10-12 (40-50 tenants): Migrar a Etapa 3. 2 nodos CX32 + LB.
  Costo: ~$400/mes. Revenue: $8.000-10.000/mes. Infra/revenue: 4-5%.
  Trigger: latencia p95 > 1.5s en las 2 semanas previas.
```

### KEDA y auto-scaling: cuándo tiene sentido

**KEDA** (Kubernetes Event-Driven Autoscaling, proyecto CNCF graduated) escala workloads en Kubernetes basándose en eventos externos: profundidad de queue de Redis, mensajes en Kafka, jobs Temporal pendientes. Para BATUTA:

- **Caso de uso válido**: los workers de auditoría se disparan en batch nocturno. Con KEDA, se escalan de 0 a N workers cuando hay auditorías en queue y vuelven a 0 cuando terminan. Elimina el costo de tener workers corriendo 24/7.
- **Prerequisito**: necesitas k3s o Kubernetes. Eso está en Etapa 4.
- **Para Etapas 1-3**: un cron job en Temporal o un scheduler manual en Hetzner cumple el mismo efecto sin la complejidad.

### Fly.io como alternativa para etapas tempranas

Fly.io ofrece scale-to-zero para contenedores: la app se detiene cuando no hay requests y arranca en ~500ms cuando llega una. Para una app de aprendizaje con tráfico discontinuo, puede reducir el costo a casi cero en periodos de inactividad. No recomendado para el agente de producción (latencia de cold start en un servicio de auditoría es inaceptable), pero válido para entornos de staging o apps secundarias.

## Errores típicos

**1. Saltar directamente a Kubernetes.**
*Síntoma*: 5 clientes, costos de €200/mes, el equipo pasa el 50% del tiempo en ops.
*Causa*: sobre-engineering motivado por el miedo a «no escalar».
*Arreglo*: Compose es suficiente hasta Etapa 3. k3s tiene sentido a partir de Etapa 4 con un SRE dedicado.

**2. No medir el ratio infra/revenue.**
*Síntoma*: el producto crece pero el margen cae.
*Causa*: la infraestructura escala más rápido que el revenue.
*Arreglo*: el ratio infra/revenue es una métrica en el dashboard FinOps. Si supera el 20%, auditar.

**3. Postgres managed caro antes de tiempo.**
*Síntoma*: PlanetScale o Supabase Pro a $50/mes con 2 tenants.
*Causa*: usar la opción más conveniente sin comparar costo.
*Arreglo*: Neon free tier aguanta hasta 0.5 GB. Hetzner CX22 como Postgres self-managed es €4.49/mes con control total. Escoge managed cuando el tiempo de administración > ahorro.

**4. Auto-scaling sin datos de tráfico.**
*Síntoma*: el autoscaler sube a 5 nodos en horas valle porque un cron job emite eventos erráticos.
*Causa*: se configuró KEDA sin un threshold calibrado contra tráfico real.
*Arreglo*: 3 meses de datos de tráfico antes de configurar autoscaling. Mientras tanto, escalado manual con schedule.

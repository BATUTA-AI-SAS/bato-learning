## Hilo conductor

En E06 (Capstone) construiste un agente multitenant que pasa un golden set de 5 ejemplos. E07 generaliza eso: cómo armas ese golden set desde cero con datos reales del cliente, cómo lo versionas en el repo, qué métricas aplicas según el tipo de output, y cómo integras las evals en CI para que un PR no pueda mergear si la calidad del agente baja. H06 (si lo tomas) extiende esto a drift detection en producción viva. Este módulo cubre la base que H06 asume.

## Idea central

Un **golden set** es una colección de pares `(input, output_esperado)` representativos del comportamiento correcto del agente en producción. Es la única forma objetiva de saber si un cambio al prompt, al modelo o al código mejora o empeora el agente. Sin golden set, «funciona» es una opinión. Con golden set, «funciona» es un número.

Las **evals** son los tests que comparan la salida real del agente contra el output esperado del golden set, usando métricas apropiadas al tipo de output: exact match para clasificaciones cerradas, LLM-as-judge para texto libre, embedding similarity para respuestas parafraseadas, o validación de schema para outputs estructurados.

## Por qué importa

El caso canónico: el agente de clasificación de facturas de ACME funcionó bien en pruebas. Tres semanas después, el modelo base actualizó sus pesos o el prompt de sistema se modificó por «mejoras». El cliente reporta que el agente está clasificando facturas de proveedor nuevo como «duplicados». Sin golden set, el equipo no puede reproducir el caso, no puede medir el impacto del bug, y no puede verificar que el fix funcionó.

Con golden set y CI de evals, el PR que causó la regresión habría sido bloqueado automáticamente: la métrica de exactitud cayó de 94% a 71%, por debajo del umbral declarado del 85%.

La ficha **F-CTA-02 (auditoría interna: anomalías y duplicados)** — el ancla del Capstone — requiere evals para ser vendible: el contrato con el cliente incluye un SLA de precisión. Sin evidencia de evals, no hay SLA.

> [!cuidado]
> El golden set es el reflejo del criterio del cliente, no del tuyo. Los 5-50 ejemplos tienen que venir del cliente (o de datos reales aprobados por el cliente), no de ejemplos que tú inventas para que el agente pase. Un golden set construido con datos sintéticos da una falsa sensación de calidad.

## Cómo funciona por dentro

### Qué va en el golden set

Un golden set es una lista de casos. Cada caso tiene:

- **input**: lo que el agente recibe (texto, JSON, o dict con los campos del request).
- **expected_output**: la respuesta correcta según criterio del cliente.
- **metadata** (opcional): tenant, categoría, caso edge, fecha de creación.

Formato canónico en el repo (`.jsonl` — JSON Lines, un objeto por línea):

```jsonl
{"input": {"description": "PAGO FACT 2026-04 ACME PROVEEDOR NUEVO", "amount": 4820.00, "vendor_id": "V-0042"}, "expected_output": {"classification": "nuevo_proveedor", "confidence": "high"}, "metadata": {"tenant": "acme", "period": "2026-04", "case": "edge_new_vendor"}}
{"input": {"description": "PAGO FACT 2026-04 ACME", "amount": 4820.00, "vendor_id": "V-0001"}, "expected_output": {"classification": "regular", "confidence": "high"}, "metadata": {"tenant": "acme", "period": "2026-04", "case": "baseline"}}
```

Ruta de versioning en el repo:

```
golden_sets/
  acme/
    invoice_classifier/
      v1.jsonl     ← primera versión
      v2.jsonl     ← tras onboarding con más datos del cliente
  globex/
    invoice_classifier/
      v1.jsonl
```

**Reglas de versionado**:
- Nunca edites un archivo de versión existente. Crea `v2.jsonl`.
- Cuando añades casos, el anterior sigue siendo la baseline.
- El CI siempre corre contra la versión más reciente del tenant activo.

### Cuántos casos necesitas

| Fase | Mínimo | Objetivo | Cuándo actualizar |
|------|--------|----------|-------------------|
| Desarrollo / pre-lanzamiento | 5 | 20 | Antes de entregar al cliente |
| Producción temprana (< 3 meses) | 20 | 50 | Cada feature nueva al agente |
| Producción estable | 50 | 100+ | Mensual; cuando cliente reporta error |

El mínimo de 5 es para tener algo. Con menos de 20 casos, una sola regresión puede mover la métrica 20 puntos sin ser estadísticamente significativa.

### Métricas según tipo de output

**Exact match**: cuando el output es una categoría cerrada (`"regular"`, `"nuevo_proveedor"`, `"duplicado"`). La métrica es simplemente exactitud: `correct / total`.

```python
def exact_match(predictions: list[str], references: list[str]) -> float:
    correct = sum(p == r for p, r in zip(predictions, references))
    return correct / len(references)
```

**LLM-as-judge**: cuando el output es texto libre (un hallazgo, un resumen, una recomendación). Un segundo modelo evalúa si la respuesta es correcta según una rúbrica.

La plantilla de judge que usamos en el curso:

```python
JUDGE_PROMPT = """Eres un evaluador de calidad para un agente de auditoría financiera.

Input del agente: {input}
Output esperado: {expected}
Output del agente: {actual}

Evalúa si el output del agente cumple estos criterios:
1. Identifica el mismo tipo de anomalía que el output esperado (0/1)
2. Menciona el monto o referencia correcta (0/1)
3. El tono es apropiado para un reporte ejecutivo (0/1)

Responde SOLO con un JSON: {{"score": <0-3>, "reasoning": "<una línea>"}}"""
```

Reglas para el judge:
- **Temperatura 0** — el judge debe ser determinístico.
- Modelo pequeño y barato (Sonnet 4.6, no Opus).
- La rúbrica tiene criterios binarios (0/1), no gradientes. Más fácil de auditar.
- El resultado es un score numérico, no «sí/no» — para calcular promedios.

**Embedding similarity**: cuando el output esperado y el real dicen lo mismo con palabras distintas. Se computan los embeddings de ambos y se calcula cosine similarity. Umbral típico: > 0.85 = pass.

**Schema validation**: cuando el output debe ser JSON con un schema específico. Si el agente devuelve un dict que no valida contra el Pydantic model esperado, fail inmediato.

```python
from pydantic import BaseModel, ValidationError

class InvoiceClassification(BaseModel):
    classification: str
    confidence: str

def validate_schema(output: dict) -> bool:
    try:
        InvoiceClassification(**output)
        return True
    except ValidationError:
        return False
```

**Latencia y costo**: siempre medir, aunque no sean el criterio de pass/fail primario.

```python
import time

start = time.perf_counter()
result = agent.run(case["input"])
latency_ms = (time.perf_counter() - start) * 1000
cost_usd = result.usage.input_tokens * 3e-6 + result.usage.output_tokens * 15e-6
```

### Frameworks disponibles

**Promptfoo** (recomendado para este curso): config en YAML, corre desde CLI o CI, soporta LLM-as-judge con una línea. Mínimo de setup:

```yaml
# promptfooconfig.yaml
description: "Invoice classifier eval — ACME v1"
prompts:
  - file://prompts/invoice_classifier.txt
providers:
  - anthropic:claude-sonnet-4-6
tests:
  - vars:
      description: "PAGO FACT 2026-04 ACME"
      amount: 4820
      vendor_id: "V-0001"
    assert:
      - type: llm-rubric
        value: "La clasificación debe ser 'regular' para un proveedor conocido con monto esperado"
```

**DeepEval** (recomendado para proyectos con pytest): integra con pytest, métricas avanzadas (G-Eval, faithfulness, hallucination). Más código, más control.

**Arize Phoenix evals**: funciona sobre las trazas de Phoenix que ya tienes de D04. Si exportas los spans, puedes correr evals directamente sobre el historial de producción.

### CI con evals: el PR no merge si la calidad baja

El pipeline de CI en D05 ya tiene tests y lint. Le añades un job de evals:

```yaml
# .github/workflows/ci.yml (añadir a lo que ya existe)
  eval:
    runs-on: ubuntu-latest
    needs: test
    steps:
      - uses: actions/checkout@v4
      - name: Run golden set evals
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          uv run python scripts/run_evals.py \
            --golden-set golden_sets/acme/invoice_classifier/v2.jsonl \
            --threshold 0.85
      # Si la exactitud < 85%, el script devuelve código de salida 1
      # GitHub Actions marca el job como failed y bloquea el merge
```

El script `scripts/run_evals.py` corre el agente contra cada caso del golden set, calcula la métrica, y falla si cae por debajo del umbral. El umbral es un parámetro declarado por el equipo — no hay valor «correcto» universal; depende del SLA con el cliente.

### A/B testing con tráfico real

Cuando tienes dos versiones del agente (prompt A vs prompt B) y quieres saber cuál es mejor en producción real:

1. Divide el tráfico: 50% de requests van a la versión A, 50% a la versión B.
2. Declara la métrica de éxito **antes** de empezar el test (tasa de aceptación humana, score del judge).
3. Corre durante suficiente tiempo para tener significancia estadística (mínimo 100 casos por rama).
4. La versión ganadora se promovea `main`; la otra se descarta.

> [!nota]
> El A/B testing de LLMs es más costoso que el de features web: cada request llama al modelo. Calcula el costo antes de activarlo: si el golden set tiene 50 casos y el modelo cobra $0.015/caso, son $0.75 por evaluación completa. Multiplicado por cada PR del equipo, el costo es manejable. Multiplicado por tráfico real, calcula antes de configurarlo.

## Ejemplo conducido

Agente de ACME que clasifica facturas entrantes. Construyes el golden set mínimo y corres la primera evaluación:

```python
# scripts/run_evals.py
import json
import sys
from pathlib import Path
from anthropic import Anthropic

client = Anthropic()

SYSTEM = """Clasificas facturas de ACME Manufacturing.
Responde siempre con JSON: {"classification": "<regular|nuevo_proveedor|duplicado|anomalia>", "confidence": "<high|medium|low>"}
"""

def classify_invoice(description: str, amount: float, vendor_id: str) -> dict:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=100,
        system=SYSTEM,
        messages=[{"role": "user", "content": f"Vendor: {vendor_id}\nMonto: {amount}\nDescripción: {description}"}],
    )
    return json.loads(response.content[0].text)

def run_evals(golden_set_path: str, threshold: float) -> bool:
    cases = [json.loads(line) for line in Path(golden_set_path).read_text().splitlines() if line.strip()]
    correct = 0
    for case in cases:
        result = classify_invoice(**case["input"])
        if result.get("classification") == case["expected_output"]["classification"]:
            correct += 1
        else:
            print(f"FAIL: {case['input']['description']!r} → got {result.get('classification')!r}, expected {case['expected_output']['classification']!r}")
    accuracy = correct / len(cases)
    print(f"Exactitud: {accuracy:.1%} ({correct}/{len(cases)})")
    return accuracy >= threshold

if __name__ == "__main__":
    import argparse
    parser = argparse.ArgumentParser()
    parser.add_argument("--golden-set", required=True)
    parser.add_argument("--threshold", type=float, default=0.85)
    args = parser.parse_args()
    passed = run_evals(args.golden_set, args.threshold)
    sys.exit(0 if passed else 1)
```

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Exact match contra golden set | determinístico | Comparación string: o es igual o no lo es |
| Validación de schema Pydantic | determinístico | Regla cerrada definida por el modelo |
| LLM-as-judge (temperatura 0) | agéntico | El modelo razona; temperatura 0 lo hace lo más repetible posible, pero no 100% determinístico |
| Calcular latencia y costo por caso | determinístico | Aritmética pura sobre números devueltos por el SDK |
| Decidir el umbral de calidad (85% vs 90%) | agéntico | Requiere contexto del SLA con el cliente, costo de falso negativo, riesgo del negocio |
| Actualizar el golden set con nuevos casos | agéntico | Requiere criterio sobre qué casos son representativos y cuál es la salida correcta |

## Errores típicos

**1. Golden set construido con datos inventados.**
*Síntoma*: el agente pasa las evals con 100% pero el cliente reporta fallos reales.
*Causa*: los casos del golden set no representan la distribución real de inputs del cliente (descripciones de facturas con tipografías raras, abreviaciones locales, montos en formato europeo).
*Arreglo*: los primeros 20 casos del golden set deben venir de datos reales del cliente, anonimizados si hay PII.

**2. Temperature > 0 en el LLM judge.**
*Síntoma*: el mismo caso del golden set pasa en una ejecución de CI y falla en la siguiente.
*Causa*: temperatura > 0 hace que el judge varíe su respuesta entre ejecuciones.
*Arreglo*: forzar `temperature=0` en todas las llamadas del judge. Si el modelo no soporta temperatura 0 exacta, usa el valor más bajo disponible.

**3. Threshold demasiado bajo para que nunca falle.**
*Síntoma*: el CI de evals siempre pasa, incluso después de cambios destructivos al prompt.
*Causa*: el threshold fue seteado al nivel de la primera medición (e.g., 60%) en lugar del SLA real.
*Arreglo*: el threshold debe ser igual o superior al nivel de precisión comprometido con el cliente. Si el SLA dice 85%, el threshold es 85%.

**4. Golden set que nunca se actualiza.**
*Síntoma*: el agente «pasa los tests» pero el cliente reporta fallos en casos nuevos que nunca estuvieron en el golden set.
*Causa*: el golden set se armó en el onboarding y nadie lo tocó desde entonces.
*Arreglo*: política explícita: todo bug reportado por el cliente que el agente reproduce → nuevo caso en el golden set antes de cerrar el ticket.

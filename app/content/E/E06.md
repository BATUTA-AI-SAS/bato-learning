## Hilo conductor

En E01–E05 construiste cada pieza por separado: el loop del SDK de Anthropic (E01), el equivalente en LangGraph (E02), skills como nodos (E03), memoria multitenant (E04), durabilidad con Temporal (E05). El Capstone es la prueba de que esas piezas funcionan juntas en un sistema real. No hay concepto nuevo. Hay integración, criterios de aceptación duros, y un entregable que un cliente podría pagar.

## Idea central

Un agente de producción multitenant tiene cuatro capas que deben estar presentes antes de que cualquier cliente lo use:

1. **Correctitud funcional**: el agente produce el resultado esperado para inputs conocidos (golden set).
2. **Aislamiento**: los datos de ACME nunca mezclan con los de Globex, ni en el agente ni en los logs ni en el storage.
3. **Durabilidad**: el workflow sobrevive a fallos del worker y no duplica side-effects en retries.
4. **Observabilidad**: cada llamada al LLM y cada span de latencia son visibles en Phoenix.

El Capstone verifica las cuatro. Si una falla, el entregable no está listo.

## Por qué importa

El skill `monthly_audit` de BATUTA corre hoy en Claude Code: loop `prompt → tool → result → ...` gestionado por el harness. Ese loop es opaco (no hay trazas estructuradas), frágil (si el proceso muere, el estado desaparece) y monocliente (no tiene `tenant_id` en los artefactos). Portarlo a LangGraph + Temporal resuelve los tres problemas y produce un servicio que se puede vender con un SLA.

El caso ancla es **F-CTA-02 (auditoría interna: anomalías y duplicados)**. Los tres casos que puedes elegir como alternativa:

- **F-FIN-01 (conciliación bancaria)**: datos más ricos, match exacto + difuso, Belvo/Plaid opcional.
- **F-OPS-01 (planeación de producción)**: schedule semanal, datos de demanda, LLM para narrativa.
- **F-CTA-02 (auditoría interna, recomendado)**: es exactamente lo que `AuditTenantWorkflow` ya hace en el repo. El delta de implementación es mínimo y el criterio de evaluación es concreto.

Elige el caso que mejor encaje con el cliente al que quieres vender primero.

## Criterios de aceptación duros

El Capstone está «listo» cuando los seis criterios siguientes son verdaderos simultáneamente. No hay «listo al 80 %».

| # | Criterio | Cómo verificar |
|---|---------|----------------|
| 1 | El agente corre contra ACME y Globex con `tenant_id` correcto | `pytest tests/test_multitenant.py` pasa para ambos tenants; los artefactos en DB tienen `tenant_id` correcto. |
| 2 | Golden set ≥ 5 ejemplos evaluados en CI | `pytest tests/test_golden.py` — cada ejemplo compara `agent_output` con `expected_output` por campos clave (no string exact match). |
| 3 | Trazas visibles en Phoenix | Abrir `http://localhost:6006`; el proyecto del tenant aparece con spans de cada nodo LangGraph y de cada llamada al LLM. |
| 4 | Backup de SQLite se restaura correctamente | `bash infra/scripts/backup.sh && bash infra/scripts/restore.sh` — la app arranca limpia; los módulos y progreso del usuario están intactos. |
| 5 | Workflow cancelable y reanudable | `python scripts/cancel_workflow.py audit-acme-2026-04` termina el workflow en CANCELLED; ejecutarlo de nuevo produce un run nuevo desde cero. |
| 6 | Deploy en CX22 con TLS válido | `curl -I https://tudominio.com/api/health` devuelve `200` y `x-powered-by: FastAPI`; `curl -I https://tudominio.com` devuelve `200`. |

Adicionalmente, el entregable incluye un archivo `CAPSTONE_REPORT.md` en la raíz del repo con:

- Caso F elegido y justificación (2–3 líneas).
- Antes/después en tokens por run y costo estimado (USD/mes para 20 tenants × 12 corridas).
- Decisiones de diseño tomadas (máximo 5 puntos).
- Riesgos identificados y mitigaciones.

## Cómo funciona por dentro

### Arquitectura objetivo

```
Cliente HTTP / Temporal UI
        |
  FastAPI router (temporal_demo.py)
        |
  AuditTenantWorkflow (@workflow.defn)
  ├── execute_activity: ingest_data(tenant_id, period)
  │     └── IO: ERP / Excel / Belvo
  ├── execute_activity: run_agent(skill_id, tenant_id, dataset_id)
  │     └── LangGraph StateGraph
  │           ├── [load_data]      determinístico: carga y valida
  │           ├── [detect_dups]    determinístico: hash exact match
  │           ├── [detect_fuzzy]   agéntico: LLM clasifica ambiguos
  │           ├── [draft_report]   agéntico: LLM redacta findings
  │           └── [human_review?]  interrupt si findings críticos > N
  └── execute_activity: persist_output(tenant_id, result, idempotency_key)
        └── DB write con upsert por idempotency_key
```

El grafo LangGraph es un `StateGraph` con `TypedDict` de estado. Los nodos determinísticos son funciones Python puras. Los nodos agénticos invocan al LLM con las tools de SHARED §3.6.

### Paso a paso

Las tareas están ordenadas. Completa cada una antes de pasar a la siguiente — cada bloque desbloquea el siguiente.

#### Bloque 0 — Setup (≈30 min)

- [ ] Clona el repo, ejecuta `uv sync`, verifica `uv run pytest` pasa en verde.
- [ ] Levanta el stack: `docker compose up -d` (app + Temporal).
- [ ] Verifica que `http://localhost:8080` muestra la Temporal UI y que `http://localhost:8000/docs` muestra la API.
- [ ] Crea las variables de entorno: `ANTHROPIC_API_KEY`, `PHOENIX_URL` (si usas Phoenix local).
- [ ] Elige el caso F y regístralo en `CAPSTONE_REPORT.md`.

#### Bloque 1 — LangGraph: portar el skill a StateGraph (≈90 min)

- [ ] Define el `TypedDict` de estado del grafo con campos: `tenant_id`, `period`, `dataset_id`, `invoices`, `findings`, `report_md`.
- [ ] Implementa los nodos:
  - `load_data`: carga `invoices` desde `dataset_id`; es determinístico; sin LLM.
  - `detect_exact_dups`: hash `(vendor_id, amount, date)` y marca duplicados exactos.
  - `detect_fuzzy_dups`: invoca al LLM con las transacciones marcadas como «candidatas»; retorna clasificación.
  - `draft_report`: invoca al LLM para redactar `report_md` con findings ordenados por riesgo.
  - (opcional) `human_review`: `interrupt_before` si `len(critical_findings) > 3`.
- [ ] Compila el grafo: `graph = builder.compile(checkpointer=MemorySaver())`.
- [ ] Ejecuta el grafo contra datos ficticios de ACME: verifica que los nodos determinísticos producen el mismo resultado en cada run.

> [!cuidado]
> El checkpointer del grafo (`MemorySaver`) vive en la activity `run_agent`, no en el workflow. El workflow solo sabe que la activity terminó y retornó un dict. El estado del grafo LangGraph es interno a la activity.

#### Bloque 2 — Temporal: envolver el grafo en un workflow (≈60 min)

- [ ] Modifica `run_agent` en `api/app/activities/agent.py` para invocar el `StateGraph` compilado en lugar del mock.
- [ ] Verifica que el heartbeat loop sigue activo mientras el LLM trabaja.
- [ ] Añade `workflow.upsert_search_attributes({"TenantId": [tenant_id], "Period": [period]})` al inicio del workflow.
- [ ] Dispara el workflow para ACME: `POST /temporal/audit` con `{"tenant_id": "acme", "period": "2026-04"}`.
- [ ] En la Temporal UI, verifica el historial: `WorkflowExecutionStarted → ActivityTaskScheduled (ingest_data) → ActivityTaskCompleted → ...`.
- [ ] Dispara para Globex: verifica que los artefactos en DB tienen `tenant_id = "globex"` y que no hay mezcla.

#### Bloque 3 — Golden set y CI (≈60 min)

- [ ] Crea `tests/fixtures/golden/` con 5 archivos JSON:
  ```json
  {
    "tenant_id": "acme",
    "period": "2026-04",
    "input": { "invoices": [...] },
    "expected": {
      "finding_count_min": 2,
      "has_duplicate": true,
      "report_contains": "PROVEEDOR-007"
    }
  }
  ```
- [ ] Escribe `tests/test_golden.py` que para cada fixture:
  1. Llama al grafo LangGraph directamente (sin Temporal, para velocidad).
  2. Compara `result` contra `expected` por los campos clave declarados.
  3. Falla con un mensaje descriptivo si algún campo no coincide.
- [ ] Ejecuta `pytest tests/test_golden.py -v` — debe pasar en verde.
- [ ] Añade el paso al CI (GitHub Actions o el runner que uses): el golden set bloquea el merge.

> [!nota]
> El golden set no exige exact match del texto del reporte. Exige campos concretos y verificables: conteo de findings, presencia de un ID de proveedor, clasificación correcta de un duplicado conocido. El LLM tiene varianza; el test debe ser estable.

#### Bloque 4 — Observabilidad en Phoenix (≈45 min)

- [ ] Verifica que `app/telemetry.py` inicializa el tracer con `openinference-instrumentation-langchain`.
- [ ] Añade instrumentación al nodo `detect_fuzzy_dups` si no está: `with tracer.start_as_current_span("detect_fuzzy_dups") as span: span.set_attribute("tenant_id", tenant_id)`.
- [ ] Levanta Phoenix: `docker run -p 6006:6006 arizephoenix/phoenix:latest`.
- [ ] Corre el workflow; abre `http://localhost:6006`; verifica que el proyecto `monthly-audit` aparece con los spans de cada nodo y de cada llamada al LLM (incluyendo `input_tokens`, `output_tokens`, `latency_ms`).
- [ ] Identifica el span de mayor latencia — el ejercicio de diagnóstico backend lo formaliza.

#### Bloque 5 — Backup y restauración (≈20 min)

- [ ] Ejecuta `bash infra/scripts/backup.sh`: debe crear `data/backup-{fecha}.db`.
- [ ] Detén la app: `docker compose down`.
- [ ] Elimina `data/learning.db`.
- [ ] Ejecuta `bash infra/scripts/restore.sh data/backup-{fecha}.db`.
- [ ] Levanta la app: `docker compose up -d` — verifica que el progreso del usuario y los módulos están intactos.

#### Bloque 6 — Deploy en Hetzner CX22 con TLS (≈60 min)

- [ ] Provisiona un CX22 (4 €/mes, 2 vCPU, 4 GB RAM). Instala Docker.
- [ ] Clona el repo en el servidor. Crea `.env` con las variables de entorno.
- [ ] Despliega con Traefik:
  ```bash
  docker compose -f compose.yaml -f infra/hetzner/compose.prod.yml up -d
  ```
- [ ] Edita `infra/hetzner/compose.prod.yml`: reemplaza `learning.example.com` y `api.example.com` con tu dominio real, y `REPLACE_ME@example.com` con tu email para Let's Encrypt.
- [ ] Verifica TLS: `curl -I https://tudominio.com/api/health` → `200`.
- [ ] Verifica cancelación remota: `python scripts/cancel_workflow.py audit-acme-2026-04` → workflow en CANCELLED en la Temporal UI.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Workflow Temporal (`AuditTenantWorkflow.run`) | determinístico | Orquesta activities; cero IO ni LLM directo. El replay debe ser idéntico. |
| `ingest_data` activity | no-determinístico | IO de red; latencia variable; resultado puede cambiar si el ERP tiene datos nuevos. |
| Nodo `detect_exact_dups` del grafo | determinístico | Hash `(vendor_id, amount, date)` es función pura; mismo input, mismo output, sin LLM. |
| Nodo `detect_fuzzy_dups` del grafo | agéntico | Transacciones con descripción ambigua no tienen regla cerrada; el LLM decide. |
| Nodo `draft_report` del grafo | agéntico | La redacción del reporte es para humano; requiere razonamiento contextual. |
| `persist_output` activity | no-determinístico | Escritura a DB; idempotency key previene duplicados; el resultado puede diferir si ya existe. |
| Schedule Temporal mensual | determinístico | La expresión cron es fija; cuándo se lanza es predecible. |

## Errores típicos

**1. El checkpointer del grafo LangGraph está en el workflow, no en la activity.**
*Síntoma*: el workflow falla con `NonDeterminismError` porque `MemorySaver` hace IO.
*Causa*: el grafo se compiló y el checkpointer se inicializó dentro de `@workflow.run`.
*Arreglo*: el `StateGraph` compilado vive dentro de la activity `run_agent`. El workflow solo recibe el resultado final como dict.

**2. El golden set usa exact match del texto del reporte.**
*Síntoma*: el test falla un día sí y un día no porque el LLM varía la redacción.
*Causa*: `assert result["report_md"] == expected["report_md"]` — el texto del LLM no es reproducible.
*Arreglo*: testea campos estructurales: `assert result["finding_count"] >= expected["finding_count_min"]`, `assert expected["report_contains"] in result["report_md"]`.

**3. Traefik en prod sin dominio real configurado.**
*Síntoma*: Let's Encrypt rechaza el certificado; el servicio arranca en HTTP pero no en HTTPS.
*Causa*: `learning.example.com` sigue en `infra/hetzner/compose.prod.yml` sin reemplazar.
*Arreglo*: edita el archivo antes de hacer `docker compose up`. El DNS del dominio debe apuntar al CX22 antes de que Traefik solicite el certificado.

**4. Tests del golden set corren contra Temporal en CI.**
*Síntoma*: el CI no tiene Temporal levantado; los tests fallan con `Connection refused`.
*Causa*: los tests llaman al workflow vía HTTP en lugar de al grafo LangGraph directamente.
*Arreglo*: los tests del golden set invocan el `StateGraph` compilado directamente, sin Temporal. El propósito es testear el agente, no la infraestructura. Temporal tiene sus propios tests de integración.

**5. `tenant_id` ausente en los search attributes del workflow.**
*Síntoma*: en la Temporal UI no puedes filtrar por tenant; todos los workflows se ven iguales.
*Causa*: falta `workflow.upsert_search_attributes({"TenantId": [tenant_id]})`.
*Arreglo*: añade la llamada al inicio de `@workflow.run`. Los search attributes son indexados por Temporal y permiten queries por tenant.

## Para profundizar

> [!profundizar]
> El workflow real que este Capstone extiende está en [`api/app/workflows/audit.py`](../../api/app/workflows/audit.py). La función `run` tiene exactamente tres `execute_activity`. Tu tarea en el Bloque 2 es reemplazar el mock en `run_agent` por la invocación del `StateGraph` compilado.

> [!profundizar]
> El trigger HTTP del workflow está en [`api/app/routers/temporal_demo.py`](../../api/app/routers/temporal_demo.py). Observa cómo `workflow_id = f"audit-{tenant_id}-{period}"` es ya una idempotency key: si el mismo workflow se dispara dos veces, Temporal rechaza el segundo con `WorkflowExecutionAlreadyStartedError`. Ese comportamiento es un feature, no un bug.

> [!profundizar]
> El `compose.prod.yml` de Hetzner está en [`infra/hetzner/compose.prod.yml`](../../infra/hetzner/compose.prod.yml). Traefik v3.2 con Let's Encrypt. Los únicos cambios obligatorios antes de hacer `up -d`: el dominio y el email. El resto funciona tal cual.

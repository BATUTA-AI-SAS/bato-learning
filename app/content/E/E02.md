## Hilo conductor

En E01 construiste un loop de agente con el SDK de Anthropic: `messages.create` → detección de `tool_use` → ejecución → siguiente turno. Ese código funciona, pero eres tú quien gestiona el loop, el historial, el estado entre turnos y cualquier interrupt. LangGraph externaliza esa gestión: el loop se convierte en un grafo con nodos, aristas y un checkpointer que persiste el estado automáticamente. E02 es la traducción 1:1. Cuando termines, tendrás un agente idéntico funcionalmente pero con infraestructura que E04 (memoria multitenant) y E05 (Temporal) construyen encima.

## Idea central

Un **StateGraph** es un grafo dirigido donde cada nodo es una función Python que recibe el estado actual y devuelve un dict parcial con las actualizaciones. El estado se define con un `TypedDict`; el campo `messages` usa `Annotated[list, add_messages]` para que cada update sea un append — nunca una sobrescritura. LangGraph gestiona el loop, las aristas condicionales y la persistencia en cada step.

## Por qué importa

Sin esto, cualquier agente que construyas con el SDK puro tiene el loop en tu código: manejas la lista de mensajes, decides cuándo salir, reinicializas si falla. Cuando el agente corre dentro de un workflow Temporal (E05), ese loop necesita ser serializable y reanudable. LangGraph hace exactamente eso con el checkpointer.

El caso **F-OPS-01 (planeación de producción semanal)** requiere un agente que puede pausarse entre el nodo `forecast` y el nodo `plan` para esperar validación humana. `interrupt_before` de LangGraph lo implementa en tres líneas. Sin LangGraph, lo reimplementas desde cero con colas y semáforos.

> [!cuidado]
> LangGraph 0.2.50+ cambió la API de checkpointers: `MemorySaver` sigue siendo el default en tests, pero en producción usas `SqliteSaver` (sync) o `AsyncSqliteSaver`. El `PostgresSaver`/`AsyncPostgresSaver` está en `langgraph-checkpoint-postgres`. No mezcles sync y async — si el grafo es async, el checkpointer también lo es.

## Cómo funciona por dentro

### El StateGraph y el estado tipado

Todo empieza con el estado. El patrón canónico del repo (`api/app/routers/langgraph_demo.py`):

```python
from typing import Annotated, Any, TypedDict
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]
```

`add_messages` es un **reducer**: cuando un nodo devuelve `{"messages": [nuevo_mensaje]}`, LangGraph no reemplaza la lista entera — la extiende. Sin el reducer, cada nodo sobreescribiría el historial. Esto mapea directamente al rol del array `messages` en la API de Anthropic.

### Nodos como funciones puras

Cada nodo recibe `state: State` y devuelve un `dict` parcial. El nodo del modelo invoca el LLM con el historial actual:

```python
def model_node(state: State) -> dict[str, Any]:
    return {"messages": [llm.invoke(state["messages"])]}
```

`llm` es un `ChatAnthropic` con tools vinculadas (`bind_tools`). LangChain-Anthropic serializa la respuesta a un `AIMessage` que puede contener `tool_calls`. El nodo no ejecuta las tools — solo genera la respuesta del modelo.

### ToolNode y tools_condition

`ToolNode` es el nodo prebuilt que ejecuta las tools que el modelo solicitó. `tools_condition` es la arista condicional que inspecciona el último mensaje: si hay `tool_calls`, va al nodo `"tools"`; si no, va a `END`.

```
START → model → [tools_condition] → tools → model → ...
                                  ↘ END (si no hay tool_calls)
```

Esto es el ReAct loop. El grafo lo gestiona; tú solo defines los nodos.

### compile() y el checkpointer

`g.compile()` sin argumentos produce un grafo en memoria: el estado no persiste entre invocaciones. `g.compile(checkpointer=checkpointer)` hace que LangGraph guarde el estado completo en cada step.

El `thread_id` es la clave de la sesión. Cada invocación lleva `config={"configurable": {"thread_id": "acme:audit:2026-04"}}`. Dos invocaciones con el mismo `thread_id` continúan la misma conversación. Dos con distintos `thread_id` son sesiones independientes — la base del aislamiento multitenant de E04.

### interrupt_before: la pausa controlada

```python
app_graph = g.compile(
    checkpointer=checkpointer,
    interrupt_before=["tools"],
)
```

Con este flag, el grafo se detiene antes de ejecutar el nodo `"tools"` y devuelve el control al llamador. El modelo ya generó sus `tool_calls` y están en el estado. Tú puedes inspeccionarlos, rechazarlos, o aprobarlos antes de reanudar. Es el equivalente exacto del hook `PreToolUse` de Claude Code.

Para reanudar después de la pausa:

```python
app_graph.invoke(None, config={"configurable": {"thread_id": tid}})
```

`None` como input indica «continúa desde donde quedaste».

### astream_events para streaming

`invoke` es síncrono y devuelve el estado final. Para SSE necesitas `astream_events`:

```python
async for event in app_graph.astream_events(
    {"messages": [HumanMessage(content=prompt)]},
    config={"configurable": {"thread_id": tid}},
    version="v2",
):
    if event["event"] == "on_chat_model_stream":
        chunk = event["data"]["chunk"].content
        yield f"data: {chunk}\n\n"
```

`version="v2"` es obligatorio en LangGraph 0.2.50+. La versión `v1` está deprecated.

## Ejemplo conducido

El grafo del repo (`api/app/routers/langgraph_demo.py`) es el punto de referencia. Aquí está la versión con checkpointer SQLite y `interrupt_before`:

```python
from typing import Annotated, Any, TypedDict

from langchain_anthropic import ChatAnthropic
from langchain_core.messages import HumanMessage
from langchain_core.tools import tool
from langgraph.checkpoint.sqlite import SqliteSaver
from langgraph.graph import END, START, StateGraph
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode, tools_condition

@tool
def sql_query(query: str, tenant: str) -> str:
    """Ejecuta una query SQL de solo-lectura sobre el warehouse del tenant."""
    return f"[{tenant}] 12 facturas, total 48 210,00 EUR"

class State(TypedDict):
    messages: Annotated[list, add_messages]

llm = ChatAnthropic(model="claude-sonnet-4-6").bind_tools([sql_query])

def model_node(state: State) -> dict[str, Any]:
    return {"messages": [llm.invoke(state["messages"])]}

g = StateGraph(State)
g.add_node("model", model_node)
g.add_node("tools", ToolNode([sql_query]))
g.add_edge(START, "model")
g.add_conditional_edges("model", tools_condition)
g.add_edge("tools", "model")

with SqliteSaver.from_conn_string("data/checkpoints.db") as checkpointer:
    app_graph = g.compile(checkpointer=checkpointer, interrupt_before=["tools"])
    tid = "acme:audit-demo:2026-04"
    state = app_graph.invoke(
        {"messages": [HumanMessage("¿Cuántas facturas hay en 2026-04?")]},
        config={"configurable": {"thread_id": tid}},
    )
    # Aquí el grafo pausó antes de ejecutar tools.
    # Inspecciona state["messages"][-1].tool_calls y aprueba:
    final = app_graph.invoke(None, config={"configurable": {"thread_id": tid}})
    print(final["messages"][-1].content)
```

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|---|---|---|
| Definir nodos, aristas y checkpointer | determinístico | El flujo del grafo es código Python fijo — no cambia por la entrada. |
| `tools_condition` decide ir a `tools` o a `END` | determinístico | Inspecciona si el último mensaje tiene `tool_calls`; es una condición booleana. |
| El nodo `model_node` elige qué tool llamar y con qué args | agéntico | El modelo decide en función del contexto; no hay regla cerrada. |
| El modelo decide cuándo parar el loop (emitir `end_turn`) | agéntico | Sin regla: depende del contenido de la respuesta y el historial. |
| `interrupt_before=["tools"]` pausa antes de ejecutar | determinístico | Regla fija en compile-time; no es una decisión del modelo. |

## Errores típicos

**1. Olvidar `add_messages` en el reducer.**
*Síntoma*: el historial siempre tiene solo el último mensaje; el modelo pierde contexto entre turnos.
*Causa*: el campo `messages` está declarado como `list` sin `Annotated[list, add_messages]`. Cada nodo sobreescribe la lista.
*Arreglo*: `messages: Annotated[list, add_messages]` en el `TypedDict`. Verificar con `state["messages"]` después de dos turnos.

**2. Mezclar `invoke` síncrono con checkpointer async.**
*Síntoma*: `RuntimeError: no running event loop` o el checkpointer no persiste.
*Causa*: usas `AsyncSqliteSaver` con `app_graph.invoke()` (no async).
*Arreglo*: si el grafo es async, usa `await app_graph.ainvoke(...)` y `async with AsyncSqliteSaver.from_conn_string(...) as cp`.

**3. Reanudar con input en lugar de `None` tras `interrupt_before`.**
*Síntoma*: el grafo ignora el estado previo y re-ejecuta desde el principio.
*Causa*: `app_graph.invoke({"messages": [...]}, config=...)` en lugar de `app_graph.invoke(None, config=...)`.
*Arreglo*: `None` como primer argumento indica «continúa desde el checkpoint»; un dict indica «empieza nuevo».

**4. Olvidar `version="v2"` en `astream_events`.**
*Síntoma*: warning de deprecation o eventos con estructura distinta a la esperada.
*Causa*: `astream_events` sin `version` usa `v1` por compatibilidad.
*Arreglo*: añadir `version="v2"` como kwarg. La estructura de eventos es más consistente en v2.

**5. `thread_id` sin `tenant_id` como prefijo.**
*Síntoma*: dos tenants con el mismo id de conversación se pisan entre sí.
*Causa*: `thread_id = "audit:2026-04"` es igual para ACME y Globex.
*Arreglo*: `thread_id = f"{tenant_id}:audit:{period}"`. El `thread_id` lleva siempre el tenant como prefijo. Regla de aislamiento multitenant (ver C03).

## Caso de uso real

**F-OPS-01 — Planeación de producción semanal** (manufactura, ACME).
El agente ingiere el plan de demanda semanal de World Office, consulta capacidad por línea con `sql_query`, y genera las órdenes de fabricación. El nodo `plan` requiere `interrupt_before` porque el jefe de planta aprueba el plan antes de liberarlo al ERP. Sin checkpointer, el estado se pierde entre la pausa y la reanudación.

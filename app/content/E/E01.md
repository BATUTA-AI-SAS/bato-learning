## 1. Hilo conductor

Vienes de A07, donde escribiste funciones `async` que paralelizan IO. Ahora usas ese patrón contra la API de Anthropic: un loop donde el modelo emite `tool_use`, tu código ejecuta el side-effect, y devuelves `tool_result`. Hacia E02 portarás este mismo loop a un `StateGraph` LangGraph; aquí aprendes la semántica base que LangGraph abstrae.

---

## 2. Idea central

Antes de entrar al código, cuatro términos que aparecen en todos los ejemplos:

- **content block**: la unidad mínima de contenido en un mensaje. Como una celda de Excel que puede contener texto, un número, o una fórmula — aquí puede ser texto, una petición de tool, o el resultado de una tool.
- **tool_use**: bloque que emite el modelo cuando quiere llamar una de tus funciones. Contiene un `id` único, el nombre de la función, y los argumentos JSON. Como una celda que llama a `=BUSCAR_PRECIO("ACME-001")` — el modelo decide cuándo llamarla y con qué argumento.
- **tool_result**: bloque que tú devuelves con el resultado de ejecutar la función. Referencia el mismo `id` del `tool_use`. Como el valor que devuelve `BUSCAR_PRECIO` — tu código lo calcula y lo pasa de vuelta al modelo.
- **thinking block**: bloque que emite el modelo cuando extended thinking está activo; contiene su razonamiento interno antes de la respuesta final. Como ver los pasos intermedios de una fórmula compleja en Excel antes del resultado final.

La **messages API** de Anthropic es un array de turnos alternos (`user` / `assistant`) con bloques de contenido tipados. Cuando el modelo quiere llamar una función, emite un bloque `tool_use` con un `id` y un `input` JSON. Tu código ejecuta la función y devuelve un bloque `tool_result` referenciando ese mismo `id`. El modelo continúa. Ese ciclo es el **loop de agente**.

## 3. Por qué importa

Sin entender el shape exacto de los mensajes, no puedes depurar un loop que se rompe en el tercer turno. El error más caro: olvidar que `tool_result` debe emparejarse por `tool_use_id`; si mandas un `id` incorrecto, el modelo responde con error de validación o ignora el resultado y alucina.

El caso F-FIN-01 (conciliación bancaria) ejecuta un agente con la tool `match_transactions` que corre en bucle; si el loop falla en el turno 4, la conciliación de ACME queda incompleta y el CFO no tiene reporte al cierre. Un loop roto en producción cuesta más que cualquier llamada extra al SDK.

## 4. Cómo funciona por dentro

### 4.1 Shape de mensajes

El SDK recibe la lista `messages` como alternancia `user → assistant → user → ...`.
Cada elemento es `{"role": "user"|"assistant", "content": <str | list[block]>}`.

Los **bloques de contenido** que importan:

| Tipo | Quién lo emite | Campos clave |
|------|----------------|--------------|
| `text` | modelo o usuario | `text: str` |
| `tool_use` | modelo | `id: str`, `name: str`, `input: dict` |
| `tool_result` | tu código | `tool_use_id: str`, `content: str\|list` |
| `thinking` | modelo (extended thinking) | `thinking: str` |

Regla dura: **si en la respuesta del modelo hay uno o más `tool_use`, el siguiente mensaje `user` debe contener un `tool_result` por cada uno de ellos**, en el mismo orden o en cualquier orden, pero todos. Uno solo que falte produce un error `400` del API («`tool_use_id` not found»).

### 4.2 Declarar tools

```python
TOOLS = [
    {
        "name": "match_transactions",
        "description": "Empareja una transacción bancaria contra el libro contable del tenant.",
        "input_schema": {
            "type": "object",
            "additionalProperties": False,   # regla dura — siempre
            "properties": {
                "transaction_id": {"type": "string"},
                "tenant":         {"type": "string", "description": "slug del tenant"},
            },
            "required": ["transaction_id", "tenant"],
        },
    }
]
```

`additionalProperties: False` es obligatorio en producción: impide que el modelo añada campos que tu código no espera y que causarían un `KeyError` silencioso o, peor, una ejecución en el tenant equivocado.

`tool_choice` controla cuándo el modelo llama tools:

| Valor | Comportamiento |
|-------|---------------|
| `{"type": "auto"}` | modelo decide (default) |
| `{"type": "any"}` | modelo debe llamar alguna tool |
| `{"type": "tool", "name": "X"}` | modelo debe llamar X concretamente |

Paralelismo: el modelo puede emitir varios `tool_use` en el mismo turno. Itera todos los bloques y ejecuta en paralelo con `asyncio.gather` antes de construir los `tool_result`.

### 4.3 Prompt caching con `cache_control`

Marcar un bloque con `cache_control` le dice a Anthropic: «almacena todo el prefijo hasta aquí». El SDK cobra el hit al precio `cache_read` (~10× más barato que `input`).

```python
system = [
    {
        "type": "text",
        "text": TUTOR_RULES,
        "cache_control": {"type": "ephemeral", "ttl": "1h"},  # explícito siempre
    },
    {
        "type": "text",
        "text": module_body_md,
        "cache_control": {"type": "ephemeral", "ttl": "1h"},
    },
]
```

> [!cuidado]
> El default de TTL **cambió de 1 h a 5 m en marzo 2026**. Desde esa fecha, si omites `ttl`, tu cache expira a los 5 minutos. El código del curso declara `ttl` explícito siempre; no hay caching «por defecto».

Qué cachear: el prefijo **estable** (reglas del tutor, body del módulo, catálogo de cuentas del banco, schema del ERP). Qué no: el historial de chat, el código actual del editor, los intentos recientes — todo lo dinámico va al final, fuera del cache.

Mínimo de tokens para que el cache funcione: **1 024 tokens** para Sonnet 4.6. Si el bloque marcado es más corto, el SDK lo acepta pero no lo almacena. El `context_builder.py` del repo resuelve esto concatenando con un footer de convenciones cuando el módulo es corto (ver `app/services/context_builder.py`).

Verificar que el cache funciona: mira `response.usage.cache_read_input_tokens` en la respuesta. Si es > 0, hubo hit. El `_estimate_cost` del repo ya lo descuenta al precio correcto.

### 4.4 Extended thinking

Extended thinking (`"thinking": {"type": "enabled", "budget_tokens": N}`) activa una cadena de razonamiento interna antes de responder. Cuándo usarlo:

- Problemas con muchos pasos interdependientes (ej.: reconciliar 500 transacciones con reglas de tolerancia complejas).
- Decisiones donde el modelo tiene que sopesar varios escenarios antes de elegir una tool.

Cuándo **no** usarlo: conversación de chat, clasificaciones simples, generación de texto con estructura clara. El costo extra en tokens (el thinking cuenta en `output_tokens`) raramente se justifica fuera de casos analíticos densos.

### 4.5 Files API y citations

La Files API permite subir documentos (PDF, texto) una sola vez y referenciarlos por `file_id`. El modelo puede citar secciones específicas con el bloque `citations`. Útil para: contratos, normativas bancarias, manuales de cuentas contables. En el contexto del curso, se menciona como recurso; el caso F-CTA-01 lo aprovecha para el catálogo de cuentas.

### 4.6 Batches (50 % de descuento, asíncrono)

La Batches API recibe hasta 10 000 requests en un JSONL, los procesa en background (hasta 24 h) y devuelve resultados. El precio es 50 % del precio normal de input/output.

Cuándo usarlo: auditorías nocturnas de miles de facturas, clasificación masiva de transacciones históricas, re-etiquetado de un dataset. Cuándo **no**: cualquier cosa que el usuario espera en tiempo real.

El patrón: enqueue en `batches.create`, poll con `batches.retrieve` hasta `status == "ended"`, descarga con `batches.results`.

### 4.7 MCP en el SDK

El SDK 0.40+ puede conectarse a servidores MCP como fuente de tools. En el curso se consume MCP, no se implementa un servidor. El harness declara `mcp_servers=[{"type": "url", "url": "..."}]` y el SDK expone las tools del servidor automáticamente al modelo.

## 5. Ejemplo conducido

El loop completo en ≤ 25 líneas, basado en `api/app/routers/anthropic_demo.py`:

```python
from anthropic import Anthropic

client = Anthropic()
messages = [{"role": "user", "content": "¿Cuántas facturas tiene ACME en 2026-04?"}]

for _ in range(6):                           # máximo 6 iteraciones
    resp = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        tools=TOOLS,
        messages=messages,
    )

    if resp.stop_reason == "end_turn":       # el modelo terminó
        print(next(b.text for b in resp.content if b.type == "text"))
        break

    messages.append({"role": "assistant", "content": resp.content})

    # recolectar tool_results — uno por cada tool_use emitido
    results = []
    for block in resp.content:
        if block.type == "tool_use":
            output = execute_tool(block.name, block.input)  # tu función
            results.append({
                "type": "tool_result",
                "tool_use_id": block.id,   # emparejamiento por id
                "content": output,
            })
    messages.append({"role": "user", "content": results})
```

Dos invariantes de este loop: (1) después de `assistant` con `tool_use` viene `user` con `tool_result`; (2) cada `tool_use_id` aparece exactamente una vez en `tool_result`.

## 7. Quiz de comprensión

**Pregunta 1.** ¿Qué pasa si mandas un `tool_result` con un `tool_use_id` que no existe en la respuesta del modelo?

- A) El modelo ignora el resultado y sigue.
- B) El API devuelve un error 400. ✓
- C) El modelo usa el resultado pero lo etiqueta como «no solicitado».
- D) El SDK lanza una excepción antes de enviar.

*Feedback A*: El API no ignora silenciosamente una discordancia de id; valida el emparejamiento y rechaza la request. *Feedback C*: No existe esa etiqueta; la validación es dura. *Feedback D*: El SDK valida algunos campos, pero el `tool_use_id` incorrecto pasa el cliente y el rechazo viene del servidor.

**Pregunta 2.** El default de TTL del `cache_control` de Anthropic a partir de marzo 2026 es:

- A) 1 hora.
- B) 24 horas.
- C) 5 minutos. ✓
- D) Sin límite mientras no se supere la cuota.

*Feedback A*: Era el default antes de marzo 2026. Desde esa fecha el default cambió; por eso el curso declara `ttl` explícito siempre. *Feedback B*: Nunca fue 24 horas. *Feedback D*: El cache siempre expira; no existe TTL indefinido.

**Pregunta 3.** ¿Cuál de estas operaciones NO debe ir en la parte cacheada del system prompt?

- A) Las reglas del tutor (invariantes del agente).
- B) El body del módulo actual (estático durante la sesión).
- C) Los últimos 5 intentos del lector en el ejercicio. ✓
- D) El catálogo de cuentas del tenant (varía por tenant pero es estable en sesión).

*Feedback A*: Las reglas son el ejemplo canónico de prefijo estable — van en cache. *Feedback B*: El body del módulo no cambia durante la sesión; es cacheable. *Feedback D*: El catálogo varía por tenant pero no cambia turno a turno; tiene sentido cachearlo por sesión, separado por tenant.

## 8. Caso de uso real

**F-FIN-01 — Conciliación bancaria multi-cuenta** (manufactura, servicios-fin).

El agente llama `match_transactions` en un loop: por cada transacción bancaria sin match, el modelo decide si intentar un fuzzy match o escalar a excepción humana. El system prompt lleva cacheado el ruleset del banco ACME (tolerancias, cuentas excluidas, formato de referencia). El historial de transacciones del día (dinámico) va al final sin cache. La combinación reduce el costo por turno en ~80 % en sesiones largas de conciliación.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Parsear extracto bancario PDF → filas | determinístico | Layout estable; `parse_bank_statement` es código puro. |
| Match exacto monto + fecha + referencia | determinístico | Regla cerrada; si coincide 100 %, es match. |
| Match difuso por descripción («PAGO ACME» ↔ «ACME INC FACT 4820») | agéntico | Variabilidad de redacción; no hay regla cerrada. |
| Decidir si una diferencia es timing o error contable | agéntico | Requiere contexto del historial del proveedor. |
| Calcular el total conciliado vs. no conciliado | determinístico | Agregación SQL; no hay ambigüedad. |
| Redactar el hallazgo del CFO | agéntico | Salida es para humano; necesita criterio editorial. |

La herramienta `match_transactions` es la bisagra: tu código la define y la ejecuta (determinístico); el modelo decide **cuándo llamarla y con qué argumento** (agéntico). Esa separación es el patrón del curso.

## 10. Errores típicos

**1. `tool_use_id` incorrecto.**
*Síntoma*: el API devuelve `400 invalid_request_error: tool_use_id not found`.
*Causa*: se usa `block.name` o un id hardcodeado en lugar de `block.id`.
*Arreglo*: `"tool_use_id": block.id` — siempre el campo `.id` del bloque.

**2. No incluir todos los `tool_result` en el mismo turno.**
*Síntoma*: el API rechaza con `missing tool_result for tool_use_id`.
*Causa*: el loop procesa solo el primer `tool_use` e ignora los demás cuando el modelo paralela.
*Arreglo*: iterar **todos** los bloques de la respuesta, no solo el primero.

**3. Omitir `ttl` en `cache_control` post-marzo 2026.**
*Síntoma*: el hit rate es 0 después de 5 minutos; el costo sube en sesiones largas.
*Causa*: el default pasó a 5 m; sin `ttl` explícito, el cache expira antes de que la sesión llegue al segundo turno en muchos flujos.
*Arreglo*: `"cache_control": {"type": "ephemeral", "ttl": "1h"}` en cada bloque estable.

**4. Cachear el historial dinámico.**
*Síntoma*: el modelo ve historial desactualizado; los `cache_read_input_tokens` suben pero el modelo no «recuerda» la última acción.
*Causa*: se marcó el bloque de intentos o del historial de chat con `cache_control`.
*Arreglo*: solo los bloques verdaderamente estables (rules + module body + catálogos) van con cache. El historial va al final sin marca.

**5. Marcar un bloque < 1 024 tokens.**
*Síntoma*: `cache_read_input_tokens` siempre 0 aunque el block esté marcado.
*Causa*: el SDK acepta la marca pero Anthropic no almacena si el prefijo es corto.
*Arreglo*: verifica el tamaño del bloque; si es pequeño, concatena con otro bloque estable (ver `context_builder.py`).

## Hilo conductor

En E02 construiste el grafo con checkpointer y `thread_id`. En E03 aprendiste a cargar skills por tenant con slots. Ahora unimos los dos: el estado del agente no solo persiste entre turnos — persiste **por tenant**, de forma aislada, con una política de retención que evita que el contexto crezca indefinidamente. E05 (Temporal) añade durabilidad externa a este estado; aquí lo diseñas en SQLite.

## Idea central

Un agente tiene tres capas de memoria con semántica distinta: **turno** (el mensaje actual, en RAM), **sesión** (el historial de una conversación, en el checkpointer), y **largo plazo** (hechos que persisten entre sesiones, en un store o en DB). El `thread_id` es la clave de la sesión. En un sistema multitenant, `thread_id` lleva siempre el `tenant_id` como prefijo — de lo contrario, dos tenants pueden compartir estado. La compactación es la política que evita que la sesión crezca hasta superar el context window.

<!-- INLINE_QUIZ: three-memory-layers -->

## Por qué importa

Sin las tres capas separadas, caes en uno de dos anti-patrones: (1) el agente sin memoria — cada mensaje empieza desde cero, el usuario repite contexto en cada turno; (2) el agente con memoria sin retención — después de 20 turnos el context window explota y el modelo pierde coherencia.

La ficha **F-CX-02 (soporte al cliente multitenant)** ilustra los dos problemas simultáneamente: un agente de soporte que olvida el problema del cliente entre mensajes es frustrante; uno que acumula 300 mensajes en una sesión de auditoría de 2 horas es inutilizable por coste y latencia.

> [!cuidado]
> En SQLAlchemy async sobre `aiosqlite`, el checkpointer es single-threaded por defecto. Para el chat tutor de la app (B06) esto es aceptable; para un servicio de producción con 10+ tenants concurrentes, el checkpointer debe ser `AsyncPostgresSaver` (necesita `asyncpg`).

## Cómo funciona por dentro

### Las tres capas de memoria

**Memoria de turno** — lo que el modelo recibe en el mensaje actual. Es la lista `messages` del estado LangGraph. Vive en RAM, no persiste. Cuando el turno termina, va al checkpointer.

**Memoria de sesión** — el historial completo de una conversación identificada por `thread_id`. El checkpointer la serializa en SQLite (o Postgres). Persiste entre llamadas; el agente puede reanudarse días después.

**Memoria de largo plazo** — hechos que trascienden una sesión: el catálogo de cuentas del tenant, el historial de auditorías anteriores, las preferencias del usuario. Vive en la base de datos de la app (`app/repos/chat.py`, `app/repos/progress.py`). El harness la inyecta en el system prompt o como `ToolMessage` cuando es relevante — el modelo no la busca solo.

### El thread_id como dimensión de aislamiento

El checkpointer identifica cada sesión por `(thread_id,)`. En un sistema multitenant:

```python
# Correcto: el tenant es parte del thread_id
thread_id = f"{tenant_id}:chat:{conversation_id}"

# Incorrecto: dos tenants con la misma conversation_id se pisan
thread_id = conversation_id
```

El `conversation_id` puede ser un UUID generado por el frontend. El `tenant_id` lo inyecta el backend, nunca el cliente. Esto garantiza que ACME no ve el estado de Globex aunque ambos tengan `conversation_id = "audit-2026-04"`.

### El checkpointer SQLite en la app

`SqliteSaver` y `AsyncSqliteSaver` guardan el estado en una tabla `checkpoints` en el archivo SQLite. La estructura relevante:

```
thread_id         TEXT    → clave de sesión (lleva tenant_id como prefijo)
checkpoint_id     TEXT    → identificador del checkpoint específico
parent_ts         TEXT    → puntero al checkpoint anterior (forma la cadena)
metadata          TEXT    → JSON con el estado serializado
```

Para el chat tutor de la app (referencia: `app/repos/chat.py`), la sesión se identifica como `f"{tenant_id}:module:{module_id}:{user_id}"`. El módulo actual y el usuario son suficientes para la granularidad de la sesión pedagógica.

### Compactación: política de retención

Cuando el historial de una sesión supera ~80% del context window, el modelo empieza a perder los mensajes más antiguos (en modelos que truncan el input) o el coste sube porque cada turno envía más tokens. La solución es la **compactación**: antes de la próxima llamada, se resume el historial antiguo en un `SystemMessage` de resumen y se descarta el detalle.

El patrón en LangGraph es un nodo `compaction_check` antes del nodo `model`:

```
START → compaction_check → model → [tools_condition] → tools → model → ...
```

`compaction_check` es determinístico: cuenta tokens del historial. Si supera el umbral, invoca una llamada separada al modelo para resumir y reemplaza los mensajes antiguos con el resumen. El modelo del nodo principal nunca ve el historial sin compactar si supera el límite.

```python
from anthropic import Anthropic

COMPACTION_THRESHOLD = 50_000  # tokens

def compaction_check(state: State) -> dict[str, Any]:
    """Nodo determinístico: compacta el historial si supera el umbral."""
    total_chars = sum(
        len(m.content) if isinstance(m.content, str)
        else sum(p.get("text", "").__len__() for p in m.content if isinstance(p, dict))
        for m in state["messages"]
    )
    # Estimación gruesa: 4 chars ≈ 1 token
    estimated_tokens = total_chars // 4
    if estimated_tokens < COMPACTION_THRESHOLD:
        return {}  # Sin cambios — LangGraph no actualiza nada

    # Compactar: resumir los mensajes más antiguos
    client = Anthropic()
    summary = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{
            "role": "user",
            "content": (
                "Resume estos mensajes de conversación en un párrafo "
                "preservando los hechos clave:\n\n"
                + "\n".join(
                    f"{m.__class__.__name__}: {m.content}"
                    for m in state["messages"][:-10]  # preserva los últimos 10
                )
            ),
        }],
    )
    from langchain_core.messages import SystemMessage
    compact_msg = SystemMessage(content=f"[RESUMEN ANTERIOR] {summary.content[0].text}")
    return {"messages": [compact_msg] + state["messages"][-10:]}
```

> [!nota]
> Este `compaction_check` usa una estimación de tokens por caracteres. Para producción usa `anthropic.count_tokens` o el método `get_num_tokens_from_messages` de LangChain para un conteo exacto.

### Memoria de largo plazo: el store de LangGraph

LangGraph 0.2.50+ incluye `InMemoryStore` y `AsyncPostgresStore` para memoria de largo plazo. La diferencia con el checkpointer:

| Checkpointer | Store (memoria largo plazo) |
|---|---|
| Indexado por `thread_id` | Indexado por `(namespace, key)` |
| Historial de mensajes | Hechos, preferencias, perfiles |
| Se limpia con la sesión | Persiste entre sesiones |
| `config["configurable"]["thread_id"]` | `store.put(namespace, key, value)` |

Para el aislamiento multitenant en el store:

```python
# Namespace incluye tenant_id — sin ello, todos los tenants comparten el store
namespace = (tenant_id, "preferences")
store.put(namespace, "audit_tone", {"value": "ejecutivo"})
pref = store.get(namespace, "audit_tone")
```

### Cómo el chat tutor de la app usa estas capas

La app `bato-learning` (referencia: `app/services/chat.py`, `app/integrations/anthropic_chat.py`) implementa este patrón con el SDK directo (no LangGraph), pero la lógica es idéntica:

- **Turno**: el mensaje actual del editor + los últimos N intentos del lector (dinámico, sin cache).
- **Sesión**: historial de mensajes `(role, content)` guardados en `chat_messages` por `(tenant_id, module_id, user_id)`.
- **Largo plazo**: el body del módulo y las reglas del tutor en el system prompt, marcados con `cache_control: {"type": "ephemeral", "ttl": "1h"}` (estáticos, cacheables).

Ese diseño cumple el mínimo de 1024 tokens para activar el prompt caching de Sonnet 4.6 en la mayoría de módulos largos.

## Ejemplo conducido

Agente de auditoría con memoria de sesión por tenant (ACME y Globex aislados):

```python
import os
from typing import Annotated, Any, TypedDict

from langchain_anthropic import ChatAnthropic
from langchain_core.messages import HumanMessage
from langchain_core.tools import tool
from langgraph.checkpoint.sqlite import SqliteSaver
from langgraph.graph import END, START, StateGraph
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode, tools_condition

@tool
def sql_query(query: str, tenant: str) -> str:
    """Ejecuta una query SQL de solo-lectura sobre el warehouse del tenant."""
    if tenant == "acme":
        return "ACME: 12 facturas, total 48 210,00 EUR"
    return "Globex: 8 facturas, total 32 400,00 EUR"

class State(TypedDict):
    messages: Annotated[list, add_messages]

llm = ChatAnthropic(model="claude-sonnet-4-6").bind_tools([sql_query])

def model_node(state: State) -> dict[str, Any]:
    return {"messages": [llm.invoke(state["messages"])]}

g = StateGraph(State)
g.add_node("model", model_node)
g.add_node("tools", ToolNode([sql_query]))
g.add_edge(START, "model")
g.add_conditional_edges("model", tools_condition)
g.add_edge("tools", "model")

# thread_id por tenant — aislamiento garantizado
acme_cfg = {"configurable": {"thread_id": "acme:audit:2026-04"}}
globex_cfg = {"configurable": {"thread_id": "globex:audit:2026-04"}}

with SqliteSaver.from_conn_string("data/sessions.db") as cp:
    app = g.compile(checkpointer=cp)

    # Sesión ACME
    app.invoke(
        {"messages": [HumanMessage("¿Cuántas facturas tiene ACME en 2026-04?")]},
        config=acme_cfg,
    )
    # Segundo turno ACME — el agente recuerda la primera pregunta
    state_acme = app.invoke(
        {"messages": [HumanMessage("¿Y cuál es el total?")]},
        config=acme_cfg,
    )

    # Sesión Globex — estado completamente independiente
    state_globex = app.invoke(
        {"messages": [HumanMessage("¿Cuántas facturas tiene Globex en 2026-04?")]},
        config=globex_cfg,
    )

# Verificar aislamiento: Globex no sabe nada de la sesión de ACME
print(f"Mensajes ACME: {len(state_acme['messages'])}")    # 4 (2 user + 2 assistant)
print(f"Mensajes Globex: {len(state_globex['messages'])}")  # 2 (1 user + 1 assistant)
```

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|---|---|---|
| Construir `thread_id = f"{tenant_id}:chat:{conv_id}"` | determinístico | Concatenación de strings — regla fija. |
| Contar tokens del historial para decidir si compactar | determinístico | Operación aritmética — regla numérica con umbral. |
| Resumir el historial antiguo para la compactación | agéntico | La síntesis coherente de una conversación exige juicio contextual. |
| Verificar que `thread_id` empieza con el `tenant_id` correcto | determinístico | Validación de string — regla de seguridad cerrada. |
| El modelo decide qué recordar de la sesión para responder | agéntico | El modelo infiere relevancia del contexto sin regla cerrada. |
| Escribir en el store con `namespace = (tenant_id, ...)` | determinístico | El namespace viene del harness, no del modelo. |

## Errores típicos

**1. `thread_id` sin prefijo de tenant.**
*Síntoma*: dos usuarios de distintos tenants con el mismo `conversation_id` comparten historial.
*Causa*: `thread_id = conversation_id` — el `tenant_id` no está en la clave.
*Arreglo*: `thread_id = f"{tenant_id}:{conversation_id}"`. Añadir una assertion en el harness: `assert thread_id.startswith(tenant_id)`.

**2. Ignorar la compactación hasta que el contexto explota.**
*Síntoma*: error `context_length_exceeded` en sesiones de auditoría largas.
*Causa*: no hay nodo `compaction_check`; el historial crece sin límite.
*Arreglo*: añadir el nodo antes del modelo con el umbral configurado por tenant. Un contador de tokens aproximado es suficiente como guardia.

**3. Confundir checkpointer con store de largo plazo.**
*Síntoma*: el agente «olvida» las preferencias del tenant entre sesiones distintas.
*Causa*: las preferencias se guardaron en el checkpointer (atadas a `thread_id`); al abrir una nueva sesión, el `thread_id` cambia y el agente empieza sin memoria.
*Arreglo*: preferencias, perfiles y hechos persistentes van en el store (indexado por `(tenant_id, key)`), no en el checkpointer.

**4. Meter memoria de largo plazo en el system prompt hardcodeado.**
*Síntoma*: el catálogo de cuentas de ACME aparece en todas las sesiones, incluidas las de Globex.
*Causa*: el system prompt está hardcodeado con datos de ACME; no hay inyección dinámica por tenant.
*Arreglo*: el system prompt base va en el system prompt; los datos del tenant se inyectan en runtime desde el store o la DB, solo para el tenant activo.

**5. Usar `MemorySaver` en producción.**
*Síntoma*: el historial se pierde cada vez que el proceso se reinicia.
*Causa*: `MemorySaver` guarda el estado solo en RAM. Es útil para tests, no para producción.
*Arreglo*: `SqliteSaver` para single-node (producción simple), `AsyncPostgresSaver` para multi-nodo.

## Caso de uso real

**F-CX-02 — Soporte al cliente multitenant** (ACME como cliente de la plataforma, con múltiples usuarios activos).
Cada conversación de soporte tiene su propio `thread_id = f"{tenant_id}:support:{ticket_id}"`. La memoria de largo plazo guarda el historial de tickets anteriores del cliente en el store. La compactación actúa cuando un ticket abierto supera 50 mensajes (auditorías complejas pueden durar horas). El fallback humano se activa cuando el agente emite `confidence < 0.6` en dos turnos consecutivos — un nodo determinístico detecta ese patrón y escala el ticket.

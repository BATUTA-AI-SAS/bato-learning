## 1. Hilo conductor

Vienes de E01, donde construiste el loop SDK con tools. Ahí el modelo decidía cuándo llamar una tool; aquí aprendes a darle al agente **recetas** codificadas (skills) que reducen su campo de decisión a lo verdaderamente agéntico. Hacia E04 necesitarás skills claros por tenant; un agente que «lo sabe todo» es el anti-patrón que este módulo desmonta.

---

## 2. Idea central

Un **skill** es un archivo Markdown con frontmatter YAML que describe una receta: qué trigger lo activa, qué pasos seguir, qué variables (`slots`) rellena el tenant. El harness lo carga selectivamente según el contexto — el modelo no elige qué skills existen; tú sí. `CLAUDE.md` define el stack e invariantes de proyecto (intent estable, global). `AGENTS.md` es el mismo concepto con nombre estándar abierto: lo lee también ChatGPT, Cursor, Windsurf. Un skill vive en `.claude/skills/` o en `app/integrations/skills/`; CLAUDE.md y AGENTS.md viven en la raíz del repo.

## 3. Por qué importa

Un agente sin skills bien definidos resuelve el mismo problema de 30 maneras distintas según cómo el usuario lo interpele. Eso es invendible: el CFO de ACME no puede auditar un proceso que cambia de forma cada mes. El skill «congela» la receta; el agente pone la inteligencia donde la receta no alcanza.

El caso F-CTA-01 (clasificación contable) ilustra el riesgo: sin un skill `classify-transaction` con el catálogo de cuentas de ACME como slot, el modelo clasifica con el catálogo genérico del preentrenamiento — que en LATAM es completamente diferente al del cliente. La clasificación incorrecta es un error contable directo, auditable, con consecuencias fiscales.

## 4. Cómo funciona por dentro

### 4.1 La jerarquía de tres niveles

```
CLAUDE.md / AGENTS.md      ← nivel repositorio / proyecto
  └── .claude/skills/*.md  ← nivel skill (cargado selectivamente)
        └── slots           ← nivel tenant (rellena el harness)
```

**CLAUDE.md** (o `AGENTS.md` — mismo contenido, distinto nombre): declaración global del proyecto. El harness lo carga siempre. Contiene stack, invariantes, comandos, qué está fuera de alcance. No contiene instrucciones específicas de tareas. Ver el `CLAUDE.md` de este repo como referencia de manual de la casa.

**AGENTS.md** es el nombre del mismo archivo según el estándar abierto que lanzaron varios editores (Cursor, Windsurf, GitHub Copilot) en 2025 para que un repo declare sus instrucciones de agente de forma interoperable. Anthropic adoptó la convención. Si tienes un repo con ambos (`CLAUDE.md` + `AGENTS.md`), Claude Code da precedencia a `CLAUDE.md`; otros agentes leen `AGENTS.md`. La recomendación del curso: un solo archivo `AGENTS.md` en proyectos multi-agente, y `CLAUDE.md` en repos exclusivamente Claude Code.

**Skills** (`.claude/skills/*.md`): cada skill describe una tarea recurrente con instrucciones concretas. El harness decide cuál cargar según el trigger; el modelo no ve todos los skills al mismo tiempo.

### 4.2 Frontmatter de un skill

```yaml
---
name: classify-transaction
description: |
  Clasifica una transacción contable según el catálogo de cuentas del tenant.
  Devuelve código de cuenta, descripción y nivel de confianza.
trigger: |
  Cuando el usuario o el workflow pida clasificar una transacción, o cuando
  el grafo entre al nodo `classify`.
slots:
  chart_of_accounts: "Catálogo de cuentas del tenant (tabla YAML o JSON)."
  currency: "Moneda funcional del tenant. Default: USD."
  tax_regime: "Régimen fiscal aplicable. Ej: IVA_CO, ISR_MX."
requires_confirmation: false
version: 1
---
```

El **cuerpo** del skill (después del frontmatter) contiene las instrucciones de qué hacer cuando el trigger se cumple: pasos, formato de salida, qué tool llamar, cuándo escalar a humano.

Los **slots** son las variables que el harness rellena en onboarding del tenant. El skill no hardcodea el catálogo de cuentas de ACME; tiene un slot `chart_of_accounts` que el harness inyecta al cargar el skill para ese tenant. Esto es **multitenancy de skills**: un solo archivo, N instancias.

### 4.3 Qué va en CLAUDE.md vs qué va en un skill

| Contenido | CLAUDE.md | Skill |
|-----------|-----------|-------|
| Stack técnico del repo (FastAPI, LangGraph, SDK 0.40+) | ✓ | ✗ |
| Comandos de desarrollo (`uv sync`, `docker compose up`) | ✓ | ✗ |
| Invariantes de seguridad (nunca hardcodear tenant, cero PII en logs) | ✓ | ✗ |
| Qué está out-of-scope (no modificar infra sin aprobación) | ✓ | ✗ |
| Receta paso a paso para clasificar una transacción | ✗ | ✓ |
| Catálogo de cuentas del cliente (slot por tenant) | ✗ | ✓ (slot) |
| Cuándo escalar a humano en la conciliación | ✗ | ✓ |
| Formato de salida del reporte de auditoría | ✗ | ✓ |

La regla operativa: CLAUDE.md es la constitución del proyecto (stable, aplica a toda tarea). Un skill es una ley específica (aplica a una tarea cuando el trigger la activa).

### 4.4 Gating: carga selectiva de skills

El harness no mete todos los skills en el system prompt. Cada turno evalúa el trigger del skill contra el contexto actual:

```python
def should_load_skill(skill: Skill, context: dict) -> bool:
    # evaluación simplificada del trigger
    trigger_keywords = extract_keywords(skill.frontmatter.trigger)
    return any(kw in context["current_intent"] for kw in trigger_keywords)
```

Solo los skills con trigger que matchea se inyectan. El resto no llega al modelo. Esto mantiene el contexto limpio y evita que el modelo confunda recetas de dominios distintos (clasificación contable ≠ conciliación bancaria).

### 4.5 El anti-patrón: el agente mayordomo

Un agente con todos los skills cargados, siempre, es el agente mayordomo. Sus síntomas:

1. El system prompt tiene > 8 000 tokens de instrucciones.
2. El modelo combina recetas de distintos skills de forma incorrecta.
3. Los skills se contradicen entre sí en casos borde.
4. No hay forma de auditar qué skill causó qué decisión.

La solución: un agente, un caso de uso. El workflow de clasificación contable tiene su propio agente con `classify-transaction`. El workflow de conciliación tiene el suyo con `match_transactions`. Se comunican por datos (base de datos), no por shared context.

### 4.6 Skills en LangGraph

En LangGraph, el skill se convierte en el **system prompt del nodo** que lo implementa:

```python
def classify_node(state: AgentState) -> AgentState:
    skill_text = load_skill("classify-transaction", tenant=state["tenant"])
    resp = client.messages.create(
        model="claude-sonnet-4-6",
        system=skill_text,          # el skill es el system del nodo
        messages=state["messages"],
        tools=CLASSIFY_TOOLS,
    )
    return {"messages": [resp]}
```

El nodo no sabe de LangGraph — es Python puro que llama al SDK. LangGraph solo orquesta cuándo se ejecuta.

## 5. Ejemplo conducido

Skill completo para clasificación contable de ACME:

```yaml
---
name: classify-transaction
description: Clasifica transacciones según el catálogo de cuentas del tenant.
trigger: |
  Cuando el nodo classify del grafo se active, o cuando el usuario pida
  "clasificar transacción" o "asignar cuenta contable".
slots:
  chart_of_accounts: "Catálogo como YAML. Claves: código, descripción, tipo."
  tax_regime: "IVA_CO | ISR_MX | IVA_PE. Default: IVA_CO."
requires_confirmation: false
version: 1
---

Clasifica la transacción usando el catálogo en `chart_of_accounts`.

Pasos:
1. Lee descripción, monto y fecha de la transacción.
2. Encuentra la cuenta más específica que aplica. Si dudas entre dos,
   elige la más específica.
3. Si la confianza es < 0.7, escala: devuelve `needs_human: true` con
   las dos cuentas candidatas y la razón de la duda.
4. Devuelve JSON: {account_code, account_name, confidence, needs_human}.

Nunca inventes un código de cuenta que no esté en el catálogo.
```

El harness lo carga así:

```python
import yaml, pathlib

def load_skill(name: str, tenant: str) -> str:
    path = pathlib.Path(f"app/integrations/skills/{name}/SKILL.md")
    raw = path.read_text()
    front, body = raw.split("---\n", 2)[1:]  # split frontmatter
    meta = yaml.safe_load(front)
    slots = get_tenant_slots(tenant, meta.get("slots", {}))
    # sustituir slots en el body
    for slot_name, slot_value in slots.items():
        body = body.replace(f"`{slot_name}`", slot_value)
    return body
```

## 7. Quiz de comprensión

**Pregunta 1.** Un repo tiene `CLAUDE.md` y `AGENTS.md`. ¿Cuál lee Claude Code?

- A) Lee ambos y fusiona las instrucciones.
- B) Lee `CLAUDE.md` con precedencia; `AGENTS.md` para otros agentes. ✓
- C) Lee `AGENTS.md` siempre; `CLAUDE.md` quedó obsoleto.
- D) Lee el que tenga más tokens de contenido.

*Feedback A*: Claude Code no fusiona automáticamente; da precedencia a `CLAUDE.md`. *Feedback C*: `CLAUDE.md` no es obsoleto; es el archivo nativo de Claude Code. `AGENTS.md` es el estándar multi-agente adoptado en 2025. *Feedback D*: La decisión no depende del tamaño.

**Pregunta 2.** ¿Cuál de estos contenidos **no** debe ir en CLAUDE.md?

- A) El comando `uv run pytest` para correr tests.
- B) «Nunca hardcodear `tenant_id` en código productivo.»
- C) Los pasos para clasificar una transacción contable según el catálogo de ACME. ✓
- D) El stack técnico del repo (FastAPI, LangGraph, Temporal).

*Feedback A*: Los comandos de desarrollo son invariantes del proyecto: van en CLAUDE.md. *Feedback B*: Una regla de seguridad global aplica a toda tarea; va en CLAUDE.md. *Feedback D*: El stack es información de contexto global; va en CLAUDE.md.

**Pregunta 3.** ¿Qué es un «slot» en un skill?

- A) Un campo del frontmatter que el lector rellena manualmente en el YAML.
- B) Una variable del body que el harness sustituye con datos del tenant en onboarding. ✓
- C) Un placeholder para el nombre del modelo que se usará.
- D) Un parámetro opcional de `messages.create`.

*Feedback A*: Los slots no los rellena el lector en el YAML; los rellena el harness automáticamente según el tenant, sin modificar el archivo del skill. *Feedback C*: El modelo se configura en el harness, no en el skill. *Feedback D*: `messages.create` no tiene parámetro `slots`; eso es una abstracción del harness.

**Pregunta 4.** Un agente tiene cargados simultáneamente los skills `classify-transaction`, `monthly-bank-recon`, `draft-audit-report`. El modelo combina instrucciones de los tres en una sola respuesta. ¿Cuál es el anti-patrón y la solución?

- A) Anti-patrón: skills con slots. Solución: hardcodear los valores por tenant.
- B) Anti-patrón: agente mayordomo. Solución: un agente por caso de uso, con gating selectivo. ✓
- C) Anti-patrón: usar CLAUDE.md. Solución: mover todo a skills.
- D) Anti-patrón: frontmatter YAML. Solución: usar JSON para que sea parseable.

*Feedback A*: Los slots son la solución al hardcoding, no el problema. *Feedback C*: CLAUDE.md y skills son complementarios; eliminar CLAUDE.md no resuelve la mezcla de instrucciones. *Feedback D*: El formato YAML del frontmatter no causa el problema de mezcla.

## 8. Caso de uso real

**F-CTA-01 — Clasificación contable de transacciones** (servicios financieros, hospitalidad).

El skill `classify-transaction` con el slot `chart_of_accounts` del cliente es lo que separa «clasificar según el PUC colombiano para Cooppopular» de «clasificar según el catálogo interno para Mesón Urbano». Sin slots, harías un skill por cliente — inescalable en un negocio LATAM donde cada PYME tiene su propio plan de cuentas personalizado. Con slots, el skill es uno; las instancias, N.

## 9. Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Parsear el frontmatter YAML del skill | determinístico | `yaml.safe_load` — función pura. |
| Sustituir slots del tenant en el body del skill | determinístico | String replacement; no requiere razonamiento. |
| Cargar el skill correcto según el trigger | determinístico (con regla) | El harness evalúa keywords; si la regla es cerrada, no necesita modelo. |
| Clasificar la transacción usando el skill cargado | agéntico | El catálogo tiene ambigüedades; la descripción de la transacción varía. |
| Decidir si escalar a humano (confianza < 0.7) | determinístico | Umbral numérico — regla cerrada. |
| Redactar la justificación del hallazgo | agéntico | Salida para humano; necesita criterio editorial. |

El skill no elimina lo agéntico — acota dónde ocurre. La clasificación sigue siendo agéntica; la receta que la guía es determinística.

## 10. Errores típicos

**1. Meter recetas de tarea en CLAUDE.md.**
*Síntoma*: CLAUDE.md tiene > 2 000 tokens con instrucciones específicas de casos de uso.
*Causa*: se fusionaron el manual del proyecto con las instrucciones de cada workflow.
*Arreglo*: mover todo lo específico de tarea a skills; CLAUDE.md queda en < 600 tokens.

**2. Skills sin slots: hardcoding del tenant.**
*Síntoma*: hay un `SKILL_acme.md`, un `SKILL_globex.md`, un `SKILL_initech.md` con el mismo body y solo el catálogo distinto.
*Causa*: no se usaron slots; el contenido del tenant se metió directo en el archivo.
*Arreglo*: un solo `SKILL.md` con `chart_of_accounts` como slot; el harness inyecta el catálogo en runtime.

**3. Cargar todos los skills siempre (agente mayordomo).**
*Síntoma*: el model confunde instrucciones de conciliación con instrucciones de clasificación. El context window se llena.
*Causa*: no hay gating; todos los skills van en el system prompt de todos los nodos.
*Arreglo*: gating por trigger en el harness; cada nodo solo recibe el skill que le corresponde.

**4. AGENTS.md y CLAUDE.md con contenidos distintos.**
*Síntoma*: Claude Code hace una cosa; Cursor hace otra; el equipo no puede reproducir comportamientos.
*Causa*: se actualizó uno y no el otro.
*Arreglo*: un solo archivo de referencia (CLAUDE.md en repos Claude Code, AGENTS.md en repos multi-agente); el otro es un symlink o redirige explícitamente.

**5. Trigger demasiado amplio.**
*Síntoma*: el skill `classify-transaction` se carga también cuando el usuario pregunta por el estado de la conciliación.
*Causa*: el trigger usa palabras como «transacción» sin especificar la acción «clasificar».
*Arreglo*: triggers con verbo + objeto: «cuando el usuario pida clasificar una transacción» en lugar de «cuando haya transacciones».

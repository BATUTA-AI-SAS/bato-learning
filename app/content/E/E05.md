## Hilo conductor

En E04 construiste un agente con memoria multitenant: el loop del modelo persiste sesiones por `tenant_id`. El problema es que ese loop vive en memoria — si el proceso muere, el progreso desaparece. E05 resuelve eso con **Temporal**: un motor de ejecución durable que puede reintentar, pausar y reanudar workflows arbitrariamente largos, incluso si el worker se cae en medio de una llamada al LLM. E06 (Capstone) toma exactamente este patrón y lo porta a un despliegue real en Hetzner.

## Idea central

Un **workflow** Temporal es una función Python anotada con `@workflow.defn` / `@workflow.run` que solo puede llamar a `workflow.execute_activity`. Todo lo no-determinístico — IO, LLM, red, tiempo real — vive en **activities** anotadas con `@activity.defn`. Temporal persiste cada paso del workflow en su historial. Si el worker muere, Temporal **replaya** el historial desde el inicio para reconstruir el estado exacto. Ese replay exige que el código del workflow produzca el mismo resultado dado el mismo historial: **determinismo absoluto**.

## Por qué importa

Un agente que corre 30 minutos procesando 500 facturas falla a los 28 minutos porque el LLM tarda 90 segundos y el cloud provider resetea la conexión. Sin Temporal, pierdes todo y reinicias desde cero. Con Temporal, la activity retrocede al último heartbeat y reintenta solo ese paso. El estado del workflow está intacto.

El caso **F-FIN-03 (cierre mensual asistido)** ilustra la presión operativa: el CFO de ACME quiere que el cierre corra cada fin de mes a las 23:00, que dure lo que dure, que se pueda pausar si falta un dato, y que nunca duplique un asiento contable aunque el worker se reinicie. Eso es exactamente lo que Temporal garantiza.

El caso **F-CTA-02 (auditoría interna)** — ancla del Capstone E06 — necesita la misma garantía: una auditoría mensual que ingiere datos, llama al agente LangGraph, persiste el reporte, todo idempotente por `tenant_id` y `period`.

> [!cuidado]
> Sin las garantías de Temporal, un agente «durable» es solo un cron job optimista. El primer timeout silencioso produce un reporte parcial o un asiento duplicado. Temporal hace que esos casos sean imposibles por construcción.

## Cómo funciona por dentro

### El ciclo de vida de un workflow

Cuando llamas a `client.start_workflow(...)`, Temporal almacena el evento `WorkflowExecutionStarted` en su historial. Cada vez que el workflow llama a `workflow.execute_activity(...)`, Temporal almacena `ActivityTaskScheduled`, luego `ActivityTaskStarted`, luego `ActivityTaskCompleted` (o `ActivityTaskFailed`). Si el worker muere en cualquier punto, Temporal asigna el trabajo a otro worker disponible y le entrega el historial. El nuevo worker **replaya** el código del workflow desde el inicio, pero en lugar de ejecutar las activities ya completadas, usa los resultados del historial. Por eso el código debe ser determinístico: si el replay produce una rama distinta, Temporal lanza `NonDeterminismError` — una señal inequívoca de que rompiste la regla.

### La regla de oro: nada no-determinístico en workflow code

```python
# MAL — viola determinismo; el replay puede producir un ID distinto
@workflow.run
async def run(self, tenant_id: str, period: str) -> dict:
    import uuid
    dataset_id = str(uuid.uuid4())   # ← cada replay genera un UUID distinto
    ...

# MAL — viola determinismo; el tiempo real cambia en cada replay
@workflow.run
async def run(self, tenant_id: str, period: str) -> dict:
    from datetime import datetime
    ts = datetime.now().isoformat()  # ← el replay tiene otra hora
    ...

# BIEN — todo no-determinístico va en la activity
@workflow.run
async def run(self, tenant_id: str, period: str) -> dict:
    result = await workflow.execute_activity(
        ingest_data,
        args=[tenant_id, period],
        start_to_close_timeout=timedelta(minutes=10),
        retry_policy=RetryPolicy(maximum_attempts=3),
    )
    ...
```

La lista completa de prohibiciones en workflow code: `datetime.now()`, `random.*`, `uuid.uuid4()`, cualquier IO (archivo, red, DB), cualquier llamada al LLM. Si necesitas tiempo actual, úsalo dentro de una activity. Si necesitas un UUID reproducible, genera en la activity y devuélvelo.

### `with workflow.unsafe.imports_passed_through()`

Los imports en el módulo del workflow se ejecutan **durante el replay**. Si importas una librería que tiene side-effects al importar (logging, apertura de socket, etc.), contaminas el replay. La solución es envolver los imports de dependencias no-determinísticas en este bloque:

```python
with workflow.unsafe.imports_passed_through():
    from ..activities.agent import ingest_data, persist_output, run_agent
```

Esto indica a Temporal que esos imports no deben ejecutarse durante el replay. El código real de `api/app/workflows/audit.py` lo usa exactamente así.

### `start_to_close_timeout` y `heartbeat_timeout`

`start_to_close_timeout` es el tiempo máximo desde que Temporal asigna la activity al worker hasta que termina. Si la activity no termina en ese tiempo, Temporal la marca como fallida y aplica la retry policy.

`heartbeat_timeout` es distinto y crítico para activities largas: si la activity no llama a `activity.heartbeat(...)` dentro de ese tiempo, Temporal asume que el worker murió y reintenta la activity desde el principio. Para un agente LLM que puede tardar varios minutos, el patrón correcto es una coroutine de heartbeat en background:

```python
@activity.defn
async def run_agent(skill_id: str, tenant_id: str, dataset_id: str) -> dict:
    async def _heartbeat() -> None:
        while True:
            activity.heartbeat({"status": "thinking", "tenant": tenant_id})
            await asyncio.sleep(15)  # cada 15 s; heartbeat_timeout debe ser > 15 s

    task = asyncio.create_task(_heartbeat())
    try:
        result = await invoke_langgraph_agent(skill_id, tenant_id, dataset_id)
        return result
    finally:
        task.cancel()
```

### Retry policy

```python
from temporalio.common import RetryPolicy

RetryPolicy(
    maximum_attempts=3,
    initial_interval=timedelta(seconds=1),
    backoff_coefficient=2.0,
    maximum_interval=timedelta(minutes=2),
    non_retryable_error_types=["ValueError"],  # errores de validación no se reintentan
)
```

Para `ingest_data` (IO de red, puede fallar transitoriamente): `maximum_attempts=3`. Para `run_agent` (caro en tokens): `maximum_attempts=2`. Para `persist_output` (escritura idempotente): sin límite explícito, la idempotency key previene duplicados.

### Idempotency keys

Una activity que escribe un reporte no debe escribirlo dos veces si Temporal la reintenta. La solución: incluir una clave única en el identificador del artefacto.

```python
idempotency_key = f"audit:{tenant_id}:{period}"
# Si persist_output se llama dos veces con la misma clave,
# la segunda llamada detecta el artefacto existente y no duplica.
await workflow.execute_activity(
    persist_output,
    args=[tenant_id, agent_result, idempotency_key],
    start_to_close_timeout=timedelta(minutes=2),
)
```

El `idempotency_key` siempre incluye `tenant_id` — un artefacto de ACME y un artefacto de Globex con el mismo período son distintos.

### Schedules para cron multitenant

```python
from temporalio.client import Client, Schedule, ScheduleActionStartWorkflow
from temporalio.client import ScheduleIntervalSpec, ScheduleSpec

async def create_monthly_audit_schedule(tenant_id: str) -> None:
    client = await Client.connect("localhost:7233")
    await client.create_schedule(
        f"monthly-audit-{tenant_id}",
        Schedule(
            action=ScheduleActionStartWorkflow(
                AuditTenantWorkflow.run,
                args=[tenant_id, "{{.ScheduledTime | formatDate \"2006-01\"}}"],
                id=f"audit-{tenant_id}-scheduled",
                task_queue="audit-queue",
            ),
            spec=ScheduleSpec(
                cron_expressions=["0 23 28-31 * *"],  # último día hábil del mes, 23:00
            ),
        ),
    )
```

Cada tenant tiene su propio schedule. El nombre del schedule incluye `tenant_id` — nunca un schedule genérico que itera sobre tenants dentro del workflow.

### Cancelación y cleanup

```python
handle = client.get_workflow_handle(f"audit-{tenant_id}-{period}")
await handle.cancel()
```

Temporal propaga la cancelación a la activity en curso en el siguiente heartbeat. La activity recibe una excepción `CancelledError` y puede limpiar recursos antes de propagarla. El workflow termina en estado `CANCELLED`, no en `FAILED`.

### Search Attributes para visibilidad operativa

```python
from temporalio.api.common.v1 import Payload
from temporalio import workflow

@workflow.run
async def run(self, tenant_id: str, period: str) -> dict:
    workflow.upsert_search_attributes({"TenantId": [tenant_id], "Period": [period]})
    ...
```

Con search attributes puedes filtrar en la UI de Temporal o hacer queries: «muéstrame todos los workflows de ACME del período 2026-04». Sin ellos, el historial es opaco.

## Ejemplo conducido

El workflow completo de `api/app/workflows/audit.py`, explicado línea por línea:

```python
from datetime import timedelta
from temporalio import workflow
from temporalio.common import RetryPolicy

with workflow.unsafe.imports_passed_through():
    from ..activities.agent import ingest_data, persist_output, run_agent

@workflow.defn
class AuditTenantWorkflow:
    @workflow.run
    async def run(self, tenant_id: str, period: str) -> dict:
        # Paso 1: ingesta — IO de red, puede fallar; 3 intentos
        ingest = await workflow.execute_activity(
            ingest_data,
            args=[tenant_id, period],
            start_to_close_timeout=timedelta(minutes=10),
            retry_policy=RetryPolicy(maximum_attempts=3),
        )
        # Paso 2: agente LangGraph — caro, heartbeat obligatorio
        agent = await workflow.execute_activity(
            run_agent,
            args=["monthly_audit", tenant_id, ingest["dataset_id"]],
            start_to_close_timeout=timedelta(minutes=30),
            heartbeat_timeout=timedelta(seconds=60),   # ← 60 s > 15 s del loop
            retry_policy=RetryPolicy(maximum_attempts=2),
        )
        # Paso 3: persistencia idempotente
        await workflow.execute_activity(
            persist_output,
            args=[tenant_id, agent, f"audit:{tenant_id}:{period}"],
            start_to_close_timeout=timedelta(minutes=2),
        )
        return {"tenant_id": tenant_id, "period": period, "agent": agent}
```

El workflow es una secuencia de tres `await`. No hay `datetime.now`, no hay IO, no hay imports sin proteger. Eso es todo.

## Determinístico vs agéntico

| Tramo | Tipo | Por qué |
|-------|------|---------|
| Código del workflow (`@workflow.run`) | determinístico | Debe producir el mismo resultado en cada replay. Cero IO. |
| `ingest_data` — descarga extracto del ERP | no-determinístico (activity) | Hace IO de red; el resultado puede variar; por eso va en activity con retry. |
| `run_agent` — loop LangGraph + LLM | no-determinístico (activity) | El modelo puede producir distintas respuestas; la latencia varía; heartbeat obligatorio. |
| `persist_output` — escribe el reporte | no-determinístico (activity) | Escribe a DB; idempotency key previene duplicados en retry. |
| Schedule cron por tenant | determinístico | La expresión cron es código fijo; cuándo se lanza es predecible. |
| Cancelación del workflow | determinístico | La señal de cancelación es un evento del historial; el comportamiento es reproducible. |

## Errores típicos

**1. `datetime.now()` dentro del workflow.**
*Síntoma*: Temporal lanza `NonDeterminismError` en el primer retry.
*Causa*: el workflow llama a `datetime.now()` directamente; en el replay la hora es distinta y el historial no coincide.
*Arreglo*: mueve la llamada a una activity; el workflow usa el resultado devuelto por la activity.

**2. Import sin `unsafe.imports_passed_through()` que tiene side-effects.**
*Síntoma*: el worker se cuelga o lanza una excepción extraña durante el replay, no durante la primera ejecución.
*Causa*: la librería importada abre un socket o escribe a un log al importarse; eso rompe el entorno de replay.
*Arreglo*: envuelve el import en `with workflow.unsafe.imports_passed_through(): ...`.

**3. `heartbeat_timeout` más corto que el intervalo de heartbeat.**
*Síntoma*: la activity se reintenta en bucle aunque no haya fallado; el agente nunca completa.
*Causa*: si `heartbeat_timeout=timedelta(seconds=10)` y el loop de heartbeat hace `asyncio.sleep(15)`, Temporal ve que la activity no latió en 10 s y la cancela.
*Arreglo*: `heartbeat_timeout` debe ser al menos 2× el intervalo de `asyncio.sleep` en el loop de heartbeat.

**4. Schedule sin `tenant_id` en el nombre.**
*Síntoma*: el segundo tenant sobreescribe el schedule del primero; solo un tenant recibe la auditoría.
*Causa*: schedule con nombre fijo `"monthly-audit"` en lugar de `f"monthly-audit-{tenant_id}"`.
*Arreglo*: siempre incluir `tenant_id` en el id del schedule y en el workflow id que lanza.

**5. Idempotency key sin `tenant_id`.**
*Síntoma*: el reporte de ACME sobreescribe el de Globex cuando ambos tienen el mismo período.
*Causa*: la clave es `f"audit:{period}"` en lugar de `f"audit:{tenant_id}:{period}"`.
*Arreglo*: `tenant_id` es dimensión obligatoria en toda clave de idempotencia.

> [!profundizar]
> Las activities con heartbeat están en [`api/app/activities/agent.py`](../../api/app/activities/agent.py). La función `run_agent` usa `asyncio.create_task` para el loop de heartbeat y lo cancela en el bloque `finally`. Ese patrón garantiza que el heartbeat se detiene aunque la activity falle.

> [!profundizar]
> El worker arranca en [`api/app/worker.py`](../../api/app/worker.py). Observa que registra tanto workflows como activities en el mismo `Worker(...)`. Si olvidas registrar una activity, Temporal la encola indefinidamente con status `SCHEDULED` — visible en la UI en `localhost:8080`.
